Theory
lca_deep_dive_master.html

Given the root of a binary tree and two nodes p and q that are guaranteed to exist in it, return their lowest common ancestor — the deepest node that has both p and q as descendants. By the standard convention, a node is allowed to be a descendant of itself, so if p is an ancestor of q, the answer is p. The tree is a plain binary tree (not a BST), and node values are unique.

Input (tree + targets)OutputWhy
tree [3,5,1,6,2,0,8,_,_,7,4], p=5, q=135 is in the left subtree of 3, 1 in the right — they split at 3
same tree, p=5, q=454 is below 5, so 5 is an ancestor of both → 5 is the LCA
tree [1,2], p=1, q=212 is a child of 1; 1 is its own ancestor

The twist is the "a node can be its own ancestor" convention, which means the LCA is not always a strict split point — it can be one of the targets itself. The clean framing: the LCA is the last node shared by the root-to-p and root-to-q paths. That observation gives two equally good solutions — a post-order recursion that bubbles the answer up, or an iterative parent-map walk-up that intersects the two paths.

2   Reading the Problem — insights from the constraints

"Lowest common ancestor": the deepest node with both targets in its subtree. Equivalently, walk from the root toward each target; the paths share a prefix and then diverge — the LCA is the last shared node. This path-intersection view directly suggests the iterative solution.
"A node can be a descendant of itself": this convention is the subtle part. If p sits on the path from the root to q, then p is the LCA — the answer need not be a strict branch point. Any solution must return a target when the other target is below it.
"Both p and q exist in the tree": you never need to handle a missing target or report "not found." This removes a whole branch of guards and lets the walk-up assume both paths reach the root.
"Plain binary tree, not a BST": you cannot use value ordering to decide which way to descend (that's the easier LC 235). You must actually locate the targets by traversal — hence either a full post-order recursion or a parent map built by one traversal.
"Unique values" but nodes are references: even with unique values, compare nodes by identity (===), not value — it is the correct and general habit, and sets must be keyed by ObjectIdentifier since TreeNode isn't Hashable.

The "ancestor of itself" cases are where solutions break. p-is-ancestor-of-q, p==q, and target-is-root all rely on a node counting as its own ancestor. The walk-up handles them naturally because p's ancestor set includes p; the recursion handles them by returning a node the moment it equals p or q, before checking subtrees. Either way, including self is the rule that covers all three.

4   Try It Yourself First

Aim for the iterative parent-map solution (the recursion is a great contrast after). Open hints one at a time.

Hint 1 — the constraint

The LCA is the last node shared by the root-to-p path and the root-to-q path. If you could walk up from each target, you'd find where the paths meet. The tree only has downward links — so how do you get upward ones?

Hint 2 — the pattern

Do one traversal (BFS or DFS) recording each node's parent in a dictionary — now you have upward links. Walk p up to the root, putting every node (including p) into a set. Then walk q up; the first node already in the set is the LCA.

Hint 3 — identity and self-ancestry

Compare nodes by reference (===) and key the set by ObjectIdentifier — TreeNode isn't Hashable. Make sure p's ancestor set includes p itself, so the case "p is an ancestor of q" correctly returns p. (Alternative: a post-order recursion where each node returns itself if it equals p or q, and the node whose left and right both return non-nil is the LCA.)

Step 2 — The plan, before any code
The plan, spoken

"This is the post-order LCA — Variant 8.2 of the Trees / LCA block. The signal is 'lowest common ancestor in a bare binary tree.' I'll frame it as path intersection: the LCA is the last node shared by the root-to-p and root-to-q paths."

"Let me pin the conditions before the structure. The key fact: the tree only links downward, so to find where the upward paths meet I first need upward links — a parent map. The invariant after building it is: parent[node] gives each node's parent, for at least all nodes on the paths to p and q. The governing condition for the answer: walking p up to the root, its ancestor set includes p itself; then walking q up, the first node already in that set is the LCA — because that is the deepest node on both paths. The self-ancestor cases (p above q, p == q, target is root) are covered precisely because the set includes the start node."

"On structures: a parent map keyed by ObjectIdentifier (nodes aren't Hashable), built by one BFS or DFS from the root, stopping once both p and q have been recorded. Then a Set<ObjectIdentifier> for p's ancestors. Reference identity throughout."

"The walk: BFS from the root filling parent until both targets are seen. Walk p upward via parent, inserting each into the ancestor set. Walk q upward, returning the first node whose identity is in the set. Time O(n) to build the map in the worst case; space O(n) for the map and set. I'll also show the elegant recursive post-order version as a contrast — it's O(n)/O(h) and doesn't need the explicit map."

6   Why This Approach — and not the others

CandidateCostVerdict
Find the full root-to-p and root-to-q paths as lists, compare prefixesO(n) time, O(n) spaceCorrect and intuitive, but stores two whole paths and compares them position by position — more bookkeeping than the set intersection, which stops at the first hit.
BST value-guided descent (LC 235 style)O(h)Inapplicable. This is a plain binary tree — there is no value ordering to decide left vs. right. Only works for BSTs.
Parent map + ancestor-set walk-up (iterative)O(n) / O(n)Chosen (primary). One traversal for upward links, then two short walk-ups intersected by a set. Explicit, no recursion, and the same technique the region-hierarchy target reuses.
Post-order recursion that bubbles the answer upO(n) / O(h)Chosen (contrast). The most elegant form — no explicit map, O(h) space. Shown in §9b; both are interview-acceptable.
The deciding signal: "common ancestor" / "where the paths meet." When you can build upward links, LCA becomes a path intersection — collect one node's ancestors, walk the other up to the first shared one. The same parent-map walk-up solves parent-pointer LCA (LC 1650) and the region-hierarchy LCA (LC 1257); learning it here transfers directly.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
parent: [ObjectIdentifier: TreeNode]Each visited node's parent (upward links)The tree only links downward; one traversal records the upward links needed to walk from a target to the root. Keyed by ObjectIdentifier because TreeNode isn't Hashable.
ancestors: Set<ObjectIdentifier>The identities of all of p's ancestors, including pA set gives O(1) "is this q-ancestor already a p-ancestor?" tests during the second walk-up. Including p itself covers the self-ancestor cases.
BFS queue (build phase)Frontier of nodes whose children still need parent links recordedA level-order sweep records parents simply; can stop early once both targets are mapped. A DFS stack works identically.
reference identity (=== / ObjectIdentifier)(no storage) node equalityNodes are reference types; identity is the correct equality and survives duplicate values, unlike comparing .val.
Why ObjectIdentifier and not the node itself. Swift Set and Dictionary keys must be Hashable; TreeNode generally isn't, and even if it were by value, duplicate values would collide. ObjectIdentifier(node) wraps the node's stable reference identity into a Hashable key — exactly "this specific node instance" — which is what LCA needs.

8   Pseudocode Dry Run

Pseudocodeparent map, then intersect the two walk-ups
parent ← { root : null }
queue ← [root]
WHILE p not in parent OR q not in parent:    stop once both mapped
    node ← dequeue(queue)
    FOR child IN [node.left, node.right] where child ≠ null:
        parent[child] ← node ; enqueue(child)

ancestors ← empty set
node ← p
WHILE node ≠ null:                  collect all of p's ancestors (incl. p)
    ancestors.insert(node) ; node ← parent[node]

node ← q
WHILE node not in ancestors:        walk q up to the first shared one
    node ← parent[node]
RETURN node

9   Swift Solution — iterative

The node is the standard LeetCode TreeNode class. The main solution is iterative (parent map + ancestor set); the recursive post-order form follows as a contrast in §9b.

A.1 Parent map + ancestor-set walk-up
O(n)O(n)
Plan
BFS from the root recording each node's parent until both targets are mapped — giving upward links. Walk p up to the root, inserting every node (including p) into an ancestor set. Walk q up; the first node whose identity is in the set is the lowest common ancestor. 
func lowestCommonAncestor(_ root: TreeNode?, _ p: TreeNode?, _ q: TreeNode?) -> TreeNode? {
    guard let root = root, let p = p, let q = q else { return nil }

    // 1. One traversal to record upward links (child → parent).
    var parent: [ObjectIdentifier: TreeNode] = [:]
    var queue: [TreeNode] = [root]
    parent[ObjectIdentifier(root)] = nil

    while parent[ObjectIdentifier(p)] == nil || parent[ObjectIdentifier(q)] == nil {
        let node = queue.removeFirst()
        if let left = node.left {
            parent[ObjectIdentifier(left)] = node ; queue.append(left)
        }
        if let right = node.right {
            parent[ObjectIdentifier(right)] = node ; queue.append(right)
        }
    }

    // 2. Collect all of p's ancestors (p included — a node is its own ancestor).
    var ancestors: Set<ObjectIdentifier> = []
    var walker: TreeNode? = p
    while let node = walker {
        ancestors.insert(ObjectIdentifier(node))
        walker = parent[ObjectIdentifier(node)]
    }

    // 3. Walk q up to the first shared ancestor — that is the LCA.
    walker = q
    while let node = walker, !ancestors.contains(ObjectIdentifier(node)) {
        walker = parent[ObjectIdentifier(node)]
    }
    return walker
}
Note: parent[ObjectIdentifier(root)] = nil is stored conceptually as "root has no parent"; in Swift assigning nil to a dictionary removes the key, which is fine — the root's walk-up simply ends when parent[...] returns nil. The stop-early while condition avoids mapping the whole tree when both targets are found shallow.

9b   Recursive post-order form — for contrast

The elegant O(h)-space version: each call returns a target on a match, and the node whose two subtrees both return non-nil is the LCA.

func lowestCommonAncestor(_ root: TreeNode?, _ p: TreeNode?, _ q: TreeNode?) -> TreeNode? {
    guard let node = root else { return nil }
    if node === p || node === q { return node }   // found a target (or its own ancestor)
    let left  = lowestCommonAncestor(node.left,  p, q)
    let right = lowestCommonAncestor(node.right, p, q)
    if left != nil && right != nil { return node }  // split point → LCA
    return left ?? right                          // both finds are on one side
}

12   Complexity

PhaseTimeSpaceWhy
Build parent map (iterative)O(n)O(n)Worst case visits every node once (targets deep/in a skewed tree); the map and queue hold up to n entries.
Two walk-upsO(h)O(h)Each walk follows a root-path of length ≤ height; the ancestor set holds one path.
Recursive post-order (§9b)O(n)O(h)One traversal visiting each node once; space is the call-stack depth = tree height.
TotalO(n)O(n) iter / O(h) recBoth are linear time. The recursion is more space-efficient (O(h)); the iterative map trades space for an explicit, no-recursion structure that generalizes to parent-pointer and region variants.
Why the iterative map is still worth knowing despite costing more space. The recursion is tighter (O(h) space), but the parent-map walk-up is the technique that transfers: LC 1650 (parent pointers given) and LC 1257 (region hierarchy) are the same walk-up with the upward links supplied differently. Learning the explicit version here is what makes those two near-instant.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The recursive post-order version (§9b) is O(n) time and O(h) space — strictly better space than the iterative map, and shorter. I'd ship that unless the interviewer specifically wants no recursion or there's a stack-depth concern on a pathologically skewed tree. One micro-note: removeFirst() on a Swift array is O(n); for a strict O(n) BFS I'd use a head index or a deque instead of removeFirst.
Follow-up: "Now each node has a parent pointer — solve it without the tree root."
That's LC 1650 — LCA III: skip building the parent map (you already have upward links) and go straight to the walk-up — collect p's ancestors in a set, walk q up to the first hit. Or do the O(1)-space two-pointer trick: advance both up, redirecting each to the other's start on reaching the root; they meet at the LCA. Same intersection idea, links supplied for free.
Follow-up: "The hierarchy is given as a list of regions, not a tree — find the smallest region containing both."
That's LC 1257 — Smallest Common Region, LCA in disguise. Build the child→parent map from the region lists, then run this exact walk-up. The only new step is deriving the parent links from the membership lists; once you have them, "smallest common region" is "lowest common ancestor."
Follow-up: "It's a BST instead of a plain binary tree — can you do better?"
Yes — that's LC 235, and it drops to O(h) with no extra space: walk down from the root comparing values. If both targets are smaller than the node, go left; if both larger, go right; the first node where they split (or that equals a target) is the LCA. The value ordering replaces the need to traverse and record parents — the easier sibling of this problem.
