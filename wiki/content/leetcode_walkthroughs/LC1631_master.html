LC 1631 — Path With Minimum Effort

Theory
dijkstra_deep_dive_master.html

You're given a rows × cols grid heights, where heights[r][c] is the height of a cell. A path goes from the top-left (0,0) to the bottom-right (rows−1, cols−1), moving 4-directionally. The effort of a path is the maximum absolute height difference between any two consecutive cells on it. Return the minimum effort over all possible paths — the smallest "worst step" you can guarantee.

InputOutputWhy
[[1,2,2],[3,8,2],[5,3,5]]2The route 1→2→2→2→5 (right along the top, down the right edge) has a max step of 2 — better than going through the 8.
[[1,2,3],[3,8,4],[5,3,5]]1A path exists whose consecutive differences never exceed 1.
[[1,2,1,1,1],[1,2,1,2,1],[1,2,1,2,1],[1,2,1,2,1],[1,1,1,2,1]]0A path staying on equal-height cells exists → effort 0.

The twist is the cost model: a path's cost is not the sum of its steps — it's the single worst step along it. That changes the shortest-path relaxation from "add the edge weight" to "take the max with the edge weight." Everything else is Dijkstra: a min-heap settles the cell of least worst-step-so-far first, and the first time you pop the destination, that effort is the answer.

2   Reading the Problem — insights from the constraints

"Effort = maximum difference along the path": the cost of a path is its worst edge, not its total. This is a "min-max" objective — minimize the maximum. It rules out plain BFS (equal-cost) and classic sum-Dijkstra, but keeps Dijkstra's skeleton with a different combine rule: max instead of +.
"Minimum effort over all paths": minimizing a max-of-edges is still a shortest-path problem — the "distance" to a cell is the minimum possible worst-step to reach it. Dijkstra's settle-cheapest-first works because max is monotonic with non-negative inputs: extending a path can never lower its worst step, so once a cell is popped its minimum effort is final.
"4-directional moves": the grid is an implicit weighted graph — each cell is a vertex, each move to an orthogonal neighbor is an edge whose weight is the absolute height difference. Never build an adjacency list; generate neighbors from a 4-direction table and compute the weight on the fly.
Heights are non-negative and differences are absolute → all edge weights ≥ 0: Dijkstra is valid (no negative edges). The effort values are bounded (≤ 10⁶ per the constraints), so an Int effort grid initialized to a large sentinel works cleanly.
Grid up to 100×100 (10⁴ cells): O(R·C·log(R·C)) Dijkstra is fast. Swift has no built-in priority queue, so you must hand-roll a min-heap — flag that out loud; it's the one real implementation cost.

The single-cell and the detour cases are the load-bearing ones. A 1×1 grid must return 0 (no steps), which falls out of initializing the start's effort to 0 and popping it first. The detour case is the whole point of using Dijkstra over a greedy walk: the cheapest worst-step route may wind around a steep cell, and only settle-cheapest-first guarantees you find it.

4   Try It Yourself First

The leap is "min-max is still Dijkstra." Open hints one at a time.

Hint 1 — what kind of cost is this?

A path's effort is the largest single step on it, not the sum. You want to minimize that maximum. Equal-cost BFS won't do it (steps differ), and you're not adding weights — so what shortest-path algorithm bends to a custom combine rule?

Hint 2 — the pattern

Dijkstra. Treat each cell as a vertex, each 4-directional move as an edge weighted by |heightDiff|. The "distance" to a cell is the minimum possible worst-step to reach it. Use a min-heap ordered by effort; the first time you pop the bottom-right cell, that effort is the answer.

Hint 3 — the one changed line

Relaxation uses max, not +: the effort to reach a neighbor through the current cell is max(effortSoFar, |height[cur] − height[neighbor]|). Keep an effort[][] grid (init to a big sentinel, start at 0), push improved cells, and skip stale heap pops. Swift has no priority queue — hand-roll a min-heap.

6   Why This Approach — and not the others

CandidateCostVerdict
Plain BFSO(R·C)Wrong. BFS finds fewest steps, but here steps have different "weights" (height diffs) and we minimize the max weight, not the count. BFS has no notion of a worst-edge cost.
Classic sum-DijkstraO(RC log RC)Wrong objective. Summing the height differences answers "least total climb," not "least worst step." Right algorithm, wrong combine rule — fix it by relaxing with max.
Binary search on effort + BFS/DFS feasibilityO(R·C·log(maxH))Correct alternative — guess an effort cap E, check if a path using only steps ≤ E exists via BFS/DFS, binary-search the smallest feasible E. Competitive complexity; more moving parts. Good follow-up.
Union-Find (Kruskal-style)O(RC·α + sort)Correct and elegant — sort edges by weight, union cells until start and goal connect; the last edge's weight is the answer. A strong alternative worth naming.
Dijkstra with max-relaxationO(RC log RC)Chosen. The most direct: Dijkstra's skeleton, one operator swapped (+ → max). Single pass, no outer search loop.
The deciding signal: "minimize the maximum" over a weighted path. That phrase is min-max Dijkstra — keep the heap-relax loop, swap + for max. Binary-search-on-answer and Union-Find are the two equally-valid alternatives to name; Dijkstra is the cleanest single-pass form.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
min-heap of (effort, row, col)The frontier, ordered by effort-so-farDijkstra needs the least-effort cell next; a heap gives O(log n) extract-min. Swift has no built-in PQ, so it's hand-rolled. A plain queue (BFS) can't order by varying effort.
effort: [[Int]] gridBest known worst-step to reach each cellThe "distance" array of Dijkstra, one entry per cell. Initialized to a large sentinel; start at 0. Lets the relax step compare and the lazy-skip detect stale pops.
4-direction tableThe up/down/left/right offsetsUniform neighbor generation on the implicit grid graph — no adjacency list to build.
lazy-deletion skip (no separate visited)(no storage) staleness testif poppedEffort > effort[cell] { continue } — a cell may sit in the heap multiple times; only its first (cheapest) pop is valid, replacing a decrease-key operation.
Why the effort grid doubles as the visited check. Classic graph traversal keeps a separate visited set, but Dijkstra folds it into the distance array: a cell is "settled" once popped with poppedEffort == effort[cell], and any later, larger-effort copy is skipped. Storing the best-known effort per cell is exactly what both relaxation and lazy deletion need, so no extra visited structure is required.

8   Pseudocode Dry Run

Pseudocodemin-max Dijkstra — relax with max, settle least-effort first
effort[*][*] ← ∞ ; effort[0][0] ← 0
heap ← min-heap with (0, 0, 0)            (effortSoFar, row, col)

WHILE heap not empty:
    (e, r, c) ← pop-min(heap)
    IF (r,c) == (rows-1, cols-1): RETURN e   destination settled = answer
    IF e > effort[r][c]: CONTINUE          stale entry
    FOR (nr,nc) IN 4-neighbors in bounds:
        stepCost ← |height[r][c] - height[nr][nc]|
        newEffort ← max(e, stepCost)          the min-max combine rule
        IF newEffort < effort[nr][nc]:
            effort[nr][nc] ← newEffort
            push(heap, (newEffort, nr, nc))
RETURN effort[rows-1][cols-1]

9   Swift Solution — iterative

A hand-rolled min-heap (Swift has no priority queue) plus the Dijkstra relax loop with max as the combine rule.

A.1 Min-max Dijkstra with a hand-rolled heap
O(RC log RC)O(RC)
Plan
Keep an effort grid (min worst-step to each cell), start cell at 0. Push the start into a min-heap ordered by effort. Pop the least-effort cell; if it's the destination, return its effort. Otherwise relax each in-bounds neighbor with max(effortSoFar, |heightDiff|), updating and pushing improved cells. Skip stale pops. 
// Minimal generic min-heap — Swift has no built-in priority queue.
struct Heap<Element> {
    private var items: [Element] = []
    private let before: (Element, Element) -> Bool
    init(_ before: @escaping (Element, Element) -> Bool) { self.before = before }
    var isEmpty: Bool { items.isEmpty }
    mutating func push(_ value: Element) {
        items.append(value)
        var child = items.count - 1
        while child > 0, before(items[child], items[(child-1)/2]) {
            items.swapAt(child, (child-1)/2); child = (child-1)/2
        }
    }
    mutating func pop() -> Element? {
        guard !items.isEmpty else { return nil }
        items.swapAt(0, items.count - 1)
        let top = items.removeLast()
        var parent = 0
        while true {
            let l = 2*parent+1, r = 2*parent+2; var best = parent
            if l < items.count, before(items[l], items[best]) { best = l }
            if r < items.count, before(items[r], items[best]) { best = r }
            if best == parent { break }
            items.swapAt(parent, best); parent = best
        }
        return top
    }
}

func minimumEffortPath(_ heights: [[Int]]) -> Int {
    let rows = heights.count, cols = heights[0].count
    let directions = [(0,1),(0,-1),(1,0),(-1,0)]

    var effort = [[Int]](repeating: [Int](repeating: Int.max, count: cols), count: rows)
    effort[0][0] = 0
    var heap = Heap<(effort: Int, row: Int, col: Int)>({ $0.effort < $1.effort })
    heap.push((0, 0, 0))

    while let (currentEffort, row, col) = heap.pop() {
        if row == rows - 1 && col == cols - 1 { return currentEffort }  // goal settled
        if currentEffort > effort[row][col] { continue }              // stale entry

        for (dr, dc) in directions {
            let nr = row + dr, nc = col + dc
            guard nr >= 0, nr < rows, nc >= 0, nc < cols else { continue }
            let stepCost = abs(heights[row][col] - heights[nr][nc])
            let newEffort = max(currentEffort, stepCost)         // min-max combine
            if newEffort < effort[nr][nc] {
                effort[nr][nc] = newEffort
                heap.push((newEffort, nr, nc))
            }
        }
    }
    return effort[rows - 1][cols - 1]   // reached via the grid (1×1 case: 0)
}
Note: the only line that distinguishes this from classic Dijkstra is max(currentEffort, stepCost) in place of currentEffort + stepCost — the cost of a path is its worst step, so the combine is max. The destination check on pop gives an early exit; the currentEffort > effort[row][col] guard discards stale heap entries (lazy deletion) so no separate visited set is needed.

12   Complexity

PhaseTimeSpaceWhy
Dijkstra peelO(R·C·log(R·C))O(R·C)Up to O(R·C) cells, each pushed a constant number of times (4 neighbors); every heap op is O(log(R·C)).
Per-cell relaxationO(4) = O(1)O(1)Fixed 4-direction table; abs-diff and compare are constant.
TotalO(R·C·log(R·C))O(R·C)The heap's log factor dominates — the price of ordering cells by varying effort. The effort grid + heap are both O(R·C). Binary-search-on-answer (§6 follow-up) is O(R·C·log maxHeight), comparable.
Why the log factor is unavoidable here. Unlike grid BFS (O(R·C) with a plain queue), the steps have different efforts, so the frontier must be ordered by effort — that's the heap, and extract-min costs O(log n). The moment edge costs vary, you pay the log to keep settling the cheapest cell first; it's the same reason Dijkstra costs more than BFS in general.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The heap is the right structure; the main note is correctness hygiene — the lazy-deletion skip avoids reprocessing settled cells, and I return on popping the goal for an early exit. If the grid were enormous and memory tight, the effort grid is unavoidable (one Int per cell), but I could encode (row,col) as a single Int in the heap entries to shrink them. Otherwise it's already O(R·C·log(R·C)).
Follow-up: "Solve it without Dijkstra."
Two clean alternatives. Binary search on the answer: guess an effort cap E, then BFS/DFS using only steps ≤ E to check if the goal is reachable; binary-search the smallest feasible E — O(R·C·log maxHeight). Union-Find: sort all edges by |height diff|, union cells in increasing weight order, and the moment start and goal are connected, the last edge's weight is the answer. Both are standard min-max-path techniques.
Follow-up: "What if the cost were the sum of the height differences instead of the max?"
Then it's classic Dijkstra — change the one line from max(currentEffort, stepCost) back to currentEffort + stepCost. The entire skeleton (heap, effort grid, lazy skip, early exit) is identical; only the combine rule differs. That's exactly the relationship between this problem and the canonical single-source shortest path.
Follow-up: "What if diagonal moves were allowed?"
Add the four diagonal offsets to the direction table — eight neighbors instead of four. Nothing else changes; the relaxation and heap are agnostic to how many neighbors a cell has. The minimum effort can only stay the same or drop, since diagonals add shortcuts.
