Given a string containing only the six bracket characters (), [], and {}, determine whether it is valid. A string is valid when every opening bracket is closed by a bracket of the same type, brackets close in the correct order, and every closing bracket has a matching opener. In short: brackets must nest properly, like well-formed math or code.

InputOutputWhy
"()[]{}"trueEach opener is immediately closed by its own type, in order
"([{}])"trueProperly nested — the innermost {} closes first, then [], then ()
"(]"falseAn opener ( is closed by the wrong type ]

The twist — and the reason this is the warm-up for the whole iterator family — is the realization that "closes in the correct order" means last opened, first closed: last-in-first-out. That is precisely a stack, and the same LIFO discipline powers the lazy nested-list iterator later in the block.

2   Reading the Problem — insights from the constraints

"Closed in the correct order": this is the whole problem. The bracket opened most recently is the one that must close next — last-in-first-out. That single phrase dictates a stack: push each opener, and when a closer arrives it must match whatever is on top.
"By the same type": a ( must be closed by ), never by ] or }. So matching is not just "is the stack non-empty" — the popped opener must be the specific partner of this closer. A small map from each closer to its opener makes the check one lookup.
Every closer needs an opener: a closing bracket with an empty stack (nothing was opened) is invalid — "]" fails immediately. The pop must be guarded: empty stack on a closer means failure, not a crash.
Every opener needs to close: after scanning the whole string, anything left on the stack is an unclosed opener — "(" alone is invalid. The final verdict is "the stack is empty," not just "we never hit a mismatch."
Only six characters, string up to 10⁴: a single O(n) pass with O(1) work per character is plenty. There is no asymptotic challenge; the value is the clean stack mechanics, which is why it anchors the family.

3   Edge Case Inventory

InputExpectedHandled by
"" — empty stringtrueThe loop never runs; the stack is empty at the end → valid (vacuously).
"(" — lone openerfalsePushed and never popped; the final "stack empty?" check fails.
"]" — lone closerfalseCloser with an empty stack → the pop guard returns false.
"(]" — wrong typefalseTop is (, closer is ]; the map says ] wants [ ≠ ( → mismatch.
"([)]" — wrong orderfalseWhen ) arrives the top is [, not ( → mismatch. Interleaved nesting is the classic trap.
"()[]{}" — flat sequencetrueEach opener pops immediately on its closer; stack returns to empty between pairs.
"((()))" — deep same-type nestingtrueThree pushes then three matching pops, LIFO order.
GOTCHA: odd-length stringalways falseAn odd count can never fully pair; falls out of the empty-stack check, though an early count % 2 != 0 short-circuit is a nice optimization.
The two failure modes are distinct and both required. A string fails either by a mismatch during the scan (wrong type or a closer with nothing to match) or by a non-empty stack at the end (unclosed openers). Checking only one misses half the invalid inputs — "(]" needs the mismatch check, "(" needs the end check.

4   Try It Yourself First

This is the family's warm-up — quick, but get the two failure modes right. Try it cold, then open hints one at a time.

Hint 1 — the constraint

"Brackets must close in the correct order" means the most recently opened bracket is the next that must close. What data structure gives you "most recent first"?

Hint 2 — the pattern

A stack. Scan the string: push every opening bracket. On a closing bracket, the top of the stack must be its matching opener — pop it. Keep a small map from each closer to its opener so the match is one lookup.

Hint 3 — the two ways to fail

Return false the instant a closer either has an empty stack (no opener) or doesn't match the top (wrong type/order). And after the whole scan, return whether the stack is empty — a leftover opener means unclosed brackets, which is also invalid.

A: A clean pass is fine. → I'll return false on the first mismatch and check emptiness at the end; an odd-length short-circuit is a cheap bonus.
Step 2 — The plan, before any code
The plan, spoken

"This is stack fundamentals — Variant 8.1 of the Iterator Design family, the warm-up that establishes the LIFO mechanics the lazy iterators reuse. The signal is 'brackets must close in the correct order,' which is last-in-first-out."

"Let me pin the conditions before the structure. The invariant the stack maintains is: it holds exactly the currently-open brackets, in the order they were opened, top = most recent. The validity condition, stated as it'll be coded, has two parts. During the scan: every closing bracket must find a matching opener on top — if the stack is empty or the top is the wrong type, the string is invalid and I stop. After the scan: the string is valid if and only if the stack is empty, because any leftover is an unclosed opener. So the loop returns false early on a mismatch, and the function returns stack.isEmpty at the end."

"The structures follow: a stack of characters for the open brackets, and a constant map from each closing bracket to its required opener — )→(, ]→[, }→{. The map turns the type-match into a single lookup instead of a three-way branch."

"The walk: for each character, if it's a closer, check the stack is non-empty and its top equals map[closer]; if so pop, else return false. If it's an opener, push it. After the loop, return whether the stack is empty. Time O(n) — one pass, O(1) per character; space O(n) for the stack in the worst case of all openers."

6   Why This Approach — and not the others

CandidateCostVerdict
Repeatedly replace "()", "[]", "{}" with "" until stableO(n²)Cute and correct — a valid string collapses to empty — but each pass is O(n) and you may need O(n) passes. Quadratic, and string replacement is costly.
Count each bracket type's opens and closesO(n)Wrong. Counts ignore order: "([)]" has balanced counts but is invalid. Counting can't capture nesting.
Stack: push openers, pop-match closersO(n) / O(n)Chosen. One pass, captures order exactly via LIFO, two clean failure checks. The canonical answer.
The deciding signal: "things must close/resolve in the reverse of the order they opened." Whenever nesting or correct-order matching appears — brackets, tags, nested scopes — a stack is the structure, because LIFO is nesting. Counting handles balance but never order; the stack handles both.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
stack: [Character]The open brackets not yet closed, most recent on topLIFO matches "last opened, first closed." An array with append/removeLast gives O(1) push/pop. A counter can't capture order; a queue would give the wrong (FIFO) discipline.
matching: [Character: Character]Each closer → its required openerTurns the type-match into one lookup (matching[closer] == stack.last) instead of a three-armed if/switch. A constant three-entry map.
the "ends empty" check(no storage) the final verdictstack.isEmpty after the loop captures the "no unclosed openers" half of validity in one expression.
Why a closer-keyed map, not an opener-keyed one. The decision point is when a closer arrives — that's when we need to know what opener it expects. Keying the map by the closer means the lookup happens exactly where the information is needed. Keying by opener would force a reverse search. Match the map's key to the moment you query it.

9   Swift Solution — iterative

A.1 Stack with a closer→opener map
O(n)O(n)
Plan
Scan the characters. Push each opening bracket. On a closing bracket, the stack must be non-empty and its top must be the matching opener (via a small map) — pop it, or return false. After the scan, the string is valid iff the stack is empty. 
func isValid(_ s: String) -> Bool {
    let matching: [Character: Character] = [")": "(", "]": "[", "}": "{"]
    var stack: [Character] = []

    for character in s {
        if let expectedOpener = matching[character] {
            // Closer: the top must be its matching opener.
            guard stack.last == expectedOpener else { return false }
            stack.removeLast()
        } else {
            // Opener: remember it for its future closer.
            stack.append(character)
        }
    }

    // Valid iff every opener was closed — nothing left over.
    return stack.isEmpty
}
On the guard. stack.last is an Optional: when the stack is empty it is nil, which is never equal to expectedOpener, so the single guard stack.last == expectedOpener handles both the empty-stack case and the wrong-type case at once — no separate isEmpty check needed before the pop.

12   Complexity

PhaseTimeSpaceWhy
ScanO(n)O(n)One pass; each character does O(1) work — a map lookup and a push or pop.
TotalO(n)O(n)Linear time, dominated by the single scan. Space is O(n) in the worst case of all openers ("((((("), when the stack holds every character.
Why the stack can't be avoided. Order-sensitive matching needs memory of which brackets are open and in what order — that is inherently O(n) space in the worst case. The counting approach is O(1) space but wrong; correctness here costs the stack. This is the trade the warm-up makes explicit before the lazy iterators reuse the same stack for descent.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
It's already O(n)/O(n), which is optimal for order-sensitive matching. Two micro-improvements: short-circuit with if s.count % 2 != 0 { return false } since an odd length can never pair, and reserveCapacity(s.count / 2) on the stack to avoid reallocations. Neither changes the asymptotics; they shave the constant on adversarial inputs.
Follow-up: "Now the string can contain other characters too — ignore non-bracket characters."
Add a guard: if the character is neither a known opener nor a closer, continue. The opener set and the closer map define what's relevant; everything else is skipped. The stack logic is unchanged — this is exactly how a real code/HTML validator ignores text between the structural tokens.
Follow-up: "Return the index of the first invalid character instead of just true/false."
Track the index in the loop (enumerate). On a mismatch or empty-stack closer, return that index. At the end, if the stack is non-empty, the first unclosed opener's index is the answer — so store indices on the stack instead of characters, and return the bottom-most leftover. The structure is the same; the stack just carries (char, index) pairs.
Follow-up: "How does this connect to the rest of the iterator family?"
This is the warm-up that proves the LIFO stack mechanic. The next problem, LC 341 — Flatten Nested List Iterator, reuses the exact same stack — but instead of matching brackets, it pushes nested lists and pops them to descend lazily. Master the push/pop discipline here and the iterator's hasNext/next becomes "the same stack, used to pause and resume a traversal."
