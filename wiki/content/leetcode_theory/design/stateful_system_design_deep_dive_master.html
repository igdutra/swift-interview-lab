Stateful system design problems ask you to implement a class — a constructor plus a set of methods — that is driven by an unknown sequence of calls, where each method reads or changes internal state. Unlike a single-function problem, there is no one input to transform; there is a lifetime of operations, and your object must answer every query correctly no matter the order. The algorithmic content is light. The real work is design: pick the data structure that makes each operation correct and fast, and keep an invariant — a property that is always true between calls — so you never have to reconstruct the past.

Every problem in this family turns on a single decision: what do you store, and what do you derive? You can keep the raw history and compute the answer on each query, or maintain a small derived summary that you update incrementally on each mutation so queries are trivial. That choice — recompute-on-read versus maintain-on-write — is the axis this page is built around, and it is what an interviewer is really probing when they ask you to "design" something. This deep dive lives under the Object & System Design hub, alongside iterator design.

4   Mental Model — store vs. derive

The one idea
Before writing a single method, ask: "what is the minimum I must store so that every query is answerable, and is there a derived summary I can keep up to date on each mutation to make the hot query O(1)?" Designing state is choosing what to remember. Everything you don't store, you must recompute — and everything you derive, you must keep consistent on every write.

The whole family is the tension between two strategies. Take tic-tac-toe: a move(row, col, player) is called repeatedly and after each you must answer "did this move win?" Strategy one, recompute-on-read: store the whole board, and on each move scan that move's row, column, and the two diagonals — O(n) per move. Strategy two, maintain-on-write: store only running totals per row, per column, and per diagonal; player 1 adds +1, player 2 adds −1; a total hitting ±n is a win — O(1) per move. Watch the second strategy run on a 3×3 board, player 1 marking a diagonal:

Tracemaintain-on-write: ±1 totals, win when |total| == n
n=3  rows=[0,0,0]  cols=[0,0,0]  diag=0  anti=0   (player 1 = +1)

move(0,0,P1)  rows[0]=1  cols[0]=1  diag+=1 → 1     |any|==3? no
move(1,1,P1)  rows[1]=1  cols[1]=1  diag+=1 → 2     anti (r+c==2) += 1 → 1
              |any|==3? no
move(2,2,P1)  rows[2]=1  cols[2]=1  diag+=1 → 3          |diag|==3? YES → P1 wins

No board scan ever happens. The diagonal total reaching 3 is the win — the derived aggregate answers the query in O(1).

Now the other side — a file system's get(path). Here there is nothing to aggregate; the answer is the stored value. The design is purely "what is the key?" A path string maps directly to its value, so the store is a dictionary and the query is one lookup:

Tracemap-backed: the path IS the key, no derivation
store = [:]   (path → value)

createPath("/a", 1)    parent "/" exists (root) → store["/a"]=1 → true
createPath("/a/b", 2)  parent "/a" in store?  yes → store["/a/b"]=2 → true
createPath("/c/d", 3)  parent "/c" in store?  no → false  (state unchanged)
get("/a/b")            → store["/a/b"] = 2

The only design decision is the validity rule: a path is creatable only if its parent already exists. The map does the rest.

Recognizing which side you are on — is the answer stored or derived? — is what lets you pick the data structure before you write any method.

5   Pseudocode Primer

Every stateful design is this shape: a constructor that sets up the store, mutators that change it (validating first), and queries that read it (recomputing or returning a derived value). The VALIDATE and UPDATE DERIVED lines are the only parts that vary.

Pseudocodethe universal stateful-object shape
CLASS System:
    INIT(config):
        store   ← empty map / array / graph     the raw state
        derived ← initial aggregates (sums, counts)  optional summary

    METHOD mutate(key, value):
        IF not VALIDATE(key, value):       preconditions first
            RETURN failure                  leave state untouched
        store[key] ← value
        UPDATE DERIVED for this change      maintain the invariant
        RETURN success

    METHOD query(key):
        RETURN derived[...]  OR  recompute from store[key]
Why validate before mutating: a method that fails its precondition must leave the object exactly as it was — partial mutations corrupt the state for every later call. Check all preconditions first, then commit the change atomically. This is the difference between a design that survives an adversarial call sequence and one that drifts into an impossible state.

6   Iterative-Only Pattern — methods, not recursion

Stateful designs are inherently call-driven rather than recursive, but two rules keep them clean:

  • Maintain the invariant on every write, never lazily. If a derived total is supposed to always be correct, update it inside the same method that mutates the underlying state — not "later" or "on next read." The invariant is only useful if it actually holds between every pair of calls. The moment you defer the update, a query in between sees stale data.
  • When a mutation cascades, drive the cascade with an explicit queue, not recursion. In the dependency-recompute variant, setting one cell can force re-evaluating its dependents, and theirs in turn. A breadth-first sweep with an explicit queue (or a topological order) is safer than recursion: it cannot overflow the call stack on a deep chain, and you can detect cycles by tracking visited nodes.
Why this matters here specifically: the bugs in this family are almost never algorithmic — they are consistency bugs. State that is updated in one place but read in another, or a cascade that recurses into a cycle and never returns. Keeping the invariant update co-located with the mutation, and making cascades explicit, is what makes those bugs impossible rather than merely unlikely.

7   Swift Foundations

Four Swift facts underlie nearly every design problem in this family. Say them out loud before coding.

Swiftthe four state-handling rules for this block
// 1. Dictionary is your default store. O(1) average insert / lookup / update.
var store: [String: Int] = [:]
store["/a"] = 1                 // insert or overwrite
if let value = store["/a"] { }    // presence test + read in one step

// 2. A class gives reference semantics — needed when nodes share/point to state.
class Cell { var value = 0; var dependents: [Cell] = [] }
// Two references to the same Cell see each other's mutations — unlike a struct.

// 3. Derived aggregates are plain arrays sized once in init.
var rowSum  = [Int](repeating: 0, count: n)
var colSum  = [Int](repeating: 0, count: n)
var diagonal = 0, antiDiagonal = 0

// 4. Validate, then mutate. Guard preconditions and return early on failure.
func transfer(_ from: Int, _ to: Int, _ amount: Int) -> Bool {
    guard isValid(from), isValid(to), balance[from] >= amount else { return false }
    balance[from] -= amount; balance[to] += amount        // commit only after all checks
    return true
}
The struct-vs-class trap. Use a struct for value-type state (totals, balances) and a class when nodes must point to and mutate each other (a dependency graph, a linked structure). Copying a struct when you needed shared references — or sharing a class when you wanted an independent copy — is the most common modeling bug in this block.

8   The Five Variants

Ordered by increasing design difficulty. The first three are taught in full walkthroughs; the last two are solve-it-yourself targets — frame and pitfalls here, no solution.

Variant 8.1 Map-backed state
O(1) / opO(keys)
When to use: the answer to a query is a stored value — there is nothing to aggregate. The design is "what is the key?" A dictionary keyed by the identifier (a path, an id, a name) gives O(1) create/set/get. The only subtlety is the validity rule on creation. Full Swift in §7; full walkthrough is LC1166.

The frame (LC 1166 — Design File System): a single [String: Int] maps each path to its value. createPath(path, value) succeeds only if the path is absent and its parent path already exists (the root / is the implicit base); get(path) returns the stored value or −1. Deriving the parent is pure string work: drop the last /-component.

The design decision: store full paths as keys rather than building an actual tree of folders. A tree would let you walk children, but no operation here needs that — every operation is keyed by a full path, so the flat map is simpler and O(1). Choosing the representation to fit the operations, not the real-world metaphor, is the lesson.

Canonical problem: LC 1166 — Design File System (warm-up). Full walkthrough.

Variant 8.2 Incremental aggregate state
O(1) / opO(n)
When to use: a query asks for an aggregate ("has anyone won?", "what's the current max?") after each mutation. Instead of rescanning, keep a running summary and update it by a delta on every write, so the query is O(1). Full Swift in the walkthrough (LC348).

The frame (LC 348 — Design Tic-Tac-Toe): keep rowSum[i], colSum[j], a single diagonal, and a single antiDiagonal. Player 1 contributes +1, player 2 contributes −1. On move(r, c, player), add the player's delta to rowSum[r], colSum[c], and (if r==c) the diagonal, and (if r+c==n−1) the anti-diagonal. A total whose absolute value reaches n means that line is full of one player — a win.

The design decision: the ±1 encoding collapses "all same player" into a single arithmetic test. Two players, one number per line, one comparison — no board scan, no per-cell inspection. Picking an encoding that turns the query into arithmetic is the move.

Canonical problem: LC 348 — Design Tic-Tac-Toe (walkthrough). Full walkthrough.

Variant 8.3 Streaming window state (queue as state)
O(1) amortized / callO(w)
When to use: you are implementing a class whose method is called repeatedly over time, and the answer to each call depends on a window of past calls that must be remembered across invocations. The object's state is the queue — it persists between calls, growing and shrinking as items enter and leave the window. Full walkthroughs at LC933 and LC362.
This is the key distinction from a one-shot function
If this were a single function over a static array, two pointers or a prefix sum would apply — the array is fixed and both ends are addressable. But here you are implementing a class whose state survives between calls. Each call adds one item and asks "how many items are in the window right now?" That streaming, call-by-call nature forces a data structure that remembers — specifically a queue that holds the live window and evicts the stale front on each call.

The frame (LC 933 — Number of Recent Calls): a RecentCounter class with a single ping(t) method. Each call enqueues t at the back, evicts any front items older than t − 3000, then returns the queue count. The queue is the entire state of the class — there is no other field needed.

SwiftVariant 8.3 skeleton — the queue IS the object's state
class RecentCounter {
    private var window: [Int] = []   // this IS the state — it persists across calls

    func ping(_ timestamp: Int) -> Int {
        window.append(timestamp)                     // enqueue — O(1) amortized
        while let oldest = window.first,
              oldest < timestamp - 3000 {
            window.removeFirst()                    // evict stale — amortized O(1)
        }
        return window.count
    }
}
// Each timestamp enqueued once, dequeued at most once → O(n) total, O(1) amortized per call.
// Compare: a one-shot function over a static array would use two pointers — no queue, no class.
What "state persists between calls" actually means. After ping(1) returns, the window still holds [1]. After ping(100), it holds [1, 100]. After ping(3001), the eviction loop removes 1 (since 1 < 3001 − 3000 = 1 is false — it stays), and after ping(3002) it finally evicts 1. The queue bridges calls; without the class wrapper the queue would vanish between calls. That is the design problem: not what algorithm to run, but what to store so the next call can pick up exactly where the last one left off.

Variant 8.4 Validated stateful operations
O(1) / opO(accounts)
Solve-it-yourself variant. This is an attempt problem — the frame and the validation rules are below, but no solution. Build it cold.
When to use: every method must reject invalid calls and leave state untouched on failure. The state is simple; the discipline is checking every precondition before committing any mutation.

The frame (LC 2043 — Simple Bank System): an array balance[1...n] of account balances. Three operations, each returning whether it succeeded:

  • transfer(from, to, money) — valid only if both accounts exist (1-indexed, in range) and balance[from] >= money.
  • deposit(account, money) — valid only if the account exists.
  • withdraw(account, money) — valid only if the account exists and balance[account] >= money.
The frame: a single isValid(account) helper checks the 1-indexed range; each operation guards its preconditions with it (plus the balance check where needed) and returns false without mutating on any failure. The whole difficulty is the off-by-one of 1-indexed accounts and never committing a partial transfer. See pitfalls §10 and edge cases §11.

Canonical problem: LC 2043 — Simple Bank System (attempt). Graduated hints in the walkthrough.

Variant 8.5 Dependency graph + recompute
O(cells) / setO(cells + edges)
Solve-it-yourself variant. This is a target problem — the model and the cascade rule are below, but no code. This is the hardest design in the block.
When to use: a value can be defined in terms of other values, and changing one must cascade to everything that depends on it. The state is a graph; a mutation triggers a recompute of all dependents.

The model (LC 631 — Design Excel Sum Formula): a grid of cells. set(cell, value) makes a cell a literal; sum(cell, ranges) makes it a formula summing a list of cells/ranges, and returns the computed value; get(cell) returns the current value. Setting or summing a cell must make every formula that (transitively) depends on it reflect the new value.

The frame: each cell stores either a literal or a formula (its list of input cells/ranges). Two viable designs — (a) recompute-on-read: get evaluates the formula by recursively reading its inputs, naturally cascading; (b) maintain-on-write: store dependents and, on a set, do a BFS over dependents recomputing each. The recompute-on-read design is simpler and sufficient at this scale; the graph and cascade are the whole challenge. See pitfalls §10.

Canonical problem: LC 631 — Design Excel Sum Formula (target). Graduated hints in the walkthrough.

9   Complexity Analysis

The metric is cost per operation, not a one-shot pass. A design is good when the hot method is O(1) or O(log n) and the others are not much worse. The recurring win is replacing a per-query rescan (O(n) or O(n²)) with a derived aggregate updated in O(1) per write — trading a little work on every mutation for cheap, constant-time queries.

VariantTime / opSpaceWhy
Map-backed (8.1)O(1)O(keys)Dictionary insert/lookup is O(1) average; deriving a parent path is O(path length).
Incremental aggregate (8.2)O(1)O(n)Each move updates a constant number of running totals; the win check is one comparison.
Streaming window state (8.3)O(1) amortizedO(w)Each item enqueued once and dequeued at most once. Window holds at most w in-range items.
Validated operations (8.4)O(1)O(accounts)Each op is a few range/balance checks plus a constant-time mutation.
Dependency recompute (8.5)O(cells + edges)O(cells + edges)A set cascades to all transitive dependents; recompute touches each affected cell once.
BottleneckO(n) per query—Rescanning the full state on every query instead of maintaining a derived aggregate — the exact mistake the incremental variant exists to avoid.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Mutating before validating (8.3). Deducting from the source account and then discovering the balance was insufficient corrupts state. Check every precondition first; commit only after all pass.
▸Stale derived state (8.2). Updating the row total but forgetting the diagonal on a diagonal move means a winning line is missed. Every mutation must update all aggregates it touches, in the same method.
▸1-indexed off-by-one (8.3). Bank accounts are numbered 1…n but arrays are 0-indexed. Size the array n+1 and index directly, or subtract 1 consistently — mixing the two is the classic bug.
▸Forgetting the parent-exists rule (8.1). Creating /a/b/c when /a/b was never created must fail. The validity check is "parent present," not just "this path absent."
▸Recompute cascade into a cycle (8.4). If cell A's formula references B and B's references A, a naive recompute recurses forever. Either forbid cycles by the problem's guarantee or track visited cells during the cascade.
▸Overwrite vs. accumulate (8.2/8.4). When a cell changes from one value to another, the derived total must be adjusted by the delta (subtract the old, add the new), not just add the new — or repeated sets double-count.

class System {
    private var store: [String: Int] = [:]   // raw state
    private var derived = 0                  // optional aggregate

    init(/* config */) { }

    func mutate(_ key: String, _ value: Int) -> Bool {
        guard /* preconditions */ true else { return false }
        store[key] = value
        // update derived by the delta here — keep the invariant
        return true
    }

    func query(_ key: String) -> Int { store[key] ?? -1 }
}

Solo problem — no solution on this page

LC 2043 — Simple Bank System (Medium). Implement a bank with n accounts (1-indexed) supporting transfer, deposit, and withdraw, each returning whether it succeeded. An operation fails — and changes nothing — if an account is out of range or a balance is insufficient. Decide your state and your validation rules before writing any method.

  • What is the state — and why size it n+1 or subtract 1 consistently for the 1-indexing?
  • What are the exact preconditions for each of the three operations?
  • Why must you validate all conditions before mutating any balance?
  • Is the balance check >= or > — what happens on an exact-balance transfer?
  • Which operations share a validity helper, and what does it check?
