A view model has to live somewhere, and SwiftUI is strict about where. @StateObject owns an object: the framework creates it once, on first render, and hands the same instance back across every subsequent redraw. @ObservedObject borrows one: it subscribes to changes but has no storage, so if the view is recreated the object goes with it.

Getting this wrong produces two distinct bugs — a view model silently rebuilt on every parent redraw, and a view model that cannot be swapped for a mock in tests. Both are avoidable with one initializer pattern.

THE ONE IDEA StateObject(wrappedValue:) takes an @autoclosure. The expression inside is evaluated only on first render — never again. Building the object at the call site instead defeats that entirely, because the enclosing init runs on every parent redraw.

2 Mental Model — how to think about it

SwiftUI views are structs, recreated constantly. Their init runs far more often than people assume — every time a parent redraws. What survives is not the struct but the storage slot SwiftUI keeps against the view's identity.

WrapperOwns storage?Use when
@StateObjectYes — created once, survives redrawsThis view is the source of truth for the object.
@ObservedObjectNo — subscribes onlyThe object is owned by an ancestor and passed down.
@StateYes, for value typesSimple local state the view owns.

Because init re-runs, anything expensive placed there runs repeatedly. That is why real work belongs in .task, which is tied to the view's lifetime rather than its construction.

4 The API — types, modifiers, signatures

SwiftThe autoclosure that makes ownership work
// The @autoclosure is the whole point: the expression is evaluated
// lazily, on first render only.
init(wrappedValue: @autoclosure @escaping () -> ObjectType)

// Assigning through the underscore-prefixed synthesized property is how
// you construct a @StateObject from an initializer.
_viewModel = StateObject(wrappedValue: MyViewModel(repository: repository))

5 Patterns That Work

Inject the dependency, build the view model inline

SwiftTestable and built exactly once
struct MyView: View {
    @StateObject private var viewModel: MyViewModel

    init(repository: ItemRepositoryProtocol) {
        _viewModel = StateObject(wrappedValue: MyViewModel(repository: repository))
    }
}

The view takes a protocol, so a test or preview can pass a stub. The object itself is constructed inside the autoclosure, so it is built once regardless of how often init runs.

Cheap init, real work in .task

Swiftinit stores references; task does the loading
final class MyViewModel: ObservableObject {
    @Published var data: ViewData?
    private let repository: ItemRepositoryProtocol

    init(repository: ItemRepositoryProtocol) {
        self.repository = repository   // cheap - just stores a reference
    }

    func load() async {
        data = await repository.fetch()   // real work happens HERE
    }
}

// In the body:  .task { await viewModel.load() }

Pass plain values down, not the whole object (iOS 16)

SwiftOnly the parent observes
struct ParentView: View {
    @ObservedObject var viewModel: ViewModel
    var body: some View {
        DetailRow(title: viewModel.title)   // a plain value goes down
    }
}

struct DetailRow: View {
    let title: String                   // NOT @ObservedObject
    var body: some View { Text(title) }
}
// Mutating an unrelated property -> DetailRow receives the same title
// -> its body call is SKIPPED entirely.

iOS 17+: @Observable tracks per property

Context only — do not use this on an iOS 16 target. With @Observable, a view redraws only if it actually read the property that changed, so passing the object directly is fine and the manual value-threading above becomes unnecessary.
SwiftPer-property tracking
@Observable
final class ViewModel {
    var title: String = ""
    var subtitle: String = ""
}

struct DetailRow: View {
    let viewModel: ViewModel        // plain reference, no wrapper
    var body: some View { Text(viewModel.subtitle) }
}
// Mutating title -> DetailRow does NOT redraw (it never read title).

6 Anti-Patterns — and the fix

Hardcoding the view model in a property initializer

SwiftUntestable by construction
// WRONG: the dependency is baked in. No test or preview can substitute
// a stub, because there is no seam to inject through.
struct MyView: View {
    @StateObject var viewModel = MyViewModel()
}

Building the object at the call site — the double-init bug

SwiftConstructed on every redraw, then discarded
// WRONG: this expression runs on EVERY parent redraw. SwiftUI keeps the
// instance from the first render and throws each new one away, so the
// work is pure waste - and any init side effect fires repeatedly.
let viewModel = MyViewModel(repository: RealRepository())
MyView(viewModel: viewModel)

// RIGHT: pass the dependency, construct inside the autoclosure.
MyView(repository: RealRepository())

Observing the whole object in every subview

On iOS 16, @ObservedObject subscribes to the entire object. A subview that reads one property still re-runs its body when any other property changes.

SwiftWasted body calls
// WRONG: mutating any property on the view model re-runs this body,
// even though it only ever reads one value.
struct DetailRow: View {
    @ObservedObject var viewModel: ViewModel
    var body: some View { Text(viewModel.title) }
}

7 Pitfalls & Gotchas

The 2x skip you throw away
→ When a subview receives plain values, SwiftUI can skip twice: the body closure never executes (CPU saved), and no diff is needed at render (redraw saved). Taking @ObservedObject or @StateObject on the object that changed bypasses both skips, every time. Under @Observable (iOS 17+) the skip is restored automatically, because tracking is per property rather than per object.
  • init runs far more often than you think. Treat it as "store references only" — anything else belongs in .task.
  • @StateObject creates once; @ObservedObject does not create at all. Using the latter where the view is the owner means the object can vanish on a redraw.
  • A view model injected as a concrete type is untestable. Depend on a protocol so the seam exists.
