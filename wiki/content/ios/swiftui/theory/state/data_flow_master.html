Once you know who owns a value, the remaining question is how it reaches the view that needs it. SwiftUI has four answers, and they are not interchangeable — each is wired into a different part of the framework.

The common mistake is reaching for a closure when a Binding would do, or contorting a Binding around something that is really an event. Choosing correctly is mostly a matter of naming what you are moving: shared state, a discrete event, a measured number, or an ambient dependency.

THE ONE IDEA Pick the tool by asking what kind of thing is moving, not just which way it goes. Binding is for state the parent owns and the child edits; a closure is for "this happened"; PreferenceKey is for numbers that only exist after layout; Environment is for values any descendant might need.

Two directions, but four tools — because "a measured height" and "the user tapped" travel upward for entirely different reasons.

Binding is the community-approved default for simple shared state — not for full dependencies like a service or a view model, which belong in @Environment or an initializer.

3 When It Applies

When you see…The data-flow question to ask
A chain of closures threaded through three view layersIs this really shared state? A Binding or @Environment may collapse the chain.
"How do I read the child's @State from the parent?"You cannot — and the question itself means the state is in the wrong place.
A parent that needs a child's rendered heightPreferenceKey. Neither a binding nor a closure is wired into the layout pass.
The same value passed through many intermediate views@Environment skips the intermediates entirely.

4 The API — types, modifiers, signatures

SwiftThe PreferenceKey protocol
// reduce() combines values when several descendants report one. It runs
// bottom-up as the layout pass resolves.
protocol PreferenceKey {
    associatedtype Value
    static var defaultValue: Value { get }
    static func reduce(value: inout Value, nextValue: () -> Value)
}

// Report a value up; observe it from an ancestor.
func preference<K: PreferenceKey>(key: K.Type, value: K.Value) -> some View
func onPreferenceChange<K: PreferenceKey>(_ key: K.Type, perform: @escaping (K.Value) -> Void) -> some View

5 Patterns That Work

Direction 1 — parent to child, with write access

Use @Binding. The child gets read and write access to state it does not own.

SwiftPass with $
struct ParentView: View {
    @State private var isOn = false
    var body: some View {
        ToggleRow(isOn: $isOn)
    }
}

struct ToggleRow: View {
    @Binding var isOn: Bool   // child can read AND write
    var body: some View {
        Toggle("Enabled", isOn: $isOn)
    }
}

Direction 2 — child to parent, reporting an event

A leaf needs to tell an ancestor "this got picked". Two idiomatic options; pick by the shape of the problem.

Option A — pass a Binding down anyway, preferred when there is one piece of shared state. The parent still owns the source of truth; the child is bound to "is this me".

Option B — a plain closure, preferred when the child fires discrete events rather than reflecting state. A tap, a swipe-to-delete, a submit.

SwiftThe converged single-selection pattern
struct PaymentInfoView: View {
    @State private var selectedPaymentType: PaymentType?

    var body: some View {
        List(paymentTypes) { paymentType in
            PaymentRow(
                name: paymentType.name,
                isSelected: selectedPaymentType?.id == paymentType.id,
                onSelect: { selectedPaymentType = paymentType }
            )
        }
    }
}

struct PaymentRow: View {
    let name: String
    let isSelected: Bool          // derived, read-only
    let onSelect: () -> Void      // event, not state

    var body: some View {
        HStack {
            Text(name)
            Spacer()
            if isSelected { Image(systemName: "checkmark") }
        }
        .contentShape(.rect)
        .onTapGesture(perform: onSelect)
    }
}

Single-selection from a list — radio-button style — is almost always modelled this way: the parent owns @State private var selection: Item?, and each row takes a derived isSelected: Bool plus an onSelect closure.

Direction 3 — leaf to ancestor, for layout geometry

Neither a binding nor a closure fits when the parent needs a number it can only know after the child is measured. PreferenceKey propagates a value up the tree during the layout pass — the mirror image of environment values.

SwiftDefine, report, observe
// 1. Define a key: how to combine values if several leaves report one.
struct HeightPreferenceKey: PreferenceKey {
    static var defaultValue: CGFloat = 0
    static func reduce(value: inout CGFloat, nextValue: () -> CGFloat) {
        value = nextValue()   // last one wins; use max/sum to merge instead
    }
}

// 2. The leaf reports its measured size.
struct InnerScrollContent: View {
    var body: some View {
        VStack { }
            .background(
                GeometryReader { proxy in
                    Color.clear
                        .preference(key: HeightPreferenceKey.self, value: proxy.size.height)
                }
            )
    }
}

// 3. The ancestor reads it and reacts.
struct OuterScrollView: View {
    @State private var measuredInnerHeight: CGFloat = 0

    var body: some View {
        ScrollView { InnerScrollContent() }
            .frame(height: measuredInnerHeight)
            .onPreferenceChange(HeightPreferenceKey.self) { newHeight in
                measuredInnerHeight = newHeight
            }
    }
}
Why not a Binding here: the child is not reporting "the user did X", it is reporting "here is my resolved geometry after layout" — a continuous, layout-phase value. PreferenceKey is specifically wired into the layout pass, with reduce running bottom-up as the tree resolves. A binding has no such mechanism.

Direction 4 — ancestor to any descendant

@Environment is the mirror image of PreferenceKey: implicit plumbing that flows top-down. An ancestor injects a value and any descendant can read it, without threading it through every intermediate view by hand.

6 Anti-Patterns — and the fix

Giving a leaf its own @State and expecting the parent to read it

A child's @State is private and unreachable from outside. There is no idiomatic way to read state back out of a child — and wanting to is a reliable signal that the state does not belong there.

SwiftState in the wrong place
// WRONG: the parent has no way to observe this, and each row's selection
// is independent - so two rows can both think they are selected.
struct PaymentRow: View {
    @State private var isSelected = false
}

// RIGHT: lift the state to the common ancestor that needs to know, and
// pass a derived Bool down plus a closure back up.
struct PaymentRow: View {
    let isSelected: Bool
    let onSelect: () -> Void
}

Using a closure where a Binding is simpler

If the child both reads and writes the same piece of state, a closure pair (value plus onChange) is a hand-rolled binding. Use the real one.

Using a Binding for geometry

Writing to a binding from inside a GeometryReader during layout invites update loops. Layout-phase values belong in a PreferenceKey, which is designed for exactly that timing.

7 Pitfalls & Gotchas

Choosing the wrong direction tool
1. A child's @State is unreachable from its parent. If you want to read it, the state is in the wrong place — lift it to the nearest common ancestor.
2. reduce decides what happens when several leaves report. The default "last one wins" is rarely what you want for sums or maxima — write it deliberately.
3. @Environment is for ambient values, not for everything. Passing a required dependency implicitly makes a view's real requirements invisible at the call site.
