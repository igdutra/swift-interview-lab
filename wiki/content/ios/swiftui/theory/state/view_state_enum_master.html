A screen that loads data has a handful of legitimate conditions: loading, loaded, failed, and often something in flight on top of loaded data. Modelled as parallel booleans, those four conditions become eight representable combinations, half of them nonsense — and every view that reads them has to defensively decide which flag wins.

Modelled as a single enum, the illegal combinations stop existing. The type system does the work that defensive if chains were doing, and the view becomes a straightforward switch.

THE ONE IDEA Give the in-flight case an associated value carrying the data it operates on. case searching(allItems: [Item]) encodes a precondition directly in the type: you cannot be searching unless you already have something to search within. A separate isSearching flag can be set at any moment, including before data ever arrives.

2 Mental Model — how to think about it

Count the states your type can represent, then count the ones that make sense. The gap is where bugs live.

FOUR BOOLEANS                          ONE ENUM
  isLoading  : true | false              .loading
  isSearching: true | false              .loaded(allItems:)
  items      : [] | [...]                .searching(allItems:)
  error      : nil | Error               .error(Error)

  2 x 2 x 2 x 2 = 16 combinations        exactly 4 states
  ~4 of them are valid                   all 4 are valid
  the other 12 need defending against    the other 12 cannot be written
The enum does not document the valid states — it makes the invalid ones impossible to construct.

The second idea is subtler. An enum case can carry data, and what it carries is a guarantee. If .searching requires an item list to construct, then "searching with nothing loaded" is not a state the program can enter — not by convention, but by construction.

3 When It Applies

Local @State in the view is fine for purely local filtering. It stops being fine as soon as any of these are true:

When…Why the enum earns its place
The operation hits a network endpointFailure becomes a real state, not an edge case.
The result feeds anything beyond this one viewThe state needs a single owner and a single shape.
You need to distinguish loading from in-flight-on-top-of-loadedBooleans cannot express "both, but one is layered on the other".
You want to unit-test the transitionsA switch over four cases is testable; eight boolean combinations are not.

4 The API — types, modifiers, signatures

SwiftThe state type
enum ViewState {
    case loading
    case loaded(allItems: [Item])
    case searching(allItems: [Item])   // data already loaded; a search is in flight on top
    case error(Error)
}

// private(set) is deliberate: only the view model transitions states,
// so the view can never put it into an inconsistent one.
@Published private(set) var viewState: ViewState = .loading

5 Patterns That Work

Store all the data; derive the filtered view

The state holds the full, unfiltered list. Filtering happens downstream in a computed property. The state is the source of truth; the filtered array is a derived view of it.

SwiftDerivation, not mutation
var visibleItems: [Item] {
    switch viewState {
    case .loaded(let allItems), .searching(let allItems):
        return searchText.isEmpty
            ? allItems
            : allItems.filter { $0.name.localizedCaseInsensitiveContains(searchText) }
    case .loading, .error:
        return []
    }
}

That separation is what lets a search be cancelled or superseded without ever having lost the underlying data — there is nothing to re-fetch, because nothing was thrown away.

The view becomes a switch

SwiftExhaustive by construction
switch viewModel.viewState {
case .loading:
    ProgressView()
case .loaded:
    itemList
case .searching:
    itemList.overlay(alignment: .top) { ProgressView() }
case .error(let error):
    ErrorView(error: error, retry: { Task { await viewModel.load() } })
}

The compiler enforces that every state is handled. Adding a fifth case later produces compile errors at exactly the places that need updating — which is the whole return on this design.

Keep rows on screen while an operation is in flight

Because .searching carries the same list .loaded had, the rows stay visible during the search. The UI layers a small progress indicator over live content instead of blanking to a full-screen spinner.

6 Anti-Patterns — and the fix

Parallel booleans

SwiftEight combinations, four of them meaningless
// WRONG: nothing stops isLoading == true alongside a populated items
// array AND a non-nil error. Every view that reads this has to decide
// defensively which flag wins.
@Published var isLoading = false
@Published var isSearching = false
@Published var items: [Item] = []
@Published var error: Error?

// RIGHT: one value, four states, no combination to defend against.
@Published private(set) var viewState: ViewState = .loading

An in-flight case that carries nothing

SwiftThe bug the associated value prevents
// WRONG: .searching with no payload can be entered before the first load
// finishes - producing a spinner over an empty screen, and losing the
// rows that were already on display.
case searching

// RIGHT: you cannot construct this case without data, so "searching with
// nothing loaded" is unrepresentable.
case searching(allItems: [Item])

Storing the filtered results in the state

If the state stored the filtered list rather than the full one, every keystroke would destroy the previous results. Cancellation would leave a partially filtered set, and clearing the query would require a re-fetch of data you already had.

7 Pitfalls & Gotchas

Modelling mistakes that reintroduce the problem
1. Dropping private(set). Without it a view can assign the state directly, which is exactly the inconsistency the enum was introduced to prevent.
2. Adding a boolean "just for this one case". One flag beside the enum reintroduces the combinatorial problem. If a new condition exists, it is a new case or a new associated value.
3. Using default: in the view's switch. It silences the compiler error that would otherwise tell you where to handle a newly added state — throwing away the main benefit.

This is the same principle as separating initial load from refresh: never discard data you still have just because a new operation started. The enum makes that principle structural rather than a matter of discipline.
