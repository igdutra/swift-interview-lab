View Identity

iOS Topic· SwiftUI· How SwiftUI decides two views are "the same view"

View identity is how SwiftUI decides whether a view on screen is the same view it drew last time or a different one. That single decision drives everything a reader cares about: whether @State survives, whether an animation continues or restarts, and whether a transition fires at all. SwiftUI never asks "did the content change?" — it asks "is this the same view?", and answers using the view's type and position in the tree, not its values.

Most SwiftUI bugs reported as "my state keeps resetting" or "my animation won't play" are identity bugs. The framework is behaving exactly as documented; the view tree is simply saying something different from what the author meant.

THE ONE IDEA Identity is a path, not a flag. Every position in the view tree gets its own identity, computed independently as (type, position-within-parent). A conditional at one position tears down only that position — it cannot reach sideways to a sibling or upward to its parent.

2 Mental Model — how to think about it

Think of the view tree as a filing cabinet where every slot has an address. The address is not the view's contents — it is which drawer, which folder, which position. SwiftUI stores your @State against that address.

Redraw the UI and SwiftUI walks the same addresses. Same address, same type, and it is the same view, so the state in that slot is handed back. A different type at that address means a different view, so the old slot and its state are thrown away and a new one is created.

HStack                          identity: (HStack, root)
├── [0] Text(name)              identity: (Text, position 0)      <- stable
├── [1] Spacer()                identity: (Spacer, position 1)    <- stable
└── [2] if selected { Image }   identity: (Image, position 2)     <- conditional
                                          ^^^^^^^^^^^^^^^^^^^
                       only THIS address is torn down when `selected` flips
Each position is diffed on its own. Position 2 changing cannot invalidate positions 0 and 1, nor the enclosing HStack.

Two vocabulary terms, from Apple's WWDC21 session Demystify SwiftUI — worth knowing by name, because the documentation uses them precisely:

TermWhat determines itIn practice
Structural identityThe view's type and its position in the treeThe default. You get it for free and never think about it — until a branch changes the type at some position.
Explicit identityIdentifiable conformance, or an .id(_:) modifierWhat ForEach and List use to track rows, and what .id() lets you set by hand.

3 When It Applies

Identity is worth thinking about the moment any of these show up:

When you see…The identity question to ask
"My @State resets whenever I toggle something"Does toggling it change the type at that position? An if/else between two different views does exactly that.
"My .transition never animates"Is the view actually being inserted or removed? A transition on a view whose identity never changes can never fire.
"The form keeps the previous record's edits"The opposite problem — identity is too stable. Reach for .id(record.id) to force a fresh view.
An if inside a List rowUsually a non-problem. Check whether the branch sits at the row's top level or nested inside an already-stable container.
A reflexive .opacity(flag ? 1 : 0)Is there actually state or an animation to preserve? On a stateless leaf this costs layout space and buys nothing.

4 The API — types, modifiers, signatures

The surface is small; the subtlety is entirely in what it means.

SwiftExplicit identity
// Attach an explicit identity to a view. Changing `value` makes SwiftUI
// treat this as an entirely different view.
func id<IdentifierValue>(_ value: IdentifierValue) -> some View
    where IdentifierValue: Hashable

// The protocol List and ForEach use to track elements across updates.
protocol Identifiable {
    associatedtype ID: Hashable
    var id: ID { get }
}
SwiftWhat a branch compiles to
// An if/else inside a ViewBuilder does NOT produce "one view that changes".
// It produces a generic wrapper whose type differs per branch:
_ConditionalContent<TrueContent, FalseContent>

// Which is why flipping the condition swaps the concrete type occupying
// that slot -- and a new type means a new identity.
Why this matters: _ConditionalContent is the whole mechanism. Once you know an if/else is a type-level choice rather than a value-level one, the state-reset behaviour stops being surprising and becomes predictable.

5 Patterns That Work

Push the condition into a value, not the structure

When both outcomes are the same view with different values, do not branch. Same type, same position, same identity — no teardown, and the change animates for free.

SwiftThe default choice
// One Text, one identity, only the value varies.
Text("Hi").foregroundStyle(isHighlighted ? .red : .blue)

.id() to force a deliberate reset

The common, intended use. A new .id() value tells SwiftUI "this is a different view now", discarding state on purpose — exactly right when moving between records.

SwiftReset a form when the record changes
EditorView(record: selectedRecord)
    .id(selectedRecord.id)   // new record -> fresh EditorView state

.id() to unify identity across branches

The rarer direction. Putting the same .id() on both sides overrides per-branch structural identity, so state can survive the flip. This is a targeted workaround for when the branches are genuinely different view types and continuity still matters — not the first tool to reach for.

SwiftBoth branches share one identity
Group {
    if isEditing {
        EditorView()
    } else {
        SummaryView()
    }
}
.id("record-detail")   // one continuous identity; less state churn

Branch when you want the insert and remove

Everything above treats teardown as a cost. .transition(_:) turns it into the feature: it describes how a view animates as it is inserted into or removed from the tree, which only happens when identity changes.

SwiftThe transition describes it; the animation drives it
if isExpanded {
    DetailPanel()
        .transition(.move(edge: .bottom).combined(with: .opacity))
}

// The state change must be animated, or the transition never runs:
Button("Toggle") {
    withAnimation(.snappy) { isExpanded.toggle() }
}

Pair .id() with .transition() to animate a replacement rather than an insertion — same slot, new identity, so SwiftUI treats it as remove-then-insert:

SwiftNew value = new identity = transition fires
Text(currentQuote)
    .id(currentQuote)
    .transition(.opacity)

6 Anti-Patterns — and the fix

Treating .id() as "preserve my state through branching"

The most common misreading. .id() is not a switch that makes SwiftUI hold on to identity through conditional code — by default it is the opposite primitive, used to discard identity deliberately. That it also happens to work in the unifying direction is what makes the confusion durable.

SwiftThe branch that quietly resets state
// WRONG: flipping `isEditing` destroys and recreates whichever branch
// appears. Each branch is a different concrete type in the same slot, so
// any @State inside resets and animations restart.
if isEditing {
    EditorView()      // its own @State resets every time isEditing flips back
} else {
    SummaryView()
}

// RIGHT, when they are the same view with different values:
//   inline the condition instead of branching structurally.
// RIGHT, when they are genuinely different types and you need continuity:
//   wrap in a Group and give both the same .id().

Reaching for .opacity() as a defensive reflex

The reason the .opacity()-swap pattern exists is to preserve state or an in-flight animation on the thing being conditionally shown. Applied to a stateless leaf view it solves a problem that does not exist — and it costs layout space, because the hidden view still reserves its frame.

SwiftA stateless leaf needs no protection
// WRONG: nothing inside Image(systemName:) can be lost. The checkmark's
// frame is now permanently reserved in the HStack, for no benefit.
Image(systemName: "checkmark")
    .opacity(isSelected ? 1 : 0)

// RIGHT: idiomatic. Only this one child position is inserted or removed;
// the row, HStack, Text and Spacer all keep their identity throughout.
if isSelected {
    Image(systemName: "checkmark")
}

.opacity() is the right call in four cases:

CaseWhy keeping the view alive wins
The branched view owns @State or @StateObjectRecreating it resets that state visibly — a scroll offset jumping back to the top, for example.
An animation or gesture is in flightA hard remove and insert interrupts it. Continuity across the toggle is the entire point for a spinner or a matchedGeometryEffect participant.
You want a cross-fade, not an insert and removeA manual opacity toggle gives direct control over timing, instead of letting SwiftUI choose an implicit insert/remove animation.
The view is expensive to constructReal work in init, or a large subtree, toggled rapidly — hiding beats repeated construction and teardown.

Promoting a branch to the row's top level

This is the shape that does destroy row identity — worth contrasting against the harmless nested version in §7.

SwiftTwo types competing for one slot
// WRONG: two different concrete types occupying the position List
// assigned to this element. The row's identity dies on every toggle.
List(paymentTypes) { paymentType in
    if isSelected(paymentType) {
        SelectedRow(paymentType)     // type A
    } else {
        UnselectedRow(paymentType)   // type B
    }
}

// RIGHT: one row type. The conditional lives one level down, scoped to a
// single child slot inside an already-stable container.
List(paymentTypes) { paymentType in
    PaymentRow(paymentType: paymentType, isSelected: isSelected(paymentType))
}

7 Worked Example — a conditional inside a list row

The question this topic tends to surface from: inside a List row whose identity is already stable via Identifiable, does an if destroy the row's identity every time the selection changes?

SwiftThe code in question
private func paymentRow(for paymentType: PaymentType) -> some View {
    HStack {
        Text(paymentType.name)
        Spacer()
        if isPaymentTypeSelected(paymentType) {
            Image(systemName: "checkmark")
        }
    }
    .onTapGesture { selectedPaymentType = paymentType }
}

List(searchResultsPaymentTypes) { paymentType in
    paymentRow(for: paymentType)
}

Answer: no. Three things are true at once, and the argument depends on keeping them distinct:

  1. The row's identity comes from paymentType.id, via List's Identifiable-based diffing. That is explicit identity — a separate mechanism from @State storage, keyed off the data element itself.
  2. paymentType is not state the view holds. It is a plain captured parameter, handed in fresh on every invocation. But it plays an equivalent role: it both defines the row's identity key and drives what the row renders. Because the branch condition and the identity key derive from the same underlying value, the row cannot get confused about which row it is.
  3. Identity is computed per position. The HStack's three children occupy three independent slots, and only slot 2 is conditional.

So flipping the if inserts or removes the Image at position 2 and touches nothing else — not positions 0 or 1, not the HStack, not the row. A sibling's branch cannot reach backward and invalidate a parent's or a sibling's identity, because each position is diffed on its own.

The generalization: branching is not inherently destructive — it is destructive only at the exact position it occupies. Nesting a conditional one level inside an already-stable container scopes the teardown to that one child slot. This is why the fix for an identity problem is usually to move the branch down the tree, not to reach for .opacity().

8 Pitfalls & Gotchas

Transitions that silently do nothing
1. No animation, no transition. .transition alone is inert. The state change must happen inside withAnimation { }, or an .animation(_:value:) modifier must be observing the driving value.
2. The view must genuinely be inserted or removed. A .transition on a view whose identity never changes will never fire. This is the one place the usual advice inverts: if you want the animation, the if is correct and .opacity() would defeat it.
  • Reach for branch-level .id() unification sparingly. Only when the branches are legitimately different view types. If they are the same view with different values, inlining into a modifier is strictly simpler and avoids the question entirely.
  • .id() is also an animation knob. Changing or removing an .id() is how you deliberately trigger or suppress a transition on replacement — not only a state-reset switch.
  • State resets are documented behaviour, not a bug. When a branch flips, the framework is doing exactly what the type system told it to.
