Almost every "why is my view the wrong size" question comes from not knowing one small model. Layout in SwiftUI is a three-step negotiation, repeated at every level of the tree: the parent proposes a size, the child chooses its own, and the parent positions the child within its bounds.

The step people do not expect is the middle one. A child is not obligated to accept the proposal, and a parent can never force a size on it. Text takes only the width it needs; Color takes everything offered; Image takes its intrinsic size unless made .resizable().

THE ONE IDEA .frame() does not resize a view. It wraps it in a new view of that size, and the original sits inside — still choosing its own size. That single fact explains why borders look "too big", why modifier order matters, and why a fixed frame does not truncate text.

2 Mental Model — how to think about it

Think of it as a conversation rather than a command:

PARENT: "You may have up to 100 x 100."        <- proposal (may be nil = "anything")
CHILD:  "Thanks, I only need 24 x 17."          <- the child decides
PARENT: "Fine, I'll centre you in my 100 x 100." <- positioning
The proposal is an offer, not an instruction. This repeats at every level down the tree.
SwiftWhy the border looks too big
// The frame PROPOSES 100x100. Text still chooses its own smaller size,
// then gets centred inside the 100x100 region.
Text("Hi")
    .frame(width: 100, height: 100)
    .border(.red)          // the border is on the 100x100 frame, not the text

This is why .background and .border sometimes look oversized: they are drawing around the frame, not around the content.

3 When It Applies

When you see…The layout question to ask
A background or border that is bigger than expectedIs it wrapping a .frame rather than the content? Check modifier order.
Text truncated to "Some long te…"The parent proposed too little and the child accepted. .fixedSize or .layoutPriority.
A view that will not fill the width.frame(maxWidth: .infinity) — the single most-used layout modifier.
Adding a GeometryReader destroyed the layoutIt accepts the full proposal. Use it as a background/overlay instead.
A stack suddenly fills its parentA Spacer makes the stack greedy along its axis.

4 The API — types, modifiers, signatures

SwiftThe two frame overloads behave differently
.frame(width: 100, height: 100)                    // fixed proposal
.frame(maxWidth: .infinity, alignment: .leading)   // flexible: "take what's offered"
ModifierWhat it does to the negotiation
.frame(width:height:)Wraps the view in a new fixed-size view and proposes that size to the original.
.frame(maxWidth: .infinity)Accepts whatever the parent offers along that axis. The workhorse for "fill available width".
.fixedSize()Tells the child to ignore the proposal and use its ideal size. Can overflow the parent, by design.
.layoutPriority(_:)When a stack must compress children, higher priority is compressed last. Default is 0.
Spacer(minLength:)A flexible view that expands along the stack's axis. minLength guarantees a floor.
ViewThatFitsiOS 16+. Evaluates children in order, uses the first that fits in both dimensions.

5 Patterns That Work

.fixedSize to rescue truncated content

SwiftThe escape hatch for compressed text
// In a tight HStack this truncates to "Some long te..."
HStack {
    Text("Some long text that gets truncated")
    Spacer()
    Button("Action") { }
}

// fixedSize tells the Text to take its full ideal width instead.
Text("Some long text that gets truncated")
    .fixedSize(horizontal: true, vertical: false)

layoutPriority to choose what survives compression

SwiftHigher priority is compressed last
HStack {
    Text(veryLongTitle)
        .layoutPriority(1)      // gets the space it needs first
    Text(subtitle)              // default priority 0 - compressed first
}

The UIKit analogue is compression resistance, though SwiftUI collapses hugging and resistance into this single number.

ViewThatFits for declarative responsive layout

SwiftNo manual measurement required
ViewThatFits {
    HStack {                      // preferred: side by side
        Label("Settings", systemImage: "gear")
        Spacer()
        Text(value)
    }
    VStack(alignment: .leading) { // fallback: stacked
        Label("Settings", systemImage: "gear")
        Text(value)
    }
}

Alignment guides for cross-view alignment

SwiftAligning differently-sized siblings
// Align text baselines rather than view edges
HStack(alignment: .firstTextBaseline) {
    Text("Big").font(.largeTitle)
    Text("small").font(.caption)
}

// A custom alignment, for non-adjacent views
extension VerticalAlignment {
    private enum BadgeAlignment: AlignmentID {
        static func defaultValue(in context: ViewDimensions) -> CGFloat {
            context[.top]
        }
    }
    static let badge = VerticalAlignment(BadgeAlignment.self)
}

6 Anti-Patterns — and the fix

Dropping a GeometryReader into a stack

GeometryReader reports the size proposed to it, and it accepts the full proposal — so it swallows all available space along both axes. Its content is also top-leading aligned by default, not centred, which is a second surprise.

SwiftGreedy by design
// WRONG: GeometryReader takes ALL available height; the Text gets shoved.
VStack {
    GeometryReader { proxy in
        Rectangle().frame(width: proxy.size.width * 0.5)
    }
    Text("Below")
}

// RIGHT: constrain it, or use it as a background/overlay where it
// inherits the size of what it is attached to.
Rectangle()
    .frame(height: 100)
    .background(
        GeometryReader { proxy in
            Color.clear.preference(key: WidthKey.self, value: proxy.size.width)
        }
    )

Prefer the alternatives where they exist: .containerRelativeFrame (iOS 17+), ViewThatFits, or a PreferenceKey for reading a child's measured size without hijacking the layout.

Assuming modifier order does not matter

SwiftEach modifier wraps the previous result
Text("Hi").padding().background(.blue)   // blue INCLUDES the padding
Text("Hi").background(.blue).padding()   // blue hugs the text; padding outside

Nesting greedy views inside each other

Spacer, Color, and GeometryReader all expand to fill what they are offered. Nesting them produces ambiguous layouts that appear to work until the content changes and one of them wins differently.

7 Pitfalls & Gotchas

Sizing surprises worth memorizing
1. .frame wraps rather than resizes. The child may still choose a smaller size and be positioned inside the frame.
2. GeometryReader fills its proposal and top-leading aligns its content. Both behaviours surprise people, and both are by design.
3. .fixedSize can overflow its parent. You have explicitly told the child to stop negotiating, so use it sparingly.
4. .layoutPriority on a Spacer is meaningless. Use Spacer(minLength:) to guarantee a floor instead.

A stack with no Spacer and no .frame sizes to fit its content. Adding a single Spacer makes it greedy along that axis — which is usually what you want in an HStack row, and usually a surprise in a VStack.

Debugging layout

SwiftFastest ways to see what is actually happening
.border(.red)                          // see the actual frame
.background(Color.blue.opacity(0.3))

// Inside a body, logs WHY it re-evaluated (iOS 15+). Useful when
// layout churn turns out to be a state problem.
Self._printChanges()
