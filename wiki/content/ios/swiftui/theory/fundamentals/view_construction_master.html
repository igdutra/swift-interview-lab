@ViewBuilder is a result builder — the same language feature behind SwiftData's query builders. Marking a function or computed property with it rewrites the block's statements into calls on a hidden builder type's static methods: buildBlock, buildEither(first:second:), buildOptional, buildArray.

The result is that a block which looks like several statements resolves to one concrete type at compile time — which is what lets it satisfy some View. It is a pure syntactic transform: the attribute itself adds no runtime cost, no boxing, and no dynamic dispatch.

THE ONE IDEA A @ViewBuilder helper function and an extracted View struct produce the same render tree. The performance question people mean to ask is not about the attribute — it is about what branching does to view identity.

2 Mental Model — how to think about it

Think of @ViewBuilder as a compiler that collapses a list of views into a single nested type. Each construct maps to a wrapper:

You writeThe builder produces
Two or more statements in a rowTupleView<(A, B)>
if / else_ConditionalContent<TrueContent, FalseContent>
if with no elseOptional<Content>
for in a ForEachAn array-backed view

Every one of those is a concrete generic type known at compile time. Nothing is erased, nothing is boxed — which is why the attribute costs nothing at runtime, and why an if/else produces two different types. That second fact is the whole basis of view identity.

3 When It Applies

You need @ViewBuilder in exactly one circumstance: the body has branching or more than one top-level view expression.

SwiftOne expression versus branching
// No @ViewBuilder needed — a single expression is already one type.
private func title(text: String) -> some View {
    Text(text).font(.title)
}

// @ViewBuilder needed — the branches produce different types.
@ViewBuilder
private func badge(isActive: Bool) -> some View {
    if isActive {
        Text("Active").foregroundStyle(.green)
    } else {
        Text("Inactive").foregroundStyle(.gray)
    }
}

Should you mark it "just in case"? There is no performance reason not to — on a single-expression body the attribute is inert. But it is not free information either: it signals "this might branch" to a reader when it does not. Add it exactly when the body needs it.

4 The API — types, modifiers, signatures

SwiftThe builder methods the transform calls
// @ViewBuilder is an instance of Swift's @resultBuilder feature. The
// compiler rewrites a marked block into calls on these:
static func buildBlock<...>(...) -> TupleView<...>
static func buildEither(first: ...) -> _ConditionalContent<...>
static func buildEither(second: ...) -> _ConditionalContent<...>
static func buildOptional(...) -> ...
static func buildArray(...) -> ...
Constraint: @ViewBuilder requires the function or property to return some View (or an explicit concrete/opaque view type). It does not work on arbitrary return types.

5 Patterns That Work

A private helper function for view code used in one place

If a chunk of view code is strictly for use inside the struct where it is declared, a @ViewBuilder function is the lower-boilerplate choice and costs nothing.

A separate struct when you need what a struct gives you

Extract to a View struct when you want one of three things a function cannot provide:

You need…Why a function cannot do it
Local @State or @StateObjectState requires a durable, SwiftUI-tracked identity slot tied to a View conformer's position in the tree. A function has no such slot, so state inside one does not persist correctly. This is the real constraint.
Reuse from another file or screenA private function is scoped to its enclosing type.
An independent preview, or a unit testBoth need a nameable type to instantiate.

6 Anti-Patterns — and the fix

Extracting a struct "for performance"

A @ViewBuilder-marked private function called directly in body produces the same render tree as extracting that content into a separate struct: View. SwiftUI's diffing is structural — based on the type and position of the resulting view nodes, not on whether a function or a struct's body produced them. Both expand to an identical hierarchy, so there is no meaningful update-skip difference.

Correction worth recording: an earlier version of these notes claimed a @ViewBuilder func has "no independent identity" and is diffed inline with its parent, implying a real cost. That was wrong for the common case. The primary source — Thinking in SwiftUI (objc.io) — states directly that there is no performance benefit to a separate struct over a helper function, because the render tree does not change.

Putting @State inside a @ViewBuilder function

This is the one place the func-versus-struct choice genuinely bites, and it is a correctness issue rather than a performance one.

SwiftState needs a struct
// WRONG: a function has no identity slot for SwiftUI to store state
// against, so this does not persist across redraws the way you expect.
@ViewBuilder
private func expandableSection() -> some View {
    @State var isExpanded = false   // does not work
    ...
}

// RIGHT: a struct conformer has a real position in the view tree, and
// therefore a durable storage slot.
private struct ExpandableSection: View {
    @State private var isExpanded = false
    var body: some View { ... }
}

Re-evaluating stored builder closures on every redraw

Distinct from a plain private func called inline. If @ViewBuilder content is captured as a closure and stored — for example a container that holds it as () -> some View — it re-runs on every body recompute. Captured once in an init, it is not re-evaluated later.

7 Pitfalls & Gotchas

The question behind the question
→ When someone asks "is a @ViewBuilder func slower than a struct?", the concern they actually have is almost always about view identity — what branching does to state and animations. That is a real effect with real consequences, and it is covered in View Identity. The attribute itself is not the variable.
  • The attribute requires an opaque or concrete view return type. It cannot be applied to a function returning something that is not a view.
  • Marking every helper @ViewBuilder is noise. It is inert on single-expression bodies, but it tells a reader "expect branching here" — misleading when there is none.
