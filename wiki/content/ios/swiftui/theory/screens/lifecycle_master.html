There are three lifecycles in an iOS app, nested inside each other, and most lifecycle bugs come from reaching for a tool that belongs to a different level. The process launches and receives push tokens. The scene moves between active, inactive, and background. An individual view enters and leaves the hierarchy.

ScenePhase has exactly three cases, and the mental model that matters is that they describe the scene — not the app, and not any view inside it.

THE ONE IDEA onDisappear never means "the app is closing". scenePhase never delivers a payload. The AppDelegate never tells you a view went away. Conflating the three is what almost every lifecycle bug turns out to be.

2 Mental Model — how to think about it

LevelToolFires for
App process@UIApplicationDelegateAdaptorLaunch, push tokens, background task registration
Scene / window@Environment(\.scenePhase)active ↔ inactive ↔ background
Individual view.task / onAppear / onDisappearA view entering or leaving the hierarchy

The three phases, with the one people forget in the middle:

PhaseMeaning
.activeForeground and receiving events.
.inactiveVisible but not receiving events. Phone calls, the app switcher, Control Center, an incoming FaceTime — all produce this, not .background.
.backgroundNot visible. The system may terminate you from here without running any more of your code.

3 When It Applies

When you see…The lifecycle question to ask
Draft data lost when the app is killedIs the save in onDisappear? It is not guaranteed to run.
Setup running twice, or not at all, on launchscenePhase cannot distinguish cold launch from resume — track it explicitly.
Background logic firing during a phone callAn interruption produces .inactive, not .background.
Needing a push token or a launch payloadThat is AppDelegate territory; scenePhase carries no data.

4 The API — types, modifiers, signatures

SwiftObserving the scene phase
@main
struct MyApp: App {
    @Environment(\.scenePhase) private var scenePhase

    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .onChange(of: scenePhase) { newPhase in
            switch newPhase {
            case .active:     break   // foreground, receiving events
            case .inactive:   break   // visible but NOT receiving events
            case .background: break   // not visible; termination can follow
            @unknown default:  break
            }
        }
    }
}
SwiftBridging to an AppDelegate when you need one
struct MyApp: App {
    @UIApplicationDelegateAdaptor(AppDelegate.self) private var appDelegate

    var body: some Scene {
        WindowGroup { ContentView() }
    }
}
The one-sentence version: scenePhase reports state; the AppDelegate receives events with data. They are not substitutes for one another.

5 Patterns That Work

Persist on .inactive, not .background

.inactive is the last phase you are guaranteed to see. The system can move you from .background to terminated without running more of your code, and .background gives only a limited execution window anyway. .inactive fires on the way out of every exit path — interruption, app switcher, backgrounding.

SwiftThe last reliably-executed moment
.onChange(of: scenePhase) { newPhase in
    if newPhase == .inactive {
        viewModel.persistDraft()
    }
}

The tradeoff worth stating out loud: .inactive fires more often than you strictly need — every phone call, every app-switcher peek. So the save must be cheap and idempotent. If it is expensive, debounce it or write incrementally rather than deferring the work to .background.

Distinguish cold launch from resume yourself

scenePhase carries no history. It is a snapshot of the current phase, not a transition log, so both a cold launch and a return from background surface as a transition into .active.

SwiftAn explicit flag is the reliable model
@Environment(\.scenePhase) private var scenePhase
@State private var hasLaunched = false

.onChange(of: scenePhase) { newPhase in
    guard newPhase == .active else { return }
    if hasLaunched {
        // returning from background - refresh stale data
    } else {
        hasLaunched = true
        // cold launch - one-time setup
    }
}

.onChange fires only on changes, so the very first .active on a cold launch may not fire at all, depending on when observation begins. That is precisely why an explicit flag — or a .task on the root view — is the dependable way to model "first launch".

6 Anti-Patterns — and the fix

Saving in onDisappear

This is the headline mistake. onDisappear is a view-hierarchy callback, not a termination hook. If the system kills a backgrounded app — memory pressure, a swipe-up in the app switcher, jetsam — the views are never torn down in a way that runs it.

SwiftData loss waiting to happen
// WRONG: "save before we die" logic that may simply never execute.
.onDisappear { viewModel.persistDraft() }

// RIGHT: the scene-level hook that fires on every exit path.
.onChange(of: scenePhase) { newPhase in
    if newPhase == .inactive { viewModel.persistDraft() }
}

Treating an interruption as backgrounding

A phone call, a notification banner, or the app switcher moves the scene to .inactive, not .background. Logic keyed on .background will not run; logic keyed on .inactive will run more often than expected. Both are worth knowing before writing either.

Expecting scenePhase to carry a payload

It reports a phase and nothing else. Push tokens, launch options, and background-task registration all live on the AppDelegate.

7 Pitfalls & Gotchas

Lifecycle facts worth memorizing
1. onDisappear is not guaranteed before termination. Never put save-before-we-die logic there.
2. Interruptions produce .inactive, not .background. The app reaches .background only if the user actually leaves.
3. scenePhase cannot distinguish cold launch from resume. Both appear as a transition into .active, with no history attached.
4. A notification banner does not background you. At most you briefly pass through .inactive.
Confidence note, carried from the source material: the macOS multi-window cases are genuinely scene-dependent and were reasoned from Apple's documented lifecycle model rather than confirmed on a running device. Treat those as reasoning to state, not facts to quote.
