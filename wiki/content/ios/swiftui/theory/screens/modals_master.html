Modal presentation covers four APIs that share one shape: a binding drives whether something is shown. .sheet and .fullScreenCover present a whole screen; .alert confirms a single action; .confirmationDialog offers a choice among several.

The decision that matters most is which kind of binding drives them. A boolean says "show something"; an optional item says "show this" — and only the second makes it impossible to present before the content exists.

THE ONE IDEA Prefer item: over isPresented: whenever the content depends on data. The boolean form lets you set the flag and the item separately, so the sheet can appear before the item is assigned — showing stale or empty content. The item: form makes that state unrepresentable.

2 Mental Model — how to think about it

A modal is not part of the navigation stack. It is presented over the current context, which has two consequences people trip on: the presenter's NavigationStack does not extend into it, and on older iOS versions its environment is not fully inherited.

Binding styleWhat it can express
isPresented: Binding<Bool>"Show something." Whether the content is ready is a separate, uncoordinated fact.
item: Binding<Item?>"Show this item." Presentation and content are the same value, so they cannot disagree.

This is the same "make invalid states unrepresentable" principle as modelling view state as an enum, applied to presentation.

4 The API — types, modifiers, signatures

SwiftThe two binding shapes
// Boolean-driven - fine when there is exactly one thing to present.
@State private var isShowingSettings = false

Button("Settings") { isShowingSettings = true }
    .sheet(isPresented: $isShowingSettings) {
        SettingsView()
    }

// Item-driven - PREFERRED when the sheet depends on a value.
@State private var selectedItem: Item?

List(items) { item in
    Button(item.name) { selectedItem = item }
}
.sheet(item: $selectedItem) { item in      // Item must be Identifiable
    DetailView(item: item)
}

.fullScreenCover is identical in API shape but covers the whole screen and cannot be swipe-dismissed — the right choice for onboarding, login, or anything requiring an explicit decision.

5 Patterns That Work

Dismiss from the child, without knowing who presented it

SwiftThe reusable way
struct SettingsView: View {
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        Button("Done") { dismiss() }
    }
}

@Environment(\.dismiss) works for sheets, covers, and navigation pops alike — the child does not need to know which. That is the point: it dismisses "whatever presented me", so the view stays reusable. Setting the binding back from the parent also works, but requires threading a Binding down.

Partial-height sheets with detents

SwiftiOS 16+
.sheet(isPresented: $isShowing) {
    FilterView()
        .presentationDetents([.medium, .large])
        .presentationDragIndicator(.visible)
}

// Track or control which detent is active:
@State private var detent: PresentationDetent = .medium
.presentationDetents([.medium, .large], selection: $detent)

// Custom heights:
.presentationDetents([.height(200), .fraction(0.75)])

Companion modifiers: .presentationCornerRadius, .presentationBackground, and .presentationBackgroundInteraction(.enabled) to let the user interact with content behind the sheet — a map beneath a search sheet, for instance.

Alerts driven by an error value

SwiftRoles drive styling and placement
.alert("Delete item?", isPresented: $isShowingDeleteConfirmation) {
    Button("Delete", role: .destructive) { delete() }
    Button("Cancel", role: .cancel) { }
} message: {
    Text("This can't be undone.")
}

.destructive renders red; .cancel is bold and always placed last. With no buttons supplied you get a default "OK". Driving the alert from an optional error value is the cleaner pattern, for the same reason item: beats isPresented:.

Confirmation dialogs for a choice among actions

SwiftThe action-sheet replacement
.confirmationDialog("Sort by", isPresented: $isShowingSortOptions, titleVisibility: .visible) {
    Button("Name") { sort = .name }
    Button("Date") { sort = .date }
    Button("Cancel", role: .cancel) { }
}

On iPhone it slides up from the bottom; on iPad and Mac it becomes a popover automatically.

6 Anti-Patterns — and the fix

A boolean plus a separately-assigned item

SwiftTwo facts that can disagree
// WRONG: two independent pieces of state. Present before the item is
// set - or fail to clear it afterwards - and the sheet shows stale or
// empty content.
@State private var isShowingDetail = false
@State private var selectedItem: Item?

// RIGHT: one value drives both presentation and content.
.sheet(item: $selectedItem) { item in DetailView(item: item) }

Two presentation modifiers on the same view

SwiftThe second one silently wins
// WRONG: only one of these will ever present. No warning, no crash.
someView
    .sheet(isPresented: $isShowingA) { AView() }
    .sheet(isPresented: $isShowingB) { BView() }

// RIGHT: attach them to different views, or drive one sheet with an
// enum of cases and switch inside the content closure.
.sheet(item: $activeSheet) { sheet in
    switch sheet {
    case .a: AView()
    case .b: BView()
    }
}

7 Pitfalls & Gotchas

Presentation quirks worth memorizing
1. One presentation modifier per view. A second .sheet on the same view silently wins, and the first never fires.
2. A sheet does not fully inherit the presenter's environment on older iOS versions. If a sheet comes up empty or crashes on an @EnvironmentObject, pass it explicitly.
3. Presenting from inside a List row can be unreliable. Attach the modifier to the List or a container instead.
4. The presenter's navigation stack does not extend into a sheet. If the sheet needs navigation, give it its own NavigationStack.
