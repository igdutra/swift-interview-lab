List carries a lot of built-in behaviour, and some of it is row-count dependent — most visibly the background of .insetGrouped, which renders differently with zero rows than with content. That is the root of the classic "my list flashes white while refreshing" complaint.

The fix is never to fight the flash where it appears. It is to stop the row count from dropping to zero in the first place, and to paint the background unconditionally rather than relying on the style's state-dependent default.

THE ONE IDEA Never discard data you still have just because a new load started. A refresh that resets state to .loading throws away the rows the user is looking at — and takes the background with them. Initial load and refresh are different operations and deserve different code paths.

2 Mental Model — how to think about it

Think of List as a UIKit table view wearing a SwiftUI coat. Several of its defaults are inherited from that implementation, and they are conditional on content:

BehaviourWhat it depends on
The grouped background colourWhether the list currently has rows
Row separator insetsThe row content's leading alignment guide
Row identity and diffingIdentifiable — the same explicit identity mechanism covered in View Identity

Because these are conditional, a list that momentarily empties will visibly change appearance and then change back. The flash is not a bug in .refreshable; it is the list correctly rendering an empty state you did not intend to enter.

3 When It Applies

When you see…The list question to ask
The background flashing white mid-refreshDoes the row count hit zero during the reload? Does the style's default background still apply?
Rows vanishing on pull-to-refreshIs the refresh path resetting state to .loading?
Separators that stop short of the row's leading edgeThe alignment guide, applied per row rather than on the list.
An empty list that looks broken rather than intentionalUse an explicit empty state via .overlay instead of the default rendering.

4 The API — types, modifiers, signatures

SwiftA stable background, whatever the row count
List(items) { item in
    Text(item.name)
}
.listStyle(.insetGrouped)
.scrollContentBackground(.hidden)                          // remove the built-in one
.background(Color(uiColor: .systemGroupedBackground))  // paint it unconditionally

.scrollContentBackground(.hidden) removes the row-count-dependent background entirely; the explicit .background behind it then paints the correct grouped colour whether the list is empty, loading, or full.

Order is load-bearing: .scrollContentBackground(.hidden) must come before .background(...). Reversed, the default background is still drawn and your custom colour ends up behind it, invisible.

5 Patterns That Work

Separate initial load from refresh

If the visible items are derived from a state enum, do not route every load through a function that resets to .loading. That collapses the item list to empty mid-refresh — which both triggers the background flash and loses the rows the user was reading.

SwiftTwo entry points, two behaviours
.task {
    await loadItems()       // first load: a loading state is fine, the list is empty anyway
}
.refreshable {
    await refreshItems()    // refresh: keep existing rows on screen
}
SwiftThe shared tail, without the reset
private func loadItems() async {
    viewState = .loading
    await applyFetchResult()
}

private func refreshItems() async {
    // No `viewState = .loading` here - existing rows and their background
    // stay on screen while the new data loads.
    await applyFetchResult()
}

private func applyFetchResult() async {
    switch await fetchItems() {
    case .success(let items): viewState = .loaded(items)
    case .failure(let error): viewState = .error(error)
    }
}

Full-width row separators

List generates a separator per row and insets it. The confirmed fix is to apply the alignment guide per row content, not on the List itself.

SwiftPer row, not per list
List(restaurants) { restaurant in
    RestaurantRowView(restaurant: restaurant)
        .alignmentGuide(.listRowSeparatorLeading) { _ in 0 }
}

An intentional empty state

If you want a distinct empty look — a centred message rather than a blank list — draw it yourself via .overlay rather than letting List fall back to its default empty rendering. ContentUnavailableView is the iOS 17+ tool; on iOS 16 a VStack in an overlay does the same job.

6 Anti-Patterns — and the fix

One load function for both first load and refresh

SwiftThe reset that causes the flash
// WRONG: pull-to-refresh routes through the same reset as the first
// load, so the rows vanish and the background flashes before the new
// data arrives.
.refreshable { await loadItems() }   // loadItems() sets .loading

// RIGHT: a refresh path that never returns to .loading.
.refreshable { await refreshItems() }

Fighting the flash with a fixed height or a delay

Both treat the symptom. The list is correctly rendering an empty state; the fix is to not enter that state, and to paint the background unconditionally.

Putting the alignment guide on the List

It has to be on the row content. Applied to the list it silently does nothing, which reads as "the fix does not work".

7 Pitfalls & Gotchas

List quirks worth memorizing
1. This is a List rendering quirk, not a .refreshable bug. The same empty-versus-loaded background mismatch happens without .refreshable at all, any time the row count can drop to zero temporarily.
2. Modifier order matters for the background. .scrollContentBackground(.hidden) before .background(...), or the custom colour is drawn behind the default and never seen.
3. .refreshable's spinner tracks a real await. For it to reflect the true duration of a completion-handler API, the call must be genuinely awaitable — bridge it with a continuation rather than returning immediately.
