.searchable adds a search field to a screen's navigation chrome and binds it to a string. The mechanics are simple; the part worth understanding is what happens on every keystroke, because a list that re-filters — or worse, re-fetches — on each character wastes work and lets stale results race back after newer ones.

The fix is a debounce, and SwiftUI has one built in without any Combine pipeline: .task(id:) restarts when its id changes, cancelling the previous run first.

THE ONE IDEA Each keystroke cancels the prior 300 ms wait and starts a fresh one. Only a value the user stops changing for 300 ms survives long enough to be committed. That is the entire debounce — no PassthroughSubject, no .debounce, no .sink, no cancellables set.

2 Mental Model — how to think about it

Keep two pieces of state, and be strict about which one filtering reads:

searchText            <- mutated by .searchable on EVERY keystroke
     |
     |  .task(id: searchText)  -- cancels the previous run, sleeps 300ms
     v
debouncedSearchText   <- the ONLY thing filtering ever reads
The gap between the two is the debounce. Filtering on the top one directly reintroduces the exact problem this pattern exists to solve.

Cancellation is the mechanism, not an error path. Task.sleep throws CancellationError when interrupted, and here that is the expected, frequent case — which is why the sleep is written with try?.

3 When It Applies

When you see…The search question to ask
No search field appears at allIs .searchable inside a NavigationStack? Outside one it renders nothing, silently.
A network call per keystrokeIs filtering reading searchText instead of the debounced copy?
Older results overwriting newer onesSame cause — without cancellation, in-flight requests can finish out of order.
The list blanking while a search runsThe in-flight state should carry the data it is filtering.

4 The API — types, modifiers, signatures

SwiftThe basic shape
struct ItemsView: View {
    @State private var searchText = ""

    var body: some View {
        NavigationStack {          // required - the field is navigation chrome
            List(items) { item in
                Text(item.name)
            }
            .searchable(text: $searchText)
        }
    }
}
APIWhat it does
placement:.automatic, .navigationBarDrawer(displayMode: .always) to keep it permanently visible, .toolbar, .sidebar.
.searchCompletion(_:)Suggestions from a trailing closure; tapping one replaces the text.
@Environment(\.isSearching)Whether the field is active. Must be read from a child view, not the one declaring .searchable.
@Environment(\.dismissSearch)Closes the search field programmatically.

5 Patterns That Work

Debounce with .task(id:)

SwiftTwo pieces of state, one of them authoritative
@State private var searchText = ""
// The only thing filtering ever reads - never written from the field.
@State private var debouncedSearchText = ""

private var filteredItems: [Item] {
    guard !debouncedSearchText.isEmpty else { return items }
    return items.filter {
        $0.name.localizedCaseInsensitiveContains(debouncedSearchText)
    }
}
SwiftThe debounce itself
List(filteredItems) { item in
    Text(item.name)
}
.searchable(text: $searchText)
.task(id: searchText) {
    try? await Task.sleep(for : .milliseconds(300))
    guard !Task.isCancelled else { return }
    debouncedSearchText = searchText
}

Use localizedCaseInsensitiveContains

Prefer it over calling .lowercased() on both sides of .contains. It is one call, it reads as intent, and it is locale-correct — naive ASCII lowercasing mishandles the Turkish dotted and dotless i.

Move search state into a view model when it grows

Local @State is fine for purely local filtering. Lift it to a view model as soon as the search hits the network, feeds more than this one view, needs to distinguish loading from searching, or has to be unit-tested — see View State as an Enum for the state shape that keeps rows on screen during an in-flight search.

6 Anti-Patterns — and the fix

Filtering on searchText directly

SwiftThe most common mistake here
// WRONG: skips the debounced copy entirely, silently reintroducing the
// per-keystroke work the pattern exists to prevent.
items.filter { $0.name.localizedCaseInsensitiveContains(searchText) }

// RIGHT: read only the debounced value.
items.filter { $0.name.localizedCaseInsensitiveContains(debouncedSearchText) }

Reaching for a Combine pipeline

A PassthroughSubject with .debounce, .sink, and a cancellables set does the same job with considerably more machinery. Reach for Combine only if you are already in a pipeline elsewhere in the same view.

Putting .searchable outside a navigation container

The field is rendered by the navigation chrome, not by the List. Outside a NavigationStack or NavigationSplitView nothing appears — and there is no warning to tell you why.

7 Pitfalls & Gotchas

Search details worth memorizing
1. try? on the sleep is deliberate. Task.sleep throws CancellationError when cancelled, and cancellation is the expected, frequent case here — not an error to surface.
2. @Environment(\.isSearching) must be read from a child view, not the view that declares .searchable. Read in the wrong place it simply never becomes true.
3. Version floors: .task(id:) needs iOS 15+, and Task.sleep(for:)'s Duration overload needs iOS 16+. Below that, use Task.sleep(nanoseconds:).
