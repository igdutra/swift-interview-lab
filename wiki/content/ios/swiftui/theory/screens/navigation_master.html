A NavigationStack has exactly two parts, and confusing them causes the most common navigation bug in SwiftUI. The root view is the trailing closure passed to the stack: fixed at the call site, living permanently at the bottom, and never a member of the path. The path is an ordered list of everything pushed on top of that root.

So path is not "which screen is showing". It is "how many screens are stacked above the root, and what they are". path.count == 0 means the root is on screen; path.count == 1 means the root plus one pushed screen, with a back button.

THE ONE IDEA Setting path = [.finished] does not mean "show .finished instead of the root". It means "push .finished on top of the root" — which is exactly why a back chevron appears. There is still something underneath to go back to.

2 Mental Model — how to think about it

path = []                 path = [.detail]           path = [.detail, .edit]

  +-----------+             +-----------+              +-----------+
  |           |             |  detail   | <- pushed    |   edit    | <- pushed
  |   ROOT    |             +-----------+              +-----------+
  |           |             |   ROOT    | <- still     |  detail   | <- still
  +-----------+             +-----------+    mounted   +-----------+
                                                       |   ROOT    | <- still
   no back button            back button               +-----------+
                                                        back button
The root never leaves. Emptying the path does not destroy anything — it just stops covering it.

This is why "pop to root" is cheap: the root was mounted the entire time, merely hidden beneath the pushed screens. It does not need to be recreated or re-fetched, because it never went away.

5 Patterns That Work

Replace the root by choosing it outside the stack

When the requirement is "go to screen X and there is no going back" — onboarding to main app, splash to home, login to dashboard, a Finish screen — mutating the path is the wrong tool. Make the choice of root itself conditional.

SwiftTwo mutually exclusive roots
var body: some View {
    switch navigation.route.last {
    case .finished:
        FinishView()          // becomes the root - no stack, no back button
    default:
        NavigationStack(path: $navigation.route) {
            ContentView(navigation: navigation)
                .navigationDestination(for : Screen.self) { screen in
                    destination(for : screen)
                }
        }
    }
}

This is view-tree replacement rather than stack manipulation. Once on FinishView there is no NavigationStack at all — so no back button, and no underlying view to pop to.

A single injected navigator

Keeping the path in one observable object, injected where it is needed, gives navigation a single source of truth that can be driven from a view model or a deep link, and inspected in tests.

SwiftOne owner for the stack
final class Navigator: ObservableObject {
    @Published var route: [Screen] = []

    func push(_ screen: Screen) { route.append(screen) }
    func popToRoot()            { route.removeAll() }
}

6 Anti-Patterns — and the fix

Mutating the path to "replace" the root

The single most common navigation bug. The requirement is "replace the main view with FinishView"; the implementation is route = [.finished], which is a push with the root surviving underneath; the symptom is a back button nobody asked for.

SwiftPush versus replace
// WRONG: this pushes FinishView ON TOP of the root. The root is still
// mounted underneath, so SwiftUI correctly offers a way back to it.
route = [.finished]

// RIGHT: choose a different root entirely, outside any NavigationStack.
// See the pattern above.

.navigationDestination on the stack instead of inside it

A silent, easy-to-miss mistake: it compiles, the path still type-checks, but the destination is never registered against the stack's navigation context. Pushes resolve to nothing — a blank screen with a stray back button, no crash and no error.

SwiftPlacement is load-bearing
// WRONG: the modifier is on the STACK, not inside its content.
NavigationStack(path: $path) {
    RootView()
}
.navigationDestination(for : MyRoute.self) { route in ... }

// RIGHT: on the root view, inside the stack's content.
NavigationStack(path: $path) {
    RootView()
        .navigationDestination(for : MyRoute.self) { route in ... }
}

The root view may register .navigationDestination for types it never pushes itself — that is fine. The modifier just needs to be visible somewhere in the content tree the stack manages; it does not need to sit on the exact view that calls append.

Still using NavigationView

Deprecated. NavigationStack with a path binding is the only shape worth writing now.

7 Pitfalls & Gotchas

Stack behaviour that surprises people
1. Emptying the path does not "clear the screen". removeAll() destroys nothing — the root was mounted the whole time, just hidden. That is why pop-to-root needs no re-fetch.
2. The root can never be swapped by mutating the path. No amount of pushing, appending, or assigning replaces it. The root is chosen in code, at the call site.
3. A misplaced .navigationDestination fails silently. No crash, no warning — just a push that lands nowhere. Check placement first when a push "does nothing".
