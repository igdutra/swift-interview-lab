SwiftUI Theory

iOS · Framework reference · Fundamentals · State & Data Flow · Building Screens

SwiftUI is declarative: you describe what the UI is for a given state, and the framework decides what to change on screen. That inversion is what makes it productive, and it is also why its failure modes are unfamiliar — the bugs are rarely rendering bugs. They are identity bugs (the framework thought two views were the same, or thought they were different) and ownership bugs (state lives in the wrong place, so it resets at the wrong moment or fails to propagate).

These pages are organized by how the knowledge is used rather than by API surface. Read Fundamentals once — it establishes the vocabulary the other two sections lean on constantly. Everything below the section cards is the recall layer: one line per page, the symptom-to-page router, and the rules that are pure memorization. Skim it the night before; open a page only where the one-liner does not already bring the whole thing back.

Where to start: View Identity. Most SwiftUI questions that begin "why did my state reset" or "why won't this animate" resolve to it, and the structural-versus-explicit vocabulary it introduces is used on nearly every other page here.

Fundamentals — how SwiftUI itself thinks
View Identity Identity is a path: (type, position), computed per slot in the view tree An if/else compiles to _ConditionalContent<A, B> — a type-level choice, so flipping it destroys @State. Push the condition into a value, not the structure. 
// WRONG: two concrete types in one slot — @State inside resets on every flip.
if isEditing { EditorView() } else { SummaryView() }

// RIGHT (same view, different value): one Text, one identity, animates for free.
Text("Hi").foregroundStyle(isHighlighted ? .red : .blue)

// .id() DISCARDS identity on purpose — the intended use:
EditorView(record: selectedRecord).id(selectedRecord.id)   // new record → fresh state

// …and the rarer direction: the SAME .id() on both branches unifies them.
Group { if isEditing { EditorView() } else { SummaryView() } }.id("record-detail")
View Construction @ViewBuilder is a compile-time transform, not a runtime feature Needed in exactly one case: the body branches or has more than one top-level view. Costs nothing at runtime — a helper function is not slower than extracting a struct. 
// No @ViewBuilder needed — a single expression is already one type.
private func title(text: String) -> some View { Text(text).font(.title) }

// @ViewBuilder needed — the branches produce different types.
@ViewBuilder
private func badge(isActive: Bool) -> some View {
    if isActive { Text("Active") } else { Text("Inactive") }
}
// The ONE case a func genuinely fails: a func has no identity slot, so it
// cannot hold @State. That is a correctness issue, not a performance one.
The Layout System Parent proposes · child chooses · parent positions A parent can never force a size. Modifier order matters: .frame then .border draws around the frame, not the content. GeometryReader swallows the full proposal and is top-leading, not centred. 
// Ignore the proposal, take the ideal size — the fix for surprise truncation.
Text(longTitle).fixedSize(horizontal: true, vertical: false)

// Choose what survives compression — higher priority is compressed LAST.
HStack {
    Text(veryLongTitle).layoutPriority(1)   // gets its space first
    Text(subtitle)                          // default 0 — compressed first
}

// Responsive without measuring: first child that fits wins.
ViewThatFits { HStack { rowContent } ; VStack(alignment: .leading) { rowContent } }
State & Data Flow — who owns a value and how it travels
State Ownership @StateObject owns and outlives redraws; @ObservedObject only borrows Views are structs and init re-runs on every parent redraw — so real work belongs in .task, never init. Inject the dependency, build the view model inline (the autoclosure runs once). 
// Testable AND built exactly once: protocol in, object built in the autoclosure.
struct MyView: View {
    @StateObject private var viewModel: MyViewModel

    init(repository: ItemRepositoryProtocol) {
        _viewModel = StateObject(wrappedValue: MyViewModel(repository: repository))
    }
}
// WRONG: `@StateObject var viewModel = MyViewModel()` — no seam to inject through.
// init stores references only; the real work goes in .task { await viewModel.load() }

// iOS 16: pass plain VALUES down, or every subview redraws on any change.
struct DetailRow: View { let title: String }   // NOT @ObservedObject → body skipped
Data Flow The direction and the kind of thing being moved pick the tool Four directions, four tools: @Binding down (read+write) · closure up (an event) · PreferenceKey up (measured geometry) · @Environment down to any descendant. 
// 1. DOWN, read+write — parent owns the truth, child mutates it.
struct ToggleRow: View { @Binding var isOn: Bool }        // parent passes $isOn

// 2. UP, an event — a closure. State stays derived and read-only in the child.
struct PaymentRow: View {
    let isSelected: Bool         // derived, read-only
    let onSelect: () -> Void     // event, not state
}

// 3. UP, measured geometry — PreferenceKey (reduce merges sibling reports).
struct HeightPreferenceKey: PreferenceKey {
    static var defaultValue: CGFloat = 0
    static func reduce(value: inout CGFloat, nextValue: () -> CGFloat) { value = nextValue() }
}

// 4. DOWN to ANY descendant, skipping the middle — @Environment.
@Environment(\.dismiss) private var dismiss
View State as an Enum One enum instead of parallel booleans, so illegal combinations cannot be constructed The in-flight case carries the data it operates on — that is the key move, and it is what keeps rows on screen during a refresh. Store the full list; derive the filtered view. 
// .searching CARRIES the list — "searching with nothing loaded" cannot be built.
enum ViewState {
    case loading
    case loaded([Item])
    case searching([Item])    // ← the key move: a precondition in the type
    case error(Error)
}

// Store everything; DERIVE the filtered view — nothing is ever thrown away.
var visibleItems: [Item] {
    switch viewState {
    case .loaded(let allItems), .searching(let allItems): return allItems
    case .loading, .error: return []
    }
}
// The body becomes an exhaustive switch — a fifth case breaks the build where it must.
Building Screens — the applied layer
Navigation The path is what sits on top of the root — never the root itself No amount of mutating the path replaces the root; it is chosen at the call site. A misplaced .navigationDestination fails silently. NavigationView is deprecated. 
// One owner for the stack — drivable from a view model, a deep link, or a test.
final class Navigator: ObservableObject {
    @Published var route: [Screen] = []
    func push(_ screen: Screen) { route.append(screen) }
    func popToRoot()             { route.removeAll() }   // root was mounted all along
}

// To REPLACE the root, choose it OUTSIDE the stack — path mutation never can.
switch navigation.route.last {
case .finished:
    FinishView()                        // no stack at all → no back button
default:
    NavigationStack(path: $navigation.route) {
        ContentView()
            .navigationDestination(for: Screen.self) { destination(for: $0) }
    }                                   // ↑ misplace this and the push silently does nothing
}
Modals Drive presentation from the data (item:), not a separate boolean One presentation modifier per view — a second one silently wins. A modal is not in the navigation stack: the presenter's NavigationStack does not extend into it. @Environment(\.dismiss) works from the child for all of them. 
// item: — the data IS the trigger, so a stale/empty sheet is unrepresentable.
.sheet(item: $selectedItem) { item in DetailView(item: item) }
// vs isPresented: + a separately-assigned item → the two can disagree.

// The child dismisses itself without knowing who presented it.
@Environment(\.dismiss) private var dismiss     // sheets, covers, AND pops

// Detents for a partial-height sheet (iOS 16+).
.presentationDetents([.medium, .large])

// Alert buttons: .destructive renders red, .cancel is bold and always last.
.alert("Delete item?", isPresented: $isConfirming) {
    Button("Delete", role: .destructive) { delete() }
    Button("Cancel",  role: .cancel) { }
}
Lists Several List behaviours are row-count dependent, so empty looks different from populated The pull-to-refresh "background flash" is the list correctly rendering an empty state you did not mean to enter — separate initial load from refresh. Fix the background with .scrollContentBackground(.hidden) plus your own .background. 
// Two entry points, ONE shared tail — only the first load resets to .loading.
private func loadItems() async {
    viewState = .loading          // fine: the list is empty anyway
    await applyFetchResult()
}
private func refreshItems() async {
    // NO .loading here — rows and their background stay on screen.
    await applyFetchResult()
}
// .task { await loadItems() }  /  .refreshable { await refreshItems() }

// A background that does not depend on the row count:
.scrollContentBackground(.hidden).background(Color.groupedBackground)

// Full-width separators — the guide goes on the ROW, never on the List.
List(restaurants) { RestaurantRowView(restaurant: $0).alignmentGuide(.listRowSeparatorLeading) { _ in 0 } }
Search .task(id:) cancels the previous run before starting a new one — that is the debounce .searchable must be inside a NavigationStack or nothing appears, with no warning. Cancellation is the mechanism, not an error path — hence try? on the sleep. A whole Combine pipeline replaced by two lines. 
// TWO pieces of state. Filtering reads only the debounced one — never the field.
@State private var searchText = ""
@State private var debouncedSearchText = ""

List(filteredItems) { item in Text(item.name) }
    .searchable(text: $searchText)           // must be inside a NavigationStack
    .task(id: searchText) {                  // id changes → previous run is CANCELLED
        try? await Task.sleep(for: .milliseconds(300))   // try? — cancelling is expected
        guard !Task.isCancelled else { return }
        debouncedSearchText = searchText
    }

// Filter with localizedCaseInsensitiveContains — one call, and locale-correct.
items.filter { $0.name.localizedCaseInsensitiveContains(debouncedSearchText) }
Lifecycle Three nested lifecycles — process (AppDelegate), scene (scenePhase), view (.task/onAppear) Persist on .inactive, the last phase you are guaranteed to see — so the save must be cheap and idempotent. onDisappear is not a termination hook and may never run. scenePhase carries no history. 
// Persist on .inactive — it fires on EVERY exit path. Keep it cheap + idempotent.
@Environment(\.scenePhase) private var scenePhase

.onChange(of: scenePhase) { newPhase in
    if newPhase == .inactive { viewModel.persistDraft() }
}

// scenePhase is a SNAPSHOT, not a transition log — cold launch and resume both
// surface as ".active". Keep your own flag to tell them apart.
@State private var hasLaunched = false

.onChange(of: scenePhase) { newPhase in
    guard newPhase == .active else { return }
    if hasLaunched { /* returning — refresh stale data */ }
    else { hasLaunched = true /* cold launch — one-time setup */ }
}
Formatting & Images Each of these has one non-obvious form that is the correct one .resizable() must go inside the AsyncImage closure — the outer view is not an Image yet. Size outside the closure or the row jumps when loading completes. "Centred" multiline text centres per line. 
// .resizable INSIDE the closure (only there do you hold a real Image);
// .frame OUTSIDE it, so the placeholder shares the size and the row never jumps.
AsyncImage(url: viewData.imageURL) { image in
    image.resizable().aspectRatio(contentMode: .fill)
} placeholder: {
    ProgressView()
}
.frame(width: 100, height: 100)   // applies to the placeholder too
.clipped()                          // required with .fill

// "Centred" multiline centres each line on ITS OWN width → looks staggered.
// Fix by left-aligning, or by pinning one shared width:
Text(viewData.name).lineLimit(2).multilineTextAlignment(.center).frame(maxWidth: .infinity)

// .locale takes a Locale, NOT a String — a raw string silently follows the device.
Text(price, format: .currency(code: "BRL").locale(Locale(identifier: "pt_BR")))

Hard rules — pure memorization

The facts with no mental model behind them. These are the ones to reread on the way in.

Silent failures — no crash, no warning, nothing on screen
1..searchable outside a NavigationStack. The field simply does not appear, and nothing tells you why.
2.A misplaced .navigationDestination. The push lands nowhere. Check placement first when a push "does nothing".
3.A second presentation modifier on the same view. One silently wins; the other never fires.
4.A transition with no animation attached. No animation, no transition — and the view must genuinely be inserted or removed, not just hidden.
5..locale given a raw String. It expects a Locale; the currency symbol quietly follows the device instead.
Placement rules that are not negotiable: .resizable() goes inside the AsyncImage closure and the .frame goes outside it · .navigationDestination goes inside the NavigationStack · .searchable goes inside the navigation chrome · the separator alignment guide goes on the row content, not the List.
The three "never" rules: never do real work in a view's init (it re-runs on every parent redraw — use .task) · never treat onDisappear as a termination hook (it may never run — use .inactive) · never add a boolean beside a state enum (it reintroduces exactly the combinatorial problem the enum removed).
Nearly every SwiftUI bug is an identity bug or an ownership bug
When something misbehaves, ask the two questions in order. (1) Did the identity change? A branch, an .id(), or a reordered ForEach makes SwiftUI treat a slot as a different view — state is discarded, animations restart, transitions fire or fail. (2) Who owns the storage? @StateObject outlives redraws, @ObservedObject does not; a view's init re-runs constantly, so anything durable must live in a slot SwiftUI keeps against the identity. Rendering is almost never the culprit — it is a faithful function of state you did not model the way you thought.
