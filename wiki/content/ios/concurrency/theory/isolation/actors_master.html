An actor is a reference type that protects its mutable state by serializing access to it. Inside the actor, code runs with exclusive access — no locks to write, no queues to manage. From outside, every access is asynchronous, because a call may need to wait its turn.

The compiler enforces this, which is the real advance: data races are eliminated by construction rather than by everyone remembering to take the right lock.

THE ONE IDEA Actors are reentrant. At every await inside an actor method the actor can service other calls, so state you read before the await may be different after it. Mutual exclusion protects each individual access — it does not make a method atomic.

2 Mental Model — how to think about it

Think of an actor as a room only one caller may occupy at a time. What surprises people is that an await is a door: the occupant steps out, and someone else may walk in before they return.

Caller A enters image(for:)        [actor is busy with A]
A reads storage[url] -> nil
A hits `await download(url)`       [A LEAVES THE ROOM]
                                   Caller B enters image(for:)
                                   B reads storage[url] -> STILL nil
                                   B hits `await download(url)`
A resumes, writes storage[url]
                                   B resumes, writes storage[url] again

  => the same URL was downloaded TWICE
Both callers saw a cache miss because the state they checked was re-read only after a suspension. This is correct actor behaviour, not a bug.

The fix is to store the in-flight task rather than only the finished result, and to publish it before the first suspension — so a second caller finds something to join.

3 When It Applies

When you see…The isolation question to ask
Shared mutable state reached from several tasksMake it an actor rather than adding a lock.
Duplicate network calls for the same resourceReentrancy across an await. Cache the task, not just the result.
DispatchQueue.main.async after every fetch@MainActor puts you back on main automatically after an await.
A property that never mutates but forces awaitnonisolated, or make it a let.

4 The API — types, modifiers, signatures

SwiftInside is synchronous; outside is async
actor ImageCache {
    private var storage: [URL: UIImage] = [:]

    func image(for url: URL) -> UIImage? {
        storage[url]                        // inside: synchronous, no await
    }

    func store(_ image: UIImage, for url: URL) {
        storage[url] = image
    }
}

// From outside, every access is asynchronous:
let cache = ImageCache()
await cache.store(image, for : url)
let cached = await cache.image(for : url)
Swiftnonisolated — opting out where there is no mutable state
actor ImageCache {
    let identifier: String                  // immutable - safe by definition
    private var storage: [URL: UIImage] = [:]

    nonisolated var debugName: String {     // no await needed at call sites
        "Cache(\(identifier))"
    }
}

let constants are implicitly non-isolated already.

@MainActor is a global actor — a singleton whose executor is the main thread. Annotating a type makes every member main-actor-isolated, which is why DispatchQueue.main.async disappears: resuming after an await inside a @MainActor type returns you to the main actor automatically.

5 Patterns That Work

Cache the task, not just the result

The direct answer to reentrancy. Publishing the in-flight task before the first await means a second caller finds it and joins rather than starting duplicate work.

SwiftRequest coalescing inside an actor
actor ImageCache {
    private enum Entry {
        case inProgress(Task<UIImage, Error>)
        case ready(UIImage)
    }
    private var entries: [URL: Entry] = [:]

    func image(for url: URL) async throws -> UIImage {
        switch entries[url] {
        case .ready(let image):
            return image
        case .inProgress(let task):
            return try await task.value       // join the existing download
        case nil:
            let task = Task { try await download(url) }
            entries[url] = .inProgress(task)  // published BEFORE the await
            let image = try await task.value
            entries[url] = .ready(image)
            return image
        }
    }
}

@MainActor at the granularity you need

SwiftType, function, property, or an explicit hop
@MainActor final class ViewModel { }        // whole type
@MainActor func updateUI() { }              // single function
@MainActor private var cachedTitle: String   // single property

// Force a hop from arbitrary context:
await MainActor.run { label.text = "Done" }

// Assert without hopping (crashes if wrong - for known-main callbacks):
MainActor.assumeIsolated { label.text = "Done" }
SwiftThe DispatchQueue.main.async replacement
@MainActor
final class ItemsViewModel: ObservableObject {
    @Published private(set) var items: [Item] = []

    func load() async {
        // Guaranteed main thread here.
        let fetched = await repository.fetch()   // hops off and back
        items = fetched                          // safe: back on main
    }

    nonisolated func makeIdentifier() -> String { UUID().uuidString }
}

SwiftUI's View is @MainActor-annotated by the framework, and ObservableObject view models are conventionally annotated too.

6 Anti-Patterns — and the fix

Assuming an actor method is atomic

SwiftCheck-then-act across a suspension
actor ImageCache {
    private var storage: [URL: UIImage] = [:]

    // WRONG: two concurrent calls for the same URL both see a cache miss
    // and both perform the download.
    func image(for url: URL) async throws -> UIImage {
        if let cached = storage[url] { return cached }

        let image = try await download(url)   // suspension: others run here

        storage[url] = image
        return image
    }
}

// RIGHT: store the in-flight Task and publish it before suspending.
// See the coalescing pattern above.

Marking everything nonisolated to silence await

nonisolated is correct only for members that genuinely do not touch mutable state. Applied to make a compiler error go away, it removes exactly the protection the actor exists to provide.

Wrapping UI updates in MainActor.run inside a @MainActor type

Redundant. Inside a main-actor-isolated type you are already on the main actor after every await. Reach for MainActor.run only from genuinely non-isolated context.

7 Pitfalls & Gotchas

Isolation facts worth memorizing
1. Mutual exclusion is per access, not per method. Any await inside an actor method is a point where other calls may interleave.
2. Re-check invariants after every await. Anything read beforehand may be stale by the time you resume.
3. MainActor.assumeIsolated crashes if you are wrong. It asserts rather than hops — use it only for callbacks documented to arrive on main.
