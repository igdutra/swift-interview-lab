Sendable marks a type as safe to pass across isolation boundaries — between actors, into a Task, across a @MainActor hop. Actors protect the state they own; Sendable governs what may travel between them. Together they are the two halves of compile-time data-race safety.

Under Swift 6's strict concurrency checking, the compiler rejects non-Sendable values crossing those boundaries. In Swift 5.x the same checking is available by opting in.

THE ONE IDEA @unchecked Sendable silences the compiler; it does not make the type safe. It is a promise that you provide synchronization the compiler cannot see. Wrapping a legacy class in it to make an error go away is precisely how data races get shipped.

2 Mental Model — how to think about it

Ask one question: if two isolation domains held this value at the same time, could they corrupt it? If the answer is no — because it cannot change, or because access is serialized — the type is Sendable.

Automatically SendableWhy
Value types whose members are all SendableCopies are independent — most structs and enums get it inferred.
actor typesAlways, by definition: access is serialized.
Immutable classes — final, all let, all SendableNothing can change, so sharing is harmless.
Primitives, String, stdlib collections of Sendable elementsValue semantics all the way down.
SwiftThe dividing line
struct Item: Sendable {              // usually inferred; write it to be explicit
    let id: Int
    let name: String
}

// NOT Sendable: a mutable class. Two domains could write `count` at once.
final class Counter {
    var count = 0
}

5 Patterns That Work

The three honest ways to make a class Sendable

SwiftImmutable, actor, or genuinely locked
// 1. Make it immutable.
final class Configuration: Sendable {
    let apiURL: URL          // all `let`, all Sendable
}

// 2. Make it an actor.
actor Counter {
    private var count = 0
    func increment() { count += 1 }
}

// 3. Assert it manually - YOU take responsibility for the locking.
final class LockedCache: @unchecked Sendable {
    private let lock = NSLock()
    private var storage: [String: Data] = [:]

    func value(for key: String) -> Data? {
        lock.lock(); defer { lock.unlock() }
        return storage[key]
    }
}

Options 1 and 2 are checked by the compiler. Option 3 is a promise, and is only defensible when the synchronization genuinely exists and is applied on every path.

Adopt strict checking incrementally

Turning on complete checking across a large codebase at once produces an unreadable wall of errors. Enable it per target, fix that target, then move on — the diagnostics are far easier to act on in small batches.

6 Anti-Patterns — and the fix

@unchecked Sendable as an error-silencer

SwiftThe promise nobody kept
// WRONG: no lock, no serialization, no safety - just a silenced warning.
// The race is still there; only the diagnostic is gone.
final class MutableCache: @unchecked Sendable {
    var storage: [String: Data] = [:]
}

// RIGHT: pick one of the three real options. An actor is usually the
// least code and the most checkable.
actor MutableCache {
    private var storage: [String: Data] = [:]
}

Making a type Sendable that holds a non-Sendable member

Conformance is structural: a struct is only Sendable if every stored property is. Declaring it anyway either fails to compile or, with @unchecked, hides a real hazard.

Reaching for a lock before considering an actor

A lock is invisible to the compiler, so it cannot help you find the path you forgot to guard. An actor gives the same serialization with compile-time enforcement.

7 Pitfalls & Gotchas

Sendable facts worth memorizing
1. Most structs and enums get it inferred. If a value type is rejected, look for a class or closure hiding among its members.
2. A @Sendable closure constrains its captures. The error often points at the closure while the real problem is the captured object.
3. Actors are Sendable automatically. Converting a shared mutable class into an actor usually resolves a cluster of errors at once.
