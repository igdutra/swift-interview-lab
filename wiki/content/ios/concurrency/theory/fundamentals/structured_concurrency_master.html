Task ≠ Thread · Cancellation

Structured means child tasks have a lifetime bound to their parent scope. The parent cannot return until its children finish, and cancelling the parent cancels every child automatically. You get that guarantee from two constructs — and lose it entirely with Task.detached.

The practical payoff is that concurrency becomes local reasoning: if a function returns, its concurrent work is done. There is no orphaned task still writing to something you thought was finished with.

THE ONE IDEA async let starts work immediately; the await only collects it. That is why two async let bindings take max(a, b) rather than a + b — and why writing the two calls as sequential awaits quietly doubles your latency.

2 Mental Model — how to think about it

SEQUENTIAL await               STRUCTURED (async let)

  |--- fetchUser 200ms ---|       |--- fetchUser 200ms ---|
                          |       |--- fetchPosts 200ms --|
  |--- fetchPosts 200ms --|
                                  total: 200ms  (they overlap)
  total: 400ms
Both requests are already in flight by the time the first await runs. The await is a collection point, not a start signal.

3 When It Applies

When you see…The structured-concurrency question
Two independent awaits in a rowDo they depend on each other? If not, async let halves the wall time.
A loop of await calls over a collectionA task group runs them concurrently instead of one at a time.
Results arriving in the wrong orderGroups yield in completion order — carry the index and sort.
Hundreds of concurrent requestsA group does not throttle. Window it explicitly.
Work continuing after its function returnedThat is a Task { }, not a child. Store and cancel it.

4 The API — types, modifiers, signatures

Swiftasync let — a fixed number of children
func loadProfile(id: User.ID) async throws -> Profile {
    async let user = api.fetchUser(id)            // starts immediately
    async let posts = api.fetchPosts(userID: id)  // also starts immediately

    // Both are already in flight; `await` just collects them.
    return try await Profile(user: user, posts: posts)
}
SwiftTask groups — a dynamic number of children
func loadItems(ids: [Item.ID]) async throws -> [Item] {
    try await withThrowingTaskGroup(of: Item.self) { group in
        for id in ids {
            group.addTask { try await api.fetchItem(id) }
        }

        var items: [Item] = []
        for try await item in group {      // yields in COMPLETION order
            items.append(item)
        }
        return items
    }
}
Throwing versus non-throwing groups: in withThrowingTaskGroup, the first child error cancels the remaining children and propagates. Use withTaskGroup with a Result payload when you want partial success instead.

5 Patterns That Work

Preserve order by carrying the index

A group yields results as they finish, not as they were added. If order matters, tag each result and sort afterwards.

SwiftTag, collect, sort
try await withThrowingTaskGroup(of: (offset: Int, item: Item).self) { group in
    for (offset, id) in ids.enumerated() {
        group.addTask { (offset, try await api.fetchItem(id)) }
    }
    var collected: [(offset: Int, item: Item)] = []
    for try await pair in group { collected.append(pair) }
    return collected.sorted { $0.offset < $1.offset }.map(\.item)
}

Bound the concurrency

A thousand addTask calls means a thousand concurrent requests. Groups do not throttle for you — window the work explicitly.

SwiftA sliding window of in-flight children
try await withThrowingTaskGroup(of: Item.self) { group in
    var index = 0
    let maxConcurrent = 5

    while index < ids.count {
        if index >= maxConcurrent {
            _ = try await group.next()    // wait for one to finish first
        }
        group.addTask { [id = ids[index]] in try await api.fetchItem(id) }
        index += 1
    }
    // drain the rest...
}

Let cancellation propagate

Cancelling the parent cancels every child automatically — no bookkeeping required. That inheritance is the main reason to prefer a structured construct over a bare Task { } whenever the work belongs to the current scope.

6 Anti-Patterns — and the fix

Sequential awaits for independent work

SwiftDoubling latency for nothing
// WRONG: sequential - 200ms + 200ms = 400ms, though neither call
// depends on the other's result.
let user = try await api.fetchUser(id)
let posts = try await api.fetchPosts(userID: id)

// RIGHT: concurrent - max(200ms, 200ms) = 200ms.
async let user = api.fetchUser(id)
async let posts = api.fetchPosts(userID: id)
let profile = try await Profile(user: user, posts: posts)

Reaching for Task.detached to "run in the background"

Detached loses cancellation propagation, priority, and isolation inheritance. It is not "the background version" of Task { } — it is an unparented root task, and it is the usual cause of "why is this suddenly off the main actor". Reserve it for genuinely independent, fire-and-forget work.

Unbounded task groups over large collections

Adding one child per element is correct only when the collection is small. For anything user-supplied or paginated, window it — otherwise you launch every request at once and may be rate-limited or run out of connections.

7 Pitfalls & Gotchas

Structured concurrency details worth memorizing
1. Every async let must be awaited, or the scope must exit. An unconsumed one is implicitly cancelled and awaited at scope exit — safe, but the work was wasted.
2. Group results arrive out of order. Completion order, not submission order. Carry the index if it matters.
3. A Task { } inside a function is not that function's child. The function can return while it is still running, so bound its lifetime yourself.
4. The first error in a throwing group cancels its siblings. That is usually what you want — but if partial success is acceptable, use a non-throwing group with a Result element.
