A Task is not a wrapper around a thread and does not own one. It is the identity and context an async call chain runs under — priority, cancellation flag, parent and child relationships, task-local values. The runtime schedules it onto threads, but it never reserves one for the task's lifetime.

Getting this backwards is the root of most Swift concurrency confusion, because it is exactly the opposite of the GCD mental model where a queue implies fixed dispatch semantics.

THE ONE IDEA Across an await you have the same Task — same identity, same cancellation flag, same priority — but potentially a different Thread. That single sentence resolves almost every "which thread am I on" question.

2 Mental Model — how to think about it

SE-0304 puts it directly: "A task is the basic unit of concurrency in the system. Every asynchronous function is executing in a task," and "A task runs one function at a time; a single task has no concurrency."

Threads still exist underneath, but neither you nor a task may claim one permanently:

COOPERATIVE THREAD POOL  (~ number of CPU cores)
  +----------+  +----------+  +----------+  +----------+
  | thread 1 |  | thread 2 |  | thread 3 |  | thread 4 |
  +----------+  +----------+  +----------+  +----------+
       ^             ^              ^
       |             |              |
   Task A        Task B         Task A again, after an await
   (job 1)       (job 1)        (job 2) -- possibly a DIFFERENT thread
A task is scheduled as a succession of jobs, each ending at a suspension point. Between jobs the thread goes back to the pool.

What happens mechanically at an await:

  1. Code before the await runs as a job, on whatever thread the pool assigned.
  2. At the await, if the runtime needs to switch context — hopping onto an actor's executor, for instance — the job's state is captured as a continuation, the thread is released back to the pool, and the task is marked suspended.
  3. When the awaited work completes, the continuation is re-scheduled as a new job, possibly on a different thread, but always the same Task.

5 Patterns That Work

Suspend, never block

Tasks cooperate by suspending at await and giving the thread back. That is the entire contract, and it is why the pool can be small. Any synchronous block starves every other task in the process.

Prefer Task { } over Task.detached

SwiftInheritance is usually what you want
Task {                    // INHERITS priority, task-locals, and actor context
    await viewModel.load()
}

Task.detached {           // inherits NOTHING - a fresh root task
    await backgroundCleanup()
}

Task { } inside a @MainActor type stays on the main actor; Task.detached does not, which is the source of "why is this suddenly off the main actor" bugs. Reach for detached only for genuinely independent, fire-and-forget work.

Rely on task identity for cancellation

Because an ordinary async call stays on the enclosing task, a cancellation check deep inside a helper function is asking about the task that started the whole chain. That is what makes cooperative cancellation composable.

6 Anti-Patterns — and the fix

Blocking a cooperative pool thread

SwiftStarving a shared resource
// WRONG: this holds a pool thread hostage. The pool is sized to core
// count, so a handful of these can deadlock the whole process.
func load() async {
    let semaphore = DispatchSemaphore(value: 0)
    legacyFetch { semaphore.signal() }
    semaphore.wait()                     // blocks the thread
}

// RIGHT: suspend instead of blocking - bridge the callback properly.
func load() async {
    await withCheckedContinuation { continuation in
        legacyFetch { continuation.resume() }
    }
}

Assuming Task { } is structured

Neither Task { } nor Task.detached is structured. A Task { } created inside a function is not a child of that function's task, so the function can return while it is still running. If its lifetime should be bounded, store it and cancel it.

Reasoning about threads instead of isolation

"Which thread is this on" is almost never the useful question. The useful one is "which actor is this isolated to" — that is what the compiler enforces and what determines whether an access is safe.

7 Pitfalls & Gotchas

Model mismatches worth memorizing
1. The pool does not grow on demand. Unlike GCD, it is sized roughly to core count. Blocking is not "slower" — it can deadlock.
2. A thread change across an await is normal. It is not a bug and not something to route around.
3. Calling an async function is ordinary control flow. No task is created, so cancellation and priority flow straight through.
