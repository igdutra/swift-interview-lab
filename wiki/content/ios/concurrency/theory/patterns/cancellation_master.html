Cancellation in Swift is cooperative. Cancelling a task sets a flag on it; nothing is interrupted, no exception is injected, and no thread is killed. Work stops only at points where someone checks that flag — either your code, or an API that observes it internally.

That design is why cancellation composes: because an ordinary async call stays on the enclosing task, a check deep inside a helper is asking about the task that started the whole chain.

THE ONE IDEA Apple's placement rule is check before starting expensive work — to avoid paying for a result already known to be unwanted. The entry-point guard is the pattern they demonstrate, not a check sprinkled through every loop iteration.

2 Mental Model — how to think about it

A cancelled task keeps running until it reaches a checkpoint. There are two kinds:

CheckpointWho provides it
try Task.checkCancellation()You — throws CancellationError if the flag is set.
Task.isCancelledYou — a plain Bool, for when you want to return early rather than throw.
Built-in observationThe API itself. URLSession's async methods and Task.sleep already do this.

Between checkpoints, a cancelled task is indistinguishable from a healthy one. That is the whole trade: no preemption means no torn state, at the cost of needing checkpoints where they matter.

3 When It Applies

A cancelled URLSession requestNothing to add: it throws URLError(.cancelled) on its own.
A debounce that should discard the previous attemptThat is exactly the .task(id:) contract.

4 The API — types, modifiers, signatures

The .task(id:) contract: when id changes, the previous task is cancelled and a new one is started. The new task is fully independent — cancelling the old one never touches it. This is a documented contract, not an implementation detail, and it is what makes the SwiftUI debounce pattern work.
SwiftTwo ways to observe the flag
// Throwing: propagates a CancellationError up the chain.
try Task.checkCancellation()

// Non-throwing: return early, no error to handle.
guard !Task.isCancelled else { return }

5 Patterns That Work

Check before expensive work — Apple's own placement

From WWDC23, Beyond the basics of structured concurrency, Apple's sample guards the entry point rather than the loop body:

SwiftThe guard goes before the group is populated
func chopIngredients(_ ingredients: [any Ingredient]) async throws -> [any ChoppedIngredient] {
    try await withThrowingTaskGroup(of: (ChoppedIngredient?).self) { group in
        try Task.checkCancellation()   // BEFORE starting expensive work

        for ingredient in ingredients {
            group.addTask { await chop(ingredient) }
        }

        var chopped: [any ChoppedIngredient] = []
        for try await item in group {
            if let item { chopped.append(item) }
        }
        return chopped
    }
}

Let structured constructs propagate for you

Cancelling a parent cancels every child automatically. Inside async let or a task group there is no bookkeeping to write — which is the strongest practical argument for preferring them over a bare Task { }.

Swallow CancellationError where cancellation is expected

In a debounce, cancellation is the normal case rather than a failure — every keystroke causes one. Writing try? around a sleep there is deliberate, not sloppy.

6 Anti-Patterns — and the fix

Assuming cancellation is preemptive

SwiftNothing stops on its own
// WRONG: cancelling this task changes nothing. The loop has no
// checkpoint, so it runs to completion regardless.
func process(_ items: [Item]) async -> [Result] {
    items.map { expensiveTransform($0) }
}

// RIGHT: give the cooperative protocol somewhere to act.
func process(_ items: [Item]) async throws -> [Result] {
    try Task.checkCancellation()
    return items.map { expensiveTransform($0) }
}

Adding manual checks around APIs that already observe cancellation

URLSession's async APIs observe task cancellation internally and throw URLError(.cancelled) on their own. A manual checkCancellation() wrapped around one is noise. The manual check is for cooperative, CPU-bound work with no built-in awareness — loops, filtering, chained awaits you wrote yourself.

Treating CancellationError as a failure to report

Surfacing "Operation cancelled" to a user who simply typed another character is a bug. Decide per site whether cancellation is expected; where it is, discard it silently.

7 Pitfalls & Gotchas

Cancellation facts worth memorizing
1. A cancelled task keeps running until a checkpoint. No thread is killed and no exception is injected.
2. The new .task(id:) task is independent. Cancelling the previous one never affects it — that separation is what makes debouncing safe.
3. A check after the work is finished is pointless. The cost has already been paid; the guard belongs before it.
