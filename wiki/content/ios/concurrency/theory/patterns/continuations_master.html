Plenty of existing code delivers its result through a completion handler — delegate patterns, legacy repositories, third-party SDKs. A continuation converts one such callback into one await, without changing the original API's signature.

The motivating case is concrete: SwiftUI's .refreshable keeps its spinner visible for exactly as long as its async closure takes. A completion-handler call returns immediately, so .refreshable believes it is done and the native spinner disappears at once.

THE ONE IDEA continuation.resume must be called exactly once. Twice is a runtime crash; never is a permanent hang — a leaked continuation with a task suspended forever. Every path through the callback must resume, and no path may resume twice.

2 Mental Model — how to think about it

async function                      legacy callback API
     |
     |-- withCheckedContinuation --> calls repository.getTypes { ... }
     |                                        |
  [SUSPENDED]                                 | (work happens)
     |                                        |
     |<----------- resume(returning:) --------+
     |
  [RESUMED, with the value]
The task suspends rather than blocking, so its pool thread is released while the callback-based work runs.

Note what this does not do: it does not make the underlying API concurrent, cancellable, or safer. It converts one shape into another so the call site can use ordinary control flow.

3 When It Applies

When you see…The bridging question to ask
A refresh spinner that disappears immediatelyIs the call genuinely awaitable, or does it return before the work finishes?
A repository protocol with completion: parametersWrap it once at the boundary rather than at every call site.
A delegate that fires once with a resultA continuation. If it fires many times, use an AsyncStream instead.
Nested completion handlers three deepBridging the innermost API flattens the whole chain.

4 The API — types, modifiers, signatures

FunctionUse when
withCheckedContinuationThe wrapped API cannot fail, or hands you a Result you want to return as-is.
withCheckedThrowingContinuationThe API can fail and you want throws rather than a Result.
continuation.resume(with: result)The callback already gives you a Result — this plugs straight in.

The "checked" variants detect misuse at runtime — a double resume traps, and a leaked continuation is reported. There are unchecked variants for hot paths; prefer checked until profiling says otherwise. Available from iOS 15, so safe for iOS 16 targets.

5 Patterns That Work

Wrap once, at the boundary

SwiftThe legacy API keeps its signature
// Existing closure-based API - do not change it.
protocol PaymentTypesRepository {
    func getTypes(completion: @escaping (Result<[PaymentType], PaymentError>) -> Void)
}

// Wrap it once with withCheckedContinuation.
private func fetchPaymentTypes() async -> Result<[PaymentType], PaymentError> {
    await withCheckedContinuation { continuation in
        repository.getTypes { result in
            continuation.resume(returning: result)
        }
    }
}

// Call site: plain await, no closures.
private func loadPaymentTypes() async {
    let result = await fetchPaymentTypes()
    // ...
}
SwiftNow the SwiftUI spinners track the real duration
.task { await loadPaymentTypes() }
.refreshable { await loadPaymentTypes() }
Why this matters beyond tidiness: without the bridge, .refreshable's native spinner vanishes instantly and only a manual overlay spinner — racing against it — ends up visible. Making the call genuinely awaitable lets the framework do its job.

Match the error type exactly

The wrapper's Result error type must match the closure's declared type — PaymentError, not Error. A mismatch here produces confusing, cascading compiler errors elsewhere in the file, far from the actual cause.

6 Anti-Patterns — and the fix

A path that never resumes

SwiftThe permanent hang
// WRONG: the failure branch returns without resuming. The task stays
// suspended forever and the continuation leaks.
await withCheckedContinuation { continuation in
    repository.getTypes { result in
        guard case .success(let types) = result else { return }
        continuation.resume(returning: types)
    }
}

// RIGHT: every path resumes exactly once.
await withCheckedContinuation { continuation in
    repository.getTypes { result in
        continuation.resume(returning: result)
    }
}

Resuming twice

A callback that can fire more than once — a delegate method invoked per update, or a handler retained across retries — will crash the second time it resumes. If the source emits repeatedly, it is an async sequence, not a continuation.

Blocking instead of bridging

Reaching for a semaphore to "wait for the callback" blocks a cooperative pool thread and can deadlock the process. The continuation suspends instead, which is the entire point.

7 Pitfalls & Gotchas

Continuation rules worth memorizing
1. Resume exactly once. Twice is a runtime crash; never is a permanent hang. Audit every branch of the callback.
2. Match the Result error type exactly. A mismatch surfaces as confusing errors elsewhere in the file.
3. A continuation adds no cancellation. The wrapped API still runs to completion unless it observes cancellation itself.
