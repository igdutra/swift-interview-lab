An AsyncSequence is a sequence whose elements arrive over time. You consume it with for await, which reads like an ordinary loop but suspends between elements instead of blocking — so the loop body runs each time a value shows up, with the thread released in between.

AsyncStream is the concrete type for building one from a callback-based source: the counterpart to a continuation, for APIs that deliver many values rather than one.

THE ONE IDEA continuation.onTermination is where you stop the underlying source. It fires on cancellation as well as normal completion — so it is the one place that reliably runs when a consumer walks away, and forgetting it leaks a location manager, a socket, or an observer.

2 Mental Model — how to think about it

The pairing with continuations is the clearest way in — same bridging job, different cardinality:

Source shapeBridgeTerminates by
Callback fires once with a resultwithCheckedContinuationresume, exactly once
Callback fires repeatedlyAsyncStreamfinish(), or the consumer stopping

Standard sequence operators work on an async sequence, and they work lazily: map, filter, compactMap, prefix, dropFirst. Nothing is materialized up front, because nothing exists up front.

5 Patterns That Work

Bridge a repeating callback into a stream

Swiftyield to emit, onTermination to clean up
func locationUpdates() -> AsyncStream<CLLocation> {
    AsyncStream { continuation in
        let delegate = LocationDelegate { location in
            continuation.yield(location)          // emit a value
        }
        locationManager.delegate = delegate
        locationManager.startUpdatingLocation()

        continuation.onTermination = { _ in       // on cancel OR finish
            locationManager.stopUpdatingLocation()
        }
    }
}

// Consumption reads like any other loop:
for await location in locationUpdates() {
    updateMap(with: location)
}

Let cancellation end the loop

Cancelling the task running a for await ends the iteration and triggers onTermination. Combined with structured concurrency, a screen disappearing tears down its stream and stops the underlying hardware without any explicit teardown call.

Use the lazy operators

map, filter, and prefix compose the same way as on a synchronous sequence, and evaluate per element as it arrives. prefix in particular is a clean way to consume a bounded number of events and let the stream terminate itself.

6 Anti-Patterns — and the fix

Omitting onTermination

SwiftThe leak that survives the screen
// WRONG: the consumer goes away, but the location manager keeps
// running - draining the battery with nobody listening.
AsyncStream { continuation in
    locationManager.startUpdatingLocation()
    // no onTermination
}

// RIGHT: stop the source when the stream ends, however it ends.
AsyncStream { continuation in
    locationManager.startUpdatingLocation()
    continuation.onTermination = { _ in
        locationManager.stopUpdatingLocation()
    }
}

Using a continuation for a repeating callback

A continuation may resume exactly once. Pointing one at a delegate that fires per update crashes on the second call. Repeating source means stream.

Reaching for Combine out of habit

If a publisher exists only to deliver values into a view, an async sequence is usually the smaller tool — no cancellables set, no subscription lifecycle to manage, and cancellation flows from the task automatically. Combine still earns its place for genuine multi-operator pipelines and multicast.

7 Pitfalls & Gotchas

Stream details worth memorizing
1. onTermination fires on cancellation too, not only on finish(). That is exactly why it is the right place for teardown.
2. A stream with no consumer still buffers. Choose a buffering policy deliberately for high-frequency sources rather than accepting the default.
3. Forgetting finish() leaves the loop waiting. A stream that never terminates keeps its consumer suspended indefinitely.
