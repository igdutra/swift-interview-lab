Stacks and queues are twin linear structures defined by one difference: which end you remove from. A stack removes from the same end you added to (LIFO — last in, first out). A queue removes from the opposite end (FIFO — first in, first out). That single difference produces completely different traversal behaviors: stacks dive deep (DFS), queues fan out level by level (BFS). Picking the wrong one gives you the wrong answer — not just a slow one.

In Swift, both structures are plain [T] arrays. append is the same for both (add to the back). The difference is in removal: removeLast() / popLast() for a stack (O(1)); removeFirst() or a head-index advance for a queue. This overview is the decision layer and Swift toolbox: all skeletons side by side, the one question that routes you to the right structure, and the mechanics you reach for every time. The per-structure theory and variant catalogs live in the deep dives linked above.

2   All Skeletons Side by Side

Six canonical skeletons — four stack, two queue. Recognizing which shape fits a problem is the entire skill.

Stack skeletons

SwiftStack skeleton 1 — push openers, pop-match closers (LC20)
var openStack: [Character] = []
let matchMap: [Character: Character] = [")":"(", "]":"[", "}":"{"]
for character in input {
    if let expectedOpener = matchMap[character] {
        guard openStack.last == expectedOpener else { return false }
        openStack.removeLast()
    } else { openStack.append(character) }
}
return openStack.isEmpty
SwiftStack skeleton 2 — push containers, pop-and-attach (LC394 / LC385)
var contextStack: [(Int, String)] = []
var currentNumber = 0; var currentString = ""
for character in input {
    if let digit = character.wholeNumberValue {
        currentNumber = currentNumber * 10 + digit
    } else if character == "[" {
        contextStack.append((currentNumber, currentString))
        currentNumber = 0; currentString = ""
    } else if character == "]" {
        let (savedCount, savedString) = contextStack.removeLast()
        currentString = savedString + String(repeating: currentString, count: savedCount)
    } else { currentString.append(character) }
}
return currentString
SwiftStack skeleton 3 — iterative DFS (trees / graphs)
var nodeStack: [Node] = [root]
while let current = nodeStack.popLast() {
    process(current)
    for neighbor in current.neighbors.reversed() {   // reversed → left-to-right visit order
        if !visited.contains(neighbor) { nodeStack.append(neighbor) }
    }
}
SwiftStack skeleton 4 — monotonic stack, next greater element (LC496)
var nextGreater: [Int: Int] = [:]
var stack: [Int] = []                       // decreasing bottom → top
for value in nums.reversed() {             // right-to-left so stack holds future candidates
    while let top = stack.last, top <= value {
        stack.removeLast()                   // dominated — evict; can never be next-greater for anything further left
    }
    nextGreater[value] = stack.last ?? -1   // record AFTER eviction, BEFORE push
    stack.append(value)
}
// each element pushed once, popped at most once → O(n) amortized
// flip to increasing stack (pop ≥) for next SMALLER element

Queue skeletons

SwiftQueue skeleton 1 — BFS level-swap (shortest path / level order)
var currentLevel: [Node] = [startNode]
var visited: Set<Node> = [startNode]
var distance = 0
while !currentLevel.isEmpty {
    var nextLevel: [Node] = []
    for node in currentLevel {
        if node == goal { return distance }
        for neighbor in neighbors(node) {
            guard !visited.contains(neighbor) else { continue }
            visited.insert(neighbor); nextLevel.append(neighbor)
        }
    }
    currentLevel = nextLevel; distance += 1
}
return -1
SwiftQueue skeleton 2 — sliding-window queue, FIFO eviction (LC933)
var window: [Int] = []
for timestamp in requests {
    window.append(timestamp)
    while let oldest = window.first, oldest <= timestamp - windowDuration {
        window.removeFirst()
    }
    answer = window.count
}
The shape is the recognition signal — before the variables. "Closest-first" or "level-by-level" → BFS queue skeleton. "Evict stale items from the front" → sliding-window queue. "Nested / matching / deepest-first" → one of the three classic stack skeletons. "Next greater / next smaller / first element to the right that…" → skeleton 4 (monotonic stack): evict dominated elements, record top-after-eviction, push current. Naming variables before identifying the skeleton is the #1 source of wiring bugs in both structures.

3   Swift Tools — the Toolbox

All the Swift mechanics for both structures in one place. Both use [T] arrays — the difference is which end you remove from and the O(1) vs O(n) tradeoff on removal.

Swift toolStructureWhat it buys you
stack.append(x) / queue.append(x)BothO(1) amortized add-to-back — identical for stack and queue
var stack: [Int] = []; stack.append(1); stack.append(2)   // [1, 2] — top = last = 2
var queue: [Int] = []; queue.append(1); queue.append(2)   // [1, 2] — front = first = 1
// Enqueue and push are the same operation. The difference is ONLY in removal.
stack.removeLast()Stack onlyO(1) pop from top — crashes on empty; use popLast() for the safe Optional form
var stack = [1, 2, 3]
stack.removeLast()     // → 3  stack=[1,2]  O(1) — no shifting
stack.popLast()        // → Optional(2)  stack=[1]  O(1) — safe, never crashes
// popLast() is the safer form: returns nil on empty instead of crashing.
// Use removeLast() when empty is a bug; popLast() when empty is a valid exit.
stack.lastStack onlyO(1) peek at the top without removing — returns Optional
var stack = [3, 7]
stack.last          // Optional(7) — non-destructive peek at the top
// Used in skeleton 1: guard stack.last == expectedOpener
// The Optional means empty-stack and wrong-type are handled in one guard.
while let item = stack.popLast()Stack onlyDrain-until-empty — safe loop, exits when the stack is empty
var stack = [1, 2, 3]
while let item = stack.popLast() { process(item) }  // 3, 2, 1 — LIFO order
// The canonical iterative DFS loop (skeleton 3). Also used to drain any stack.
queue.removeFirst()Queue onlyDequeue from front — correct semantics, O(n) on Array (shifts all elements)
var queue = [1, 2, 3]
queue.removeFirst()   // → 1  queue=[2,3]  but O(n) — every element shifts left
// Use this for small inputs (n < ~1000) or when clarity matters.
// Replace with the head-index trick for large-n BFS to avoid TLE.
Head-index trickQueue onlyO(1) amortized dequeue on a plain Array — advance a head index instead of shifting
var queue = [startNode]; var head = 0
while head < queue.count {
    let node = queue[head]; head += 1     // O(1) dequeue
    for neighbor in neighbors(node) { queue.append(neighbor) }  // O(1) enqueue
}
// The array grows but never shrinks. Fine for interviews where n ≤ 10⁵.
// For production: swift-collections' Deque gives true O(1) at both ends.
queue.firstQueue onlyO(1) peek at the front — used in the sliding-window stale-eviction check
while let oldest = window.first, oldest <= cutoff {
    window.removeFirst()    // evict stale items from the front
}
// window.first is Optional — the `let` handles the empty-queue case for free.
children.reversed() before pushingStack (DFS) onlyPushes children in reverse so the leftmost child is on top — matches recursive DFS visit order
// Recursive DFS: calls left child, then right child → left is processed first.
// Stack DFS: last pushed is first popped (LIFO). To visit left before right:
for child in node.children.reversed() { stack.append(child) }
// children=[A,B,C] → push C,B,A → pop A first → visit order A,B,C ✓
Multi-source BFS seedQueue (BFS) onlySeed all sources at distance 0 — finds minimum distance from any source in one pass
// Running BFS once per source is O(sources × V+E). Instead:
var currentLevel: [Cell] = allSourceCells   // all sources at distance 0
var visited = Set(allSourceCells)
// The rest is identical to single-source BFS. Total cost: O(V+E).

4   When to Use Each Structure

The one question that picks the structure
Do I need the most recent thing, or the oldest thing? Most recent → stack (LIFO). Oldest → queue (FIFO). Restate it in terms of traversal: do I need to go as deep as possible first (DFS / nesting / matching) → stack. Do I need to process everything at distance d before anything at d+1 (BFS / shortest path / level order) → queue. If neither applies but you need FIFO for scheduling or rate-limiting → queue.

Confusion pair — the single most important distinction

The mix-upAsk……and route
Stack vs Queue (DFS vs BFS)Do I need the shortest path, or any / deepest path?Shortest → queue (BFS). Any / deepest / topological → stack (DFS) or Kahn's queue.
Stack skeleton 1 vs 2Am I validating balance, or building a nested object?Validate → skeleton 1 (push openers, pop-match). Build → skeleton 2 (push containers, pop-and-attach).
Queue vs DijkstraAre all edge weights equal?Equal → queue BFS (simpler, same O(V+E)). Unequal → Dijkstra (min-heap).
Stack DFS vs recursive DFSCan the input be deeply nested (> a few hundred levels)?Deep → explicit stack (no overflow risk). Shallow → either works.
Sliding-window queue vs sliding-window arrayIs the input a stream or a static array?Stream → queue with front-eviction. Static array → two-pointer window (see sliding window deep dive).
Monotonic stack vs suffix-maxDo I want the nearest qualifying element, or just the maximum value to the right?Maximum value only → suffix precomputation (one pass). Nearest greater/smaller → skeleton 4 (monotonic stack). The word "first" or "nearest" in the problem is the tell.
Monotonic stack vs two pointersAm I resolving pending queries for many elements simultaneously, or steering two indices?Pending multi-element queries → skeleton 4. Converging / both-rightward pair → two pointers / sliding window.

InputExpected behaviorStack or Queue
Empty inputStack: return the empty-stack result (valid brackets = true, decode = ""). Queue: frontier is empty, loop never runs, return sentinel.Both
Extra closers ("]" with empty stack)Stack skeleton 1: guard fails immediately — return false before poppingStack
Unclosed openers ("(" alone)Stack skeleton 1: stack is non-empty at the end — return falseStack

Goal unreachableBFS queue empties without finding the goal — return −1 or the problem's sentinelQueue

Mark visited at dequeue (not enqueue)GOTCHA: same node enqueued multiple times — corrupts distances, O(n²) workQueue

9   Ultimate Cheat Sheet

The night-before skim. Pick the structure, pick the skeleton, know the one pitfall.

Pick the structure

You see…Use…Because…

Complexity at a glance

Structure / PatternTimeSpaceWhy
Stack — skeletons 1 / 2 / 3O(n)O(n)Each element pushed and popped at most once. Stack depth ≤ n.
Stack — skeleton 4 (monotonic)O(n)O(n)Same push-once / pop-at-most-once amortized argument. The inner while-loop can pop many per step, but across the full sweep each element exits at most once.
Queue — BFS (graph)O(V + E)O(V)Each vertex and edge visited once. Visited set holds up to V nodes.
Queue — BFS (grid n×m)O(n · m)O(n · m)Each cell visited once; up to 4/8 directional edges per cell.
Queue — sliding-windowO(n) amortizedO(w)Each item enqueued and dequeued at most once.
Hidden cost (queue)O(n²)—removeFirst() on a plain Array is O(n). In a BFS hot loop, this makes traversal O(V²). Use head-index or Deque.
One pitfall per skeleton
▸Stack skeleton 1: check the top before popping. Pop first and you discard the opener before confirming it matches — the mismatch becomes invisible.
▸Stack skeleton 2: the pop AND the attach happen on ] — two steps. Miss the attach and the popped container is discarded, never reaching the root.
▸Stack skeleton 3 (DFS): push children in reversed order so the leftmost is on top and processed first — matching recursive DFS visit order.
▸Queue BFS: mark nodes visited at enqueue time, not dequeue time — otherwise the same node is enqueued multiple times via different paths, corrupting distances.
▸Queue sliding-window: enqueue the new item before running the eviction loop — if you evict first, the current item may be incorrectly removed at the boundary.
▸Stack skeleton 4 (monotonic): record nextGreater[value] = stack.last ?? -1 after the eviction while-loop but before pushing the current value — reversing those two steps makes an element its own next-greater.
▸Stack skeleton 4 — variant choice: decreasing stack (pop ≤) for next-greater; increasing stack (pop ≥) for next-smaller. Mixing them produces a valid-looking stack that answers the wrong question silently.
