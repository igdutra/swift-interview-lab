Theory
stack_overview_master.html

You are given an encoded string like "3[a2[c]]". The encoding rule is: k[encodedString] means the encodedString inside the brackets should be repeated exactly k times. Brackets can be nested. Decode the string and return it.

InputOutputWhy
"3[a]2[bc]""aaabcbc"Two separate groups: "a" repeated 3 times, then "bc" repeated 2 times
"3[a2[c]]""accaccacc"Inner 2[c] → "cc"; outer 3[acc] → "accaccacc"
"2[abc]3[cd]ef""abcabccdcdcdef"Two encoded groups followed by a plain suffix "ef"

The twist that makes this Medium: the brackets nest, which means the string you are building at one level must be paused while you descend into an inner level — then resumed when that inner level closes. That pause-and-resume is exactly what a stack is for.

2   Reading the Problem — insights from the constraints

"k[encodedString]": the integer always comes before the bracket. This means digits always indicate an upcoming repetition count — never a count after the bracket. You must collect digits into a number before you push.
"Brackets can be nested": this is the phrase that rules out a simple counter or a single-pass replace. When you see [, you cannot yet finish building anything — you may still be inside an outer group. You need to remember the state of the outer group while you build the inner one. That memory is a stack.
"No negative k, valid input guaranteed": you don't need to handle malformed strings, negative repeats, or unmatched brackets. Every [ has a matching ] and every number precedes a bracket.
Plain letters outside brackets: letters that appear outside (or before) any bracket just get appended to whatever string you are currently building. They are part of the result at their nesting level.

"k[encodedString]"Accumulate digits into a number; use it as the repeat count on [

3   Edge Case Inventory

InputExpectedHandled by
"abc" — no brackets"abc"All characters are letters; they append to currentString. Stack is never pushed. Return currentString directly.
"3[a]" — single flat group"aaa"Stack push on [, pop and repeat on ]. The outer level's currentString was empty; result is the repetition only.
"3[a2[c]]" — nested groups"accaccacc"Inner [ saves ("a", 3); inner ] restores it, appends "cc" repeated 2 times to "a" → "acc"; outer ] repeats "acc" three times.
"2[abc]3[cd]ef" — plain suffix"abcabccdcdcdef"Plain letters after all brackets just stay in currentString; returned as-is after the loop.
"10[a]" — multi-digit repeat count"aaaaaaaaaa"Digit accumulator builds currentNumber character by character: currentNumber = currentNumber * 10 + digit. Multi-digit numbers work naturally.
GOTCHA: currentNumber not reset after [wrong repeat counts on siblingsMust reset currentNumber = 0 after pushing it onto the stack — or the next group inherits the previous group's count.
GOTCHA: string concatenation in a loopO(n²) timeBuild with a [Character] array (or String in Swift which uses value semantics efficiently), but be aware that += on strings in Swift is O(n) per call in a tight loop on large inputs. Prefer appending to a buffer.

4   Try It Yourself First

This is the canonical Skeleton 2 problem — the one that makes the push-containers/pop-and-attach mechanic click. Try it cold before reading the plan.

Hint 1 — the constraint

Brackets can nest. When you see [, you are not done with the current level — you need to pause it and start building an inner level. What data structure stores "the state I want to come back to"?

Hint 2 — what to push

When you see [, push two things onto the stack: the repeat count you just finished reading, and the string you have built so far at this level. Then reset both to zero and empty string — you are starting fresh inside the inner group.

Hint 3 — what happens on ]

Pop the saved (count, string) pair. Take the string you just built inside the brackets, repeat it by the saved count, then append that to the saved string. That combined value becomes your new current string — you are back at the outer level, with the inner group's contribution folded in.

5   Interview Simulation

The interviewer has stated the problem. The script starts at your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can the repeat count k be multi-digit, like 10 or 100?"
A: Yes. → I need to accumulate digits into an integer: currentNumber = currentNumber * 10 + digit. A single-digit assumption would break on "10[a]".
Q: "Can brackets be nested? And to what depth?"
A: Yes, nested. Depth is not bounded in a special way. → I need a stack to save and restore context at each nesting level. A single-pass counter won't work.
Q: "Can there be plain letters outside all brackets?"
A: Yes. → Plain letters just go into the string I'm currently building — no special case needed; they belong at whatever nesting level they appear.
Q: "Is the input always valid — every k has a bracket, every bracket has a matching closer?"
A: Yes, input is always valid. → No need to guard against empty-stack pops or missing numbers. I can trust the structure.
Step 2 — The plan, before any code
The plan, spoken

"This is a stack problem — specifically Stack Skeleton 2: push containers, pop and attach. The signal is the nesting: when I enter a bracket I need to pause what I'm building and start fresh inside; when I close a bracket I need to pick up where I left off. That pause-and-resume is exactly what a stack does."

"Let me trace the sequence before naming any variable. I scan left to right. When I see a digit, I'm accumulating a repeat count — could be multi-digit, so I multiply by 10 and add. When I see [, I push the repeat count I just finished reading and the string I have built so far onto the stack, then reset both to zero and empty — I'm now inside a new level. When I see a letter, I append it to whatever string I'm currently building. When I see ], I pop the saved (count, string) pair, take the string I just built inside these brackets, repeat it by the saved count, then append it to the saved outer string — that becomes my new current string. At the end of the loop, currentString is the answer."

"From that sequence, the state is: a stack of (Int, String) pairs — the saved repeat count and the saved partial string at each outer level; a currentNumber: Int accumulator for the digit being read; and a currentString: String for the string being built at the current level."

"The invariant: at all times, currentString holds the decoded output of the current nesting level so far, and the stack holds the saved contexts of every outer level. The pop on ] restores the outer context and folds in the completed inner result."

"Walk on \"3[a2[c]]\": I see '3' → currentNumber=3. I see '[' → push (3, \"\"), reset currentNumber=0, currentString=\"\". I see 'a' → currentString=\"a\". I see '2' → currentNumber=2. I see '[' → push (2, \"a\"), reset. I see 'c' → currentString=\"c\". I see ']' → pop (2, \"a\"): \"c\" repeated 2 = \"cc\", then \"a\" + \"cc\" = \"acc\" → currentString=\"acc\". I see ']' → pop (3, \"\"): \"acc\" repeated 3 = \"accaccacc\", then \"\" + \"accaccacc\" → currentString=\"accaccacc\". End of input — return \"accaccacc\". Correct."

"Time is O(n · k_max) because string repetition expands the output; space is O(n) for the stack depth proportional to nesting depth."

6   Why This Approach — and not the others

CandidateCostVerdict
Recursion — parse from index, return (result, nextIndex)O(n · k_max) / O(depth)Correct and natural — each [ calls into a sub-parse, each ] returns. But uses the call stack for depth, which risks overflow on deeply nested inputs. The iterative stack makes the same computation explicit and safe.
String replace in a loop — find innermost k[s], expand, repeatO(n² · k_max)Works for flat cases but is O(n²) because each pass re-scans the whole string. Nesting requires multiple passes — quadratic.
Explicit stack — push (count, string) on [, pop-and-attach on ]O(n · k_max) / O(n)Chosen. One pass, O(1) work per character except at repetition. Handles arbitrary nesting depth without call-stack risk. Clean separation of the four character cases.
The deciding signal: nesting + "I need to come back to where I was." The moment you recognize you need to pause and resume a computation, a stack is the tool. Recursion is the same idea using the call stack implicitly; the explicit stack is safer on unbounded depth.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
contextStack: [(Int, String)]At each nesting level: the repeat count for the bracket we just entered, and the string built so far at the outer levelA tuple because two pieces of state travel together — they were produced at the same [ and consumed at the same ]. Separate stacks for counts and strings would work but are easier to misalign.
currentNumber: IntThe repeat count being accumulated digit by digitMulti-digit counts require accumulation: currentNumber = currentNumber * 10 + digitValue. Reset to 0 after each [ so it doesn't bleed into the next group.
currentString: StringThe decoded output being built at the current nesting levelAppended to on letters, replaced with the repeated+attached result on ]. The final value of currentString after the loop is the answer.
Why a (Int, String) pair and not two separate stacks. The repeat count pushed on [ and the string it should repeat are semantically one unit of context — they're created together and consumed together. A tuple keeps them physically co-located. Two separate stacks are equivalent but create an implicit coupling that's easy to break if you accidentally push to one and forget the other.

contextStack ← []
currentNumber ← 0

9   Swift Solution — iterative

A.1 Explicit stack of (count, string) context pairs
O(n · k_max)O(n)
Plan
Scan the characters. Accumulate digits into currentNumber. On [, push (currentNumber, currentString) onto the stack and reset both. On a letter, append to currentString. On ], pop the saved context, repeat the just-built string by the saved count, append to the saved string — that becomes the new currentString. Return currentString after the loop. 
func decodeString(_ s: String) -> String {
    var contextStack: [(Int, String)] = []
    var currentNumber = 0
    var currentString = ""

    for character in s {
        if let digit = character.wholeNumberValue {
            // Accumulate multi-digit repeat counts.
            currentNumber = currentNumber * 10 + digit
        } else if character == "[" {
            // Descend one nesting level: save context, start fresh.
            contextStack.append((currentNumber, currentString))
            currentNumber = 0
            currentString = ""
        } else if character == "]" {
            // Ascend one level: pop context, repeat inner result, attach to outer.
            let (savedCount, savedString) = contextStack.removeLast()
            currentString = savedString + String(repeating: currentString, count: savedCount)
        } else {
            // Plain letter: belongs to the current nesting level.
            currentString.append(character)
        }
    }

    return currentString
}
On character.wholeNumberValue. This is Swift's idiomatic way to check if a Character is a decimal digit and get its integer value simultaneously — cleaner than character >= "0" && character <= "9" and no ASCII arithmetic needed. Returns nil for non-digits, so the if let doubles as a type-check and an extraction in one line.

12   Complexity

PhaseTimeSpaceWhy
Character scanO(n)O(n)Each character processed once. Stack depth ≤ nesting depth ≤ n.
String repetition on ]O(|inner| · k)—String(repeating:count:) expands the inner string by a factor of k. In the worst case (single deeply nested group), total work is O(n · k_max).
TotalO(n · k_max)O(n)Where k_max is the largest repeat count. O(n) if all k are treated as constants. Space is O(n) for the context stack — one frame per nesting level, at most n/2 levels deep.
The bottleneck is the output size, not the input size. The decoded string can be exponentially longer than the encoded input in theory (e.g. 2[2[2[a]]] → "aaaaaaaa"). The time complexity tracks the output length, which is bounded by n · k_max in practice given LeetCode's constraints.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
Two micro-improvements: pre-reserve the result string's capacity if the final decoded length is known (avoids repeated reallocations), and avoid building intermediate strings during repetition by tracking counts lazily. Neither changes the asymptotic bound; the O(n · k_max) is irreducible because you must materialize the output.
Follow-up: "What if the brackets can also appear inside letter sequences — e.g. letters before the digit?"
The problem guarantees a digit always precedes [, so this doesn't arise here. If it did, the rule for what constitutes a repeat count would need redefinition — the digit accumulator and the push condition would both need updating, but the stack skeleton itself would be unchanged.
Follow-up: "Can you solve it recursively instead?"
Yes — instead of an explicit stack, use a recursive helper that takes the string and a current index, parses until it sees ] or end-of-string, and returns (result, nextIndex). Each [ calls into a sub-parse. This is equivalent to the iterative approach but uses the call stack for depth. On deeply nested inputs (depth in the thousands), it risks a stack overflow in Swift — the explicit stack version is always safer for production.
Follow-up: "How does this relate to LC 385 — Mini Parser?"
Both are Skeleton 2: push a new container on [, pop and attach on ]. LC 394 builds a decoded string; LC 385 builds a nested integer object. The pop-and-attach step is structurally identical — only the container type changes. Solve 394 first to internalize the push/pop rhythm, then 385 to see the same skeleton applied to a richer data structure.
