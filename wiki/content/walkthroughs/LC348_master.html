Design a tic-tac-toe game on an n × n board for two players. Implement move(row, col, player), called once per turn with player being 1 or 2, which places that player's mark and returns the winner: 1 or 2 if that move wins the game, or 0 if there is no winner yet. A player wins by filling an entire row, an entire column, or one of the two main diagonals with their own marks. Assume every move is valid (an empty cell) and that no move is made after someone has already won.

Calls (n = 3)ReturnsWhy
move(0,0,1), move(0,2,2), move(2,2,1)0, 0, 0No line is yet complete for either player
… move(1,1,1), move(2,0,1)0, then 1Player 1's anti-diagonal (0,2)?-no; the main diagonal (0,0),(1,1),(2,2) completes → 1 wins
First move on an empty board0One mark can never complete a line on a board larger than 1×1

The twist is the efficiency expectation: the naive answer rescans the move's row, column, and diagonals after each move — O(n) per move. The intended design keeps running totals so each move is O(1). The trick that makes it clean is encoding the two players as +1 and −1.

2   Reading the Problem — insights from the constraints

"Returns the winner after each move": the query is an aggregate — "is some line now entirely one player?" — evaluated after every mutation. That is the incremental-aggregate signal: rather than rescan, maintain a running summary per line and update it by a delta on each move.
A win is a full line of one player: a row, a column, or a diagonal containing n marks all from the same player. If you track, per line, a signed count where one player adds +1 and the other subtracts 1, then "all n cells are player 1" is exactly "the count equals +n," and "all player 2" is "−n." One comparison replaces an n-cell scan.
Only the move's own lines can change: placing a mark at (r, c) can only complete row r, column c, the main diagonal (if r == c), or the anti-diagonal (if r + c == n−1). You never need to check any other line — so each move updates at most four totals and tests at most four.
"player is 1 or 2": two players, so a single signed total per line suffices. Map player 1 → +1 and player 2 → −1. There's no need for separate per-player counts; the sign carries the player identity.
"No move after a win" and "every move valid": you don't need to guard against moves on occupied cells or post-game moves. That removes validation from the hot path — the whole method is update-four-totals-and-test.

GOTCHA: mixed players on a linenever a false winPlayer 1's +1 and player 2's −1 cancel, so a mixed line's total can never reach ±n. The ±1 encoding makes this automatic.
Why the ±1 encoding can't produce a false positive. A line reaches +n only if all n contributions are +1 (all player 1), and −n only if all are −1 (all player 2). Any mix sums to something strictly between −n and +n. So |total| == n is true if and only if the line is full of a single player — exactly the win condition, with no separate "are they all the same?" check.

4   Try It Yourself First

Try for the O(1)-per-move design before reading on. Open hints one at a time.

Hint 1 — the constraint

You're asked for the winner after every move. Rescanning the affected row, column, and diagonals each time is O(n) per move. Can you keep a small summary that you update in O(1) per move so the win test is a single comparison?

Hint 2 — the pattern

Incremental aggregate. Keep a running total for each row, each column, the main diagonal, and the anti-diagonal. A move at (r, c) only affects row r, column c, and possibly the two diagonals — update just those.

Hint 3 — the ±1 trick

Encode player 1 as +1 and player 2 as −1. Add that delta to the move's row total, column total, and (if on a diagonal) the diagonal totals. A line is fully one player exactly when its total's absolute value equals n. Return the player when any of the four updated totals hits ±n, else 0.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Is every move guaranteed valid — always an empty cell, and never called after the game is won?"
A: Yes. → No validation on the hot path; the method is just update-and-test.
Q: "There are exactly two players, 1 and 2?"
A: Yes, 1 and 2. → I can encode them as +1 and −1 and keep a single signed total per line instead of per-player counts.
Q: "Is O(1) per move expected, or is an O(n) check after each move acceptable?"
A: Aim for O(1) per move. → I'll maintain running totals per row, column, and the two diagonals, updated incrementally.
Q: "Do I need to detect a draw, or only a win?"
A: Only a win — return the player or 0. → No move counter for draws; just the four total checks.
Step 2 — The plan, before any code
The plan, spoken

"This is incremental aggregate state — Variant 8.2 of the Stateful System Design family. The signal is 'report an aggregate — the winner — after each move,' which says: maintain a running summary, don't rescan."

"Let me pin the conditions before the structure. The invariant I maintain on every move is: each line's stored total always equals the signed count of marks on that line, where player 1 counts +1 and player 2 counts −1. The win condition, stated exactly as it'll be coded, is that a line is fully one player if and only if the absolute value of its total equals n — because reaching +n requires n contributions of +1, and −n requires n of −1, while any mix falls strictly inside. The locality condition is that a move at (r, c) can only change row r, column c, the main diagonal when r == c, and the anti-diagonal when r + c == n−1 — so I update at most four totals and test at most four."

"The structures follow from those conditions: an array rowSum of length n, an array colSum of length n, and two scalars diagonal and antiDiagonal. The ±1 encoding is why a single signed number per line suffices rather than two per-player counters — the sign carries the player. No board array is needed at all, because the totals are the only state any query consults."

"The walk: on move(r, c, player), compute delta = player == 1 ? 1 : -1. Add it to rowSum[r] and colSum[c]; if r == c add it to diagonal; if r + c == n−1 add it to antiDiagonal. Then if any of those four updated totals has absolute value n, return player; otherwise return 0. Time is O(1) per move — a constant number of updates and comparisons; space is O(n) for the two arrays."

6   Why This Approach — and not the others

CandidateCostVerdict
Store the full board, rescan the move's lines each turnO(n) per moveCorrect and simple, but does redundant work — it re-reads up to 4n cells per move when only one cell changed. Fine for tiny n, but not the intended design.
Per-line, per-player counts (two numbers per row/col/diag)O(1) per moveWorks and is O(1), but twice the state and a two-part "did player p fill it?" check. The ±1 encoding folds both players into one number and one comparison.
±1 signed totals per row, column, and two diagonalsO(1) / move, O(n) spaceChosen. Minimal state, one comparison per line, no board, no per-player split. The cleanest expression of the win condition.
The deciding signal: "report an aggregate after each mutation." Whenever a design repeatedly asks for a summary that changes by a small delta each write, maintain the summary incrementally instead of recomputing it. And when there are exactly two contributors, a signed ±1 total collapses "which one, and are they all the same?" into a single number and a single comparison.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
rowSum: [Int] (length n)Signed mark count for each rowOne number per row, updated by ±1 per move. Replaces scanning a row's n cells with a single comparison. An array indexed by row is the natural fit.
colSum: [Int] (length n)Signed mark count for each columnSymmetric to rows. Indexed by column.
diagonal: IntSigned count on the main diagonal (cells where r == c)A single scalar — there is only one main diagonal. Updated only when the move lands on it.
antiDiagonal: IntSigned count on the anti-diagonal (cells where r + c == n−1)One scalar for the single anti-diagonal. Updated only when r + c == n−1.
n: IntBoard size — the win thresholdStored so the win test abs(total) == n has its bound. No board array is kept at all.
Why no board is stored. Every query (the winner) is answered from the four kinds of totals, and the problem guarantees moves are valid, so we never need to know what's in an arbitrary cell. Dropping the n² board for O(n) totals is the space win that accompanies the time win — the aggregate is strictly more compact than the raw state it summarizes.

8   Pseudocode Dry Run

Pseudocode±1 totals, win when |total| == n
CLASS TicTacToe(n):
    rowSum ← array of n zeros
    colSum ← array of n zeros
    diagonal ← 0, antiDiagonal ← 0

    METHOD move(r, c, player):
        delta ← (player == 1) ? +1 : -1
        rowSum[r] += delta
        colSum[c] += delta
        IF r == c:        diagonal += delta
        IF r + c == n-1:  antiDiagonal += delta

        IF |rowSum[r]| == n OR |colSum[c]| == n
           OR |diagonal| == n OR |antiDiagonal| == n:
            RETURN player
        RETURN 0

9   Swift Solution — iterative

A.1 ±1 signed totals per line
O(1) / moveO(n)
Plan
Keep a signed total per row, per column, and one each for the two diagonals. Player 1 adds +1, player 2 adds −1. A move updates only its row, column, and the diagonals it lies on, then checks whether any of those four totals has absolute value n — that's a full line of one player, a win. 
class TicTacToe {
    private let size: Int
    private var rowSum: [Int]
    private var colSum: [Int]
    private var diagonal = 0
    private var antiDiagonal = 0

    init(_ n: Int) {
        size = n
        rowSum = [Int](repeating: 0, count: n)
        colSum = [Int](repeating: 0, count: n)
    }

    func move(_ row: Int, _ col: Int, _ player: Int) -> Int {
        let delta = (player == 1) ? 1 : -1

        rowSum[row] += delta
        colSum[col] += delta
        if row == col { diagonal += delta }
        if row + col == size - 1 { antiDiagonal += delta }

        // A line full of one player has |total| == size.
        if abs(rowSum[row]) == size || abs(colSum[col]) == size
            || abs(diagonal) == size || abs(antiDiagonal) == size {
            return player
        }
        return 0
    }
}

12   Complexity

OperationTimeSpaceWhy
init(n)O(n)O(n)Allocates the two length-n total arrays.
moveO(1)O(1)A constant number of additions (at most four totals) and at most four comparisons. Independent of n.
Per-move bottleneckO(1)O(n) total stateThe aggregate eliminates the O(n) line scan entirely. The only n-dependence is the one-time array allocation and the O(n) resting state.
Why this is strictly better than the board scan. The naive design stores n² cells and does up to 4n work per move re-reading lines. This design stores 2n + 2 numbers and does constant work per move. Both are correct; the difference is the incremental aggregate — we pay a tiny, fixed cost on every write so the "winner?" read is free. That trade is the entire lesson of the variant.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
It's already O(1) per move and O(n) space, which is optimal for this query. The one robustness gap is the assumption that moves are always valid; in production I'd guard against a move on an occupied cell or a move after the game is won, which means either keeping a small board/occupied-set or a won flag. That trades a bit of the space win for safety — a reasonable call outside a controlled judge.
Follow-up: "Generalize to k players, or to 'k in a row wins' on a large board."
For k players the ±1 trick breaks — a single signed total can't distinguish three or more contributors — so I'd revert to per-line, per-player counts (a small map per line) and win when one player's count hits the line length. For "k-in-a-row" on a board larger than k, the line totals no longer suffice because a win is a contiguous run; that needs scanning a window around the move (O(k) per move) or maintaining run-length state per direction.
Follow-up: "Also report when the game is a draw."
Add a move counter incremented each move. After updating the totals and finding no win, if the counter equals n² the board is full → it's a draw. That's one extra integer and one comparison, keeping the move O(1). The win check still takes precedence, since a board-filling move can also be a winning move.
Follow-up: "How does this connect to the broader family — what's the reusable idea?"
The reusable idea is maintain-on-write: keep a derived aggregate updated by a delta on each mutation so the query is O(1). The same shape powers a running-median structure, a sliding-window sum, or a scoreboard. The sibling warm-up, LC 1166 — Design File System, is the opposite pole — pure store-on-write with nothing to derive — so the two together bracket the family's central store-vs-derive axis.
