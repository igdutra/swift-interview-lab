Theory
topological_sort_deep_dive_master.html

There are numCourses courses labeled 0 … numCourses−1. You're given prerequisites, where each pair [a, b] means "you must take course b before course a". Return true if it's possible to finish all courses, false otherwise. In graph terms: can you find an order that respects every prerequisite?

InputOutputWhy
numCourses=2, prerequisites=[[1,0]]trueTake 0, then 1 — one valid order exists
numCourses=2, prerequisites=[[1,0],[0,1]]false0 needs 1 and 1 needs 0 — a cycle, no valid order
numCourses=3, prerequisites=[]trueNo prerequisites — any order works

The twist is recognizing that "can I finish all courses?" is exactly "is the prerequisite graph acyclic?" Each pair is a directed edge; a cycle is a group of courses that all depend on each other, with no valid starting point. So the whole problem reduces to cycle detection on a directed graph — and topological sort (Kahn's algorithm) detects a cycle as a side effect of trying to order the courses, in one O(V+E) pass.

2   Reading the Problem — insights from the constraints

"[a, b] means take b before a": this is a directed edge b → a — the prerequisite points to the course that needs it. Pin this concretely before coding, because reversing it gives a wrong-but-plausible answer. It is a's in-degree (its count of unmet prerequisites) that increases.
"Finish all courses" → acyclic: a valid completion order exists if and only if the prerequisite graph is a DAG. If courses form a dependency loop, none of them can ever be the "first" one taken, so you can't finish them. The question is therefore pure cycle detection.
Only a yes/no is asked: you don't need to produce the order — just whether one exists. Kahn's algorithm answers that by counting how many courses it manages to place: if it places all of them, the graph was acyclic. The order itself can be discarded (that's the LC210 variant).
Courses may have no prerequisites: a course that appears in no pair (or only as a prerequisite) has in-degree 0 and is immediately takeable. Such courses must be seeded into the queue at the start and counted — forgetting isolated vertices undercounts and falsely reports a cycle.
Bounds (numCourses ≤ 2000, prerequisites ≤ 5000): an O(V+E) Kahn's pass is trivially fast. Build an adjacency list, not a matrix — the graph is sparse, and a 2000×2000 matrix would waste memory and make the peel O(V²).

The cycle and the isolated-vertex cases are the load-bearing ones. A cycle is detected only because its vertices never reach in-degree 0 and so never get counted — that's the whole correctness argument. Isolated courses must be seeded and counted, or the processed total falls short and you falsely report a cycle. Both hinge on getting the initial queue and the final count exactly right.

4   Try It Yourself First

Aim for Kahn's algorithm — it gives the cycle check for free. Open hints one at a time.

Hint 1 — reframe the question

"Can I finish all courses?" is the same as "does the prerequisite graph have no cycle?" Each pair [a,b] is a directed edge (b before a). A dependency loop = impossible. So this is cycle detection on a directed graph.

Hint 2 — the pattern

Topological sort. Track each course's in-degree (number of remaining prerequisites). Repeatedly take a course with in-degree 0 (nothing left blocking it), "complete" it, and decrement its dependents' in-degrees. That's Kahn's algorithm.

Hint 3 — the cycle test

Count how many courses you manage to complete. If it equals numCourses, the graph was acyclic → return true. If some courses never reach in-degree 0 (they're stuck in a cycle), the count falls short → return false. Build the edge as b → a and increment a's in-degree. Use a head-index queue, not removeFirst().

Step 2 — The plan, before any code
The plan, spoken

"This is a topological sort — Variant 8.1 of the Graphs / Topological Sort block. The signal is 'prerequisites' and 'finish all courses,' which I'll reframe as: can I order the courses so every prerequisite comes first? That's possible exactly when the prerequisite graph is acyclic, so the problem is cycle detection, and Kahn's algorithm gives me that for free."

"Let me pin the conditions before the structures. I model each pair [a,b] as a directed edge b → a — b is the prerequisite — and that increments a's in-degree, its count of unmet prerequisites. The validity condition for placing a course: its in-degree is 0, meaning nothing is left blocking it. The invariant: the queue holds exactly the currently-takeable courses; when I complete one and decrement a dependent to 0, that dependent becomes takeable and joins the queue. The stop/answer condition: I count how many courses I place — if that equals numCourses, every course was placeable, so the graph is acyclic and I return true; if it falls short, the unplaced courses are stuck in a cycle, so I return false."

"On structures: an adjacency list [[Int]] mapping each course to the courses it unblocks; an in-degree array [Int]; a queue of in-degree-0 courses with a head index so dequeues are O(1); and a processed counter — which I can read straight off the head index."

"The walk: build the adjacency list and in-degrees from the pairs. Seed the queue with every in-degree-0 course. Peel: dequeue a course, decrement each neighbor's in-degree, enqueue any that hit 0. At the end, compare the processed count to numCourses. Time and space O(V+E)."

6   Why This Approach — and not the others

CandidateCostVerdict
Try every course ordering, check validityO(V!)Hopeless. Factorial permutations — only works for a handful of courses. The structure (a DAG) makes brute force unnecessary.
DFS with three-color cycle detectionO(V+E)Correct and equally optimal — White/Gray/Black marking, a Gray neighbor is a back-edge → cycle. Recursive; needs explicit on-path tracking and risks stack depth on a long chain. A great alternate (§9b).
Union-Find~O(E·α)Wrong tool. Union-Find detects cycles in undirected graphs; prerequisites are directed, where it can't distinguish a back-edge from a cross-edge. Doesn't apply here.
Kahn's algorithm (BFS topo sort)O(V+E)Chosen. Iterative, no recursion-depth risk, and the processed-count comparison is the cycle test — cleanest correct solution for a yes/no question.
The deciding signal: "prerequisites / finish all" = ordering under dependencies = topological sort. Between the two topo algorithms, Kahn's wins here because it's iterative (no stack-depth worry on a 2000-long chain) and its cycle detection is a free byproduct of the count — no extra bookkeeping. DFS post-order is the equally-valid recursive alternative.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
adjacency: [[Int]]For each course, the courses it unblocks (its out-neighbors)Sparse graph → list, not a 2000×2000 matrix (wasteful and makes the peel O(V²)). Lets the decrement step touch only real edges.
inDegree: [Int]Each course's count of remaining prerequisitesThe peel's driver — a course is takeable exactly when this hits 0. An array indexed by course id is O(1) to read and decrement.
queue: [Int] + headThe currently-takeable (in-degree-0) coursesFIFO peel order; head index makes dequeue O(1). removeFirst() would make the peel O(V²).
processed counter (= head)How many courses have been placedThe cycle test: compare to numCourses. Reading it off the head index avoids a separate variable.
Why in-degree, not a visited set. Plain graph traversal uses a visited set, but topo sort needs to know when a course becomes takeable — which is only after all its prerequisites are done, not after any one neighbor is seen. The in-degree counter captures exactly that "how many prerequisites are still pending" condition; a boolean visited flag can't express "almost ready, one prerequisite to go."

8   Pseudocode Dry Run

PseudocodeKahn's — peel in-degree-0 courses, count them
build adjacency and inDegree from pairs   [a,b] → edge b→a, inDegree[a]++
queue ← all courses with inDegree == 0
processed ← 0

WHILE queue not empty:
    course ← dequeue(queue)
    processed ← processed + 1
    FOR next IN adjacency[course]:
        inDegree[next] ← inDegree[next] - 1
        IF inDegree[next] == 0: enqueue(queue, next)

RETURN processed == numCourses           all placed → acyclic

9   Swift Solution — iterative

Kahn's algorithm. The processed count (read off the head index) is the cycle test. The recursive DFS three-color form follows as a contrast in §9b.

A.1 Kahn's algorithm — in-degree peel
O(V+E)O(V+E)
Plan
Build an adjacency list and in-degree array (edge b → a per pair [a, b]). Seed a queue with every in-degree-0 course. Peel: dequeue a course, decrement each neighbor's in-degree, enqueue any that hit 0. The number dequeued equals the number of courses that could be ordered — if it's all of them, the graph is acyclic, so return true. 
func canFinish(_ numCourses: Int, _ prerequisites: [[Int]]) -> Bool {
    // 1. Build adjacency (prereq → course) and in-degree (unmet prereqs per course).
    var adjacency = [[Int]](repeating: [], count: numCourses)
    var inDegree  = [Int](repeating: 0, count: numCourses)
    for pair in prerequisites {
        let course = pair[0], prereq = pair[1]   // take prereq before course
        adjacency[prereq].append(course)              // edge prereq → course
        inDegree[course] += 1
    }

    // 2. Seed the queue with every course that has no prerequisites.
    var queue: [Int] = []
    for course in 0..<numCourses where inDegree[course] == 0 {
        queue.append(course)
    }

    // 3. Peel in-degree-0 courses; head index doubles as the processed count.
    var head = 0
    while head < queue.count {
        let course = queue[head]; head += 1        // O(1) dequeue
        for next in adjacency[course] {
            inDegree[next] -= 1                       // one prerequisite of next satisfied
            if inDegree[next] == 0 { queue.append(next) }   // next is now takeable
        }
    }

    // 4. All courses placed ⇔ acyclic ⇔ you can finish them all.
    return head == numCourses
}
Note: head serves double duty — it's the dequeue cursor and the count of processed courses, since every dequeued course advances it exactly once. The head-index queue keeps each dequeue O(1); removeFirst() would make the peel O(V²). Edge direction is the one thing to get right: [course, prereq] → edge prereq → course, incrementing the course's in-degree.

9b   DFS three-color form — for contrast

The recursive alternative: a cycle exists iff DFS reaches a vertex still on the current path (Gray). 0 = unvisited, 1 = on current path, 2 = fully done.

func canFinish(_ numCourses: Int, _ prerequisites: [[Int]]) -> Bool {
    var adjacency = [[Int]](repeating: [], count: numCourses)
    for pair in prerequisites { adjacency[pair[1]].append(pair[0]) }
    var color = [Int](repeating: 0, count: numCourses)   // 0=white 1=gray 2=black

    func hasCycle(_ node: Int) -> Bool {
        if color[node] == 1 { return true }            // back-edge to current path → cycle
        if color[node] == 2 { return false }           // already proven safe
        color[node] = 1                              // mark on current path
        for next in adjacency[node] where hasCycle(next) { return true }
        color[node] = 2                              // done; off the path
        return false
    }

    for course in 0..<numCourses where hasCycle(course) { return false }
    return true
}

12   Complexity

Build adjacency + in-degreesO(V+E)O(V+E)One pass over E pairs; the adjacency list and in-degree array are O(V+E) and O(V).
Peel (Kahn's loop)O(V+E)O(V)Each course dequeued once (V); each edge triggers one decrement (E). Queue holds up to V courses.

Why the cycle check is free. The processed-count comparison costs nothing beyond the peel you'd do anyway — there's no separate cycle-finding pass. That's why this yes/no problem (LC207) and the order-producing one (LC210) have identical O(V+E) complexity: detecting feasibility and producing the order are the same traversal.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
It's already O(V+E), which is optimal — you must read every edge. The main robustness note is the queue: I used a head index so dequeues stay O(1); a naive removeFirst() would silently make it O(V²). If memory were tight on a huge sparse graph I'd confirm the adjacency list (not a matrix) is used. Otherwise I'd ship it as is.
Follow-up: "Now return a valid order, not just whether one exists."
That's LC 210 — Course Schedule II: record each course into an order array as I dequeue it, instead of only counting. At the end, if order.count == numCourses return order; otherwise a cycle truncated it, so return []. The dequeue sequence is a valid topological order by construction — same algorithm, one line added.
Follow-up: "What if the prerequisites came as sorted words in an unknown alphabet?"
That's LC 269 — Alien Dictionary: I'd first derive the edges by comparing each adjacent word pair (the first differing character gives one precedence edge), guard the prefix-invalid case, then run this exact Kahn's. The topo sort is unchanged; the new skill is building the graph from the data.
Follow-up: "Could you solve it recursively instead?"
Yes — DFS with three-color marking (§9b): a cycle exists iff DFS reaches a vertex still on the current path (Gray). It's also O(V+E). I lead with Kahn's because it's iterative — no stack-depth risk on a 2000-long prerequisite chain — and its cycle detection is a free count rather than explicit on-path bookkeeping. Both are interview-acceptable.
