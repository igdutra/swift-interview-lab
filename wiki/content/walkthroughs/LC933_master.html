Implement a RecentCounter class with a single method ping(t: Int) → Int. Each call records a new request at time t and returns the total number of requests that have occurred in the closed interval [t − 3000, t]. The problem guarantees that calls arrive in strictly increasing timestamp order.

Calls in orderReturn valueWhy
ping(1)1Window [−2999, 1] — only request 1 is in range
ping(100)2Window [−2900, 100] — requests 1 and 100 both fit
ping(3001)3Window [1, 3001] — requests 1, 100, 3001 all fit
ping(3002)3Window [2, 3002] — request 1 is now stale; requests 100, 3001, 3002 remain

The twist: the window boundary moves with every call, so a request that was valid last time may be stale this time. You need a structure that can efficiently add new items and remove old ones — and the strictly-increasing constraint tells you exactly where the oldest item always is: the front.

2   Reading the Problem — Recognition Signals

How to recognize this is a queue problem
Three signals fire at once, and together they route you to a sliding-window queue before you write a single variable:

Signal 1 — "how many requests in the last T milliseconds?" Any time a problem asks for a count (or sum, or presence) within a rolling time window, that window has two ends: a back where new items arrive and a front where old items expire. Two ends with different operations = queue.

Signal 2 — "strictly increasing timestamps." This is the critical simplifier. It means the oldest item is always at the front — you never need to search for stale items. The front of the queue is the only place a stale item can be. The while-loop that evicts from the front runs in O(1) amortized because each item is evicted at most once across all calls.

Signal 3 — stateful object with repeated queries. You are implementing a class (RecentCounter), not a function. The state persists between calls. That state is the window itself — a queue that survives across pings. If this were a one-shot function over a static array, two pointers or a prefix sum might apply. The streaming, call-by-call nature forces a data structure that remembers.

Why not a stack? A stack removes from the same end you add to (LIFO). Here you add new requests to the back and evict old ones from the front — those are opposite ends. LIFO would force you to remove the most recent request first, which is exactly wrong. FIFO (queue) removes the oldest first, which is exactly what "stale" means.
Why not a sliding-window array (two pointers)? The standard two-pointer sliding window runs over a static array you can index freely. Here the input is a stream — requests arrive one at a time, and you must answer after each one without seeing future values. A queue is the natural fit for streams because it accumulates items as they arrive and discards them as they expire, with no random access needed.
Why not a sorted set or binary search? Sorted structures give O(log n) insertion and O(log n) eviction. The queue gives O(1) amortized for both — a strict improvement. Binary search would only help if timestamps could arrive out of order, but the problem guarantees they don't.

4   Try It Yourself First

Hint 1 — constraint

Timestamps are strictly increasing. Where is the oldest request always located? Does that eliminate the need to search for stale items?

Hint 2 — pattern

You need to add new items at one end and remove old items from the other end. Which linear structure does that describe — a stack or a queue? What is the eviction condition?

Hint 3 — structure

Store all in-window timestamps in a queue. On each ping(t): (1) append t to the back; (2) while the front is less than t − 3000, remove it; (3) return queue.count. The window boundary is [t − 3000, t] inclusive, so use strict less-than for eviction.

5   Interview Simulation

Interview Simulation — Say This Out Loud

5.1   Clarifying Questions

Q: Are timestamps guaranteed to be strictly increasing?
A: Yes. → Then the oldest request is always at the front of whatever structure I use — I never need to search for stale items. Front-eviction is O(1) amortized.
Q: Is the window inclusive on both ends — does a request at exactly t − 3000 count?
A: Yes, [t − 3000, t] is closed. → My eviction condition must be strict less-than: remove the front only when it is strictly less than t − 3000. Equal means it's still valid.
Q: What should ping return — just the count, not the actual requests?
A: Just the integer count. → I only need to store the timestamps themselves, not any payload. The queue length after eviction is the answer.

5.2   The Plan, Spoken

"This is a sliding-window queue problem. The three signals that tell me that: first, 'how many requests in the last 3000ms' — that's a rolling time window, which means I add new items at one end and evict old items from the other. Two different ends, two different operations — that's a queue, not a stack. Second, 'strictly increasing timestamps' — the oldest item is always at the front, so I never need to search for stale items; front-eviction is the only eviction path. Third, it's a stateful class with repeated calls — the window has to survive between pings, so it's an instance variable. The sequence: on each ping(t), I first enqueue t at the back. Then I evict from the front while the front timestamp is strictly less than t minus 3000. The window is inclusive, so I must use strict less-than — if the front equals t minus 3000, it still counts. After eviction, the queue holds exactly the valid requests, so I return the queue's count. The state I need: a single array used as a queue — a window of Int timestamps. The invariant: after every ping returns, the queue contains exactly the timestamps in [t − 3000, t] and nothing else. The condition: evict while window.first! < t − 3000. Record inside the condition? No — I just return window.count after the loop exits. Let me walk the example: pings arrive at 1, 100, 3001, 3002. ping(1): append 1 → window=[1]. Evict while front < 1−3000 = −2999: 1 is not < −2999 → no eviction. Return 1. ✓ ping(100): append 100 → window=[1,100]. Evict while front < −2900: 1 is not < −2900 → no eviction. Return 2. ✓ ping(3001): append 3001 → window=[1,100,3001]. Evict while front < 1: 1 is not < 1 → no eviction. Return 3. ✓ ping(3002): append 3002 → window=[1,100,3001,3002]. Evict while front < 2: 1 < 2 → remove 1 → window=[100,3001,3002]. Front 100 is not < 2 → stop. Return 3. ✓ Time is O(1) amortized per ping — each timestamp is appended once and removed at most once across all calls. 6   Why This Approach

ApproachTime per pingSpaceVerdict
Store all timestamps; filter on every pingO(n) — scan all previous pingsO(n)Correct but wasteful — rescanning pings that were already stale last call
Binary search on sorted listO(log n) — find left boundaryO(n)Better, but requires sorted access and still O(n) deletion cost
Sliding-window queue (front-eviction)O(1) amortizedO(w)Optimal. Monotonically increasing timestamps make front = oldest, always.
The strictly-increasing guarantee is the whole trick. If timestamps could arrive out of order, the oldest item might be anywhere in the structure — you'd need a min-heap or sorted set (O(log n)). The guarantee that timestamps increase means the queue is always sorted for free: the front is always the oldest. Front-eviction requires no search at all.
Why the amortized argument holds. Across all n total pings, each timestamp is appended exactly once and removed at most once. So even though the while-loop can run many times on a single ping, it cannot run more than n times total. Total work across all pings is O(n), so amortized cost per ping is O(1).

Pseudocode
class RecentCounter:
    window ← empty queue          // holds timestamps of in-window pings

    function ping(t):
        window.enqueue(t)          // 1. record the new request

9   Swift Solution

A.1 Sliding-window queue — removeFirst() form
O(1) amortizedO(w)
PLAN: Instance stores a queue of in-window timestamps. Each ping appends the new timestamp, evicts from the front while stale, returns the count.
Swift
private class RecentCounter {
    private var window: [Int] = []

    func ping(_ timestamp: Int) -> Int {
        window.append(timestamp)

        while let oldest = window.first, oldest < timestamp - 3000 {
            window.removeFirst()             // O(n) per call — acceptable for n ≤ 10⁴
        }

        return window.count
    }
}
A.2 Head-index optimization — O(1) amortized dequeue
O(1) amortizedO(n)
PLAN: Replace removeFirst() with a head index. The array grows but never shrinks — elements before head are logically evicted. Eliminates O(n) shifting at the cost of holding evicted timestamps in memory.
Swift
private class RecentCounterFast {
    private var window: [Int] = []
    private var head = 0               // index of the current front (logical dequeue)

    func ping(_ timestamp: Int) -> Int {
        window.append(timestamp)

        while window[head] < timestamp - 3000 {
            head += 1                   // O(1) logical eviction — no shifting
        }

        return window.count - head    // elements from head to end are in-window
    }
}

13   Follow-Up Bank

Follow-up: Before you push this to production — can you see any improvements you'd make?

The removeFirst() call shifts every remaining element in the array — O(current window size) per eviction step. In production with high-throughput pings, I'd swap to the head-index trick (solution A.2): advance a head index instead of shifting, and return window.count - head. Each ping is then a true O(1) operation. The tradeoff is that the array now holds all timestamps ever seen (O(n) space) rather than just the in-window ones (O(w)), but for a rate-limiter that's usually acceptable.

Follow-up: What if pings could arrive out of order?

The strictly-increasing guarantee is what makes front-eviction O(1) amortized. Without it, the oldest item is no longer guaranteed to be at the front — stale items could be anywhere. You'd need a min-heap to always find the oldest in O(log n), or a sorted structure. The queue degrades from O(1) amortized to O(log n) per ping in that case.

Follow-up: What if the window size isn't fixed at 3000ms but is passed per query?

The same queue structure works — just replace timestamp - 3000 with timestamp - windowSize in the eviction condition. If different callers can use different window sizes simultaneously, you'd need one queue per window size. The algorithm is unchanged; only the threshold becomes a parameter.

Follow-up: How would you implement this in a thread-safe way?

Wrap the queue in an actor (Swift concurrency) or protect the array and head index with a lock. The queue's single-writer / single-reader pattern maps cleanly to an actor: actor RecentCounter with func ping(_ t: Int) async → Int. The lock-based approach works too but requires careful unlock-on-return handling. In a high-throughput system, a lock-free ring buffer (fixed-size circular buffer) is the production choice — O(1) at both ends with no allocation after init.

14   Theory Decoder

This problem instantiates one concept from the parent queue theory: the sliding-window queue — Pattern 2 of the queue theory page.

Theory conceptHow it appears in LC 933
FIFO eviction (oldest first)The front of the queue is always the oldest timestamp. When it falls outside [t−3000, t], it is the first to be removed — and the only candidate. No other position needs to be checked.
Enqueue at back, dequeue at frontwindow.append(timestamp) is enqueue; window.removeFirst() is dequeue. These are opposite ends — the defining property of a queue versus a stack.
Amortized O(1) per itemAcross all pings, each timestamp enters the queue once (append) and leaves at most once (removeFirst or head advance). Total work is O(n); amortized per call is O(1). The while-loop running multiple evictions on one ping is paid for by earlier pings that did no eviction at all.
Strictly-increasing constraint → natural sorted orderBecause pings arrive in order, the queue is always sorted oldest-to-newest from front to back. This is the property that makes front-eviction sufficient — it is the entire reason O(1) amortized is achievable here.
Window size = queue.count after evictionThe queue maintains the invariant that it contains exactly the in-window timestamps. No separate counter needed — the structure itself is the count.
