Theory
intervals_overview_master.html

You are given a list of non-overlapping intervals already sorted by start, and a new interval to insert. Return the merged list after inserting the new interval — merging any overlaps it creates.

InputOutputWhy
intervals=[[1,3],[6,9]] newInterval=[2,5][[1,5],[6,9]][1,3] and [2,5] overlap → merge to [1,5]; [6,9] is disjoint after
intervals=[[1,2],[3,5],[6,7],[8,10],[12,16]] newInterval=[4,8][[1,2],[3,10],[12,16]][4,8] overlaps [3,5],[6,7],[8,10] → merge all to [3,10]
intervals=[] newInterval=[5,7][[5,7]]Empty list — new interval is the only output

The twist: the existing list is already sorted and non-overlapping — so you get O(n) instead of the O(n log n) of a full re-sort. The insight is that relative to the new interval, the existing list splits into exactly three non-overlapping regions. The most common mistake is forgetting Region 3 — the tail after the merge — which silently truncates the output.

2   Reading the Problem — insights from the constraints

"Already sorted and non-overlapping": the most important constraint. You do not re-sort. Instead the existing list splits into three clean regions relative to newInterval: intervals entirely before it, intervals that overlap it, and intervals entirely after it. These regions are contiguous in the sorted list — you never need to backtrack.
"Before" region condition — interval.end < newInterval.start: an existing interval is entirely before the new one if it ends before the new one starts. These intervals need no modification — append them as-is.
"Overlapping" region condition — interval.start ≤ newInterval.end: an existing interval overlaps the new one if it starts at or before the new one's end. Expand newInterval by taking min(starts) and max(ends). This accumulates all overlapping intervals into one growing merged interval.
"After" region — everything remaining: once the overlap region is exhausted, all remaining intervals start strictly after newInterval.end. Append them unchanged. Forgetting this tail is the single most common bug — the output gets silently truncated.
O(n) not O(n log n): because the list is pre-sorted, the sort step that costs O(n log n) in LC 56 is free here. This is Variant 8.3's defining property and worth stating aloud in an interview.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

The input is already sorted and non-overlapping. No re-sort needed — this gives you O(n). Think about how the existing intervals relate to the new one: can you categorize each existing interval into one of three buckets without backtracking?

Hint 2 — the three regions

Every existing interval falls into exactly one of: (1) entirely before the new interval, (2) overlapping the new interval, (3) entirely after the new interval. The conditions are contiguous in the sorted list — once you exit a region, you never go back.

Hint 3 — the structure

Three sequential while-loops, one per region. Region 1: append as-is while interval.end < newInterval.start. Region 2: expand newInterval while interval.start ≤ newInterval.end, then append newInterval. Region 3: append remaining as-is. Don't forget the tail.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Is the existing list guaranteed to be sorted and non-overlapping?"
A: Yes. → I get O(n) instead of O(n log n). I can do a single linear pass — no sort step needed.
Q: "Should the new interval be merged with touching adjacent intervals — like [1,3] and [3,5]?"
A: Yes, touching counts as overlap. → The Region 1 boundary is strict: an existing interval is "before" only if its end is strictly less than the new interval's start. Equal means adjacent — that belongs in the overlap region.
Q: "Can the existing list be empty?"
A: Yes. → All three loops skip; I just return [newInterval]. I'll handle this naturally — no special guard needed.
Q: "Can the new interval be entirely before or after all existing intervals?"
A: Yes. → In the before case, Region 1 never runs and Region 3 appends everything. In the after case, Region 1 appends everything and Regions 2 and 3 are empty. Both fall out naturally from the three-region structure.
Step 2 — The plan, before any code
The plan, spoken

"This is a three-region split problem — Variant 8.3. The key observation: because the existing list is pre-sorted and non-overlapping, every existing interval falls into exactly one of three regions relative to the new interval, and those regions are contiguous in the list. I never need to backtrack."

"The three conditions. Region 1 — entirely before: interval.end < newInterval.start. Strict less-than — touching endpoints belong in Region 2. I append these intervals unchanged. Region 2 — overlapping: interval.start ≤ newInterval.end. I expand newInterval by taking min(newInterval.start, interval.start) and max(newInterval.end, interval.end). When the Region 2 loop exits, newInterval is fully merged — I append it once. Region 3 — entirely after: everything remaining. I append these unchanged."

"The invariant: after processing Region 2, newInterval represents the union of itself and all overlapping existing intervals. It is ready to append. Region 3 starts exactly where the index is when Region 2 ends."

"The most common bug: forgetting Region 3. After appending newInterval, I must also append all remaining intervals. Skipping that tail silently truncates the output."

"Data structures: a single index variable index, the mutable newInterval, and a result array. No sorting, no maps. Time: O(n) — each interval is visited exactly once. Space: O(n) for the output."

6   Why This Approach — and not the others

CandidateCostVerdict
Append + re-sort + merge (LC 56)O(n log n)Correct, but wastes the pre-sorted guarantee. You're paying O(n log n) for a sort you didn't need. Never re-sort a pre-sorted input.
Binary search for insertion pointO(log n) find + O(n) insertFinds the position in O(log n) but still needs a linear sweep to merge all overlapping intervals. Net O(n) — same as the three-region approach, more complex to implement.
Three-region linear splitO(n)Chosen. Single pass, three sequential loops, no extra data structures. Exploits the pre-sorted guarantee for the only O(n) algorithm in the interval family.
The deciding signal: "insert into an already-sorted non-overlapping list." The pre-sort is a gift — use it. The three-region split is the canonical O(n) technique. If the input were unsorted you'd fall back to LC 56's O(n log n) sort-then-sweep.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
index: IntCurrent position in intervalsSingle pointer that moves monotonically forward through all three regions. No backtracking — the sorted order guarantees region membership is decided once per interval.
newInterval: [Int]The growing merged interval — starts as the input new interval, expands as Region 2 absorbs overlapping existing intervalsMutable local copy. In Swift, function parameters are immutable — declare a var copy to enable in-place expansion.
result: [[Int]]Accumulated output — Region 1 intervals, then the merged newInterval, then Region 3 intervalsPlain array; appended to sequentially. At most n + 1 entries.
Why three loops instead of one loop with conditionals. Three sequential while-loops make the three-region structure explicit and verifiable — each loop's termination condition is exactly its region's boundary. One loop with three if/else branches is equivalent but harder to audit: it's easy to miss the implicit transition from Region 2 to Region 3 and forget to append newInterval before continuing.

9   Swift Solution — iterative

A.1 Three-region linear split
O(n)O(n)
Plan
Three sequential loops: append intervals entirely before newInterval, expand newInterval through all overlapping intervals and append it once, then append all remaining intervals. The pre-sorted guarantee makes this O(n) — no sort needed. 
func insert(_ intervals: [[Int]], _ newInterval: [Int]) -> [[Int]] {
    var result: [[Int]] = []
    var index = 0
    var mergedInterval = newInterval

    // Region 1 — entirely before: append unchanged.
    while index < intervals.count && intervals[index][1] < mergedInterval[0] {
        result.append(intervals[index])
        index += 1
    }

    // Region 2 — overlapping: expand mergedInterval, then append it once.
    while index < intervals.count && intervals[index][0] <= mergedInterval[1] {
        mergedInterval[0] = min(mergedInterval[0], intervals[index][0])
        mergedInterval[1] = max(mergedInterval[1], intervals[index][1])
        index += 1
    }
    result.append(mergedInterval)

    // Region 3 — entirely after: append unchanged.
    while index < intervals.count {
        result.append(intervals[index])
        index += 1
    }

    return result
}

Output—O(n)Result holds at most n + 1 intervals (n existing plus the new one if it doesn't overlap anything).
TotalO(n)O(n)No sort. Each existing interval is visited by exactly one of the three loops. This is the only O(n) variant in the interval family — the pre-sorted guarantee eliminates the O(n log n) sort cost.
Why O(n) and not O(n log n). LC 56 costs O(n log n) because the input is unsorted and sorting is unavoidable. Here, the problem guarantees sorted order — the sort is already done. The three-region sweep pays only O(n) for the linear pass. Always state this distinction explicitly in an interview: "because the input is pre-sorted, I can solve this in O(n) instead of O(n log n)."

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The algorithm is already optimal at O(n) — you must visit every interval at least once. A minor improvement: result.reserveCapacity(intervals.count + 1) to avoid repeated reallocations when most intervals are disjoint. For a streaming or online use case where intervals arrive one at a time, you'd maintain the sorted invariant with a binary-search insertion, making each insert O(log n) find + O(n) shift — the three-region logic still applies, just on a live array.
Follow-up: "What if the existing list is not sorted?"
You lose the O(n) guarantee. Sort first (O(n log n)), then apply the same three-region logic — but now it degenerates to the LC 56 sort-then-sweep approach. The insert step after sorting is still O(n) for the sweep, but you've paid O(n log n) upfront. See LC56_master.html for the full walkthrough of that variant.
Follow-up: "What if you need to insert multiple new intervals?"
Two options. (1) Insert one at a time: call this function k times — O(k × n) total. (2) Merge the new intervals together first using LC 56's sort-then-sweep (O(k log k)), then run a modified two-pointer walk against the existing list — O(n + k). Option 2 is better when k is large. Option 1 is fine when k is small (say, k ≤ 5).
Follow-up: "How does this relate to LC 56 — Merge Intervals?"
LC 56 and LC 57 use the same core insight — the overlap condition incoming.start ≤ lastEnd and the merge rule max(ends) — but differ in what they start from. LC 56 starts from an unsorted input and pays O(n log n) to sort. LC 57 starts from a pre-sorted list and exploits it for O(n). The three-region split in LC 57 is structurally identical to the "only compare against result.last" invariant in LC 56 — both are enabled by sorted order. See LC56_master.html.
