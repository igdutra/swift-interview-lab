Theory
sliding_window_master.html

You are given a string s. Find the length of the longest contiguous slice of it that has no character appearing twice.

InputOutputWhy
s = "abcabcbb"3"abc" — the 4th char repeats 'a', so the window can't grow past length 3
s = "bbbbb"1Every char is the same; the longest unique slice is any single "b"

The twist that defines this problem: the window is variable and you are chasing the largest valid one, not the smallest. That single word — "longest" — flips the entire control structure relative to the minimum-window family: you only shrink when the window is broken, and you record the best after the shrink, never inside it.

2   Reading the Problem — insights from the statement

"Longest substring": the word "longest" is the whole signal. The window size is variable and you are searching for the maximum — so the natural move is to grow the window greedily and only pull the left edge in when a repeat forces it. Record the size after each repair, when the window is guaranteed valid again. This is the exact inverse of LC 76's "minimum".
"Substring" — contiguous: a substring is a contiguous run, not a subsequence. That contiguity is what makes a two-pointer window the right tool: the answer is always a single [left, right] range, never scattered indices.
"Without repeating characters": validity is a no-duplicate test. A character either is or is not already in the window — so a Set of present characters, or a count map where every count is ≤ 1, captures validity. There are no multiplicities to track as in LC 76; "valid" simply means "no count reached 2".
"English letters, digits, symbols and spaces": the alphabet is the full printable ASCII range, not just lowercase letters. A Character-keyed dictionary handles any alphabet without assuming a fixed size — though if the interviewer pins the alphabet to ASCII you can swap in a 128-slot array (see §13).

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

n up to 5·10⁴ makes enumerating all O(n²) substrings and checking each for duplicates far too slow. What pattern delivers "best contiguous range" in a single O(n) pass?

Hint 2 — the pattern

Sliding window — the "longest" variant. Ask the inversion question: does my while loop move the window toward validity or away from it? Here it repairs a broken window, so it moves toward validity — which means the answer is recorded after the loop, not inside it.

Hint 3 — the structure

Validity is "no repeats". Track the characters currently in the window. The classic way: a count map; when adding a character makes its count 2, shrink from the left until it is 1 again. The slight optimization: store each character's last-seen index, and when you hit a repeat jump left straight past it instead of stepping one by one.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Substring meaning a contiguous slice, not a subsequence — correct?"
A: Contiguous. → a single sliding window over one [left, right] range is the right tool, not subsequence DP.
Q: "Do I return the length, or the substring itself?"
A: Just the length. → I only need to track a max integer, not a best range.
Q: "What's the alphabet — lowercase letters only, or anything?"
A: Letters, digits, symbols, and spaces. → a Character-keyed dictionary; no normalization. If you'd rather I assume ASCII I can use a fixed 128-int array.
Q: "Is the comparison case-sensitive, and can the string be empty?"
A: Case-sensitive; empty is allowed. → 'a' and 'A' are distinct keys; empty input returns 0 because the loop never runs.
Step 2 — The plan, before any code
The plan, spoken

"This is a sliding window problem — specifically the longest-window variant. The signal is the word 'longest' over a contiguous substring: I grow a window rightward greedily, and the only time I pull the left edge in is when a character repeats and breaks the no-duplicate rule. Because the loop repairs the window — moving it back toward validity — I record the best length after the repair, not inside it. That's the inversion that separates this from the minimum-window problems, where you shrink while valid and record inside."

"Before I name any data structure, let me pin the conditions, because they decide what I track and how the loop runs. The validity condition is: every character in the current window appears exactly once — no count reaches 2. The invalid condition is the mirror: the window goes invalid the instant the character I just added is already inside it, pushing its count to 2. The shrink loop runs while invalid — evicting from the left until that duplicate is gone — and the record happens after that loop, when the window is guaranteed valid. The invariant: at the bottom of each outer iteration, the window [left, right] is always valid, so its length is always a legitimate candidate for the max."

"Those conditions tell me what to store. The classic version: a frequency map from character to its count inside the window. When I add the right character and its count becomes 2, I shrink from the left — decrementing each evicted character's count and advancing left — until the offending character is back to 1. Then I record. The slight optimization removes the inner stepping entirely: instead of counts, I store each character's last-seen index. When the right character was seen before and inside the current window, I jump left straight to one past that index. One subtlety I'll guard: I only move left forward — left = max(left, lastSeen[c] + 1) — so a stale index from outside the window, like the second 'a' in 'abba', can never drag left backwards."

"The walk: for each right character, either repair the window (shrink while the count hits 2, or jump left past the prior occurrence) then record right − left + 1 against the running max. Each character enters once and the left pointer only ever moves forward, so it's a single amortized pass."

"So I expect O(n) time and O(k) space, where k is the number of distinct characters — bounded by the alphabet size. I'll code the classic version first, then mention the index-jump optimization, and trace 'abcabcbb' to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: every substring, check uniquenessO(n²) substrings × O(n) check = O(n³)~10¹⁴ at n = 5·10⁴ — dead on arrival, but name it as the baseline
Every start, extend with a Set until repeatO(n²)Drops the inner uniqueness scan but still re-scans from every start; borderline and wasteful
Sliding window + count map (classic)O(n)Chosen baseline. Left and right each move forward at most n times; shrink one char at a time on a repeat
Sliding window + last-seen index (jump)O(n)Chosen optimization. Same O(n) class but skips the inner stepping — left jumps straight past the duplicate in O(1)
The deciding signal: "longest contiguous window under a constraint that is incrementally checkable" — adding one character can only break validity in one way (a repeat), and that break is repairable by moving the left edge forward. That is exactly the shape variant 8.2 was built for.

7   Data Structures — choice and justification

Classic version — count map

StructureHoldsWhy this and not weaker
windowCounts: [Character: Int]How many times each character appears inside the windowA plain Set would tell you whether a char is present but not let you decrement cleanly as the left edge passes it; counts make the eviction exact. (A Set works too if you remove on eviction — counts generalize better.)
leftPointer: IntThe window's left edgeAdvanced one char at a time inside the shrink loop; only ever moves forward
maxLength: IntBest valid window length seenThe problem asks for a length, so a single Int — no range needed

Optimized version — last-seen index map

StructureHoldsWhy this and not weaker
lastSeenIndex: [Character: Int]The most recent index at which each character was seenLets the left edge jump straight to lastSeen[c] + 1 instead of stepping; replaces the inner shrink loop with one O(1) assignment
leftPointer: IntThe window's left edgeMoved by max(left, lastSeen[c] + 1) — the max is mandatory
maxLength: IntBest valid window length seenSame as above
The one trap in the jump version: always move left with left = max(left, lastSeenIndex[c] + 1), never a bare assignment. On "abba", when the second 'a' arrives at index 3, its last-seen index is 0 — but left has already advanced to 2 (past the b's). A bare left = 0 + 1 would drag the window backwards, re-admitting the b's and corrupting the count. The max clamps left so it only ever moves forward.
Swift cost note: Array(s) once up front gives O(1) indexing (String isn't randomly indexable). Dictionary reads/writes are O(1) amortized; the whole thing is one linear pass.

8   Pseudocode Dry Run

Pseudocodeclassic — shrink-while-invalid with a count map
windowCounts ← empty; left ← 0; maxLength ← 0

FOR right FROM 0 TO n − 1:
    windowCounts[s[right]] += 1

    WHILE windowCounts[s[right]] > 1:        // window is INVALID — a repeat
        windowCounts[s[left]] −= 1
        left += 1

    maxLength ← max(maxLength, right − left + 1)  // record AFTER repair

RETURN maxLength

And the slight optimization — the same algorithm, but the inner shrink loop collapses into a single jump:

Pseudocodeoptimized — jump left past the last occurrence
lastSeen ← empty; left ← 0; maxLength ← 0

FOR right FROM 0 TO n − 1:
    IF s[right] IN lastSeen AND lastSeen[s[right]] ≥ left:
        left ← lastSeen[s[right]] + 1        // jump, never step

    lastSeen[s[right]] ← right
    maxLength ← max(maxLength, right − left + 1)

RETURN maxLength

9   Swift Solution — iterative

A.1 Classic — shrink-while-invalid with a count map
O(n)O(k)
Plan
One pass of the right pointer. Add the right character to the window counts. While that character's count is 2 — the only way validity breaks — evict from the left one char at a time until it is back to 1. Then record the window length against the running max. Return the max. 
func lengthOfLongestSubstring(_ s: String) -> Int {
    let characters = Array(s)

    var windowCounts: [Character: Int] = [:]
    var leftPointer = 0
    var maxLength = 0

    for (rightPointer, rightCharacter) in characters.enumerated() {
        windowCounts[rightCharacter, default: 0] += 1

        // INVALID while the new character now appears twice — shrink until it doesn't
        while windowCounts[rightCharacter]! > 1 {
            let leftCharacter = characters[leftPointer]
            windowCounts[leftCharacter]! -= 1
            leftPointer += 1
        }

        // VALID here — record after the repair
        maxLength = max(maxLength, rightPointer - leftPointer + 1)
    }

    return maxLength
}
A.2 Optimized — jump left past the last occurrence
O(n)O(k)
Plan
Same window, same invariant — but instead of stepping the left edge one character at a time, store each character's last-seen index and jump left straight to one past it. The max guard keeps left from ever moving backwards when the prior occurrence sits outside the current window. 
func lengthOfLongestSubstring(_ s: String) -> Int {
    let characters = Array(s)

    var lastSeenIndex: [Character: Int] = [:]
    var leftPointer = 0
    var maxLength = 0

    for (rightPointer, rightCharacter) in characters.enumerated() {
        // Seen before AND still inside the window → jump left past it
        if let previous = lastSeenIndex[rightCharacter], previous >= leftPointer {
            leftPointer = previous + 1   // guaranteed forward: previous ≥ left
        }

        lastSeenIndex[rightCharacter] = rightPointer
        maxLength = max(maxLength, rightPointer - leftPointer + 1)
    }

    return maxLength
}
The previous >= leftPointer test is the max guard from §7, expressed as a branch: it only jumps when the prior occurrence is still inside the window, so left never moves backwards. Writing left = max(left, previous + 1) unconditionally is the equivalent one-liner.

12   Complexity

VariantTimeSpaceWhy
Classic (count map)O(n)O(k)Right moves n times; left moves ≤ n times total across the whole run (amortized — each character is evicted at most once). The inner while looks nested but is paid for globally, not per iteration.
Optimized (last-seen jump)O(n)O(k)Same O(n) class — the jump replaces a run of single steps with one O(1) assignment, so it does the same left-movement work in fewer operations. A constant-factor win, not an asymptotic one.
TotalO(n)O(k)k = distinct characters in the window, bounded by the alphabet (≤ 128 for ASCII), so space is effectively O(1) for a fixed alphabet. Reading the input is the floor — this is optimal.
The O(n²) trap to avoid out loud: the original instinct is to test validity by scanning the whole window (windowCounts.values.contains(2) or .max()) every step. That hidden full scan turns the single pass into O(n²). The fix is to test only the one character that could have broken validity — the one just added — which is exactly what windowCounts[rightCharacter]! > 1 does.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "The alphabet is plain ASCII. Can you drop the dictionary?"
Yes — replace the map with an [Int] of size 128 indexed by ASCII value (last-seen index, initialized to −1, or a count). The algorithm is identical; space becomes a fixed 128 ints and the constant factor drops because array indexing beats dictionary hashing. State this as a constant-factor optimization, not an asymptotic one.
Follow-up: "Return the substring itself, not just its length."
Track a best range instead of a best length: whenever right − left + 1 beats the current best, record left ..< right + 1. At the end, String(characters[bestRange]). One extra variable, same O(n) — the same upgrade LC 76 makes by default.
Follow-up: "Now it's the longest substring with at most k distinct characters."
Same variant 8.2 shape, looser validity: the window is invalid when distinctCount > k rather than when any count hits 2. Keep a count map, increment distinctCount when a char's count goes 0→1, decrement it when a count returns to 0 during a shrink, and shrink while distinctCount > k. Record after the loop, exactly as here. (LC 340.)
Follow-up: "How would the structure change if it asked for the shortest window meeting a condition instead?"
That flips it to variant 8.3: the while loop would run while valid (shrinking toward invalidity) and the record would move inside the loop. Saying this — that I'd invert the loop condition and move the record — shows I own the inversion rule, not just this one problem. That's exactly LC 76.
