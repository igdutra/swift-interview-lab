A robot starts at the origin of an infinite plane, facing North. It receives a string of instructions: 'G' means move forward one unit, 'L' means turn left 90°, and 'R' means turn right 90°. The robot will repeat the instruction string forever. Return true if the robot is guaranteed to be bounded within a circle (stays in a finite region), false if it drifts to infinity.

InputOutputWhy
instructions = "GGLLGG"trueAfter one cycle: position (0,2) facing South → four cycles trace a square, returns to origin
instructions = "GG"falseAfter one cycle: position (0,2) still facing North → drifts north indefinitely
instructions = "GL"trueAfter one cycle: position (0,1) facing West → four cycles close a square at origin

The twist is a mathematical insight, not a simulation marathon. You only ever need to simulate one cycle of the instruction string — the bounded test reads two fields of the final pose and requires understanding why those two fields are sufficient.

2   Reading the Problem — insights from the statement

"Repeats the instruction string forever": you are not simulating an infinite repetition. You are proving a property of one cycle. The key observation is that the robot's trajectory after k cycles is fully determined by its net displacement and net rotation after one cycle.
"Bounded within a circle": bounded does not mean "returns to origin" — it means the trajectory stays within some finite radius. A robot that traces a square of radius 1 is bounded even though it visits four distinct points. This is why "facing a different direction" is sufficient even without being at the origin.
Three-instruction alphabet G/L/R: the alphabet is fixed and tiny. No parsing complexity, no multi-char tokens. A switch statement over three cases is the natural expression. The direction table handles the geometry cleanly.
String length up to 100: the constraint is tiny — O(n) simulation of one cycle is trivially fast. There is no optimization pressure here. The only difficulty is the correctness of the bounded test.
Starting facing North: the initial direction is fixed. The direction table only needs to encode which of the four compass headings corresponds to each index, and the robot starts at index 0 (North).

3   Edge Case Inventory

InputExpectedHandled by
"G" — single step northfalseAfter one cycle: position (0,1), direction North (d=0) → not at origin, not rotated → unbounded
"L" — only a turntruePosition stays at (0,0), d=1 (West) → at origin → bounded
"R" — only a right turntruePosition stays at (0,0), d=3 (East) → at origin → bounded
"LLLL" — four left turnstrueAfter four turns: back at origin, d=0 again → at origin → bounded (trivially)
"GG" — drift northfalsePosition (0,2), d=0 (North) → not at origin AND not rotated → unbounded
"GLGLGLGL" — traces a squaretrueAfter one cycle: position (0,0), d=0 → at origin → bounded
GOTCHA: "GL" — not at origin but boundedtruePosition (0,1), d=3 (East, rotated) → not at origin BUT facing non-North → bounded. Forgetting the rotation check fails this case.
The rotation insight is the whole problem. A robot that ends one cycle facing East will, over four cycles, trace a closed path (each cycle is a 90° rotation of the last). A robot facing South closes after two cycles. A robot facing West closes after four. Only a robot that ends facing North and is not at the origin drifts — every "forward" cycle sends it in the same direction. The Boolean test (posX == 0 && posY == 0) || dirIndex != 0 captures this exactly.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

Simulating forever is impossible, but you only need to decide whether the robot eventually closes a loop. Think about what happens to the robot's path after one full cycle of the instructions.

Hint 2 — the pattern

State-machine simulation. Represent the robot's pose as (x, y, direction). Apply each instruction once to transform the pose. The answer depends on two fields of the final pose after one cycle.

Hint 3 — the bounded test

After one cycle, the robot is bounded if: (a) it is back at the origin — it trivially loops, or (b) it is not facing North — because then after 2 or 4 repetitions, the net displacements cancel (rotated vectors sum to zero). The only unbounded case is "moved AND still facing North".

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Is the instruction string guaranteed to be non-empty?"
A: Yes, length ≥ 1. → No empty-string guard needed; the simulation loop always runs at least once.
Q: "Are the only valid characters G, L, and R — no other characters in the string?"
A: Yes, only those three. → A switch with three cases and a default:break is complete. No character parsing or validation needed.
Q: "Does 'bounded in a circle' mean returning to the origin, or just staying within some finite distance?"
A: Staying within a finite distance — not necessarily at the origin. → The bounded test is two conditions: at the origin, OR facing a different direction after one cycle. I need to be careful not to require a return to origin as the only true case.
Q: "Can I assume integer arithmetic — the plane is an infinite integer grid?"
A: Yes, moves are unit steps. → Plain Int x and y variables suffice; no floating-point geometry needed.
Step 2 — The plan, before any code
The plan, spoken

"This is a state-machine simulation — Variant 8.2 of the Text Formatting and Simulation pattern. The signal is 'an agent follows a fixed instruction sequence and we need a property of its final state.' The phrase 'repeats the instructions forever' looks scary, but it's the key insight: I only need to simulate one cycle."

"Let me state the governing condition before I touch any data structure. The validity condition — the test that decides the return value — is: the robot is bounded if and only if, after one cycle, either (a) it is back at the origin, or (b) it is not facing North. Why? If it's back at the origin, it trivially loops. If it's rotated — say, facing East — then after four cycles its net displacement vectors point North, East, South, and West respectively, and they sum to zero. The only unbounded case is: the robot drifted away from the origin AND is still facing North, because then every subsequent cycle adds the same net displacement northward. The Boolean (posX == 0 && posY == 0) || dirIndex != 0 is the entire answer."

"The data structures follow directly: I need (posX, posY) for position and a direction index dirIndex into a four-vector table. The table encodes the (dx, dy) for each of the four compass headings in clockwise order: North (0,1), East (1,0), South (0,−1), West (−1,0). A left turn is (dirIndex + 3) % 4 — not (dirIndex − 1) % 4, which gives negative indices in Swift. A right turn is (dirIndex + 1) % 4. Forward is posX += dirs[dirIndex].0; posY += dirs[dirIndex].1."

"I loop once over the instruction string, apply each instruction to mutate the pose, then read the final pose and return the Boolean. Time is O(n) where n is the instruction length; space is O(1) — four scalar variables and a four-element constant array."

6   Why This Approach — and not the others

CandidateCostVerdict
Simulate many cycles until a repeat or divergenceO(k·n) for some kWorks but overkill — you need to decide how many cycles to run (4 is sufficient, since 4 cycles always close any bounded path). But simulating one cycle with the mathematical test is simpler and has no magic constant.
Track visited positions across cyclesO(k·n) time, O(positions) spaceSpace blows up and still requires knowing when to stop. The mathematical insight makes this unnecessary.
Simulate one cycle, read final poseO(n) / O(1)Chosen. The bounded test is a two-field Boolean on the final pose after one cycle. The proof is short and covers all cases.
The deciding signal: "bounded" in an infinite repetition problem. Whenever a problem says "repeats forever" and asks about convergence, ask whether one-cycle behavior determines the answer. For a constant-turn automaton on an integer grid, it does — the 4-rotation periodicity is the key.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
posX: Int, posY: IntRobot's current position on the integer planeUnit-step moves on an integer grid — plain Int, no Float. Initialized to (0,0) per the problem.
dirIndex: IntDirection index (0=N, 1=E, 2=S, 3=W, clockwise)An index into the direction table avoids a Direction enum and keeps turn arithmetic to one modular expression. Initialized to 0 (North).
dirs: [(Int,Int)]The four unit displacement vectors, in clockwise orderA constant array of four tuples maps an index to a (dx,dy) pair. Clockwise ordering makes right-turn (d+1)%4 and left-turn (d+3)%4. A dictionary or switch over direction names is slower and more verbose.
Why left turn is (dirIndex + 3) % 4. In Swift, (0 − 1) % 4 returns −1 — a negative remainder, not the expected 3. Adding the modulus first (dirIndex + 3) keeps the expression always non-negative. This is the single most common bug in direction-table code: writing (dirIndex − 1) % 4 and getting an out-of-range index.

9   Swift Solution — iterative

A.1 State-machine simulation — one-cycle pose read
O(n)O(1)
Plan
Represent the robot's full state as (x, y, direction index). Apply each instruction once — G moves by the direction vector, L/R adjust the index with modular arithmetic. After one cycle, return true if the robot is at the origin or is no longer facing North. Those are the only two conditions under which repetition stays bounded. 
func isRobotBounded(_ instructions: String) -> Bool {
    var posX = 0, posY = 0, dirIndex = 0
    let dirs = [(0, 1), (1, 0), (0, -1), (-1, 0)]  // N E S W

    for instruction in instructions {
        switch instruction {
        case "G": posX += dirs[dirIndex].0; posY += dirs[dirIndex].1
        case "L": dirIndex = (dirIndex + 3) % 4   // (dirIndex-1)%4 gives -1 in Swift
        case "R": dirIndex = (dirIndex + 1) % 4
        default: break
        }
    }

    // Bounded iff at origin OR rotated off North after one cycle
    return (posX == 0 && posY == 0) || dirIndex != 0
}
A.2 Direction enum with computed properties — readable rotation
O(n)O(1)
Plan
Same one-cycle simulation, but direction is an enum with computed properties for the displacement vector and for turning left/right. The rotation logic lives on the type itself — the main loop never sees a switch over direction. Trades the compact index math of A.1 for self-documenting case names and zero risk of the (d−1)%4 negative-remainder bug. 
func isRobotBounded(_ instructions: String) -> Bool {
    enum Direction {
        case north, south, east, west

        var delta: (x: Int, y: Int) {
            switch self {
            case .north: (0,  1)
            case .south: (0, -1)
            case .east:  (1,  0)
            case .west:  (-1, 0)
            }
        }

        var turnedLeft: Direction {
            switch self {
            case .north: .west
            case .west:  .south
            case .south: .east
            case .east:  .north
            }
        }

        var turnedRight: Direction {
            switch self {
            case .north: .east
            case .east:  .south
            case .south: .west
            case .west:  .north
            }
        }
    }

    var position: (x: Int, y: Int) = (0, 0)
    var direction: Direction = .north

    for instruction in instructions {
        switch instruction {
        case "G":
            position.x += direction.delta.x
            position.y += direction.delta.y
        case "L": direction = direction.turnedLeft
        case "R": direction = direction.turnedRight
        default: break
        }
    }

    // Not facing north → each repeat rotates the net displacement 90°,
    // so 4 cycles always cancel out and the robot returns to origin.
    return (position.x == 0 && position.y == 0) || direction != .north
}
A.1 vs A.2
A.1 compresses direction into an Int index — short, interview-fast, but the (d+3)%4 trick needs a comment. A.2 is longer but every case is named; a reader never has to mentally map an index to a compass heading. Both are O(n) / O(1). In an interview, A.1 is faster to write; in production, A.2 is easier to maintain.

12   Complexity

PhaseTimeSpaceWhy
Instruction scanO(n)O(1)One pass over the n-character string. Each character is processed in O(1) via a three-case switch. No additional storage grows with input size.
TotalO(n)O(1)The input scan is the bottleneck. The constant-size direction table (four tuples) and the four scalar variables x, y, d, and the return Boolean are the only allocations.
Why simulating one cycle is sufficient. Suppose after one cycle the robot's net displacement is vector v and it has rotated by angle θ. If θ = 0° (still facing North), each subsequent cycle adds the same v — the robot drifts along v, unbounded, unless v = 0. If θ = 90°, 180°, or 270°, then after 4, 2, or 4 cycles respectively, the rotated copies of v cancel: v + R(90°)v + R(180°)v + R(270°)v = 0. So the path always closes in at most four cycles. Exactly these two cases — v=0 or θ≠0 — are what the bounded test checks.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The solution is already O(n) time and O(1) space, which is optimal. One minor constant-factor improvement: convert the String to an Array of Characters once at the start — Swift's String iterates by Unicode scalar, which is fine here, but explicit Array(instructions) avoids repeated String indexing overhead in hot loops. For strings of length ≤ 100 as in this problem, the difference is negligible.
Follow-up: "What if the robot can also move diagonally — instruction 'D' moves one step northeast?"
The direction table generalizes to eight compass headings. Left/right turns become ±45° rotations (step of 2 in an eight-element table). The bounded test remains the same: bounded iff at origin or rotated after one cycle — but now "rotated" means any of the seven non-North headings, and the number of cycles to close is at most 8. The code changes to an eight-element direction table and (d + 7) % 8 / (d + 1) % 8 for turns.
Follow-up: "What if the instructions can repeat a variable number of times — you're given k and the robot runs the instructions k times. Is it bounded after exactly k cycles?"
Simulate one cycle to get the net displacement v and net rotation θ. If θ = 0: bounded after k cycles only if v = 0. If θ = 90° or 270°: bounded after k cycles iff k is a multiple of 4. If θ = 180°: bounded iff k is even. You can answer in O(n) time (one cycle) plus O(1) per query, making this trivially efficient for multiple k queries.
Follow-up: "Can you prove — not just assert — that 4 cycles always suffice for a bounded robot?"
After one cycle with rotation θ ≠ 0°, the net displacement vectors over 4 cycles are v, R(θ)v, R(2θ)v, R(3θ)v. Their sum equals v(1 + R(θ) + R(2θ) + R(3θ)). For any rotation R that is a non-trivial element of the cyclic group of order 4 (i.e., R ≠ identity), the geometric series 1 + R + R² + R³ = 0 as a sum of unit complex numbers evenly spaced on the unit circle. Therefore the total displacement after 4 cycles is zero — the robot returns to the origin. QED.
