A topological sort of a directed graph is a linear ordering of its vertices such that for every directed edge u → v, u appears before v. It answers "in what order can I do these tasks so every task comes after the things it depends on?" — course prerequisites, build dependencies, package install order. Crucially, a topological order exists if and only if the graph has no directed cycle (it's a DAG): a cycle means a set of tasks that all depend on each other, with no valid starting point.

The dominant algorithm is Kahn's algorithm: track each vertex's in-degree (how many prerequisites remain), repeatedly take a vertex with in-degree 0 (nothing left blocking it), output it, and decrement its neighbors' in-degrees — which may free them next. If you manage to output every vertex, the graph was acyclic; if some vertices never reach in-degree 0, they're trapped in a cycle. This makes Kahn's a two-for-one: it produces the order and detects cycles in the same pass. This deep dive lives under the Graphs hub; it pairs with the DFS deep dive, whose §8.4 covers the DFS-flavored cycle detector that Kahn's complements.

4   Mental Model — peel the layers with no prerequisites

The one idea
Picture a to-do list where each task lists what must come first. Repeatedly do every task that currently has nothing left blocking it (in-degree 0), then cross those off — which may unblock the next batch. The order you complete them in is a valid topological order. If you ever get stuck with tasks remaining but none unblocked, those tasks form a cycle — a dependency deadlock with no valid order.

Watch Kahn's algorithm peel a small course graph. Edges point prerequisite → course (0→1 means "take 0 before 1"):

DiagramKahn's peeling in-degree-0 vertices
edges: 0→1, 0→2, 1→3, 2→3        (take 0, then 1 and 2, then 3)

       0            in-degrees:  0:0  1:1  2:1  3:2
      / \
     1   2          queue starts with in-degree-0 vertices → [0]
      \ /
       3

step 1: pop 0 → output [0]; decrement 1→0, 2→0 ; queue [1,2]
step 2: pop 1 → output [0,1]; decrement 3→1     ; queue [2]
step 3: pop 2 → output [0,1,2]; decrement 3→0   ; queue [3]
step 4: pop 3 → output [0,1,2,3]                ; queue []

processed 4 of 4 → no cycle. order: 0 1 2 3 (also valid: 0 2 1 3)

Each vertex leaves the queue exactly when its last prerequisite is removed. Processing all 4 means the graph was a DAG; a leftover vertex would signal a cycle.

Now the cycle case — the thing topo sort is built to catch. Add an edge 3→0, making a loop:

Tracea cycle leaves vertices unprocessable
edges: 0→1, 1→2, 2→0   (every vertex depends on another in the loop)

in-degrees: 0:1  1:1  2:1     ← NObody has in-degree 0
queue starts EMPTY → loop never runs → processed = 0

processed (0) ≠ total (3) → cycle detected, no valid order

When every vertex has a prerequisite, none can start — the queue is empty from the outset and the processed count falls short. That short count is the cycle signal.

So the whole algorithm is: seed the queue with in-degree-0 vertices, peel and decrement, and compare the processed count to the total. Equal → DAG, here's the order. Less → cycle, no order.

5   Pseudocode Primer

Kahn's algorithm. The processed-count comparison at the end is both the cycle test and the correctness check.

PseudocodeKahn's — peel in-degree-0 vertices, count them
FUNCTION topoSort(V, edges):
    adj ← adjacency list ; indeg ← in-degree of each vertex
    queue ← every vertex with indeg == 0
    order ← []

    WHILE queue not empty:
        node ← dequeue(queue)
        order.append(node)                  safe to place — no prereqs left
        FOR next IN adj[node]:
            indeg[next] ← indeg[next] - 1   one prereq of next is now satisfied
            IF indeg[next] == 0:
                enqueue(queue, next)        next is now free

    IF order.count == V: RETURN order    all placed → DAG
    ELSE: RETURN "cycle — no valid order"   some vertex never freed
Why the count detects cycles for free: a vertex is appended to order only when its in-degree hits 0, i.e. all its prerequisites were already placed. Vertices inside a cycle each depend on another cycle member, so none of their in-degrees can ever reach 0 — they're never enqueued, never counted. If the final count equals V, every vertex was placeable, which is exactly the definition of a DAG. No separate cycle-detection pass is needed.

6   Kahn vs DFS — two routes to the same order

There are two standard topological sorts; both are O(V+E), and knowing the trade-off is interview gold:

  • Kahn's (BFS-flavored, iterative). In-degree array + a queue of free vertices. Cycle detection is automatic (processed count). The order comes out front-to-back. This is the default — no recursion, no stack-depth risk, and the cycle check is a one-liner. It's what the LC207/LC210 walkthroughs use.
  • DFS post-order (recursive). DFS each vertex; after exploring all its descendants, push it onto a stack. Reverse the stack for the topological order. Cycle detection needs the three-color / on-path marking from the DFS deep dive §8.4 — a Gray neighbor is a back-edge. Elegant, and natural when you already need finish-time information.
Why lead with Kahn's: the standard prefers iterative solutions, and Kahn's is inherently iterative with built-in, count-based cycle detection — no recursion-stack tracking, no overflow on a 10⁵-vertex chain. Reach for DFS post-order when the problem also wants finishing times or you're already doing a DFS pass; otherwise Kahn's is cleaner and harder to get wrong.

7   Swift Foundations

Five Swift facts underlie nearly every topological sort. The edge-direction one is where most bugs start.

Swiftthe five topo-sort building blocks for this block
// 1. Adjacency list + in-degree array, sized to the vertex count.
var adjacency = [[Int]](repeating: [], count: numNodes)
var inDegree  = [Int](repeating: 0, count: numNodes)

// 2. Edge direction is the #1 gotcha. "b must come before a" → edge b → a.
//    LC207 gives [a, b] meaning "take b before a", so add b → a:
for pair in prerequisites {
    let course = pair[0], prereq = pair[1]
    adjacency[prereq].append(course)        // prereq → course
    inDegree[course] += 1
}

// 3. Seed the queue with every in-degree-0 vertex (use a head index, not removeFirst).
var queue: [Int] = []
for node in 0..<numNodes where inDegree[node] == 0 { queue.append(node) }
var head = 0

// 4. removeFirst() is O(n); head index keeps the peel O(V+E).
while head < queue.count { let node = queue[head]; head += 1 /* ... */ }

// 5. Cycle test = processed count vs total (no separate pass).
let processed = head            // how many we dequeued
return processed == numNodes
The edge-direction trap. Reversing an edge silently produces a valid-looking but wrong answer. Pin it concretely: "take b before a" means b is the prerequisite, so the edge runs b → a and it is a's in-degree that increases. Always restate the dependency in words before writing the edge, and double-check which index is the prerequisite in the problem's pair format.

8   The Four Variants

Ordered by what the problem asks for. Variant 8.1 is taught in a full walkthrough (LC207); the rest are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Kahn's — can it be ordered? (cycle check)
O(V+E)O(V+E)
When to use: a yes/no question — "can all tasks be finished," "is the dependency set consistent." Run Kahn's and compare the processed count to the total. Full Swift in the walkthrough (LC207).

The frame (LC 207 — Course Schedule): build an adjacency list and in-degree array from the prerequisite pairs (edge prereq → course). Queue all in-degree-0 courses, peel them, decrement neighbors, enqueuing any that hit 0. If every course gets processed, there's no cycle → you can finish all of them; otherwise a cycle blocks some → return false.

The design decision: you don't need to find the cycle or produce an order — just whether one exists. The processed-count comparison answers that in the same pass that would build the order, so a yes/no problem costs nothing extra over the full sort.

Canonical problem: LC 207 — Course Schedule (walkthrough). Full walkthrough.

Variant 8.2 Output a valid order
O(V+E)O(V+E)
Solve-it-yourself variant. This is an attempt problem — the frame is below, but no solution. It's a one-line change from §8.1.
When to use: the problem wants an actual ordering, not just feasibility — "return any valid build/course order."

The frame (LC 210 — Course Schedule II): run the exact same Kahn's algorithm, but record each vertex as you dequeue it into an order array. At the end, if order.count == numCourses return order; otherwise a cycle exists, so return an empty array. The removal sequence is a valid topological order by construction.

The frame: §8.2 is §8.1 with the output recorded instead of just counted — the "yes/no" and "give me the order" questions share one algorithm. The only subtlety is returning [] (not the partial order) when a cycle truncates the result. See pitfalls §10.

Canonical problem: LC 210 — Course Schedule II (attempt). Graduated hints in the walkthrough.

Variant 8.3 DFS post-order topological sort
O(V+E)O(V)
Solve-it-yourself variant. This is an attempt problem — the model is below, but no code. It's the recursive alternative to Kahn's.
When to use: you prefer recursion, or already need DFS finish-time structure. Push each vertex after its descendants; reverse the result.

The model: DFS from each unvisited vertex. After recursing into all of a vertex's out-neighbors (so everything that must come after it is already on the stack), push the vertex. The reversed push order is a topological order. Cycle detection uses the three-color marking from the DFS deep dive §8.4: hitting a Gray (on-current-path) vertex is a back-edge → a cycle.

The frame: post-order guarantees a vertex is recorded only after all its dependents, so reversing yields "dependencies first." The catch versus Kahn's: cycle detection isn't free — you must track the on-path (Gray) set explicitly, or a back-edge slips through. See pitfalls §10 and the DFS deep dive.

Canonical framing: the DFS counterpart to LC 207/210 (attempt). Hints in the DFS deep dive.

Variant 8.4 Build the graph from input, then sort
O(input + V + E)O(V + E)
Solve-it-yourself variant. This is a target problem — the edge-derivation model is below, but no code. The hardest topo-sort framing: the graph is hidden in the data.
When to use: the ordering constraints aren't given as edges — you must derive them from the input (sorted words, version strings, ranking lists), then topo sort.

The model (LC 269 — Alien Dictionary): given words sorted in an unknown alphabet's order, compare each adjacent pair of words; the first position where they differ gives one ordering edge (char in the earlier word → char in the later word). Collect all such edges, then run Kahn's over the distinct characters. A cycle means the input is inconsistent (no valid alphabet); also guard the "prefix" invalid case (a longer word before its own prefix).

The frame: the topo sort is the easy half; the skill is recognizing that adjacent sorted items leak exactly one precedence each, and turning those into edges. Once the graph is built, it's plain Kahn's (§8.2). The two extra traps are the prefix edge case and remembering to seed every character as a vertex, even isolated ones. See pitfalls §10.

Canonical problem: LC 269 — Alien Dictionary (target). Graduated hints in the walkthrough.

9   Complexity Analysis

Topological sort is O(V + E): building the in-degree array touches every edge once, and the peel dequeues every vertex once while decrementing along every edge once. No vertex or edge is handled twice.

VariantTimeSpaceWhy
Kahn's cycle check (8.1)O(V+E)O(V+E)Build in-degrees over E edges; peel V vertices, decrementing along E edges. Adjacency list is O(V+E), queue O(V).
Output the order (8.2)O(V+E)O(V+E)Identical to 8.1 plus an O(V) result array — recording is free.
DFS post-order (8.3)O(V+E)O(V)One DFS visits each vertex/edge once; space is the recursion stack + color array, O(V), lighter than Kahn's adjacency if edges are streamed.
Build graph then sort (8.4)O(L + V + E)O(V+E)L = total input size to scan for edges (e.g. all characters of all words); then plain Kahn's on the derived graph.
BottleneckO(V+E)—You must read every dependency at least once, so O(V+E) is a lower bound for any topo sort — Kahn's and DFS both hit it exactly. For §8.4, deriving the edges (O(L)) can dominate if the input is large.
Why cycle detection adds nothing. The processed-count comparison (Kahn's) or the on-path marking (DFS) piggybacks on the single traversal you already do to build the order — no second pass. Detecting "can it be ordered" is therefore the same O(V+E) as producing the order, which is why a yes/no problem like LC207 and an order problem like LC210 have identical complexity.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Reversing the edge direction (all variants). "Take b before a" is edge b → a, and a's in-degree increases. Getting this backward yields a plausible-but-wrong order. Restate the dependency in words first.
▸Using a plain visited set for cycle detection in DFS (8.3). A finished (Black) vertex isn't a cycle; only a vertex still on the current path (Gray) is. Plain "visited" can't tell them apart — track the on-path set.
▸Returning the partial order instead of [] on a cycle (8.2). If a cycle truncates Kahn's, the recorded order is incomplete and invalid. Check order.count == V and return empty otherwise.
▸Forgetting isolated vertices (8.1/8.4). A vertex with no edges has in-degree 0 and must still be enqueued and counted. In §8.4, every character is a vertex even if it never appears in a derived edge.
▸The prefix invalid case (8.4). If a longer word precedes its own prefix (e.g. "abc" before "ab"), the input is invalid with no edge to detect it — guard this explicitly while deriving edges.
▸removeFirst() as the dequeue (Swift). O(n) per call turns the O(V+E) peel into O(V²). Use a head index or deque.

Swiftcopy-paste Kahn's skeleton (cycle check + order)
func topoSort(_ numNodes: Int, _ edges: [[Int]]) -> [Int]? {
    var adjacency = [[Int]](repeating: [], count: numNodes)
    var inDegree  = [Int](repeating: 0, count: numNodes)
    for edge in edges {                       // edge = [from, to], from → to
        adjacency[edge[0]].append(edge[1])
        inDegree[edge[1]] += 1
    }
    var queue: [Int] = []
    for node in 0..<numNodes where inDegree[node] == 0 { queue.append(node) }
    var head = 0, order: [Int] = []
    while head < queue.count {
        let node = queue[head]; head += 1
        order.append(node)
        for next in adjacency[node] {
            inDegree[next] -= 1
            if inDegree[next] == 0 { queue.append(next) }
        }
    }
    return order.count == numNodes ? order : nil   // nil = cycle
}
