Given a string, determine whether it is a palindrome after converting every uppercase letter to lowercase and removing all non-alphanumeric characters (anything that is not a letter or digit). An empty string — or a string that reduces to empty after cleaning — is considered a valid palindrome.

InputOutputWhy
"A man, a plan, a canal: Panama"trueCleaned: "amanaplanacanalpanama" — reads the same forward and backward
"race a car"falseCleaned: "raceacar" — r≠r at the ends but e≠a at the middle mismatch
" "trueCleaned: "" — a string of pure whitespace has no alphanumeric characters; empty is a palindrome by convention

The twist that makes this problem worth studying despite its Easy rating: the naive approach — build a cleaned string, reverse it, compare — works but allocates two extra strings. The elegant solution skips non-alphanumeric characters on the fly with inner skip loops, comparing pairs in a single left-to-right/right-to-left pass without ever building a cleaned copy.

2   Reading the Problem — insights from the statement

"Only alphanumeric characters": this phrase does double duty. First, it defines what to skip — any character that is neither a letter (isLetter) nor a digit (isNumber) must be ignored, including spaces, punctuation, and special characters. Second, it implies no sorting, no counting, no frequency map — we only need a yes/no comparison. That Boolean output is the signal that two pointers with early return dominates any map-based approach.
"Converting all uppercase to lowercase": case folding must happen after we decide the character is alphanumeric — there is no point lowercasing a comma. In Swift, Character.lowercased() returns a String, not a Character — the comparison stringArray[left].lowercased() != stringArray[right].lowercased() is a String-vs-String comparison, which still works correctly but surprises readers expecting a Character-vs-Character result.
"Considering only alphanumeric characters" vs pre-filtering: you can either (a) build a filtered, lowercased array upfront and then run two pointers on it, or (b) advance the pointers past non-alphanumeric characters on the fly. Option (b) avoids a full preprocessing pass and keeps space O(1) — but both approaches are O(n). The key interview insight: whenever the problem says "consider only X characters," ask whether you can skip non-X characters in-line rather than in a separate pass.

The empty-string gotcha: Array("").count - 1 = -1, and var rightPointer = -1 makes the outer while left < right condition (0 < -1) immediately false. Swift's Int is signed, so this is safe — no underflow, no crash. If you used an unsigned index type, this would wrap around.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the output type

The function returns a Bool. What does that tell you about the data structure you need? Is there any reason to build a frequency map or count characters?

Hint 2 — the pattern

You need to compare characters at opposite ends and work inward. What pointer setup gives you that? What do you do when the character at either pointer is not alphanumeric — skip it in a preprocessing step, or skip it inside the loop?

Hint 3 — the inner skip loops

Use two nested while loops — one to advance the left pointer past non-alphanumeric characters, one to retreat the right pointer. The guard in each inner loop must be leftPointer < rightPointer && !isAlphanumeric(…) — the bounds check comes first, not last. After both inner loops settle, if leftPointer < rightPointer still holds, compare the pair.

A: O(1) space would be ideal, but O(n) is acceptable. → I'll do the two-pointer solution that skips characters on the fly. I'll note where the O(1) vs O(n) tradeoff lives.

The plan, spoken

"This is a two-pointer problem — the converging variant, comparing pairs from the outside in. The signal is: Boolean output, comparing characters at opposite positions. No frequency counting, no sorting — just equality checks."

The invalid condition is: any pair of alphanumeric characters that doesn't match — I can return false immediately. The loop runs while the left pointer is strictly less than the right pointer — when they meet or cross, all pairs checked, return true."

"Expected complexity: O(n) time — each character is visited at most once across all three loops; O(n) space from the character array. I'll code it now."

6   Why This Approach — and not the others

CandidateCostVerdict
Build cleaned string, reverse, compareO(n) time, O(n) spaceCorrect and readable, but allocates two extra strings — the reversed copy and the filtered copy. Fine as a first pass; not the interview-optimal answer.
Build cleaned array, then two-pointer on itO(n) time, O(n) spaceSame asymptotic cost, but at least eliminates the reverse copy. The filtering pass and the comparison pass are separate — the on-the-fly solution collapses them into one.
Converging two pointers with inner skip loopsO(n) time, O(n) space*Chosen. Single pass: skip + compare happens in one outer loop. Clearest demonstration of the converging-pointer pattern. (* O(1) space achievable with String.Index — see §13.)
Hashmap of character frequenciesO(n) time, O(1) space†Works for even-length palindromes; for odd-length, exactly one character may have an odd count. More code, more edge cases, same complexity. Never reach for a map when a Boolean comparison suffices.
THE ONE IDEA: the Boolean return type tells you everything. When a function returns Bool, any data structure that accumulates information to be inspected later — a map, a set, a frequency table — is already over-engineering. You need exactly one comparison per pair, and you can return false the instant that comparison fails. Two pointers with early return is the minimal structure that expresses this.
When to use converging two pointers over preprocessing + reverse: whenever you need to compare a sequence against its own mirror and you care about space — or whenever the interviewer asks "can you do it in one pass?" The skip-in-place pattern extends naturally to any "consider only X characters" variant.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker
stringArray: [Character]The input string as a random-access array of charactersSwift's String does not support O(1) integer indexing — s[2] is illegal. Array(string) gives a [Character] with O(1) subscript access. Cost: O(n) extra space for the copy. Alternative: use String.Index directly on the original string for O(1) space, at the cost of more verbose index arithmetic.
leftPointer: Int, rightPointer: IntCurrent positions of the two converging pointersTwo integers replace any map, set, or auxiliary array. The outer loop terminates when they cross; the inner skip loops advance each one past non-alphanumeric characters without needing any storage.
isAlphanumeric(_:)Helper predicate: character.isLetter || character.isNumberExtracted to a named function so the inner loop conditions read as intent, not as a compound boolean expression. In Swift, isLetter covers Unicode letters; on printable ASCII inputs, it behaves identically to a range check on ASCII values.
Swift gotcha — .lowercased() returns String, not Character: calling character.lowercased() on a Character in Swift returns a String, not a Character. The expression stringArray[left].lowercased() != stringArray[right].lowercased() is therefore a String comparison — which works correctly, but surprises anyone who expects a Character-to-Character comparison. This is by design: lowercased() can expand a character to multiple code points in some Unicode locales.
Swift gotcha — cast the Array conversion: Array(string) returns [Character], but Swift's type inference sometimes needs a hint. Writing let stringArray = Array(string) as [Character] prevents the compiler from inferring an ambiguous type and documents the intent explicitly.

8   Pseudocode Dry Run

Pseudocodeconverging two pointers with in-place skip
chars ← Array(string) // [Character], O(n) space
left  ← 0
right ← chars.count − 1

WHILE left < right:

    // INNER SKIP LEFT: advance past non-alphanumeric, bounds-check first
    WHILE left < right AND NOT isAlphanumeric(chars[left]):
        left += 1

    // INNER SKIP RIGHT: retreat past non-alphanumeric, bounds-check first
    WHILE left < right AND NOT isAlphanumeric(chars[right]):
        right −= 1

    // COMPARE: both pointers are on alphanumeric characters
    IF lowercase(chars[left]) ≠ lowercase(chars[right]):
        RETURN false   // early exit — mismatch found

    left += 1
    right −= 1

RETURN true   // all pairs matched (or no pairs to check)

9   Swift Solution

A.1 Converging two pointers — in-place skip  (canonical)
O(n)O(n)
Plan
Convert the string to a [Character] array for O(1) index access. Run left/right pointers from the ends. Two inner while loops advance each pointer past non-alphanumeric characters — the bounds check (left < right) must precede the character test to avoid out-of-bounds access. After both skip loops settle, compare the pair lowercased; return false on any mismatch. Advance both pointers and repeat. Return true when the pointers meet. 
private func isAlphanumeric(_ character: Character) -> Bool {
    character.isLetter || character.isNumber
}

private func isValidPalindrome(_ string: String) -> Bool {
    let stringArray = Array(string) as [Character]
    var leftPointer = 0
    var rightPointer = stringArray.count - 1

    while leftPointer < rightPointer {
        // Bounds check FIRST, then character test — order matters.
        while leftPointer < rightPointer && !isAlphanumeric(stringArray[leftPointer]) {
            leftPointer += 1
        }
        while leftPointer < rightPointer && !isAlphanumeric(stringArray[rightPointer]) {
            rightPointer -= 1
        }

        if stringArray[leftPointer].lowercased() != stringArray[rightPointer].lowercased() {
            return false
        }

        leftPointer += 1
        rightPointer -= 1
    }

    return true
}

12   Complexity

PhaseTimeSpaceWhy
Array conversionO(n)O(n)Array(string) copies every character into a new buffer — one allocation, one pass
Outer loop + inner skip loopsO(n)O(1)Each character is visited by at most one pointer, at most once — total work across all three loops is bounded by n; the two pointer variables are the only extra state
TotalO(n)O(n)The array copy dominates space; time is linear. The O(1)-space variant (using String.Index directly) is shown in §13.
Why each character is visited at most once: the left pointer only moves right; the right pointer only moves left. A character visited by the left skip loop is never revisited — the outer loop advances leftPointer past it. Same for the right skip loop. The three loops (outer, left inner, right inner) collectively perform at most 2n pointer moves total, not n × n.
The O(n) space from Array(string): Swift strings are not random-access collections by integer index — s[i] is illegal when i is an Int. Converting to [Character] is the straightforward way to get O(1) subscript access, at the cost of O(n) extra memory. The O(1)-space solution keeps the original String and uses String.Index with index(after:) / index(before:) — more verbose, but allocates nothing beyond the two index variables.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The array copy in Array(string) allocates O(n) extra memory. The improvement is to drop it entirely and use String.Index directly on the original string: start with var left = string.startIndex and var right = string.index(before: string.endIndex), advance with string.index(after: left) and string.index(before: right). Same O(n) time, now O(1) space — no copy, no allocation. The trade-off is verbosity: Swift's index arithmetic is more explicit than integer arithmetic, which can hurt readability in an interview setting.
Follow-up: "What if the definition of 'palindrome' changes — at most one character can differ (the 'Valid Palindrome II' variant, LC 680)?"
On the first mismatch, instead of immediately returning false, you try two branches: skip the left character and check the remainder, or skip the right character and check the remainder. If either sub-range is a palindrome, return true. This is LC 680 — the key insight is that you only ever get one "skip" allowance, so you only branch on the first mismatch. Time stays O(n); no extra space needed beyond the recursive (or inline) sub-check.
Follow-up: "Could you implement isAlphanumeric without using Swift's isLetter and isNumber — using ASCII values only?"
Yes — every printable ASCII alphanumeric falls in three ranges: digits 48–57 ('0'–'9'), uppercase letters 65–90 ('A'–'Z'), lowercase letters 97–122 ('a'–'z'). In Swift: let ascii = character.asciiValue and check the three ranges with ||. This avoids any Unicode-aware property lookup and is slightly faster on ASCII-only inputs — relevant if the constraint "printable ASCII" is part of the problem spec.
Follow-up: "What if the string can contain Unicode — accented letters, CJK characters? Does your solution still work?"
Swift's Character.isLetter is already Unicode-aware — it returns true for accented characters like 'é' and CJK ideographs. Similarly, lowercased() handles Unicode case folding. The algorithm is correct as-is for Unicode. The only concern is that some Unicode characters lower-case to multiple code points (e.g., 'ß' → "ss") — lowercased() returns a String precisely to handle this, so the String-vs-String comparison remains correct. The ASCII range check from the previous follow-up would break on Unicode — isLetter is the right tool for the general case.
