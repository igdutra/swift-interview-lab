You are given two lists of closed intervals, firstList and secondList, each already sorted by start and containing only non-overlapping intervals. Return the intersection of the two lists — every closed interval that appears in both.

InputOutputWhy
firstList=[[0,2],[5,10],[13,23],[24,25]]
secondList=[[1,5],[8,12],[15,24],[25,26]]		 [[1,2],[5,5],[8,10],[15,23],[24,24],[25,25]] [0,2]∩[1,5]=[1,2]; [5,10]∩[1,5]=[5,5]; [5,10]∩[8,12]=[8,10]; etc.
firstList=[[1,3]] secondList=[[4,6]][]No overlap between the lists — returns empty
firstList=[] secondList=[[1,5]][]Empty list — while condition false from the start

The twist: you have two sorted lists and need their intersection. The overlap formula is symmetric — overlapStart = max(a.start, b.start), overlapEnd = min(a.end, b.end) — but the pointer-advance rule is asymmetric: you always advance the pointer whose current interval has the smaller end. Understanding why that rule is correct is the whole problem.

2   Reading the Problem — insights from the constraints

"Both lists are already sorted and non-overlapping": this is a gift. You do not need to sort or pre-merge either list. The sorted order guarantees that when you advance a pointer, everything behind it is permanently done — the same invariant the sort step buys in Variant 8.1.
"Intersection of two interval lists": you need every point that appears in at least one interval from each list. This is set intersection, not union. The formula: two intervals [a,b] and [c,d] intersect if and only if max(a,c) ≤ min(b,d). If max(starts) > min(ends), there is no overlap.
"Closed intervals": both endpoints are included. The intersection of [1,3] and [3,5] is [3,3] — a single point, which is a valid closed interval. A half-open convention would produce an empty intersection here. Always clarify which convention the problem uses.
Two separate sorted lists: you cannot flatten and re-sort — that would destroy the structure and lose the information about which list an interval belongs to. You need a two-pointer walk: one pointer per list, advancing through them simultaneously.
0 ≤ firstList.length, secondList.length ≤ 1000: at most 1000 intervals per list — 10⁶ total comparisons in brute force. O(m + n) is straightforwardly achievable and expected.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

Both lists are already sorted and non-overlapping. You have two separate lists. O(m×n) brute force checks every pair — can you do it in O(m + n)?

Hint 2 — the pattern

Two-pointer walk: one pointer per list. At each step, compute the intersection of the current pair. Then ask: which pointer should advance? The interval that ends first is permanently done — it cannot overlap any future interval.

Hint 3 — the overlap formula and advance rule

Intersection exists when max(a.start, b.start) ≤ min(a.end, b.end). Advance the pointer whose current interval has the smaller end. If they tie, it doesn't matter which — but pick one consistently.

The plan, spoken

"This is a two-pointer interval overlap problem — Variant 8.2. I have two sorted lists and need their intersection. The O(m×n) brute force checks every pair — I can beat that to O(m + n) by walking both lists simultaneously with one pointer each."

"The key conditions. The overlap condition: two closed intervals [a,b] and [c,d] intersect when max(a,c) ≤ min(b,d). If that holds, I record the intersection [max(a,c), min(b,d)]. "The invariant: at each step, the current pair of intervals is the only pair that could produce a new intersection. Every pair involving an already-advanced interval has been accounted for."

"Data structures: two integer index variables indexA and indexB, both starting at 0. A result array built by appending. No sorting, no maps, no auxiliary structures."

"Time: O(m + n) — each pointer advances at most its list's length. Space: O(k) for the output where k is the number of intersections. I'll trace the canonical example now."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check every pair (i, j)O(m × n)At most 10⁶ pairs — feasible for this problem's constraints, but misses the structure. Name it to show you know the baseline.
Flatten + sort + sweepO((m+n) log(m+n))Wrong approach: flattening loses list membership. You can't tell which list an interval came from after merging the two arrays — you'd end up merging intra-list intervals instead of finding inter-list intersections.
Binary search per intervalO(m log n)For each interval in A, binary-search B for overlapping intervals. Correct, but the two-pointer approach is simpler and achieves the same or better asymptotic cost.
Two-pointer walk — advance smaller endO(m + n)Chosen. Each pointer moves monotonically forward; the advance rule is provably correct; no preprocessing needed since both lists are pre-sorted.
The deciding signal: two separately sorted lists. You cannot merge them — that destroys list membership. The two-pointer walk is exactly the tool for walking two sorted sequences simultaneously.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
indexA: IntCurrent position in firstListDirect index into the sorted list. Starts at 0, increments when firstList's current interval has the smaller-or-equal end.
indexB: IntCurrent position in secondListSame pattern. Advances when secondList's current interval has the smaller end.
result: [[Int]]Accumulated intersection intervalsAppended to when overlapStart ≤ overlapEnd. At most min(m, n) entries in the worst case.
Why no sort step. Both input lists are pre-sorted — the problem guarantees this. A sort step would be wasted work. The two-pointer technique only works because both lists are sorted: the advance rule's correctness proof depends on monotonically increasing starts.

9   Swift Solution — iterative

A.1 Two-pointer walk — advance smaller end
O(m+n)O(k)
Plan
Walk both sorted lists with one pointer each. At each step compute the overlap. If it exists (overlapStart ≤ overlapEnd), record it. Always advance the pointer whose current interval ends first — that interval cannot overlap anything later in either list. 
func intervalIntersection(_ firstList: [[Int]], _ secondList: [[Int]]) -> [[Int]] {
    var indexA = 0
    var indexB = 0
    var result: [[Int]] = []

    while indexA < firstList.count && indexB < secondList.count {
        let overlapStart = max(firstList[indexA][0], secondList[indexB][0])
        let overlapEnd   = min(firstList[indexA][1], secondList[indexB][1])

        if overlapStart <= overlapEnd {
            result.append([overlapStart, overlapEnd])
        }

        // Advance the pointer whose interval ends first — it's exhausted.
        if firstList[indexA][1] <= secondList[indexB][1] {
            indexA += 1
        } else {
            indexB += 1
        }
    }

    return result
}

12   Complexity

PhaseTimeSpaceWhy
Two-pointer walkO(m + n)O(1) extraEach pointer advances monotonically. indexA moves at most m times, indexB at most n times. Each iteration is O(1) — two max/min ops, one optional append.
Output—O(k)k = number of intersections, at most min(m, n). Each intersection is a constant-size two-element array.
TotalO(m + n)O(k)No sort step (input pre-sorted). The walk visits each interval exactly once. Output space is proportional to the answer size — unavoidable.
Why no sort cost. In Variant 8.1 (LC 56), sorting costs O(n log n) and dominates. Here the input is pre-sorted — the problem guarantees it. The two-pointer walk pays only O(m + n). If the inputs were unsorted you'd pay O(m log m + n log n) for two separate sorts before the walk.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The algorithm is already optimal: O(m + n) time, O(k) space, with no unnecessary work. One practical improvement: pre-allocate result with reserveCapacity(min(firstList.count, secondList.count)) if the caller knows the output will be large. This avoids repeated array reallocations. Asymptotically it changes nothing.
Follow-up: "What if the input lists are unsorted?"
Sort each list independently by start — O(m log m + n log n). Then apply the same two-pointer walk. The advance rule's correctness proof requires sorted order: it relies on the fact that all future intervals in each list start at or after the current interval's start. Without sorted order, advancing the smaller-end pointer could skip intersections.
Follow-up: "What if you have k lists instead of two?"
Two approaches. (1) Reduce pairwise: intersect lists 1 and 2, then intersect that result with list 3, and so on — O(k × n) total if each list has n intervals. (2) Use a k-way merge with a min-heap keyed by current interval's start — O(n log k) per interval. The pairwise reduction is simpler to implement and often sufficient; the heap approach is better when k is large.
Follow-up: "What if the intervals can overlap within a single list — it's not guaranteed non-overlapping?"
Pre-merge each list using the LC 56 approach before running the two-pointer walk. Without that guarantee, the advance rule breaks: advancing the smaller-end pointer could skip intervals in the same list that overlap the current one. Merge first (O(n log n) per list), then intersect (O(m + n)).
