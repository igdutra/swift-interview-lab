Theory
dfs_deep_dive_master.html

You're shopping for a fixed basket. price[i] is the unit price of item i; needs[i] is exactly how many of item i you must end up with. There's also a list of special offers: each special[j] is a vector [c₀, c₁, …, c_{n−1}, cost] meaning "pay cost to receive c₀ of item 0, c₁ of item 1, …". Return the lowest price to buy exactly needs — no more of any item. You may use any offer any number of times, but never in a way that overshoots a need.

InputOutputWhy
price=[2,5], special=[[3,0,5],[1,2,10]], needs=[3,2]14Apply offer1 ([1,2]→10) leaving needs [2,0], then buy 2×A at base 2×2=4 → 14. Beats offer0 ([3,0]→5)+2×B base 10 = 15, and beats all-base 16.
price=[2,3,4], special=[[1,1,0,4],[2,2,1,9]], needs=[1,2,1]11Buy 1 A + 1 B with offer 0 (cost 4), then 1 B + 1 C at base 3+4=7 → 11. The greedy "use the biggest offer" would overshoot.
price=[2,5], special=[[3,0,5]], needs=[0,0]0Nothing needed → cost 0 (the base case).

The twist is the "exactly, no overshoot" rule. It rules out a greedy "always take the cheapest-per-item offer" — an offer that's cheap per unit can still be wrong because it forces you past a need. The clean framing: the remaining needs vector is a state, and from any state you either apply one still-affordable offer (recursing on the smaller leftover) or stop using offers and buy everything left at base price. That's a DFS over states, and because many offer-orderings reach the same leftover vector, you memoize on the vector.

2   Reading the Problem — insights from the constraints

"Buy exactly needs, no overshoot": this is the constraint that kills greedy. An offer is only legal from a state if every item it grants is ≤ the corresponding remaining need. So at each step you can only apply a subset of offers, and that subset shrinks as needs deplete. This "applicable only if it fits" rule is what makes it a search, not a formula.
"Any offer any number of times": there's no per-offer limit, so you don't track how many times you've used an offer — only the resulting needs vector matters. That's the key to memoization: two different sequences of offers that leave the same remaining needs cost the same from here on, so the answer depends only on the current needs vector.
"Return the lowest price": a minimization over choices = DFS that returns the min over (each applicable offer → recurse) versus (buy the rest at base price). The base-price option is always available and is the floor that guarantees the recursion terminates with a real number.
Small constraints (n ≤ 6 items, needs[i] ≤ 10, ≤ 100 offers): the state space is at most 11⁶ ≈ 1.7M vectors, but in practice far fewer are reachable. Small enough that DFS + memo over the needs vector is comfortably fast — no need for clever bitmask packing, a dictionary keyed by the vector suffices.
Offers can be "bad": an offer may cost more than buying its items individually. You don't pre-filter — the recursion's min naturally ignores a bad offer because the base-price branch (or a better offer) wins. Trying every applicable offer and taking the min is what makes correctness automatic.

The overshoot check and the base case are the load-bearing lines. Rejecting any offer that exceeds even one remaining need is what enforces "exactly needs." The all-zero base case is what stops the recursion and what every applicable-offer branch eventually bottoms out into. Get those two right and the min-over-branches handles bad and useless offers for free.

4   Try It Yourself First

Resist greedy — the "exactly needs" rule breaks it. Open hints one at a time.

Hint 1 — why greedy fails

"Pick the cheapest offer per item" can overshoot a need, and you're forbidden to overshoot. A cheap offer might force you past a need you can't undo. So you must consider keeping and skipping each offer — that's a search over choices, not a formula.

Hint 2 — the state

What changes as you shop is the remaining needs vector. From any remaining-needs state, your options are: (a) for each offer that fits (every count ≤ the matching remaining need), apply it and recurse on the smaller leftover, paying its cost; or (b) stop using offers and buy everything left at base price. Return the minimum.

Hint 3 — memoize

Many different offer orders reach the same remaining-needs vector, and the cost from there on depends only on that vector. Cache results keyed by the vector (e.g. a [Int] as a dictionary key) so each distinct state is solved once. That turns exponential DFS into polynomial-ish top-down DP.

The plan, spoken

"This is a DFS over states with memoization — top-down DP. The signal is 'minimize cost' under a choice at each step (which offer, if any), where greedy fails because of the no-overshoot rule. So I search."

The state is the remaining needs vector — that's all that matters, because offers are unlimited and the future cost depends only on what's left to buy. The validity condition for applying an offer: every item count in the offer must be ≤ the matching remaining need — otherwise it overshoots and is rejected. "The walk: at a state, first compute the base-price answer (buy all remaining individually) as the running minimum. Then for each offer that fits, build the leftover vector, recurse, and take cost + sub-answer if it beats the running min. Memoize and return. Time is O(offers · reachable-states); space is the memo plus recursion depth. I'll guard against useless offers that don't shrink any need to avoid non-termination."

6   Why This Approach — and not the others

CandidateCostVerdict
Greedy — repeatedly take the best per-unit offerO(offers)Wrong. The no-overshoot rule means a per-unit-cheap offer can force you past a need; skipping it is sometimes optimal. Greedy can't look ahead, so it returns wrong answers.
Pure DFS over offer choices (no memo)exponentialCorrect but recomputes the same remaining-needs vector along many offer orders — exponential blow-up. The fix is caching, not a different search.
Bottom-up DP over all needs vectorsO(Πᵢ(needsᵢ+1) · offers)Valid — iterate every reachable vector in increasing order. Correct, but the iteration order over a multi-dimensional grid is fiddly to write; top-down only visits reachable states.
DFS over the needs vector + memoizationO(states · offers)Chosen. The state is naturally the remaining needs; memoizing on it dedupes overlapping subproblems and only visits reachable states. Cleanest correct solution.
The deciding signal: "minimize cost" + "a constrained choice at each step where greedy can overshoot" + "overlapping subproblems (same leftover via different orders)." That triad is top-down DP: DFS the choices, and memoize on the state that fully determines the remaining cost — here, the needs vector.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
needs: [Int] (the recursion argument)The remaining quantity of each itemThe full state — it alone determines the optimal remaining cost. A scalar can't capture a multi-item basket; the vector is the minimal faithful state.
memo: [[Int]: Int]Solved minimum cost for each needs vector seenMany offer orders reach the same leftover; caching by the vector turns exponential DFS into polynomial. Swift arrays are Hashable (elements are), so [Int] is a legal key.
price: [Int]Base unit pricesUsed to compute the base-price fallback (dot product with remaining needs) — the always-available branch and the recursion's floor.
special: [[Int]]Offers — counts then trailing costIterated at each state; the applicability check reads the first n entries, the cost is the last.
leftover vector (per branch)needs[i] − offer[i] for each itemThe smaller state to recurse on after applying an offer; rebuilt fresh per branch so siblings don't interfere.
Why the needs vector is the right memo key. The cost to finish shopping depends on exactly one thing: what you still need. It does not depend on which offers you used to get here or in what order — offers are unlimited and stateless. That independence is what licenses memoizing on the vector alone, and it's the difference between an exponential search and a fast one.

8   Pseudocode Dry Run

PseudocodeDFS over needs, memoized; min(base-price, each applicable offer)
FUNCTION solve(needs):
    IF needs in memo: RETURN memo[needs]

    best ← Σ price[i] * needs[i]          buy everything at base price

    FOR offer IN special:
        leftover ← []
        FOR i IN 0..<n:
            IF offer[i] > needs[i]: leftover ← INVALID; BREAK   overshoot → skip offer
            leftover.append(needs[i] - offer[i])
        IF leftover is INVALID: CONTINUE
        best ← min(best, offer.cost + solve(leftover))

    memo[needs] ← best
    RETURN best

9   Swift Solution

Top-down: a recursive helper over the needs vector with a dictionary memo. The base-price fallback is the running minimum; each applicable offer can only lower it.

A.1 DFS over needs + memoization
O(states·offers)O(states)
Plan
Recurse on the remaining needs vector. Start the answer at the base price (buy everything left individually). For each offer that fits every remaining need, form the leftover vector, recurse, and take its cost + sub-answer if smaller. Memoize the result by the needs vector and return. 
func shoppingOffers(_ price: [Int], _ special: [[Int]], _ needs: [Int]) -> Int {
    let itemCount = price.count
    var memo: [[Int]: Int] = [:]

    func solve(_ needs: [Int]) -> Int {
        if let cached = memo[needs] { return cached }

        // Always-available branch: buy every remaining item at base price.
        var best = 0
        for item in 0..<itemCount { best += price[item] * needs[item] }

        // Try each offer that fits within the remaining needs (no overshoot).
        for offer in special {
            var leftover = [Int](repeating: 0, count: itemCount)
            var fits = true
            for item in 0..<itemCount {
                if offer[item] > needs[item] { fits = false; break }   // overshoot → reject
                leftover[item] = needs[item] - offer[item]
            }
            if !fits { continue }

            let offerCost = offer[itemCount]                          // trailing entry is the price
            best = min(best, offerCost + solve(leftover))
        }

        memo[needs] = best
        return best
    }

    return solve(needs)
}
Note: the base-price computation doubles as the base case — when needs is all zeros, the loop adds nothing and best stays 0, so no explicit zero check is needed. An offer that grants nothing useful (all counts 0 but positive cost) makes leftover == needs; it can't lower best (it only adds cost), and because the result is memoized the same state isn't re-expanded — so it neither helps nor loops. If an offer could have a non-positive cost and zero counts you'd guard against the no-progress recursion explicitly, but LC638 offers have positive cost.

12   Complexity

PhaseTimeSpaceWhy
Per state (one solve call body)O(offers·n)O(n)Iterate all offers; each does an n-item fit check and builds an n-item leftover.
Distinct states (memoized)O(Πᵢ(needsᵢ+1))O(Πᵢ(needsᵢ+1)·n)Each reachable needs vector is solved once; the memo stores one entry (an n-vector) per state.
TotalO(states · offers · n)O(states · n)Bounded by Πᵢ(needsᵢ+1) ≤ 11⁶ states × 100 offers × 6 — comfortably small in practice since only reachable states are visited. The state count, not the per-state work, is the cap.
Why memoization is what makes it tractable. Without the memo, the same leftover vector is recomputed along every offer order that reaches it — exponential. The cache collapses all those repeats into one solve per distinct needs vector, so the cost becomes "number of reachable states × work per state." This is the defining move of top-down DP: a DFS whose state fully determines the answer, cached on that state.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
Two. First, I'd pre-filter offers that are strictly worse than buying their items at base price (offerCost ≥ Σ price[i]·offer[i]) — they can never help, so dropping them shrinks the branching. Second, the [Int] dictionary key hashes the whole vector each call; since needs are small (≤ 10, ≤ 6 items) I could pack the vector into a single Int (mixed-radix, 4 bits per item) for an O(1) integer key and faster memo lookups.
Follow-up: "Why not just buy greedily using the cheapest-per-item offers?"
Because you must buy exactly the needs — no overshoot. A per-item-cheap offer can push you past a need, and you can't return the surplus. So sometimes the optimal solution skips a locally-cheap offer. Greedy has no way to see that; only a search over keep-vs-skip (with memo) is guaranteed correct.
Follow-up: "What if you were allowed to overshoot — buy extra if it's cheaper?"
Then the fit check changes from "offer[i] ≤ needs[i] for all i" to "applicable as long as at least one need is still positive," and the leftover clamps at zero (max(0, needs[i] - offer[i])). The DFS+memo skeleton is identical — only the applicability rule and the leftover formula change. Memoizing on the clamped vector still works.
Follow-up: "Could you write this bottom-up instead of recursive?"
Yes — iterate every needs vector from [0,…,0] upward in each dimension, filling a table where dp[v] = min cost for vector v, computed from base price and from offerCost + dp[v − offer] for fitting offers. Same recurrence, explicit order. The top-down version is easier to write and only touches reachable states, which is why I'd lead with it.

14   Theory Decoder — the theory as used here

This problem is DFS over an implicit state graph with memoization — i.e. top-down dynamic programming, Variant 8.3 of the DFS deep dive. The general concepts, instantiated:

DFS vs BFS here. This is a shortest-cost search where edges (offers) carry different prices and you must explore all the way to the zero state to compare totals — so depth-first recursion with memo fits naturally, not a level-by-level BFS. BFS shines for fewest-steps on equal-cost edges; this is fewest-cost over a DAG of states, which is DP territory.
