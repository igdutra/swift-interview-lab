sliding_window_master.html

You are given two strings: a haystack s and a requirement list t. Find the shortest contiguous piece of s that contains every character of t — counting duplicates — and return it. If no such piece exists, return the empty string.

InputOutputWhy
s = "ADOBECODEBANC", t = "ABC""BANC"Shortest window containing A, B, and C
s = "a", t = "a""a"Entire string is the only valid window
s = "a", t = "aa"""t needs two A's; s only has one — no valid window exists

The twist that makes it Hard: every other sliding window you have practiced shrinks when the window breaks. This one shrinks when the window works — you squeeze each valid window down to its smallest form before letting the right pointer move on. That single inversion is the whole problem.

2   Reading the Problem — insights from the statement

"Minimum window": the word "minimum" does all the work. The window size is variable and you are searching for the smallest — so once a window becomes valid you must keep shrinking it, recording every size you pass. Compare LC 567, which only asks whether a fixed-size window exists.
"All characters of t, including duplicates": order is irrelevant; multiplicity is everything. The validity check is a frequency comparison, not a membership test — a Set cannot represent "two A's", so the requirement must live in a count map.
"Window in s": the window may contain characters that are not in t at all — "ADOBEC" is a valid window for t = "ABC". Non-target characters are ignored by the counting but still sit inside the window, which kills any hard-reset strategy.

s, t up to 10⁵O(n·k) per-step validity too slow → matchedCount makes it O(1)
"return the substring"Track the best range, not just the best length

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

n up to 10⁵ rules out checking all O(n²) substrings. What pattern gives you "optimal contiguous range" in O(n)?

Hint 2 — the pattern

Sliding window — but the "minimum" variant. Ask: does my while loop move toward validity or away from it? Where does the answer get recorded in each case?

Hint 3 — the structure

Two count maps (required vs in-window) make validity O(k) per step — still too slow. Track one integer: how many distinct characters currently meet their required count. Valid ⇔ that integer equals the number of distinct chars in t.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can t contain duplicate characters — say t is 'AAB'? Does my window then need two A's?"
A: Yes, duplicates count. → I need frequency maps, not a set.
Q: "If no valid window exists, what do I return — empty string, or something else?"
A: Empty string. → I'll keep the best answer optional and guard the return.
Q: "Is the comparison case-sensitive? Are we limited to letters, or any character?"
A: Case-sensitive, mixed letters. → no normalization; a Character-keyed map handles any alphabet.
Q: "How large can s and t get — should I optimize for time first?"
A: Up to 10⁵ each; time first. → O(n²) substring checks are out; I'm aiming for one pass.
Step 2 — The plan, before any code
The plan, spoken

"This is a sliding window problem — specifically the minimum-window variant. The signal is the word 'minimum' over a contiguous substring: I expand a window rightward until it satisfies t, then shrink it from the left while it still satisfies t, recording every size on the way down. That inversion — shrink while valid, record inside the shrink — is what distinguishes this from the 'longest' problems."

"Before I name my data structures, let me pin the conditions, because they determine what I need to track. The validity condition is: the window contains every character of t at its required multiplicity. The invalid condition is the mirror: the window goes invalid the instant an eviction drops some character's count below its requirement. The loop runs while valid — shrinking toward a smaller window — and the answer is recorded inside that loop, before each eviction, because that is the moment we are guaranteed a valid window. The invariant: every pass through the shrink loop holds a valid window."

"Those conditions tell me exactly what data I need to track. I need to know — at O(1) — whether the window is currently valid. Comparing two full frequency maps each step would be O(k), so I'll maintain one integer, matchedCount: how many distinct characters currently meet their required count. The validity condition becomes simply matchedCount equals the number of distinct characters in t. matchedCount only moves at boundary crossings: up by one when a character's count reaches exactly its requirement on entry, down by one when an eviction drops a count just below it. For the maps themselves: one built once from t for required counts, one maintained live as the window changes."

"The walk follows directly: for each right character, fold it into the window counts and increment matchedCount if its count just hit the requirement; then, while the validity condition holds, record the window if it beats the current best, and evict the leftmost character, decrementing matchedCount if that eviction breaks a requirement. Each character enters once and leaves at most once."

"So I expect O(n + m) time — one pass over s plus building t's map — and O(k) space, where k is the distinct characters in t. I'll code it now and trace 'ADOBECODEBANC' / 'ABC' to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check every substringO(n²) windows × O(k) check10⁵ input makes ~10¹⁰ checks — dead on arrival, but name it to show the baseline
Prefix counts + binary search per startO(n log n · k)Workable but strictly worse than the window, and far more code
Sliding window, naive validityO(n · k)Correct shape, but comparing maps each step hides a factor k
Sliding window + matchedCountO(n + m)Chosen. Monotonic pointers give amortized O(n); the integer makes validity O(1)
The deciding signal: "minimum contiguous window over a constraint that is incrementally updatable" — counts can be added and removed one character at a time, which is precisely what the window needs to stay O(n).

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker
targetCounts: [Character: Int]Required multiplicity of each char in tA Set can't express "two A's"; built once, never mutated
windowCounts: [Character: Int]Live counts of target chars inside the windowOnly target characters are stored — non-target chars pass through uncounted
matchedCount: IntHow many distinct chars currently meet their requirementReplaces an O(k) map comparison with one O(1) equality; updated only at the exact-match boundary crossings
bestRange: Range<Int>?The smallest valid window seenOptional encodes "none found" without sentinels; a range beats two ints because the return is a substring, not a length
Swift cost note: Array(s) once up front for O(1) indexing; dictionary reads/writes are O(1) amortized; the final String(characters[range]) is O(answer length), paid once.

8   Pseudocode Dry Run

Pseudocodeshrink-while-valid with matchedCount
targetCounts ← counts of t          // e.g. {A:1, B:1, C:1}
windowCounts ← empty; matched ← 0; left ← 0; best ← none

FOR right FROM 0 TO n − 1:
    IF s[right] IN targetCounts:
        windowCounts[s[right]] += 1
        IF windowCounts[s[right]] == targetCounts[s[right]]: matched += 1

    WHILE matched == |targetCounts|:            // window is VALID
        IF (right − left + 1) < |best|: best ← [left, right]
        IF s[left] IN targetCounts:
            windowCounts[s[left]] −= 1
            IF windowCounts[s[left]] < targetCounts[s[left]]: matched −= 1
        left += 1

RETURN substring(best) OR ""

9   Swift Solution — iterative

A.1 Shrink-while-valid window with matchedCount
O(n+m)O(k)
Plan
One pass of the right pointer over s. Expand: count target characters and bump matchedCount on exact requirement hits. While valid: record the window if smallest, evict from the left, and downgrade matchedCount when an eviction breaks a requirement. Return the best range as a substring, or "". 
func minWindow(_ s: String, _ t: String) -> String {
    let characters = Array(s)

    var targetCounts: [Character: Int] = [:]
    for character in t { targetCounts[character, default: 0] += 1 }

    var windowCounts: [Character: Int] = [:]
    var matchedCount = 0           // distinct chars meeting their requirement
    var leftPointer  = 0
    var bestRange: Range<Int>? = nil

    for (rightPointer, rightCharacter) in characters.enumerated() {
        // EXPAND — only target characters are counted
        if let required = targetCounts[rightCharacter] {
            windowCounts[rightCharacter, default: 0] += 1
            if windowCounts[rightCharacter] == required { matchedCount += 1 }
        }

        // SHRINK while valid — record INSIDE: every shrink step may be a new best
        while matchedCount == targetCounts.count {
            let windowLength = rightPointer - leftPointer + 1
            if bestRange == nil || windowLength < bestRange!.count {
                bestRange = leftPointer ..< rightPointer + 1
            }
            let leftCharacter = characters[leftPointer]
            if let required = targetCounts[leftCharacter] {
                windowCounts[leftCharacter]! -= 1
                if windowCounts[leftCharacter]! < required { matchedCount -= 1 }
            }
            leftPointer += 1
        }
    }

    guard let range = bestRange else { return "" }
    return String(characters[range])
}

13   Follow-Up Bank — the interviewer's second act

Follow-up: "The alphabet is only uppercase/lowercase letters. Can you drop the dictionaries?"
Yes — replace both maps with Int arrays of size 128 indexed by ASCII value, and matchedCount works identically. Same O(n + m) time, but space becomes a fixed 128 ints and the constant factor drops noticeably. The structure of the algorithm does not change at all.
Follow-up: "What if s arrives as a stream and you can't index backwards?"
The right side already consumes the stream in order; the problem is the left evictions. Keep the window's characters in a FIFO buffer (a deque) so eviction pops from the buffer instead of re-reading s. Space grows to O(window) but the one-pass property survives.
Follow-up: "Return all minimum windows, not just one."
Track bestLength first (same pass), or collect candidates during the shrink: whenever a recorded window equals the current best length, append its range; if it beats the best, clear and restart the list. Still O(n) plus output size.
Follow-up: "How would the code change for 'longest substring with at most k distinct characters'?"
That flips the variant: the while loop becomes "while distinct count > k" (shrinking toward validity), and the record moves after the loop. Saying this out loud — that I'd switch from shrink-while-valid to shrink-on-invalid and move the record — shows I own the pattern, not just this problem. See variant 8.2 in the theory page's catalog.
