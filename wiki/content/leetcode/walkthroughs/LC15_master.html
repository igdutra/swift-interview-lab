Given an integer array, find every unique triplet of elements whose values sum to zero. Return all such triplets; the answer must contain no duplicate triplets regardless of how many duplicate values appear in the input.

InputOutputWhy
[-1, 0, 1, 2, -1, -4][[-1,-1,2],[-1,0,1]]Two distinct triplets sum to zero; both -1s may appear together
[0, 1, 1][]No three elements sum to zero
[0, 0, 0][[0,0,0]]Three zeros — one unique triplet, not counted multiple times

The twist that makes it Medium: 2Sum is a sub-problem you already know, but the deduplication requirement adds a second dimension of correctness. A brute-force set-based approach works but costs O(n) extra space per outer iteration. The elegant solution eliminates all duplicates structurally — no set, no post-processing — by sorting once and skipping equal values in-place at three different points in the loop.

2   Reading the Problem — insights from the statement

"All unique triplets": the word "unique" is the deduplication contract. The input may contain duplicate values, but the output must not contain duplicate triplets. This is not just a correctness edge case — it is the central design challenge. A solution that ignores it will fail silently on inputs like [-2,0,0,2,2].
"Three elements": immediately suggests the 2Sum sub-problem. Fix one element as an anchor, then find two more in the remaining array that sum to −anchor. This reduces the O(n³) brute force to O(n²) with any 2Sum solver. The choice of 2Sum solver (hashmap vs converging pointers) is where the two approaches diverge.
"Not necessarily distinct" (implicit): the constraints allow −10⁵ ≤ nums[i] ≤ 10⁵ with no uniqueness guarantee. This means the same value may appear multiple times and must be counted independently — but the same triplet of values must appear only once in the output, regardless of how many ways those values can be selected from the array.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

n ≤ 3000 means O(n²) is your budget. What two-pass structure gives you exactly that with a 2Sum sub-problem?

Hint 2 — the pattern

Fix an anchor, then run 2Sum on the rest targeting −anchor. Now ask: which 2Sum variant avoids the O(n) extra space per iteration? What precondition does it require?

Hint 3 — deduplication

Sort the array. If nums[i] == nums[i−1], you've already found all triplets anchored at this value — skip. After recording a valid triplet, if the new left or right equals the pointer you just moved away from, skip those too. Three skip conditions, zero extra data structures.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can the same value appear in multiple triplets? For example, if there are two −1s, can they both appear in a single triplet?"
A: Yes — elements are identified by index, not value. → I cannot skip by value alone; I must skip by index or by sorted structure.
Q: "The output must contain no duplicate triplets — does that mean the same combination of values should appear at most once, even if multiple index combinations produce it?"
A: Exactly — value-level uniqueness on the output. → I need a deduplication strategy, either a Set on the output or structural skip conditions.
Q: "Can the input be empty or have fewer than three elements?"
A: The constraint says n ≥ 3. → No empty-input guard needed, but good to mention.
Q: "Are there any time or space preferences — is O(n²) time with O(1) extra space acceptable?"
A: Yes, that's ideal. → This confirms the two-pointer approach over the hashmap variant.
Step 2 — The plan, before any code
The plan, spoken

"This is a two-pointers problem — specifically the converging variant with a fixed anchor. The signal is: find all unique triplets summing to zero in an unsorted array. That 'all unique triplets' language collapses into 2Sum once I fix one element as an anchor and look for two more summing to its negative."

"Before naming data structures, let me pin the conditions. The validity condition is: three elements at distinct indices whose sum is exactly zero. The deduplication condition is: no two triplets in the output may share the same multiset of values — so [-1,0,1] and [1,0,-1] are the same triplet. The governing decision: I need to sort the array. Sorting gives me two things at once — it lets me use converging pointers instead of a hashmap for the inner 2Sum, and it groups duplicate values together so I can skip them structurally with a single comparison to the previous element."

"The three skip conditions follow directly from sorting. First, the outer skip: if the current anchor equals the previous anchor, I've already explored every triplet rooted there — continue. Second, the left skip: after recording a valid triplet and advancing the left pointer, if the new left value equals the old left value, it would generate an identical triplet with the same anchor and same right — advance again. Third, the right skip: same reasoning for the right pointer moving inward. These three conditions guarantee I never produce a duplicate — no Set, no post-processing."

"The walk: sort the array. For each anchor at index i, set left = i+1, right = n−1. Compute the sum. If the sum equals zero, record the triplet and advance both pointers, applying the two inner skip conditions. If the sum is too small, advance left. If too large, retreat right. An early exit fires if the anchor itself is positive — since the array is sorted, every subsequent sum will also be positive."

"I expect O(n log n) for the sort plus O(n²) for the nested loops — dominated by O(n²) total time, and O(1) auxiliary space beyond the output. I'll code it now and trace the canonical example to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: three nested loopsO(n³) time, O(output) spacen=3000 → ~27 billion iterations — dead on arrival; name it to show the baseline
Fix anchor + 2Sum hashmap (inner loop)O(n²) time, O(n) spaceCorrect, but allocates a Set per outer iteration for deduplication and a Set<Int> per inner iteration for pair lookup — two allocation sources per call
Fix anchor + converging two pointersO(n²) time, O(1) auxiliaryChosen. Sort once; structural skip conditions replace both Sets; the sorted order gives us direction (sum too large → shrink right, too small → grow left) — no storage needed
THE ONE IDEA: A sorted array gives you direction. When the two-pointer sum overshoots, only one move is rational — shrink the larger side. When it undershoots, only one move is rational — grow the smaller side. This deterministic steering is what makes the inner loop O(n) without any storage, and it is also what makes deduplication free: equal consecutive values are provably redundant, so skipping them is correct by construction, not by bookkeeping.
When to reach for converging two pointers over a hashmap: whenever the array can be sorted without breaking the problem, and the inner loop's goal is to find a pair summing to a target. The sort is O(n log n) — dominated by the O(n²) outer loop — and buys you O(1) space and structural deduplication for free.

7   Data Structures — choice and justification

A.2 Two Pointers (optimal)

StructureHoldsWhy this and not weaker
sorted: [Int]The input sorted ascendingEnables both converging pointers (direction is known) and structural deduplication (equals mean redundant)
left: Int, right: IntThe two inner pointersTwo ints replace an entire Set<Int> for pair lookup — direction replaces storage
validTriplets: [[Int]]Output accumulatorPlain array; no Set needed because skip conditions guarantee uniqueness structurally

A.1 Hashmap (reference approach)

StructureHoldsCost vs two pointers
possiblePairElements: Set<Int>Complement values seen so far in the inner loop — insert target − innerNumber, look up innerNumberAllocated and discarded once per outer iteration — O(n) total allocations
validTriplets: Set<[Int]>Sorted triplets, deduplicated by Set membershipEach insert requires sorting the triplet — O(3 log 3) per hit, plus hashing overhead
The deduplication contrast in one sentence: the hashmap approach deduplicates reactively — it generates duplicates and then removes them via a Set; the two-pointer approach deduplicates proactively — the three skip conditions prevent duplicates from ever being generated.

8   Pseudocode Dry Run

Pseudocodetwo-pointer approach — optimal
sorted ← SORT(nums)

FOR outerIndex FROM 0 TO n − 3:
    // OUTER SKIP: duplicate anchor → all its triplets already found
    IF outerIndex > 0 AND sorted[outerIndex] == sorted[outerIndex − 1]: CONTINUE
    // EARLY EXIT: anchor positive → every sum will be positive
    IF sorted[outerIndex] > 0: BREAK

    left ← outerIndex + 1
    right ← n − 1

    WHILE left < right:
        sum ← sorted[outerIndex] + sorted[left] + sorted[right]

        IF sum == 0:
            RECORD [sorted[outerIndex], sorted[left], sorted[right]]
            left += 1; right −= 1
            // LEFT SKIP: same left value → same triplet with same anchor and right
            WHILE left < right AND sorted[left] == sorted[left − 1]: left += 1
            // RIGHT SKIP: same right value → same triplet with same anchor and left
            WHILE left < right AND sorted[right] == sorted[right + 1]: right −= 1
        ELSE IF sum < 0: left += 1    // too small → need larger number
        ELSE: right −= 1              // too large → need smaller number

RETURN validTriplets

9   Swift Solutions

A.1 Hashmap — 2Sum inner loop  (understand this first)
O(n²)O(n)
Plan
For each outer element, run a 2Sum hashmap scan on the rest of the array targeting −outerNumber. Insert target − innerNumber into a seen-set; when innerNumber appears in the set, a valid pair exists. Sort each triplet before inserting into a Set<[Int]> — this collapses [-1,0,1] and [1,0,-1] to the same key. Reactive deduplication: duplicates are generated, then removed. 
func threeSumHashmap(_ nums: [Int]) -> [[Int]] {
    var validTriplets: Set<[Int]> = []

    for (outerIndex, outerNumber) in nums.enumerated() {
        let target = -outerNumber
        var possiblePairElements: Set<Int> = []

        for (innerIndex, innerNumber) in nums.enumerated() {
            guard innerIndex != outerIndex else { continue }

            if possiblePairElements.contains(innerNumber) {
                // ✦ DEDUPLICATION: sort before insert so [-1,0,1] and [1,0,-1]
                //   collapse to the same key in the Set.
                let triplet = [innerNumber, target - innerNumber, -target]
                validTriplets.insert(triplet.sorted())
            } else {
                // Insert complement: (target − innerNumber) + innerNumber = target
                possiblePairElements.insert(target - innerNumber)
            }
        }
    }

    return Array(validTriplets)
}
A.2 Two Pointers — converging with skip conditions  (optimal)
O(n²)O(1)
Plan
Sort the array. Fix an anchor; run converging left/right pointers over the remainder targeting −anchor. Three structural skip conditions prevent any duplicate from ever being generated: outer skip (same anchor), left skip (same left value after a match), right skip (same right value after a match). An early exit fires once the anchor turns positive. Proactive deduplication: duplicates are structurally impossible, so no Set is needed. 
func threeSumTwoPointers(_ nums: [Int]) -> [[Int]] {
    let sorted = nums.sorted()
    var validTriplets: [[Int]] = []

    for outerIndex in 0 ..< sorted.count - 2 {
        // OUTER SKIP: same anchor value was already processed —
        // every triplet it can form is a duplicate.
        if outerIndex > 0 && sorted[outerIndex] == sorted[outerIndex - 1] { continue }

        // EARLY EXIT: anchor > 0 → every sum will be positive, no solution possible.
        if sorted[outerIndex] > 0 { break }

        var left = outerIndex + 1
        var right = sorted.count - 1

        while left < right {
            let sum = sorted[outerIndex] + sorted[left] + sorted[right]

            if sum == 0 {
                validTriplets.append([sorted[outerIndex], sorted[left], sorted[right]])
                left += 1
                right -= 1
                // LEFT SKIP: new left equals the one we just used → identical triplet.
                while left < right && sorted[left] == sorted[left - 1] { left += 1 }
                // RIGHT SKIP: same reasoning for the right pointer.
                while left < right && sorted[right] == sorted[right + 1] { right -= 1 }
            } else if sum < 0 {
                left += 1   // sum too small → need a larger number
            } else {
                right -= 1  // sum too large → need a smaller number
            }
        }
    }

    return validTriplets
}

12   Complexity

A.1 — Hashmap approach

PhaseTimeSpaceWhy
Outer loopO(n)—Iterates over every element as anchor
Inner 2Sum scanO(n)O(n)Full scan per anchor; possiblePairElements grows up to n/2
Triplet sort + Set insertO(1)O(output)3-element sort is constant; Set<[Int]> holds at most output-count entries
TotalO(n²)O(n)The n-per-outer inner scan dominates; O(n) space from possiblePairElements re-allocated per outer iteration

A.2 — Two Pointers approach

PhaseTimeSpaceWhy
SortO(n log n)O(log n)Swift's Timsort; log n stack space for sort recursion
Outer loop × converging pointersO(n²)O(1)For each of n anchors, left and right together traverse at most n steps (each pointer moves inward, never outward)
TotalO(n²)O(1) auxSort is dominated by O(n²); O(1) auxiliary — no Set, no complement map, nothing beyond the two pointer variables and the output array itself
Why the sort's O(n log n) disappears: O(n log n) + O(n²) = O(n²). The sort is a one-time cost; the nested loop runs n × n / 2 times. At n = 3000, the sort does ~35,000 comparisons; the loop does ~4.5 million. The sort is invisible in the total.
The one real difference between A.1 and A.2: same asymptotic time, but A.1 allocates O(n) extra space per call (the two Sets) while A.2 allocates O(1). For n = 3000, A.1 creates and GCs thousands of Set entries; A.2 moves two integer pointers. On this problem the practical difference is moderate, but the pattern generalizes: whenever a sorted order is available, converging pointers dominate hashmaps in space.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The early-exit if sorted[outerIndex] > 0 { break } is already in the solution. A further constant-factor win: skip any anchor where sorted[outerIndex] + sorted[outerIndex+1] + sorted[outerIndex+2] > 0 (smallest possible triplet from this anchor is already too large) or sorted[outerIndex] + sorted[n-2] + sorted[n-1] < 0 (largest possible triplet from this anchor is still too small). Both are O(1) checks that can prune many outer iterations on skewed inputs.
Follow-up: "What if the target is not zero but an arbitrary integer k — 'find all triplets summing to k'?"
The algorithm is identical — replace the sum == 0 check with sum == k and the early-exit condition sorted[outerIndex] > 0 with sorted[outerIndex] > k (only valid when k > 0). The three skip conditions are unchanged. This is the general form; 3Sum is just the k = 0 special case.
Follow-up: "What about 4Sum — find all unique quadruplets summing to a target?"
Add one more outer loop: fix two anchors (i and j, where j > i), then run the same converging two-pointer inner loop on the remainder targeting k − nums[i] − nums[j]. Time becomes O(n³), space stays O(1). The same skip conditions apply at all three levels (outer i, middle j, and the two pointers). This generalizes to k-Sum in O(n^(k-1)) time recursively.
Follow-up: "What if the input array is already sorted and you must not sort it again — can you still avoid O(n) extra space?"
Yes — the two-pointer solution works unchanged on a pre-sorted array. The sort line becomes a no-op; the rest of the algorithm is identical. This is actually the best case: O(1) extra space and no sort overhead at all.
