Theory
lca_deep_dive_master.html

Given an integer array nums sorted in ascending order, build a height-balanced binary search tree from it. Height-balanced means that for every node, the heights of its left and right subtrees differ by at most one. Return the root of the tree. Many valid trees exist; any height-balanced one is accepted.

InputOutput (a valid tree)Why
[-10,-3,0,5,9]root 0, left subtree {−10,−3}, right {5,9}Middle element 0 is the root; halves recurse the same way → balanced
[1,3]root 3, left child 1 (or root 1, right child 3)Either midpoint choice is height-balanced for two elements
[]nilEmpty array → empty tree

The twist — and the reason this is the warm-up for the tree block — is the realization that "sorted array ↔ balanced BST" is the inverse of "in-order traversal of a BST is sorted." Pick the middle as the root and the two halves become the left and right subtrees; recurse. There is no LCA here yet, just the pure recurse-and-combine reflex every tree problem leans on.

2   Reading the Problem — insights from the constraints

"Sorted in ascending order": this is the gift that makes it easy. A BST's in-order traversal is sorted, so a sorted array is already the in-order layout of some BST — you just need to pick which element is the root at each level. The sortedness means no comparisons are needed to place values; position determines BST order.
"Height-balanced": this dictates which element is the root — the middle. Choosing the midpoint splits the remaining elements into two halves of nearly equal size, so the left and right subtrees have heights within one of each other. Any non-middle choice risks an unbalanced tree.
"Build a tree / return the root": the output is a tree structure, built by recursion that mirrors it. build(range) returns the subtree for an index range; the root's left and right come from build(left half) and build(right half). This is the recurse-and-combine skeleton.
"Many valid answers": with an even-length range there are two middles; either works. The grader checks for a height-balanced BST, not a specific one — so you don't need to agonize over (lo+hi)/2 vs (lo+hi+1)/2, only be consistent.
Recurse on index ranges, not array slices: passing (lo, hi) indices into the original array avoids allocating sub-arrays at every level, which would turn O(n) space of copies into O(n log n). Slice by indices, build over the shared array.

GOTCHA: recursing on overlapping rangesno duplicated/dropped nodeLeft half is [lo, mid-1], right is [mid+1, hi] — the root index is excluded from both.
The half-range boundaries are the only real trap. The root sits at mid; its left subtree must come from [lo, mid-1] and its right from [mid+1, hi]. Off-by-one here — including mid in a half, or skipping mid±1 — either duplicates the root or drops a neighbor. Excluding exactly the root index from both halves is what makes every element appear once.

4   Try It Yourself First

This is the tree block's warm-up — build the recurse-and-combine reflex. Try it cold, then open hints one at a time.

Hint 1 — the constraint

The array is sorted, and you need a balanced BST. A BST read in-order is sorted — so the array already is a valid in-order layout. Your only job is choosing the root at each level so the tree stays balanced. Which element splits the rest into two equal halves?

Hint 2 — the pattern

Recursive construction. The middle element is the root; everything to its left becomes the left subtree, everything to its right the right subtree. Write a helper build(lo, hi) that returns the subtree for that index range: pick mid, make a node, set its children to build(lo, mid-1) and build(mid+1, hi).

Hint 3 — boundaries and base case

Base case: when lo > hi, the range is empty → return nil. Recurse on [lo, mid-1] and [mid+1, hi] so the root index is excluded from both halves (no element used twice). Pass indices into the original array — don't copy sub-arrays.

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Is the array guaranteed sorted ascending, and can it be empty?"
A: Sorted ascending; can be empty. → Position encodes BST order; the empty case returns nil via the base case.
Q: "Does the tree need to be exactly balanced, or just height-balanced (subtree heights differ by ≤ 1)?"
A: Height-balanced. → Picking the middle as root at every level guarantees that automatically.
Q: "If the range has even length and two middles, is either acceptable?"
A: Yes, any valid height-balanced tree. → I'll consistently take the lower (or upper) middle; either is fine.
Q: "Can the values repeat, and do duplicates affect placement?"
A: They may repeat; placement is by position, not value. → I never compare values — sortedness already fixes order.
Step 2 — The plan, before any code
The plan, spoken

"This is recursive tree construction — Variant 8.1 of the Trees / LCA block, the warm-up that builds the recurse-and-combine reflex. The signal is 'build a balanced BST from sorted data,' and the key fact is that a BST's in-order traversal is sorted, so the array is already a valid in-order layout."

"Let me pin the conditions before the structure. The invariant each call maintains is: build(lo, hi) returns a height-balanced BST containing exactly the elements nums[lo...hi], in BST order. The governing choice that makes it balanced is picking the middle index as the root, because that splits the range into two halves whose sizes differ by at most one — so their subtree heights differ by at most one. The base case is the empty range, lo > hi, returning nil. The non-overlap condition: the left subtree is built from [lo, mid-1] and the right from [mid+1, hi], excluding the root so every element is placed exactly once."

"On structures: I work over index ranges into the original array, never copying slices — that keeps it O(n) space instead of O(n log n). I'll present the iterative form per our standard: an explicit stack of frames, each frame a (lo, hi) range plus where to attach the node it produces. The recursive version is identical in shape and I'll show it as a contrast."

"The walk: push the full range with a 'this is the root' attach target. Pop a frame, compute mid, create the node, attach it to its parent slot, and push frames for the two halves with attach targets pointing at the new node's left and right. Repeat until the stack empties. Time O(n) — each element becomes one node once; space O(log n) for the stack depth on a balanced tree."

6   Why This Approach — and not the others

CandidateCostVerdict
Insert elements one by one into a BSTO(n log n) best, O(n²) worstInserting sorted values naively produces a degenerate (linked-list) tree unless you balance — wrong shape and slow. Defeats the purpose.
Recurse but pass array slices (Array(nums[lo...mid-1]))O(n log n) time & spaceCorrect shape, but every slice copies — O(n) copying at each of log n levels. Index ranges avoid all of it.
Divide on index ranges, middle-as-root (iterative stack or recursion)O(n) / O(log n)Chosen. Each element becomes one node exactly once; the middle choice guarantees balance; no slice copies.
The deciding signal: "build a balanced structure from already-sorted data." Whenever sorted input must become a balanced tree (or you must binary-search-build anything), pick the middle as the pivot/root and recurse on the halves — the same divide-and-conquer that powers binary search, here producing a tree instead of finding an element.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
nums: [Int] (shared)The sorted source, accessed by index rangeWorking over (lo, hi) indices into the one array avoids slice copies — O(1) extra per level instead of O(n).
frame stack [(lo, hi, parent, isLeft)]Pending index ranges and where to attach the node they'll produceThe explicit stand-in for the recursion stack. Each frame says "build this range and hang the result on that parent's left/right." LIFO order reproduces the depth-first build.
TreeNode (class)A built node: value + left/right child referencesReference type so a parent frame can hold a pointer and a later frame can fill in its child slot.
a dummy parent node(setup) a holder whose left becomes the real rootLets the root be attached by the same "attach to parent's slot" rule as every other node — no special case for the root.
Why a frame carries an attach target. In recursion, the return value is wired into the caller automatically. The iterative build has no call to return to, so each frame must record where its produced node should be hung — a reference to the parent node and which side. A dummy root holder means even the very first node attaches by the same mechanism, eliminating a special case.

9   Swift Solution — iterative

The tree node is the standard LeetCode TreeNode class. The main solution is iterative (a frame stack); the recursive form follows as a contrast in §9b.

A.1 Iterative build — frame stack of index ranges
O(n)O(log n)
Plan
Push the full index range with an attach target on a dummy root. Pop each frame: skip empty ranges, else build a node from the middle element, hang it on its parent's slot, and push the two half-ranges pointing at this new node's left and right. When the stack empties, the dummy's left child is the balanced BST root. 
func sortedArrayToBST(_ nums: [Int]) -> TreeNode? {
    if nums.isEmpty { return nil }

    let dummy = TreeNode(0)            // dummy.left becomes the real root
    // Each frame: the index range to build, its parent, and which side to attach.
    var stack: [(low: Int, high: Int, parent: TreeNode, isLeft: Bool)] =
        [(0, nums.count - 1, dummy, true)]

    while let frame = stack.popLast() {
        if frame.low > frame.high { continue }   // empty range → no node
        let mid = (frame.low + frame.high) / 2
        let node = TreeNode(nums[mid])

        if frame.isLeft { frame.parent.left = node }
        else           { frame.parent.right = node }

        // Build the two halves; the root index is excluded from both.
        stack.append((frame.low, mid - 1, node, true))
        stack.append((mid + 1, frame.high, node, false))
    }

    return dummy.left
}

9b   Recursive form — for contrast

The same divide, expressed as recursion. The standard prefers the iterative version above; this clarifies the shape.

func sortedArrayToBST(_ nums: [Int]) -> TreeNode? {
    func build(_ low: Int, _ high: Int) -> TreeNode? {
        if low > high { return nil }       // empty range
        let mid = (low + high) / 2
        let node = TreeNode(nums[mid])
        node.left  = build(low, mid - 1)
        node.right = build(mid + 1, high)
        return node
    }
    return build(0, nums.count - 1)
}

12   Complexity

PhaseTimeSpaceWhy
BuildO(n)O(log n)Each array element is turned into exactly one node once; empty-range frames are O(1) and number O(n) total. The stack holds at most one root-to-leaf path of frames → height = O(log n) when balanced.
TotalO(n)O(log n)Linear, dominated by node creation. Space is the stack depth, not slice copies — working over index ranges is what keeps it O(log n) instead of O(n log n).
Why index ranges beat slices. Passing Array(nums[low...mid-1]) copies up to n elements, and across the log n levels that totals O(n log n) time and space in copies alone. Passing the two integers (low, high) is O(1) per frame, so the whole build stays O(n) time and O(log n) space. The output tree itself is O(n) nodes — irreducible.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The build is already O(n) time and O(log n) auxiliary space, which is optimal — you must create n nodes. The recursive version is arguably cleaner and just as efficient (its call stack is the same O(log n)); I'd choose it for readability unless stack-overflow on pathological depth is a concern, which it isn't for a balanced build. The one robustness note is asserting the input is actually sorted if that's not guaranteed.
Follow-up: "Now the input is a sorted linked list, not an array — LC 109."
You lose O(1) random access to the middle. Two options: copy the list into an array first (back to this exact problem), or use the elegant in-order simulation — count the length, then build the tree with an in-order recursion that consumes list nodes left-to-right, advancing a shared pointer. The second is O(n) time, O(log n) space, and avoids the array copy.
Follow-up: "Verify the result is actually a valid, height-balanced BST."
Two checks. BST validity: an in-order traversal must yield strictly increasing values (or non-decreasing if duplicates allowed). Balance: a post-order recursion returning each subtree's height, failing if any node's children differ by more than one. Both are O(n) single passes — useful as test oracles against the builder.
Follow-up: "How does this set up the rest of the tree block?"
It drills the recurse-and-combine reflex with no combine logic yet — just "build a subtree from each half." The next problem, LC 236 — LCA of a Binary Tree, keeps the same recursion shape but makes the combine step the whole point: each node combines what its two subtrees report to decide if it is the lowest common ancestor. Same skeleton, richer return value.
