The lowest common ancestor of two nodes p and q in a tree is the deepest node that has both p and q as descendants — where, by convention, a node is allowed to be a descendant of itself. Intuitively, it is the point where the root-to-p path and the root-to-q path last coincide before splitting apart. LCA is the canonical "combine results from two subtrees" tree problem and a staple of interview tree questions because it forces you to think in terms of what each subtree reports up to its parent.

Every problem in this family turns on one decision: do you have a way to walk up the tree, or only down? With child→parent pointers (or a parent map you build), you can collect one node's ancestors into a set and walk the other up until you hit it — an O(height) path intersection. Without them, you recurse top-down and combine bottom-up: each node asks its two subtrees "did you find p or q?", and the first node that hears "yes" from both sides (or is itself a target) is the LCA. That down-vs-up choice is the axis this page is built around. This deep dive lives under the Trees hub, which carries the general tree vocabulary.

2   Recognition Signals — what variant is this?

What you are given — a bare tree, parent pointers, or a hierarchy of regions — tells you which variant you are in.

The tell: the question asks where two nodes' paths to the root meet. The instant you see "common ancestor" or "smallest containing region," ask: can I walk up (parent pointers / a map → intersect ancestors) or only down (bare tree → post-order recursion)? That picks your variant.

4   Mental Model — let the answer bubble up

The one idea
In a bare tree, LCA is a post-order question: each node asks its left and right subtrees "did you contain p or q?" A node is the LCA exactly when it hears "yes" from both sides, or when the node itself is one target and the other lies below it. The recursion's return value carries "I found a target / the LCA" upward, and the split point is where the two finds meet.

Picture the tree below; we want LCA(4, 5). Each call returns non-nil if it found a target (or the LCA) in its subtree. The node where left and right both return non-nil is the answer:

DiagramLCA(4,5): the split node is the answer
            3
          /   \
         5      1
        / \    / \
       6   2  0   8
          / \
         7   4

find(4,5) at 3: left subtree (rooted 5) → non-nil, right (rooted 1) → nil → answer is on the left
find at 5: 5 IS a target → returns 5 immediately (4 is below it, so 5 is the LCA)
result: LCA(5,4) = 5     LCA(6,4) = 5     LCA(7,8) = 3

A node returns non-nil if it is a target or either subtree found one. The LCA is the shallowest node that gets non-nil from both directions — or a target that is itself an ancestor of the other.

Now the other side of the fork — when nodes have parent pointers, you do not need the whole tree at all. Walk one node up to the root, recording every ancestor in a set; then walk the other up until you hit a node already in the set. That first hit is the LCA, exactly like finding where two linked lists merge:

Traceparent-pointer walk-up: first shared ancestor wins
p = 7, q = 8  (using the tree above, with parent pointers)

walk p=7 up, recording: { 7, 2, 5, 3 }        (7→2→5→3→root)
walk q=8 up, checking each against the set:
   8 in set? no
   1 in set? no
   3 in set? YES → LCA = 3

With upward links the tree shrinks to two paths that must intersect at the root at the latest — the first common node is the LCA.

Recognizing whether you can move up or only down is the whole decision: parent links → set-intersection walk-up; bare tree → post-order recursion (or build a parent map first to enable the walk-up).

5   Pseudocode Primer

The post-order LCA on a bare tree is this shape. The return value is the message a subtree sends its parent.

Pseudocodepost-order LCA — the answer bubbles up
FUNCTION lca(node, p, q):
    IF node is null OR node == p OR node == q:
        RETURN node            found a target (or hit the bottom)

    left  ← lca(node.left,  p, q)
    right ← lca(node.right, p, q)

    IF left ≠ null AND right ≠ null:
        RETURN node            p and q split here → this node is the LCA
    RETURN left ≠ null ? left : right   both finds (if any) are on one side
Why returning the node on a match works: if a node equals p or q, it returns itself without recursing further — so if the other target is below it, this node legitimately becomes the LCA (a node is its own ancestor). If the two targets are in different subtrees, the first node whose left and right both come back non-null is, by definition, the deepest node that has both below it. The recursion never needs an explicit "found both" flag — the two non-null returns are the flag.

6   Iterative-Only Pattern — explicit stack and a parent map

Tree problems read naturally as recursion, but the standard's iterative discipline pays off here too, and the iterative LCA is genuinely instructive:

  • Turn the recursion into an explicit parent map. Do one traversal (DFS with a stack, or BFS with a queue) recording each node's parent in a dictionary, stopping once both p and q are seen. Now you have parent pointers even on a bare tree — walk p up to the root into a set, then walk q up until you hit the set. This is the iterative twin of the post-order recursion and the exact technique the §8.4 region problem uses.
  • For construction (LC 108), the explicit work is an index-range stack. Building a balanced BST is "pick the middle as root, recurse on halves." The iterative form pushes (lo, hi, parent-slot) frames onto a stack instead of recursing — the same divide, made explicit.
Why this matters here specifically: the parent-map traversal is not just an anti-recursion exercise — it is the bridge between the two halves of the fork. It converts a down-only tree into an up-walkable one, unifying the bare-tree case (§8.2) with the parent-pointer case (§8.3) and the region-hierarchy case (§8.4). Master the parent map and three of the four variants become the same walk-up.

7   Swift Foundations

Four Swift facts underlie nearly every tree problem in this family. Say them out loud before coding.

Swiftthe four tree-handling rules for this block
// 1. The node is a class (reference type) with optional children.
class TreeNode {
    var val: Int; var left: TreeNode?; var right: TreeNode?
    init(_ val: Int) { self.val = val }
}

// 2. The base case is the nil child — every recursion bottoms out there.
guard let node = node else { return nil }

// 3. Identity (===) compares references — use it to match "this is node p".
if node === p || node === q { return node }

// 4. A parent map / ancestor set keyed by ObjectIdentifier (nodes aren't Hashable).
var parent: [ObjectIdentifier: TreeNode] = [:]
var ancestors: Set<ObjectIdentifier> = []
ancestors.insert(ObjectIdentifier(node))   // stable identity for a reference
The identity trap. Tree nodes are reference types and often not Hashable or comparable by value (values may repeat). Compare nodes with === (reference identity), not ==, and key sets/maps by ObjectIdentifier(node). Comparing by .val breaks the instant the tree has duplicate values.

8   The Four Variants

Ordered by increasing difficulty. The first two are taught in full walkthroughs; the last two are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Recursive tree construction
O(n)O(log n)
When to use: the warm-up that builds the "recurse and combine" reflex without LCA yet — construct a tree by recursively choosing a root and building its subtrees. Full Swift in the walkthrough (LC108).

The frame (LC 108 — Convert Sorted Array to BST): a sorted array maps to a height-balanced BST by picking the middle element as the root (so equal counts go left and right), then recursively building the left subtree from the left half and the right subtree from the right half. The middle-as-root choice is what guarantees balance.

The design decision: because in-order traversal of a BST is sorted, the inverse holds — a sorted array is the in-order reading of some BST, and choosing the midpoint as root at every level keeps the two halves equal, hence balanced. The recursion mirrors the divide: build(lo, hi) returns the subtree for that index range.

Canonical problem: LC 108 — Convert Sorted Array to BST (warm-up). Full walkthrough.

Variant 8.2 Post-order LCA (bare tree)
O(n)O(h)
When to use: you have only the tree (no parent pointers) and must find the LCA of two nodes. Recurse, and let the answer bubble up. Full Swift in the walkthrough (LC236).

The frame (LC 236 — LCA of a Binary Tree): recurse on left and right. Base case: a null node, or a node equal to p or q, returns itself. If both the left and right calls return non-null, the current node is the split point — return it as the LCA. Otherwise return whichever side was non-null (the side that contains both targets, or the single target found so far).

The design decision: the return value doubles as the message "I found a target / the LCA below me." No separate found-flags or path lists are needed — two non-null subtree returns at the same node is the LCA condition, and a target returning itself handles the ancestor-of-each-other case for free.

Canonical problem: LC 236 — LCA of a Binary Tree (walkthrough). Full walkthrough.

Variant 8.3 Ancestor-set intersection (parent pointers)
O(h)O(h)
Solve-it-yourself variant. This is an attempt problem — the frame and the walk-up rule are below, but no solution. Build it cold.
When to use: each node carries a pointer to its parent, so you can walk up without the whole tree. The LCA is the first ancestor shared by both nodes.

The frame (LC 1650 — LCA of a Binary Tree III): two clean approaches:

  • Set walk-up: walk p up to the root inserting each node into a set; then walk q up, returning the first node already in the set.
  • Two-pointer (O(1) space): like linked-list intersection — advance two pointers up; when one hits the root, redirect it to the other's start. They meet at the LCA after at most two passes.
The frame: upward links turn the tree into two paths to the root that must intersect. The set version is O(h) space and obvious; the two-pointer version is O(1) space and reuses the exact "redirect to the other head" trick from intersecting linked lists. See pitfalls §10.

Canonical problem: LC 1650 — LCA III (attempt). Graduated hints in the walkthrough.

Variant 8.4 LCA via parent-map paths (region hierarchy)
O(n)O(n)
Solve-it-yourself variant. This is a target problem — the model and the path-intersection rule are below, but no code. This is the canonical Airbnb LCA framing.
When to use: the "tree" is given as a hierarchy of named regions (each region lists its sub-regions), and you must find the smallest region containing two given regions — LCA in disguise.

The model (LC 1257 — Smallest Common Region): build a child→parent map from the region lists. Walk one region up to the root, collecting its full ancestor chain into a set (it includes itself). Then walk the other region up until you reach a region in that set — the first such is the smallest common region.

The frame: this is exactly the parent-map walk-up of §8.3, but the parent links are derived from the region-membership lists rather than given. Once the map exists, "smallest common region" is "lowest common ancestor." Recognizing the disguise is the whole problem. See pitfalls §10 and edge cases §11.

Canonical problem: LC 1257 — Smallest Common Region (target). Graduated hints in the walkthrough.

9   Complexity Analysis

The bare-tree LCA is O(n) because it may visit every node once; the parent-pointer versions are O(h) because they only walk two root-paths. Height h is O(log n) for a balanced tree but O(n) in the worst (degenerate) case.

VariantTimeSpaceWhy
Recursive construction (8.1)O(n)O(log n)Each array element becomes one node exactly once; recursion depth is the tree height (balanced → log n).
Post-order LCA (8.2)O(n)O(h)One traversal visits each node once; the call stack is as deep as the tree's height.
Ancestor-set intersection (8.3)O(h)O(h)Walks two root-paths of length ≤ h; the set holds one path. The two-pointer variant drops space to O(1).
Parent-map paths (8.4)O(n)O(n)One pass builds the parent map over all regions; the walk-up is O(h) on top.
BottleneckO(n)—A degenerate (linked-list-shaped) tree makes h = n, so even the "O(h)" walk-ups become O(n); balance is what keeps tree algorithms logarithmic.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Comparing nodes by value (all variants). Trees can hold duplicate values; matching node.val == p.val finds the wrong node. Use reference identity === and key sets by ObjectIdentifier.
▸Forgetting a node is its own ancestor (8.2). If p is an ancestor of q, the LCA is p. Returning the node when it equals p or q (before recursing deeper) handles this; a solution that requires finds in both subtrees misses it.
▸Off-by-one picking the middle (8.1). mid = (lo + hi) / 2 vs (lo + hi + 1) / 2 shifts which element is root; either is valid but be consistent, and recurse on [lo, mid-1] and [mid+1, hi] so the root is excluded from both halves.
▸Not stopping the parent-map traversal early (8.3/8.4). You only need parents up to both targets; for huge trees, stop the build once both are recorded rather than mapping the entire tree.
▸Empty intersection assumed impossible (8.4). In a valid LCA problem the paths always meet at the root at worst — but if regions can be disjoint forests, guard for "no common ancestor." Confirm the tree is connected with one root.
▸Including/excluding self inconsistently in the ancestor set (8.3/8.4). The walk-up set must include the starting node itself, or LCA(p, q) where one is the other's ancestor will be missed.

Swiftcopy-paste post-order LCA skeleton
func lowestCommonAncestor(_ root: TreeNode?, _ p: TreeNode?, _ q: TreeNode?) -> TreeNode? {
    guard let node = root else { return nil }
    if node === p || node === q { return node }
    let left  = lowestCommonAncestor(node.left,  p, q)
    let right = lowestCommonAncestor(node.right, p, q)
    if left != nil && right != nil { return node }  // split → LCA
    return left ?? right
}

LC 1257 — Smallest Common Region (Medium). Given a list of regions where each entry is [parent, child1, child2, …], and two region names, return the smallest region that contains both. Recognize this as LCA, decide your representation, and build the walk-up before writing any code.

  • What hidden tree is this — and what becomes a node, what becomes a parent link?
