Intervals

Interval problems operate on ranges defined by a start and an end, and ask you to combine, compare, or query them. The canonical technique is to sort intervals by their start value once, then sweep through them linearly — keeping only the last merged interval in hand — so the entire sweep is a single O(n) pass with no backtracking and no nested loops.

Sorting unlocks a key invariant: after sorting by start, an incoming interval can only overlap the most recently merged interval, never an earlier one. This reduces what would otherwise be an O(n²) pairwise comparison to a single comparison per interval. The same sort-then-sweep backbone powers all four core variants — merge, overlap walk, insert, and gap-find — and a fifth, more advanced shape (split/handoff coverage) reduces a coverage problem to prefix and suffix precomputation.

2   Recognition Signals — when to reach for it

Interval problems rarely announce themselves by name. They say one of these:

When the problem says…It means…
"merge all overlapping intervals"Variant 8.1 — sort by start, sweep and fold overlapping intervals into the last merged one
"intersection of two interval lists"Variant 8.2 — two-pointer walk; advance whichever pointer has the smaller end
"insert this interval into a sorted list"Variant 8.3 — three-region split: before the new interval / overlapping / after
"find the free time" / "find gaps between schedules"Variant 8.4 — flatten all intervals, merge (8.1), then return the gaps between merged intervals
"can two teams / segments cover the full window?"Variant 8.5 — precompute how far each team covers from the left (prefixEnd) and from the right (suffixStart); check pairs in O(1)
"does point X lie in any interval?"Not an interval sweep — a hash set or binary search on sorted endpoints is enough
The disqualifier: if the problem asks about isolated point membership with no notion of ranges overlapping or covering each other, the interval machinery is overkill. Reach for a hash set or a sort + binary search instead.

4   Mental Model — sort once, never look back

The one idea
Sort by start. Then sweep left to right, holding only the last merged interval in hand. Each incoming interval either extends that last interval (overlap → merge) or starts a new one (no overlap → append). You never need to look backward — sorting guarantees that.

Watch the sweep merge [[1,3],[2,6],[8,10],[15,18]] into its minimal form:

Tracesorted by start; hold last merged interval at each step
start: result = []

interval [1,3]  result is empty → append
             result = [[1,3]]

interval [2,6]  last=[1,3], incoming start 2 ≤ last end 3 → overlap
             merge: last end = max(3,6) = 6
             result = [[1,6]]

interval [8,10] last=[1,6], incoming start 8 > last end 6 → no overlap
             append new interval
             result = [[1,6],[8,10]]

interval [15,18] last=[8,10], incoming start 15 > last end 10 → no overlap
             append new interval
             result = [[1,6],[8,10],[15,18]]  done — 4 intervals, 3 comparisons, 1 pass

Each interval is examined exactly once. The sort is the only expensive step; the sweep is O(n).

5   Pseudocode Primer

Every sort-then-sweep interval solution is this loop. Variants differ in what they do with overlaps and gaps.

Pseudocodecanonical sort + sweep merge
FUNCTION mergeIntervals(intervals):
    IF intervals is empty: RETURN []

    sorted ← sort intervals by start
    result ← [sorted[0]]              seed with the first interval

    FOR each interval IN sorted[1..]:
        last ← result[last index]

        IF interval.start ≤ last.end:   overlap — extend
            last.end ← max(last.end, interval.end)
        ELSE:                            gap — start fresh
            result.append(interval)

    RETURN result
Why it works: after sorting, any interval that overlaps the current last merged interval must start ≤ last.end. An interval starting after last.end cannot overlap anything already in result — so appending is always safe. This is the invariant the sort buys.

6   Iterative-Only Pattern — no recursion needed

Interval sweeps are inherently iterative. The algorithm is one sort followed by one for loop with a single comparison per step. There is no recursive formulation worth writing:

  • Recursion adds nothing. A recursive merge would require passing the sorted array and an index through call frames — identical logic with call-stack overhead and a risk of stack overflow at n = 10⁵.
  • The invariant is visible. "result.last always holds a valid merged interval" is a one-line comment in iterative code; in recursion it disappears into return values.
  • Variants compose cleanly. The flatten-then-merge shape (8.4) just prepends a collect step before the same loop. Recursion would require restructuring the entire call graph for each variant.

7   Swift Translation

Swiftannotated merge-intervals template — compilable starting point
func merge(_ intervals: [[Int]]) -> [[Int]] {
    guard !intervals.isEmpty else { return [] }

    // Sort by start value — the single most important line.
    let sorted = intervals.sorted { $0[0] < $1[0] }
    var result = [sorted[0]]

    for interval in sorted.dropFirst() {
        // result is never empty here — we seeded it above.
        let lastEnd = result[result.count - 1][1]

        if interval[0] <= lastEnd {
            // Overlap: extend the last interval's end if needed.
            result[result.count - 1][1] = max(lastEnd, interval[1])
        } else {
            // Gap: the new interval starts fresh.
            result.append(interval)
        }
    }
    return result
}
Swift-specific costs to say out loud: sorted { $0[0] < $1[0] } allocates a new array — if the input is guaranteed sorted, skip it. Mutating result[result.count - 1][1] in place is O(1); never removeLast() and re-append, which is two allocations. dropFirst() is a lazy slice — O(1), not O(n).

8   Variant Catalog

Variant 8.1 Sort + sweep merge
O(n log n)O(n)
When to use: "merge all overlapping intervals." The input is unsorted; you own the sort step. After sorting by start, maintain one running last interval and fold each incoming interval into it if it overlaps, or append it if it doesn't.
sorted ← sort by start
result ← [sorted[0]]
FOR each interval:
    IF interval.start ≤ result.last.end:
        result.last.end ← max(result.last.end, interval.end)
    ELSE:
        result.append(interval)

Canonical problem: LC 56 — Merge Intervals.

Variant 8.2 Two-pointer interval overlap
O(n log n)O(n)
When to use: "find the intersection of two interval lists." You have two independently sorted lists A and B. Walk them with two pointers; record the overlap if one exists; advance whichever pointer has the smaller end (that interval is exhausted and can't overlap anything later in the other list).
i ← 0, j ← 0
WHILE i < A.length AND j < B.length:
    overlapStart ← max(A[i].start, B[j].start)
    overlapEnd   ← min(A[i].end,   B[j].end)
    IF overlapStart ≤ overlapEnd:
        result.append([overlapStart, overlapEnd])
    IF A[i].end < B[j].end: i += 1
    ELSE:                    j += 1
Why advance the smaller end? The interval with the smaller end is fully covered by the current window. Any future interval in either list starts at or after the current position — so this interval cannot contribute another overlap. Advancing it is the only safe move.

Canonical problem: LC 986 — Interval List Intersections.

Variant 8.3 Insert interval into sorted list
O(n)O(n)
When to use: "insert a new interval into an already-sorted list and merge any overlaps." The list is pre-sorted, so no re-sort is needed — O(n). The key insight is that the existing intervals split cleanly into three non-overlapping regions relative to the new interval.
result ← []
i ← 0

Region 1 — entirely before new interval (end < newStart)
WHILE i < n AND intervals[i].end < newInterval.start:
    result.append(intervals[i]); i += 1

Region 2 — overlapping (start ≤ newEnd): merge them all into newInterval
WHILE i < n AND intervals[i].start ≤ newInterval.end:
    newInterval.start ← min(newInterval.start, intervals[i].start)
    newInterval.end   ← max(newInterval.end,   intervals[i].end)
    i += 1
result.append(newInterval)

Region 3 — entirely after (start > newEnd)
WHILE i < n:
    result.append(intervals[i]); i += 1
Don't forget Region 3. The tail of the list (intervals entirely after the new one) must be appended — it's the most frequently dropped piece in a rushed implementation.

Canonical problem: LC 57 — Insert Interval.

Variant 8.4 Flatten + merge + find gaps
O(n log n)O(n)
When to use: "given several schedules (each a list of intervals), find all time slots where nobody is working." The input is nested — multiple lists of intervals. Flatten all of them into one list, apply the 8.1 merge, then walk the merged result and collect the gaps between consecutive intervals.
all ← flatten all interval lists into one array
merged ← mergeIntervals(all)          ← Variant 8.1

gaps ← []
FOR i FROM 1 TO merged.length − 1:
    gapStart ← merged[i−1].end + 1
    gapEnd   ← merged[i].start − 1
    IF gapStart ≤ gapEnd:
        gaps.append([gapStart, gapEnd])
The insight: "free time" is just the complement of the merged union. Once you have the merged union, finding the gaps is a single additional pass — no new algorithmic work.

Canonical problem: LC 759 — Employee Free Time.

Variant 8.5 Split/handoff coverage — prefix-suffix precompute
O(t·d)O(t)
When to use: "can exactly two teams (or segments) together cover the full window [startDay, endDay] with no gap?" The teams don't provide interval objects — they provide raw day sets. A naïve O(t²·d) brute-force tries every split point for every pair. Precomputing prefix and suffix coverage reduces pair-checking to O(1) per pair.

Two precomputed values per team:

  • prefixEnd[team] — the last day the team covers contiguously starting from startDay. Walk the team's sorted days; stop the moment a gap appears. If the team doesn't cover startDay at all, prefixEnd is undefined (skip this team as a first segment).
  • suffixStart[team] — the first day the team covers contiguously ending at endDay. Walk backward from endDay; stop the moment a gap appears. If the team doesn't cover endDay, suffixStart is undefined (skip as a second segment).

Valid pair condition: pair (A, B) is valid if and only if:

suffixStart[B] <= prefixEnd[A] + 1

B's contiguous-to-endDay reach starts at or before the handoff point — meaning B already covers day prefixEnd[A] + 1 and everything through endDay contiguously. Note: the condition is <=, not ==. A team may cover a wider suffix than strictly needed (e.g. B covers days 3–8, so it can serve as the second team for any A that hands off on day 3, 4, 5, 6, or 7).

Pseudocodeprefix-suffix precompute + pair lookup
Step 1 — precompute for each team
FOR each team:
    prefixEnd[team]   ← walk startDay→endDay; stop at first gap; nil if startDay not covered
    suffixStart[team] ← walk endDay→startDay; stop at first gap; nil if endDay not covered

Step 2 — build lookup: suffixStart value → list of team names
suffixMap ← Dictionary(suffixStart value → [team names])

Step 3 — for each valid first team, collect matching second teams
FOR each team A where prefixEnd[A] is defined:
    handoffPoint ← prefixEnd[A] + 1
    candidates ← all teams B where suffixStart[B] ≤ handoffPoint
    ↑ B's suffix covers from suffixStart[B] to endDay — if suffixStart[B] ≤ handoffPoint,
      B covers [handoffPoint…endDay] contiguously
    FOR each B IN candidates where B ≠ A:
        result.append((A, B))
Why prefix + suffix, not just sweep? The sweep variants (8.1–8.4) work on interval objects with explicit start/end fields. Here the teams provide raw day sets, and the "interval" — the contiguous segment each team can anchor — must be derived. The prefix/suffix precompute is that derivation: it extracts the only interval that matters for each team's role (left anchor or right anchor) before any pair comparison happens.

Motivating problem: Relay Handoff Coverage (custom) — given teams with availability day sets, find all ordered pairs (A, B) that together cover [startDay, endDay] contiguously. See CUSTOM_Relay_Handoff_Coverage_master.html for the brute-force and optimal Swift implementations.

9   Complexity Analysis

Why the sort is the irreducible bottleneck. The sweep itself is O(n) — each interval is examined exactly once and either merged or appended. But before the sweep can run, the input must be sorted, which costs O(n log n) by comparison sort. No interval sweep algorithm can avoid this: without sorted order, the "only check the last merged interval" invariant breaks and you'd need O(n) comparisons per step.

VariantTimeSpaceWhy
8.1 Sort + sweep mergeO(n log n)O(n)Sort dominates; one comparison per interval in the sweep; output holds up to n intervals.
8.2 Two-pointer overlapO(n log n)O(n)Sort each list independently; two-pointer walk is O(n+m); output is at most min(n,m) intersections.
8.3 Insert into sortedO(n)O(n)Input is pre-sorted — no sort step. Single pass through three regions. This is the one variant that beats O(n log n).
8.4 Flatten + find gapsO(n log n)O(n)Flattening is O(n); merge is O(n log n); gap pass is O(n). Sort dominates.
8.5 Split/handoffO(t·d)O(t)t = number of teams, d = window length. Precompute is O(t·d); pair check with suffixMap is O(t). No sort needed — day sets are walked linearly from the anchored end.
BottleneckO(n²)—Any O(n) operation inside the sweep loop (e.g. checking all past intervals for overlap instead of just the last one) destroys the O(n) sweep guarantee and returns you to brute force.

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Forgetting to sort before sweeping. If the input is unsorted and you skip the sort, an out-of-order interval can fail the overlap check against the last merged interval even though it overlaps an earlier one. The output silently has too many intervals, not an obvious crash.
▸Wrong overlap condition: < instead of <=. Adjacent intervals like [1,3] and [3,5] share the point 3 — they should be merged. Using interval.start < lastEnd instead of <= lastEnd leaves adjacent intervals unmerged and produces wrong output on any test with touching endpoints.
▸Dropping the tail in the insert variant (8.3). After merging the overlapping region, the loop exits — but the remaining intervals (Region 3, entirely after the new interval) still need to be appended. Forgetting this tail truncates the output list.
▸Mutating while iterating. In the merge sweep, never remove the last element and re-append a merged version — that is two allocations. Mutate in place: result[result.count - 1][1] = max(...).
▸Split/handoff direction confusion. The pair (A, B) means A covers the left segment and B covers the right — order matters. prefixEnd[A] + 1 == suffixStart[B] is not symmetric: swapping A and B tests a different (and usually wrong) split point.
▸Split/handoff: not guarding startDay == endDay. A single-day mission window cannot be split between two teams by definition — both teams would need to cover the same single day, but the problem requires a contiguous handoff. Guard this case explicitly and return empty.
