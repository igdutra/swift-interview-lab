Object & System Design

Theory Hub · Stateful Systems · Iterators

Design problems hand you an API to implement, not a single function to write: a class with a constructor and a set of methods that are called in sequence, each one reading or mutating internal state. There is rarely an algorithmic trick — the difficulty is choosing the internal representation so that every method is correct and cheap, and keeping that state consistent across an unknown stream of calls. The whole game is "what do I store, and what do I derive?"

These problems split along one axis: are you mutating and querying evolving state, or are you exposing a traversal? A stateful system (a file system, a scoreboard, a bank) holds state that methods change and read — the art is picking the data structure and deciding which answers to precompute versus recompute. An iterator exposes a hasNext/next contract over a structure — the art is advancing a cursor lazily so each call is amortized O(1). This hub teaches you to name which kind you are designing, because that choice picks the deep dive and, with it, the internal state. The Stateful System Design deep dive covers mutate-and-query objects; the Iterator Design deep dive covers traversal contracts.
