Iterator design problems ask you to implement a class that exposes a traversal: a next() that returns the following element and a hasNext() that reports whether one remains. Unlike a stateful system, there is no rich set of mutators — there is a contract, and your object must honor it across any interleaving of hasNext and next calls. The whole game is keeping a cursor — a position, a stack, or a small buffer — that always knows where the next element is without re-walking what you have already produced.

Every problem in this family turns on one decision: how much do you advance eagerly versus lazily? The discipline is to do the minimum work per call and to keep hasNext and next consistent — hasNext must never claim an element that next can't deliver, and calling hasNext repeatedly must not consume anything. That balance — advance just in time, peek without consuming — is the axis this page is built around. This deep dive lives under the Object & System Design hub, alongside stateful system design; both are "design a class with a clean API," and iterators all lean on the stack mechanics the warm-up establishes.

4   Mental Model — advance just in time

The one idea
An iterator is a paused traversal. Keep exactly the state needed to resume — a cursor — and on each next() do the smallest amount of work to produce one element and re-pause. The eternal trap is hasNext/next consistency: make one of them do the advancing and the other merely report, so calling hasNext ten times never skips an element and next never returns something hasNext denied.

The family's centerpiece is flattening nested data lazily. Picture [[1,1],2,[1,3]] exposed as the stream 1,1,2,1,3. The clean design pushes the top-level list onto a stack in reverse so the front element is on top, and defers descending into sub-lists until hasNext needs to. Watch hasNext do the unwinding — popping a sub-list and pushing its reversed contents — until an integer sits on top:

Tracelazy flatten: hasNext unwinds lists until an integer is on top
input [[1,1],2,[1,3]]   stack (top → right): [ [1,3], 2, [1,1] ]   (pushed reversed)

next() → hasNext: top is list [1,1] → pop, push reversed → [ [1,3], 2, 1, 1 ]
                  top is int 1 → return 1, pop → [ [1,3], 2, 1 ]
next() → hasNext: top is int 1 → return 1, pop → [ [1,3], 2 ]
next() → hasNext: top is int 2 → return 2, pop → [ [1,3] ]
next() → hasNext: top is list [1,3] → pop, push reversed → [ 3, 1 ]
                  top is int 1 → return 1, pop → [ 3 ]
next() → hasNext: top is int 3 → return 3, pop → [ ]
hasNext: stack empty → false

No descent happens until it is needed. hasNext guarantees an integer is on top; next simply pops it. Each element is pushed and popped once → amortized O(1).

The simpler members are pure cursors. A two-level (jagged) structure like [[1,2],[],[3]] needs only an (outer, inner) pair that knows to skip empty inner lists:

Tracetwo-level cursor: advance inner, roll over outer, skip empties
rows=[[1,2],[],[3]]   outer=0  inner=0   (positioned at first valid element)

next() → return rows[0][0]=1 ; inner→1
next() → return rows[0][1]=2 ; inner→2 → past row 0 end → outer→1,inner→0
         row 1 is empty → skip → outer→2,inner→0
hasNext → outer(2) < 3 and rows[2] has element → true
next() → return rows[2][0]=3 ; inner→1 → past end → outer→3
hasNext → outer(3) == 3 → false

The only subtlety is the "skip empty rows" advance — folded into one helper both methods call before they trust the cursor.

Recognizing whether your next element is one cursor-step away or buried under nesting is what picks between a plain index and an explicit stack.

5   Pseudocode Primer

Every iterator is this shape: a constructor that sets up the cursor positioned at the first element, a hasNext that reports (and, in the lazy variants, prepares) the next element, and a next that returns it and advances. The ADVANCE line is the only part that varies.

Pseudocodethe universal iterator shape
CLASS Iterator:
    INIT(source):
        cursor ← state positioned at / before the first element
        (stack, indices, or a cached lookahead)

    METHOD hasNext() -> Bool:
        PREPARE: advance the cursor past anything empty/nested
                 until it points at a real element (lazy variants)
        RETURN cursor still has an element

    METHOD next() -> Element:
        IF not hasNext(): error / undefined
        element ← element at cursor
        ADVANCE cursor past it
        RETURN element
Why one method owns the advancing: the single most common iterator bug is hasNext and next disagreeing. The fix is to make the "skip to the next real element" logic live in one place — usually called from hasNext (which is idempotent) and relied on by next. Then repeated hasNext calls converge to the same state, and next can trust that an element is ready.

6   Iterative-Only Pattern — explicit stack, never recursion

Iterators are fundamentally incompatible with recursion, and that is the lesson:

  • A recursive traversal runs to completion; an iterator must pause. You cannot suspend a recursive descent halfway and return control to the caller between two elements. So the recursion must be turned inside out into an explicit stack of frames that you can stop and resume across next calls. The stack is the paused call stack, made into data you own.
  • Prepare lazily, in hasNext. Do not flatten the entire structure in the constructor — that defeats the point and costs O(total) up front. Push only the top level, and unwind nesting one layer at a time when a call actually needs the next element. This keeps the constructor O(1)–O(top-level) and the work spread across the calls that consume it.
Why this matters here specifically: the bugs in this family are consistency and laziness bugs — eagerly flattening (wrong complexity, and breaks if the structure is huge or infinite), or advancing in both methods (skips elements). An explicit stack prepared on demand in a single place makes both impossible by construction.

7   Swift Foundations

Four Swift facts underlie nearly every iterator in this family. Say them out loud before coding.

Swiftthe four iterator-handling rules for this block
// 1. An array used as a stack: append / removeLast are O(1) amortized.
var stack: [Int] = []
stack.append(1)                  // push
let top = stack.removeLast()    // pop (precondition: not empty)

// 2. Push children REVERSED so the front element ends up on top.
for element in list.reversed() { stack.append(element) }

// 3. Peek without popping — read .last, don't remove.
if let top = stack.last { /* inspect, decide */ }

// 4. Cache one element ahead for peek()/lookahead wrappers.
private var cached: Int?       // the next element, fetched eagerly once
func peek() -> Int { cached! }   // return without consuming
The removeFirst trap. Using an array as a queue with removeFirst() is O(n) per call — it shifts every remaining element. For LIFO traversal (the natural fit for nested descent) use removeLast(), which is O(1). If you genuinely need FIFO, reverse once and pop from the back, or keep a head index — never removeFirst in a loop.

8   The Four Variants

Ordered by increasing design difficulty. The first two are taught in full walkthroughs; the last two are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Stack fundamentals
O(n)O(n)
When to use: the warm-up that builds the stack mechanics every iterator leans on — push on one signal, pop-and-check on another, and the structure is valid iff the stack ends empty. Full Swift in the walkthrough (LC20).

The frame (LC 20 — Valid Parentheses): scan the string; push each opening bracket; on a closing bracket, the top of the stack must be its matching opener — pop it, or fail. The string is valid iff every closer matched and the stack is empty at the end (no unclosed openers).

The design decision: a stack is the right structure because the most recently opened bracket must close first — last-in-first-out is exactly nesting. This LIFO discipline is the same one the flatten iterator uses to descend and resume, which is why this is the family's warm-up.

Canonical problem: LC 20 — Valid Parentheses (warm-up). Full walkthrough.

Variant 8.2 Lazy flattening with a stack
amortized O(1)/callO(depth)
When to use: expose the integers of a nested list as one flat stream via hasNext/next, descending into sub-lists only when asked. Full Swift in the walkthrough (LC341).

The frame (LC 341 — Flatten Nested List Iterator): push the top-level list onto a stack in reverse (front element on top). In hasNext, while the top is a list, pop it and push its contents reversed; stop when the top is an integer (or the stack empties). next calls hasNext to ensure an integer is on top, then pops and returns it.

The design decision: push reversed so popping yields front-to-back order, and put the descent logic in hasNext so it is idempotent — calling it repeatedly leaves an integer on top without consuming. Each element is pushed and popped exactly once across the whole traversal, so per-call cost is amortized O(1).

Canonical problem: LC 341 — Flatten Nested List Iterator (walkthrough). Full walkthrough.

Variant 8.3 Lookahead wrapper (peek)
O(1)/callO(1)
Solve-it-yourself variant. This is an attempt problem — the frame and the caching rule are below, but no solution. Build it cold.
When to use: you are given an existing iterator and must add peek() — return the next element without consuming it — while keeping next and hasNext correct.

The frame (LC 284 — Peeking Iterator): wrap the underlying iterator and keep a one-element cache:

  • peek() — if nothing is cached, pull one from the underlying iterator and store it; return the cached element without clearing it.
  • next() — if something is cached, return and clear it; otherwise delegate to the underlying next().
  • hasNext() — true if something is cached OR the underlying iterator has more.
The frame: the cache is a single optional holding "the element already pulled but not yet consumed." Every method consults it first. The whole difficulty is keeping the three methods consistent around that one cached slot. See pitfalls §10.

Canonical problem: LC 284 — Peeking Iterator (attempt). Graduated hints in the walkthrough.

Variant 8.4 Two-level cursor (flatten 2D)
amortized O(1)/callO(1)
Solve-it-yourself variant. This is a target problem — the model and the skip rule are below, but no code. This is the canonical Airbnb iterator design.
When to use: iterate a jagged 2D structure (a list of lists) as one flat sequence, where some inner lists may be empty. The cursor is two indices; the art is skipping empties cleanly.

The model (LC 251 — Flatten 2D Vector): keep an outer index over the rows and an inner index within the current row. A shared "advance to the next valid position" step moves inner forward, rolls outer over at a row's end, and skips any empty rows. Both hasNext and next run that step first so the cursor always points at a real element or past the end.

The frame: the only subtlety is the skip-empty-rows loop — it must be a while, not an if, because several empty rows can sit in a row. Centralize it in one helper both methods call; do not duplicate the advance logic. See pitfalls §10 and edge cases §11.

Canonical problem: LC 251 — Flatten 2D Vector (target). Graduated hints in the walkthrough.

9   Complexity Analysis

The metric is amortized cost per call, and total work across the traversal. A good iterator touches each element a constant number of times over its whole lifetime, so even if one next unwinds several stack frames, the average per call is O(1). The goal is never to do O(total) work in the constructor or in any single call.

VariantTime / callSpaceWhy
Stack fundamentals (8.1)O(1)O(n)One scan, push/pop per character; stack holds at most all open brackets.
Lazy stack flatten (8.2)amortized O(1)O(depth + width)Each element pushed and popped once; one occasional call unwinds several frames but the average is constant.
Lookahead wrapper (8.3)O(1)O(1)One cached element; each method does constant work around it.
Two-level cursor (8.4)amortized O(1)O(1)Two indices; skipping empty rows is amortized over the elements eventually returned.
BottleneckO(total) up front—Eagerly flattening the whole structure in the constructor — the exact anti-pattern the lazy design exists to avoid; it wastes work if the caller stops early and breaks on very large or infinite sources.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸hasNext / next disagreeing (all variants). If both advance, hasNext can consume an element next then misses. Put the advance in one place — usually hasNext — and have next rely on it.
▸Eager flattening (8.2). Walking the whole nested structure in the constructor is O(total) up front and defeats laziness. Push only the top level; descend on demand.
▸Pushing children in forward order (8.2). The stack is LIFO, so pushing a list front-to-back yields it back-to-front. Push reversed so the front element pops first.
▸Skipping empties with if not while (8.4). Several empty inner lists can be adjacent; a single if skips only one and leaves the cursor on an empty row. Loop until a non-empty position or the end.
▸peek() consuming (8.3). If peek calls the underlying next without caching the result, the element is lost. Cache it; peek returns the cache, next clears it.
▸Calling removeLast/next without checking emptiness. A crash on an exhausted iterator. Gate every consume behind hasNext, and never removeFirst() in a loop (O(n) each).

class MyIterator {
    private var stack: [Element] = []   // or indices / a cache

    init(_ source: [Element]) {
        for element in source.reversed() { stack.append(element) }  // front on top
    }

    func hasNext() -> Bool {
        // PREPARE: advance/unwind until top is a real element
        return !stack.isEmpty
    }

    func next() -> Element {
        // precondition: hasNext() == true
        return stack.removeLast()
    }
}

15   Appendix — open questions to deep-dive later

Two short notes that connect this family to neighboring ideas. Each is a thread worth pulling on its own; here we only sketch the answer.

Flatten 2D — pointers or a stack? Both work, and the choice is about depth. A jagged 2D vector is nesting of depth exactly one (a list of lists), so a two-index cursor — (outer, inner) — is enough and is O(1) extra space; there is nothing to "descend" into beyond one level, so a stack would be overkill. The moment nesting becomes arbitrary (lists inside lists inside lists, as in LC 341), a fixed number of indices can no longer name your position — you need a stack to remember every level you paused at. Rule of thumb: bounded depth → cursor/pointers; unbounded depth → stack. Flatten 2D could absolutely be solved with the LC 341 stack approach (treat each row as a sub-list), but the pointer version is simpler precisely because the depth is capped at one.
Is the stack iterator the same design as graph DFS? Essentially yes — and this is the deeper point. The lazy-flatten iterator is an iterative depth-first traversal: stack.append to descend, stack.removeLast (pop the top) to visit, push children to go deeper. That is byte-for-byte the iterative DFS skeleton from the graphs material — the nested list is a tree, the sub-lists are edges, the integers are leaves. The only thing the iterator adds is pausing: graph DFS runs the whole traversal in one loop, while the iterator suspends between visits so the caller can pull one element at a time. So "iterator over nested structure" and "iterative DFS that yields lazily" are the same algorithm wearing two interfaces. The one caveat: trees have no cycles, so the iterator needs no visited-set; a general graph DFS does. We can deep-dive the full correspondence — visited-sets, BFS-vs-DFS iterators, and why removeLast gives DFS while a queue gives BFS — later.
