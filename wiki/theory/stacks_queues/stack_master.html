A stack is a single-ended list where you can only interact with the top. You push items onto the top and pop them off the top — Last In, First Out. That one rule is surprisingly powerful: any problem where the most recent thing you saw governs what to do next is a stack problem. Nested brackets, arithmetic expressions, flattening recursive structures, and iterative DFS all reduce to this mechanic.

The stack pattern appears across several domains in this wiki: Strings & Parsing (matching brackets, evaluating expressions, decoding nested encodings), Design (iterators that lazily unwind nested structures), and Graphs (iterative DFS over trees and graphs). The LIFO mechanic is the same in all three — only the type stored on the stack changes. This overview is the decision layer and Swift toolbox: the three canonical loop skeletons, when to pick each variant, and the Swift mechanics you reach for every time.

2   The Three Standard Skeletons

Every stack problem is one of three loop shapes. Recognize the shape before naming any variable — the skeleton determines the variables, not the other way around.

SwiftPush-openers / pop-match-closers — bracket validation (LC20)
var openStack: [Character] = []
let matchMap: [Character: Character] = [")": "(", "]": "[", "}": "{"]
for character in input {
    if matchMap.keys.contains(character) {             // it's a closer
        guard openStack.last == matchMap[character] else { return false }
        openStack.removeLast()
    } else {
        openStack.append(character)                   // it's an opener — push
    }
}
return openStack.isEmpty                              // valid iff nothing left open
SwiftPush-on-open / pop-and-attach-on-close — nested structure parse (LC385 / LC394)
var containerStack: [NestedContainer] = []
var tokenAccumulator = ""
for character in input {
    switch character {
    case "[":
        containerStack.append(NestedContainer())       // descend one level
    case "]":
        if !tokenAccumulator.isEmpty { flushToken() }
        let finished = containerStack.removeLast()      // pop the completed level
        containerStack.last?.add(finished)              // attach to new top
    case ",":
        if !tokenAccumulator.isEmpty { flushToken() }
    default:
        tokenAccumulator.append(character)             // accumulate digits/chars
    }
}
return containerStack.last                            // the root container
SwiftIterative DFS — explicit stack replaces the call stack (trees / graphs)
var nodeStack: [Node] = [root]
while let current = nodeStack.popLast() {           // pop = visit
    process(current)
    for neighbor in current.neighbors.reversed() {   // reversed so left child processes first
        if !visited.contains(neighbor) {
            nodeStack.append(neighbor)
        }
    }
}
// The explicit stack IS the call stack — same visit order as recursive DFS,
// but no risk of stack overflow on deep inputs.
The shape is the recognition signal. Match openers to closers → skeleton 1. Descend on [, ascend and attach on ] → skeleton 2. "Visit this node, then its neighbors" → skeleton 3. Naming variables before identifying the skeleton is the #1 stack bug — you end up with the right pieces wired in the wrong order.

3   Swift Tools — the Toolbox

The Swift-specific mechanics that appear in every stack problem. Stacks in Swift are plain [T] arrays — no special type needed.

Swift toolWhat it buys youSeen in
stack.append(x)O(1) amortized push — adds to the end, which is the topevery stack problem
var stack: [Int] = []
stack.append(3)    // stack = [3]
stack.append(7)    // stack = [3, 7]   ← 7 is now the top
// the "top" of the stack is always the LAST element of the array
stack.removeLast()O(1) pop — removes and returns the top; crashes on empty stackLC20, LC227, LC341, LC385
var stack = [3, 7, 5]
let top = stack.removeLast()   // top = 5,  stack = [3, 7]
// removeLast() crashes if the array is empty — always guard first:
guard !stack.isEmpty else { /* handle underflow */ }
// OR use popLast() which returns Optional — never crashes:

var stack = [3, 7]
stack.last          // Optional(7) — non-destructive peek
stack.last == 7    // true — use for the "does the top match?" check in LC20
// stack is unchanged — peek never pops

var stack = [1, 2, 3]

    process(item)           // called with 3, then 2, then 1 — LIFO order

// stack is now empty. This is the canonical "drain the stack" pattern — used
// whenever you need to process everything that has been queued up.
stack.reversed() before pushing childrenPreserves left-to-right processing order in iterative DFS — right child pushed last is popped firstiterative tree / graph DFS
// Recursive DFS visits children left-to-right because it calls left then right.
// An iterative stack visits children in REVERSE push order (LIFO).
// To match recursive order: push children REVERSED so the leftmost is on top.
for child in node.children.reversed() {
    nodeStack.append(child)
}
// children = [A, B, C] → push C, B, A → pop A first → visit A before B before C ✓
stack.isEmptyThe "valid iff stack ends empty" terminal check — the only correct exit condition for bracket matchingLC20 return statement
// After scanning the whole string, unmatched openers remain on the stack.
// Valid iff nothing was left unmatched:
return openStack.isEmpty   // true = balanced, false = unclosed brackets remain
// Common bug: returning true early (inside the loop) — an opener at the END
// would be missed. The isEmpty check must come AFTER the full scan.

4   When to Use Each Pattern

Three stack variants, each solving a structurally different problem. The question that routes you is: what relationship am I tracking between the current token and something I saw earlier?

The one question that picks the stack variant
Match: "does this closer pair with the most recent opener?" → push openers, pop-match closers (skeleton 1). Nest: "this token opens a new level that I'll attach to its parent when it closes" → push containers on open, pop-and-attach on close (skeleton 2). Traverse: "visit this node, then its unvisited neighbors" → explicit DFS stack (skeleton 3). If you feel the urge to recurse on a string or list, you probably want skeleton 2 — the stack is the recursion, made into data.

Pattern 1 — Push openers, pop-match closers

Pattern 1 Push openers / pop-match closers
O(n)O(n)
Reach for it when: every opener must be closed by the most recent unmatched opener of the same type. The LIFO order of the stack exactly models nesting: the last thing opened is the first thing that must close. Valid iff the stack is empty at the end.
Wrong tool—Why not
Counter (e.g. depth += 1 / -= 1)—Works for a single bracket type (( ) only), but breaks immediately with mixed types — a counter cannot check that ] matches the most recent [ and not a (.
Recursion—Technically correct but allocates O(n) call-stack frames. The iterative stack is the same idea made explicit and safe on deep inputs.
The closer map is the key. Store [closer: opener] — looking up ")" gives "(". Then the check is just stack.last == matchMap[closer]. You never need an if/else chain per bracket type.

Pattern 2 — Push containers, pop-and-attach

Pattern 2 Push containers / pop-and-attach
O(n)O(n)
Reach for it when: brackets delimit nested containers (a list of integers and sub-lists, a run-length encoded string). On [, push a new empty container — you are now one level deeper. On ], pop the finished container and attach it to the new stack.last (its parent). Accumulate tokens between brackets into the current top.
Wrong tool—Why not
Recursion—Feels natural for nested structures, but the iterative stack is safer on deep inputs (no O(n) call-stack risk) and forces you to make the container-attach step explicit — which is exactly the step that is easy to get wrong.
Skeleton 1 (match-only)—Skeleton 1 validates balance — it does not build anything. If you need to construct the nested object, you need skeleton 2 with a container type on the stack.
The pop IS the attach. When you pop on ], two things happen atomically: the completed container is removed from the stack, and it is added to the new stack.last (its parent). Missing the attach step is the most common bug — you build the levels correctly but they never connect to the root.

Pattern 3 — Explicit DFS stack

Pattern 3 Explicit DFS stack
O(V + E)O(V)
Reach for it when: you need depth-first traversal over a tree or graph — pre-order, post-order, topological sort — without risking stack overflow on deep inputs. The explicit stack replaces the call stack; the loop replaces the function call.
Wrong tool—Why not
BFS queue—BFS explores level by level (shortest path); DFS explores depth-first (cycle detection, topological order, connected components). Swapping them changes the traversal order and breaks algorithms that depend on it.
Recursive DFS—Correct for shallow inputs, but O(n) call-stack frames on a path-shaped graph risk a stack overflow on large inputs. Swift's default call-stack limit (~8 MB) is surprisingly easy to hit on interview-size inputs with deeply nested trees.
Push in reverse, visit in order. A recursive DFS naturally visits left-then-right because it calls the left child first. An iterative stack visits in LIFO order — so push right first, then left, so left is popped (visited) first. This is the only non-obvious wiring detail; get it wrong and your traversal order flips.

Confusion pairs — the one question that decides

The mix-upAsk……and route
Stack vs recursionCan the input be deeply nested (> a few hundred levels)?Yes → explicit stack (no overflow risk). No → either works; stack is still cleaner for strings.
Stack vs queue (DFS vs BFS)Do I need depth-first (deepest path) or breadth-first (shortest path)?Deepest / topological → stack (DFS). Shortest path → queue (BFS).
Skeleton 1 vs Skeleton 2Am I just validating balance, or am I building a nested object?Validate only → skeleton 1 (push openers, pop-match). Build → skeleton 2 (push containers, pop-and-attach).

5   LIFO — the Core Mechanic

Every stack problem reduces to one idea: the most recently added item is the first one removed. Understanding why that matters is the entire theory.

Why LIFO? Nesting is inherently LIFO: the innermost bracket closes before the outer one. An expression stack evaluates the most recently deferred sub-expression first. A DFS explores the most recently discovered path before backtracking. In every case, "what I saw most recently" determines what to do right now — that is Last In, First Out.
DomainWhat's pushedWhat triggers a popWhat the pop produces
Bracket matchingAn opener character ( [ {Seeing a closer characterA match check — did the closer pair with the top?
Nested structure parseA new empty containerA closing bracket ]A completed container, attached to its parent
Expression evaluationA partial result or operandAn operator of equal or lower precedenceA combined sub-result pushed back
Iterative DFSAn unvisited nodeEach loop iteration (popLast)The next node to visit
Lazy iterator (LC341)Reversed sub-list contentshasNext() — unwind until an integer is on topAn integer ready to return via next()
The call stack is a stack. When you write a recursive function, the language runtime uses a call stack: each function call pushes a frame, each return pops one. An explicit iterative stack is literally the same thing with you in control of push and pop. Skeleton 2 and skeleton 3 are both "manual recursion" — the stack holds the state that would otherwise live in local variables of recursive calls.

10   Ultimate Cheat Sheet

The night-before skim. Pick the skeleton, wire the Swift ops, and know the one pitfall per variant.

Pick the skeleton

"valid parentheses" / "matching brackets"Skeleton 1 — push openers, pop-match closersLIFO ensures the most recent opener is checked first; valid iff empty at end
"evaluate arithmetic expression" (no parens)Skeleton 1 — deferred operator + operand stackpush for +/−, modify top for ×/÷, sum at end (LC227)

Complexity at a glance

PatternTimeSpaceWhy
Skeleton 1 — bracket matchO(n)O(n)Each character pushed or checked once. Stack holds up to n/2 openers.
Skeleton 1 — expression evalO(n)O(n)Each digit/operator processed once. Stack holds up to n/2 operands.
Skeleton 2 — nested parseO(n)O(n)Each character processed once. Stack depth = nesting depth ≤ n.
Skeleton 2 — decode string (LC394)O(n · kmax)O(n)String repetition expands the output; O(n) if repetition factors are treated as constant.
Skeleton 3 — iterative DFSO(V + E)O(V)Each node and edge visited once. Stack holds the frontier.
Hidden costO(n²)—String concatenation inside a loop (result += str) is O(n) per step → O(n²) total. Always build into an array and join at the end.
One pitfall per variant
▸Skeleton 1 (bracket match): check the top before popping — if the top doesn't match, return false immediately. Popping first and then checking means you discard the opener and can no longer detect the mismatch.
▸Skeleton 2 (nested parse): the pop AND the attach happen on ] — two steps, not one. Forgetting the attach means the popped container is discarded and never reaches the root.
▸Skeleton 2 (final flush): the last token has no trailing delimiter. Flush the accumulator after the loop, not only at , and ]. The final-flush omission is the #1 parsing bug across all four string-parsing variants.
▸Skeleton 3 (DFS order): push children in reversed order so the leftmost child is on top and gets popped first — matching recursive DFS's left-to-right order.
▸All skeletons: removeLast() crashes on an empty stack. Use popLast() when the empty case is legitimate; guard before removeLast() when it is not.
