A queue is a double-ended line where you add to the back and remove from the front — First In, First Out. That single rule is the foundation of breadth-first search: every node at distance d is processed before any node at distance d+1, because the nodes discovered first are the ones at the front of the queue. Any problem asking for the shortest path, minimum steps, or level-by-level processing is a queue problem.

The queue is the direct twin of the stack. A stack (LIFO) dives as deep as possible before backtracking — it finds any path. A queue (FIFO) fans out level by level — it finds the shortest path. This distinction determines which structure you reach for the moment you read "minimum" or "level order" in a problem statement.

2   The Two Standard Skeletons

Every queue problem is one of two shapes. Identify the shape before naming any variable.

SwiftBFS — level-order traversal, level-swap form (exact distances)
var currentLevel: [Node] = [startNode]
var visited: Set<Node> = [startNode]
var distance = 0

while !currentLevel.isEmpty {
    var nextLevel: [Node] = []
    for node in currentLevel {              // process every node at this distance
        if node == goal { return distance }
        for neighbor in neighbors(node) {
            guard !visited.contains(neighbor) else { continue }
            visited.insert(neighbor)
            nextLevel.append(neighbor)      // enqueue for the NEXT level
        }
    }
    currentLevel = nextLevel                // swap: advance one level
    distance += 1
}
return -1                                  // goal unreachable
SwiftSliding-window queue — evict stale items from the front (LC933)
// Keep only items within the window. Enqueue new, dequeue stale from front.
var window: [Int] = []

for timestamp in requests {
    window.append(timestamp)                    // enqueue — O(1) amortized
    while let oldest = window.first,
          oldest <= timestamp - windowDuration {
        window.removeFirst()                    // evict stale — O(n) on plain Array
    }                                           // use head-index trick for O(1)
    answer = window.count
}
The shape is the recognition signal. Processing all nodes at distance d before any at d+1 → BFS level-swap skeleton. Evicting items that fall outside a time or size window → sliding-window queue. If you find yourself needing the element you added most recently, that is a stack, not a queue.

4   When to Use Each Pattern

Two queue variants, each solving a structurally different problem. The routing question is: do I need shortest distance, or do I need to evict items that are too old?

The one question that picks the queue variant
"Shortest path / minimum steps / level order" → BFS skeleton (level-swap). "Only keep items within a time or size window" → sliding-window queue (enqueue back, evict front). If neither applies but you need FIFO order for some other reason (task scheduling, print queues), the sliding-window skeleton generalizes — it is just "enqueue + conditional evict from front."

Pattern 1 — BFS (breadth-first search)

Pattern 1 BFS — breadth-first search
O(V + E)O(V)
Reach for it when: the problem asks for the shortest path, minimum steps, or processes nodes level by level. BFS guarantees that the first time you reach a node is via the shortest path — because you always process nearer nodes before farther ones.
Wrong tool—Why not
DFS (stack)—DFS finds a path, not the shortest path. On an unweighted graph, DFS may reach the goal via a long detour while the short path was right there. Only BFS guarantees shortest-path on unweighted graphs.
Dijkstra—Correct but overkill when all edge weights are equal (unweighted graph). BFS is Dijkstra with unit weights — simpler, no heap needed, same O(V+E) result.
DFS + memoization—Memoized DFS counts paths or finds optimal values in DAGs. It does not give shortest hops in a general graph — cycles and revisits require the visited-set discipline that BFS enforces naturally.
First arrival = shortest path. BFS expands nodes in non-decreasing distance order. The moment a node is dequeued for the first time, the distance recorded is optimal — marking it visited at enqueue time (not dequeue time) is what prevents re-processing it via a longer path.

Pattern 2 — Sliding-window queue

Pattern 2 Sliding-window queue (FIFO eviction)
O(n)O(w)
Reach for it when: you process a stream of items and must answer "how many items are within the last T seconds / k requests?" The queue holds exactly the in-window items; the front is always the oldest. Enqueue each new item, evict any front items that are now outside the window, then read the count.
Wrong tool—Why not
Array counter with binary search—Binary search gives O(log n) per query instead of amortized O(1). The queue's front-eviction is already in-order, so no search is needed.
Sliding-window array (grow/shrink both ends)—The standard sliding-window uses two pointers moving rightward on a static array. The queue variant handles a stream where items arrive one by one and are never revisited — a different access pattern.
Evict from the front, enqueue at the back — always in order. Because items arrive in non-decreasing timestamp order, the front is always the oldest. You never need to search for stale items — just peek and evict until the front is fresh. Amortized O(1) per item because each item is enqueued once and dequeued at most once.

Confusion pairs — the one question that decides

The mix-upAsk……and route
BFS queue vs DFS stackDo I need the shortest path, or any path?Shortest → queue (BFS). Any / all / deepest → stack (DFS).

Sliding-window queue vs. traditional sliding window — they share a name, not a tool

Both patterns use the word "sliding window," but they solve structurally different problems with different data structures. The table above has one routing row for this; the two snippets below make the difference concrete.

The one question that separates them
Is your input a static array you can index into both directions? → Traditional sliding window (two pointers, no queue needed). Is your input a live stream arriving one item at a time, where you can only evict from the front? → Sliding-window queue (FIFO, enqueue back / evict front).
SwiftProblem A — "longest contiguous subarray with sum ≤ k" — static array → two-pointer window, no queue
// Input: a fixed array. Both pointers index into it freely.
// Tool: two pointers (left + right), O(n) time, O(1) space.
func longestSubarrayWithSumAtMost(_ numbers: [Int], _ limit: Int) -> Int {
    var left = 0
    var windowSum = 0
    var best = 0

    for right in numbers.indices {
        windowSum += numbers[right]
        while windowSum > limit {
            windowSum -= numbers[left]   // shrink from LEFT — direct index access
            left += 1
        }
        best = max(best, right - left + 1)
    }
    return best
}
// Key point: we jump left directly to any index. No queue needed — the array is static.
// The "window" is just a pair of indices, not a data structure holding the items.
SwiftProblem B — "how many pings in the last 3000 ms?" — live stream → sliding-window queue, two-pointer cannot apply
// Input: timestamps arrive ONE BY ONE. We never "go back" to a previous index.
// Tool: a queue — enqueue new item at back, evict stale items from front.
var window: [Int] = []

func ping(_ timestamp: Int) -> Int {
    window.append(timestamp)               // enqueue — always at the back
    while let oldest = window.first,
          oldest <= timestamp - 3000 {
        window.removeFirst()               // evict stale — always from the front
    }
    return window.count
}
// Key point: items are APPENDED — past items cannot be revisited by a left pointer.
// The queue IS the window; its count IS the answer. Two pointers have nothing to index.

5   FIFO — the Core Mechanic

Every queue problem reduces to one idea: the item that has been waiting longest is processed first. Understanding why that guarantees shortest paths is the entire theory.

Why FIFO guarantees shortest paths. When you enqueue a node at distance d, all nodes at distance d−1 are already in the queue ahead of it (they were enqueued earlier). So the queue always holds nodes in non-decreasing distance order: all distance-0 nodes first, then all distance-1 nodes, and so on. The first time any node reaches the front of the queue, it arrived via the shortest possible path — there is no shorter route that could have been missed.
DomainWhat's enqueuedWhat triggers a dequeueWhat the dequeue produces
Graph / grid BFSAn unvisited node at distance dEach loop iteration (head advance or pop)The next node to visit — guaranteed shortest distance
Level-order tree traversalA tree node at depth dProcessing each node in the current levelAll nodes at depth d, in left-to-right order
Multi-source BFSAll source nodes at distance 0Same as single-source BFSMinimum distance from any source — one pass replaces running BFS once per source
State-space BFSA (position, state-flags) pair at cost dEach loop iterationShortest number of moves to reach a target state
Sliding-window queueA timestamped itemWhen the front item falls outside the windowThe current in-window count — O(1) amortized per item
Mark visited at enqueue, not at dequeue. A common BFS bug is to mark a node visited when you pop it rather than when you push it. If you mark at dequeue, the same node can be enqueued multiple times via different paths before it is first processed — this corrupts the distance and wastes work. Mark it visited the instant you add it to the queue, so it is never added again.

One pitfall per variant
▸BFS (all variants): mark nodes visited at enqueue time, not at dequeue time. If you mark at dequeue, the same node is added to the queue multiple times via different paths — corrupting distances and causing O(n²) work.
▸BFS (all variants): removeFirst() on a plain Swift Array is O(n). For n > ~1000, use the head-index trick or swift-collections' Deque.
▸Sliding-window queue: enqueue the new item before running the eviction loop. If you evict first, the current item may be incorrectly evicted if its timestamp happens to be exactly at the boundary.
▸State-space BFS: the visited set must key on the full state, not just the position. Two visits to the same cell with different collected keys are different states — keying only on position would incorrectly skip valid paths.
