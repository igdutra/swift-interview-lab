Arrays & Hashing is the foundational interview block: a small set of single-pass techniques that together solve the vast majority of array and string problems. Hash maps trade O(n) space for O(1) lookup, turning "have we seen this before?" into a constant-time question. Two pointers exploit the sorted order of an array to converge from both ends, eliminating the O(n²) brute-force pair search. Sliding windows maintain an incremental summary of a contiguous range, finding optimal subarrays in a single pass. Prefix/suffix precomputation stores an accumulated value at every index so each later query costs O(1).

These tools share one property: each element is visited at most a constant number of times. That amortized linearity — not any particular data structure — is what they have in common, and it is also why they are constantly confused. This overview is the hub's decision layer and Swift toolbox: the loop skeletons that implement them, the language mechanics they lean on, and — most importantly — how to pick the right one and rule out the wrong ones. The per-technique theory lives in the four deep dives linked above.

2   The Four Standard Skeletons

Memorize these loop shapes and the variable names come for free. Each technique has exactly one canonical control-flow skeleton — recognizing the problem is reaching for the right skeleton.

SwiftHash map — look up the complement BEFORE you store (LC1)
var complementMap: [Int: Int] = [:]      // value → index
for (currentIndex, number) in nums.enumerated() {
    if let firstIndex = complementMap[number] {  // lookup FIRST — avoids self-match
        return [firstIndex, currentIndex]
    }
    complementMap[target - number] = currentIndex  // store AFTER
}
return []
SwiftTwo pointers — converge with while left < right (LC11 / LC125)
var leftPointer = 0
var rightPointer = nums.count - 1
while leftPointer < rightPointer {
    // evaluate nums[leftPointer] and nums[rightPointer]
    if /* steer toward the answer */ shouldMoveLeft {
        leftPointer += 1           // l →   (converges; never moves back)
    } else {
        rightPointer -= 1          //   ← r
    }
}
SwiftSliding window — for right { while invalid { shrink } } (LC3 / LC209)
var leftPointer = 0
var windowState: [Character: Int] = [:]   // or a running sum / counter
for (rightPointer, item) in Array(string).enumerated() {
    windowState[item, default: 0] += 1     // admit the right element

    while isInvalid(windowState) {            // LONGEST: shrink while INVALID
        windowState[Array(string)[leftPointer], default: 0] -= 1
        leftPointer += 1
    }
    best = max(best, rightPointer - leftPointer + 1)  // record AFTER the shrink
}
// SHORTEST flips it: `while isValid { record INSIDE; shrink }`
SwiftPrefix / suffix — build once with reduce(into:), most idiomatic form
// prefix sums with the sentinel already seeded at [0]
let prefix = nums.reduce(into: [0]) { accumulator, value in
    accumulator.append(accumulator.last! + value)
}
// nums=[2,3,1] → prefix=[0,2,5,6];  sum of nums[l...r] = prefix[r+1] - prefix[l]
The shape is the recognition. If you reach for while left < right you have committed to converging two pointers; if you reach for for right { while … } you have committed to a window. Writing the wrong skeleton is the same mistake as picking the wrong pattern — the loop header is the tell. And if you feel the urge to recurse on any of these, you have misidentified the tool: all four are inherently iterative single passes. Recurse for trees, graphs, and divide-and-conquer — not here.

4   When to Use Each Pattern

All five tools run in roughly O(n) over an array, so they are constantly mistaken for one another. Picking the right one is the entire skill — and the wrong choices are predictable. For each pattern: the one positive signal that selects it, and the specific tools people wrongly reach for with the exact reason each fails.

The three questions that pick the pattern
Before writing a line, answer these out loud — the three answers alone narrow it to one tool: (1) Is the answer contiguous? (a range, not a scatter of indices). (2) Is the array sorted, or sortable without losing what you must return? (3) Can the values be negative? Contiguous + constraint → sliding window. Sorted + a pair/area/symmetry → two pointers. "Seen this before?" → hash. Repeated range queries, "except self", or a quantity that needs both sides of each index → prefix/suffix. And a "no" to question 3 is what keeps a sliding window legal at all.

Pattern 1 — Hash Map / Hash Set

Pattern 1 Hash Map / Hash Set
O(n)O(n)
Reach for it when: the array is unsorted and the question is "have we seen this value? / does a complement exist? / did a value repeat within distance k?" You trade O(n) space for O(1) lookup — turning a nested scan into one pass.
Wrong tool—Why not
Two pointers—Only legal if the array is sorted. On unsorted Two Sum you would have to sort first — which destroys the original indices the problem asks you to return.
Sort, then anything—Sorting throws away index identity. LC1 must return positions; once sorted, "where was this number?" is unrecoverable.
Dynamic programming—"Have we seen this?" is a single O(1) set lookup, not a recurrence. Misreading LC1 as DP is over-engineering a one-pass problem.
Look up before you store. Query the map for the complement first, insert the current element after — otherwise a single element matches itself (input [0, 0], target 0 would falsely pair index 0 with index 0).

Pattern 2 — Two Pointers (converging)

Pattern 2 Two Pointers (converging)
O(n)O(1)
Reach for it when: the array is sorted (or sortable without losing the answer) and you want a pair, a triple, an area, or a palindrome check that you can steer by comparing the two ends. O(1) extra space — just two indices.
Wrong tool—Why not
Sliding window—A window moves both pointers rightward; two pointers converge (l→, r←). If you catch yourself moving both right, you are writing a window — name the difference out loud.
Sort, then two-pointer (LC11)—Sorting destroys the width. Container-With-Most-Water's area is (right − left) × min(height) — sorting scrambles the x-positions that are the width. (Contrast LC15, where sorting is the right move — see below.)
Hash map—Correct but costs O(n) space. When the array is already sorted, two pointers matches its speed at O(1) space — prefer it.
A sorted array gives you direction. Overshoot the target → shrink the larger side; undershoot → grow the smaller side. That deterministic steering is what makes the walk O(n) with zero storage — and in 3Sum (LC15) it also makes deduplication free: equal neighbors are provably redundant, so skipping them is correct by construction, no Set needed.

Pattern 3 — Sliding Window

Pattern 3 Sliding Window
O(n)O(k)
Reach for it when: you want the longest, shortest, or count of a contiguous subarray or substring satisfying a constraint. Both pointers move right only; the array need not be sorted.
Wrong tool—Why not
Window on a subsequence—If the problem lets you skip elements (a subsequence), there is no contiguous range to slide — the add/remove logic has nothing to summarize. → DP or greedy.
Window with negatives (LC560/LC209)—The shrink invariant dies. Adding an element no longer reliably increases the sum, so "if the sum is too big, shrink" has no valid trigger. This is the most common wrong instinct on these problems. → prefix-sum.
Window on Trapping Rain Water—A window looks in one direction. Water at a bar needs the tallest wall on both the left and the right, simultaneously, and those walls can be anywhere — there is no fixed range to slide. → prefix/suffix max, or converging two pointers.
The inversion rule decides where you record. Longest shrinks only while the window is invalid and records the best after the shrink (LC3). Shortest shrinks while the window is valid and records inside the shrink loop (LC76, LC209). Swapping the two is the #1 sliding-window bug — and the entire difficulty of LC76.

Pattern 4 — Prefix / Suffix Precomputation

Pattern 4 Prefix / Suffix Precomputation
O(n) buildO(1) query
Reach for it when: you face repeated range aggregates, an "except self" computation, or a quantity that depends on both sides of each index (trapped water, coverage). Build once in O(n); answer every query in O(1).
Wrong tool—Why not
Sliding window (range sum, signed)—No monotonic shrink condition exists with mixed signs — you cannot infer the sum of [l, r] from [l−1, r] when values can be negative.
Total product ÷ nums[i] (LC238)—Division is disqualified. It breaks on zeros (÷0; multiple zeros give wrong results) and division isn't a monoid — there's no universal identity. The prefix×suffix "gap" excludes index i structurally: neither pass ever sees it, so no zero-casing is needed.
Re-sum on every query—Brute force is O(k·n) total; for k ≈ n that is the O(n²) you precomputed to avoid.
Static prefix array on a mutating array—Any update invalidates the whole prefix array. If the data changes between queries → Fenwick (BIT) or segment tree for O(log n) updates.
It overlaps DP but isn't DP. Prefix/suffix is two passes storing a monoid (sum, product, max, boolean OR — all associative with an identity). Where each cell is a recurrence over the previous cell (running max in LC42) it is DP-flavored; where it just accumulates a set (boolean coverage) it's a plain sweep with no overlapping subproblems.

Pattern 5 — Prefix-Sum + Hash Map (patterns compose)

Pattern 5 Prefix-Sum + Hash Map
O(n)O(n)
Reach for it when: you must count the contiguous subarrays with a sum property and the values can be negative (LC560). Plain prefix-sum gives you the running total; the hash map turns the pair-search into an O(1) lookup. This is the canonical example that patterns compose — neither tool alone is enough.
Wrong tool—Why not
Sliding window—Negatives kill the shrink trigger (see Pattern 3). There is no valid point to move the left edge.
Prefix-sum, then scan all pairs—Searching every (i, j) for a difference of k is O(n²). The algebraic flip — fix the right end, ask "how many earlier prefixes equal current − k?" — converts the pair search into one O(1) map lookup per step.
Using a Set instead of a count map—A prefix value can recur at several positions, each a distinct valid left endpoint. A Set records presence, not multiplicity — silent undercount. Use [sum: count] with sentinel {0: 1}, and look up before inserting.

Confusion pairs — the one question that decides

The mix-upAsk……and route
Two pointers vs Sliding windowDo the pointers converge, or do they both run right?Converge (l→, r←) ⇒ two pointers. Both rightward ⇒ window.
Sliding window vs Prefix-sumCan the values be negative?All non-negative ⇒ window (monotonic shrink). Any negative ⇒ prefix-sum (+ map if counting).
Hash map vs Two pointersIs it sorted — or can I sort without losing what I must return?Sorted & free to sort ⇒ two pointers (O(1) space). Otherwise ⇒ hash map.
Prefix/suffix vs DPIs each cell a recurrence over the prior cell, or just an accumulated set?Recurrence ⇒ DP-flavored. Accumulation ⇒ plain sweep. (Mostly a naming distinction in the interview.)
When more than one pattern fits — pick by the tie-breaker:
▸LC15 (3Sum) — fixed anchor + hash map (O(n) space) vs. + two pointers (O(1) space). Two pointers wins: sorting gives free deduplication and direction; same O(n²) time, but no per-iteration Set allocation.
▸LC560 (Subarray Sum = K) — sliding window vs. prefix-sum + map. Only prefix-sum + map works: negatives kill the window's shrink invariant, so the window isn't a valid option at all.
▸LC42 (Trapping Rain Water) — prefix-max/suffix-max arrays (O(n) space) vs. converging two pointers (O(1) space). Same idea, compressed: the two-pointer version is the O(1)-space collapse of the two max arrays — pick it when space matters.
▸LC209 (Min Size Subarray Sum) — two-pointer window vs. prefix-sum + binary search. Window wins here: all values are positive, so the sum is monotonic and the window runs O(n) versus O(n log n).

5   Hash Map — Definition

Two pointers, sliding windows, and prefix/suffix are techniques; the hash map is the one data structure underneath them all worth defining precisely, because interviewers ask how it works.

MechanicWhat it doesWhy it matters
Hash functionConverts any key into a fixed-size integer (the bucket index)Makes the O(1) jump possible — no search, just arithmetic
Collision resolutionTwo keys may hash to the same bucket; resolved by chaining (linked list per bucket) or open addressing (probe to next empty slot)Worst-case O(n) if all keys collide, but rare with a good hash function and load factor ≤ 0.75
RehashingWhen the table exceeds its load factor, it allocates a larger array and re-inserts all entriesKeeps average bucket length short — this is the "amortised" in O(1)
The interview answer in one sentence: "A hash map uses a hash function to turn each key into a direct array index, giving O(1) average-case lookup, insert, and delete at the cost of O(n) extra space and an unordered iteration order."

Hash Set — same structure, no value

A hash set is a hash map where the key is the value — there is nothing else stored in the bucket. Same hash function, same collision resolution, same O(1) amortised operations. In Swift, Set<Element> is the hash set. It is marginally faster than Dictionary for pure membership tests because each bucket stores only one thing instead of a key-value pair, which means less memory touched per lookup. The rule of thumb: if you only need to answer "have I seen this?" with no associated data, use Set — the code is cleaner and the intent is clearer. If you need to store or retrieve a value alongside the key, use Dictionary. (The Swift mechanics for both — contains/insert, the default: subscript, grouping — are in the Toolbox.)

Prefix / suffixO(n)O(n)Two passes to build, then O(1) per query. O(1) extra if you reuse the output array.
Prefix-sum + mapO(n)O(n)One pass; complement lookup is O(1). Map holds one count per distinct prefix.
Hidden O(n²) trapO(n²)—A nested full scan, comparing whole maps each step, calling .max() in the loop, or re-summing per query — silently destroys every bound above.
Instant disqualifiers — rule a pattern out on sight
▸Unsorted and you must return indices → not two pointers (sorting kills the indices).
▸Values can be negative → not sliding window (the shrink invariant dies).
▸Answer is a subsequence (non-adjacent picks) → not sliding window (no range to slide) → DP / greedy.
▸Quantity needs the max on both sides of each index → not a single-direction window.
▸Array mutates between queries → not a static prefix array → Fenwick / segment tree.
▸Division proposed for product-except-self → disqualified (breaks on zeros; not a monoid).

Prefix / suffix array idioms

The two-pass building blocks for every prefix/suffix problem — sentinel placement is the whole trick. (The two forms of reduce are in the Toolbox; this section is the prefix/suffix-specific assembly.)

Swiftsentinel — prefix (identity at the front), two ways
// Explicit: allocate n+1 slots, sentinel lands at index 0, loop fills index+1
var prefix = [Int](repeating: 0, count: nums.count + 1)
for index in 0 ..< nums.count {
    prefix[index + 1] = prefix[index] + nums[index]
}
// nums=[-2,0,3] → prefix=[0,-2,-2,1]  ← 0 is the sentinel at index 0

// Idiomatic: [0] IS the starting array — sentinel is already there before reduce runs
let prefix = nums.reduce(into: [0]) { acc, value in
    acc.append(acc.last! + value)
}
// same result: [0, -2, -2, 1]
Swiftsentinel — suffix (identity at the end)
// Idiomatic: build right-to-left, flip back — [1] seeds the right end
let suffix = nums.dropFirst().reversed().reduce(into: [1]) { acc, value in
    acc.append(acc.last! * value)
}.reversed()
// nums=[1,2,3,4] → [24, 12, 4, 1]   (products of everything RIGHT of each index)
// reverse twice: once to feed reduce right-to-left, once to restore order
When to drop the boundary — the one question: should position i be ABSENT from its own accumulated value? Yes (product-except-self) → dropLast() for prefix, dropFirst() for suffix; the exclusion is structural. No (range sum, LC303) → no drop, every element is included and the query formula prefix[right+1] − prefix[left] handles the boundary.
Swiftfull product-except-self pipeline (idiomatic O(n))
// nums = [1, 2, 3, 4]
let prefixes = nums.dropLast().reduce(into: [1]) { acc, value in
    acc.append(acc.last! * value)
}
// → [1, 1, 2, 6]   (sentinel [1] + products of everything LEFT of each index)

let suffixes = nums.dropFirst().reversed().reduce(into: [1]) { acc, value in
    acc.append(acc.last! * value)
}.reversed()
// → [24, 12, 4, 1]  (products of everything RIGHT of each index + sentinel [1])

zip(prefixes, suffixes).map(*)   // [24, 12, 8, 6]  ✓
