Arrays & Hashing — Overview

Arrays & Hashing is the foundational interview block: three single-pass techniques that together solve the vast majority of array and string problems. Hash maps trade O(n) space for O(1) lookup, turning "have we seen this before?" into a constant-time question. Two pointers exploit the sorted order of an array to converge from both ends, eliminating the O(n²) brute-force pair search. Sliding windows maintain an incremental summary of a contiguous range, finding optimal subarrays in a single pass without ever examining all O(n²) subarrays.

The three techniques share one property: each element is visited at most a constant number of times. That amortized linearity — not any particular data structure — is what they have in common. Recognizing which tool a problem calls for is the core skill of this block.

2   Recognition Signals — which tool to reach for

These three tools are often confused because they all run in O(n) and all operate on arrays. The signal words that separate them:

When the problem says…Reach for…Why
"have we seen this value before?" / "contains duplicate"Hash set membershipO(1) lookup vs O(n) scan
"find a pair that sums to target" (unsorted)Complement mapstore target − x, look it up in one pass
"duplicate within distance k" / "index of last seen"Sliding index maphash map keyed by value, storing last index
"detect a cycle" / "have we visited this state?"Hash set of seen statesO(1) membership check per step
"pair/triplet that sums to" + sorted or sortableTwo pointers (converging)sorted order lets you steer left or right
"is it a palindrome?" / "two walls, maximize area"Two pointers (converging)problem symmetry → shrink from both ends
"longest/shortest contiguous subarray/substring"Sliding windowcontiguous = range, range = two rightward pointers
"every window of size k"Sliding window (fixed)left trails right by exactly k − 1
The key disqualifier for two pointers: the array must be sorted (or sortable without destroying information). If the problem forbids sorting, reach for a hash map or sliding window instead.
The key disqualifier for sliding window: the answer must be contiguous. Non-adjacent elements (subsequences, subsets) have no range to slide — reach for DP or greedy.

4   Mental Model — three tools, one family

The one idea
All three tools are the same idea at different levels of structure: instead of re-examining the full array for each element (O(n²)), each tool maintains a small piece of state that makes each element's "question" answerable in O(1). The state is a hash table, a pair of indices, or a window summary — but the principle is identical.

Here is how each tool answers the same core question — "does this element's answer depend on something we've seen before?" — differently:

Diagramthe three tools side by side

────────────────────        ──────────────────────     ──────────────────────

O(n) space                  O(1) space                  O(k) space

Same input structure, three different questions. The state each tool maintains (map / two indices / window summary) is exactly what lets it answer in O(1) per element.

5   Pseudocode Primer

One template per tool — the skeleton that never changes regardless of the specific problem.

Pseudocodehash map — complement lookup (LC1 shape)
FUNCTION hashMapSolve(input, target):
    seen ← empty map          // key: value seen; value: its index
    FOR i FROM 0 TO length(input) − 1:
        complement ← target − input[i]
        IF complement IN seen:
            RETURN answer using seen[complement] and i
        seen[input[i]] ← i
    RETURN not found
Pseudocodetwo pointers — converging (LC11 shape)
FUNCTION twoPointers(input):
    sort(input)               // required for convergence to be meaningful
    l ← 0,  r ← length(input) − 1
    best ← 0
    WHILE l < r:
        candidate ← evaluate(input[l], input[r])
        best ← max(best, candidate)
        IF should move left pointer:   l ← l + 1
        ELSE:                          r ← r − 1
    RETURN best
Pseudocodesliding window — longest valid range (LC3 shape)
FUNCTION slidingWindow(input):
    state ← empty
    left ← 0,  best ← 0
    FOR right FROM 0 TO length(input) − 1:
        add input[right] TO state
        WHILE state is invalid:
            remove input[left] FROM state
            left ← left + 1
        best ← max(best, right − left + 1)
    RETURN best

6   Iterative-Only Pattern

All three tools are inherently iterative. There is no recursive formulation worth writing for any of them:

  • Hash map: a single forward pass storing and querying a map. A recursive version would just be a manual call-stack simulation of the same loop.
  • Two pointers: a while l < r convergence loop. Recursion would add call-stack overhead with no structural benefit — the invariant "answer lies in [l, r]" is clearest as an explicit loop condition.
  • Sliding window: a for right loop with an inner while left. A recursive substring enumeration is exactly the O(n²) brute force the window exists to replace.
Why this matters in an interview: if you feel the urge to recurse on any of these three patterns, you have likely misidentified the tool. Recurse for trees, graphs, and divide-and-conquer. Iterate for single-pass array techniques.

7   Swift Translation

The canonical Swift idiom for each tool's starting point:

Swifthash map — set membership skeleton
var seen = Set<Int>()
for value in nums {
    if seen.contains(value) { return true }
    seen.insert(value)
}
return false
Swifttwo pointers — converging skeleton
var l = 0, r = nums.count - 1
while l < r {
    // evaluate nums[l] and nums[r]
    if /* move left */ false { l += 1 } else { r -= 1 }
}
Swiftsliding window — longest valid skeleton
var left = 0, best = 0
var state: [Character: Int] = [:]
for (right, item) in Array(s).enumerated() {
    state[item, default: 0] += 1
    while /* invalid */ false {
        let lc = Array(s)[left]; state[lc]! -= 1; left += 1
    }
    best = max(best, right - left + 1)
}
Swift-specific costs to say out loud: Set.contains and Dictionary subscript are O(1) amortized — not guaranteed O(1). Array(s) converts a String once for O(1) indexing; never subscript a Swift String inside a loop. nums.sorted() is O(n log n) and must be called before two-pointer convergence.

8   Variant Catalog

Variant 8.1 Hash Map / Hash Set
O(n)O(n)
When to use: the problem asks whether a value has been seen before, whether a complement exists, or whether an element appeared within a certain index distance. The array does not need to be sorted. You are trading O(n) space for O(1) lookup.

Four sub-patterns, each covered in the deep dive: set membership (LC217), seen-state tracking (LC202), complement map (LC1), sliding index map (LC219). See hash_map_deep_dive_master.html §8.

Variant 8.2 Two Pointers (converging)
O(n)O(1)
When to use: the array is sorted (or can be sorted without destroying the answer), and the problem asks for a pair, triple, or boundary condition that can be steered by comparing the two ends. O(1) extra space — no auxiliary structure.

Four sub-patterns: converging sum (LC11), fixed-anchor + inner window (LC15 / 3Sum), palindrome compare (LC125), left/right-max accumulator (LC42). See two_pointers_deep_dive_master.html §8.

Variant 8.3 Sliding Window
O(n)O(k)
When to use: the problem asks for the longest, shortest, or counted contiguous subarray or substring satisfying a constraint. Both pointers move rightward only — unlike two pointers, the array need not be sorted.

Three sub-patterns: fixed window (size k given), longest / grow on expand (LC3, LC1004), shortest / shrink while valid (LC76, LC209). See sliding_window_master.html §8.

9   Complexity Analysis

ToolTimeSpaceWhy
Hash map / setO(n)O(n)One pass; each operation is O(1) amortized. Space: up to n entries in the map.
Two pointersO(n log n)O(1)Sort dominates; pointer walk is O(n). No auxiliary structure — only two indices.
Two pointers (no sort)O(n)O(1)When the array is already sorted or sorted order is given (palindrome, rain water).
Sliding windowO(n)O(k)Each element enters and exits at most once (amortized ≤ 2n moves). State holds at most k distinct elements.
Hidden O(n²) trapO(n²)—Scanning the full array inside a loop (brute force pair check), or any per-iteration full-map comparison, silently destroys the bound.

10   Common Pitfalls

Pitfalls — the most common mistakes across all three tools
▸Using a hash map when a hash set suffices. If you only need to check membership (not store associated data), a Set is cleaner and makes intent explicit. LC217 needs a set, not a map.
▸Forgetting to sort before two-pointer convergence. Converging pointers only work correctly on a sorted array. If you skip the sort, the "steer left or right" decision has no meaning and the algorithm produces garbage.
▸Applying sliding window to a non-contiguous problem. If the problem allows skipping elements (a subsequence), there is no contiguous range to maintain. The window's add/remove logic has nothing to summarize.
▸Reaching for DP when a hash map does the job in O(n). Any "have we seen this?" question is a hash set lookup, not a recurrence. Two Sum (LC1) is O(n) with a map; misidentifying it as DP costs you an interview.
▸Subscripting a Swift String inside a loop. String[index] is O(n) per access. Always convert with Array(s) before any loop over characters.
