Sliding Window

The sliding window is a traversal technique that finds an optimal contiguous subarray or substring by maintaining a range between two pointers — a right pointer that only ever advances, and a left pointer that only ever follows — while keeping an incremental summary of the range's contents.

Because both pointers move monotonically, every element is added to the window once and removed at most once, giving linear time even when the code contains a nested loop. The technique converts a naive "examine all O(n²) subarrays" search into a single pass, and it is the standard tool whenever a problem asks for the longest, shortest, or counted contiguous range satisfying a constraint.

2   Recognition Signals — when to reach for it

Sliding window problems rarely say "sliding window." They say one of these:

When the problem says…It means…
"longest substring/subarray such that…"Variant 8.2 — grow right, shrink left on invalid, record after the shrink loop
"shortest/minimum subarray such that…"Variant 8.3 — shrink while valid, record inside the shrink loop
"…of size k" / "every window of length k"Variant 8.1 — fixed window, left trails right by exactly k − 1
"contains all characters of t"Frequency-map window + matched-count validity (LC 76)
"at most k distinct / at most k replacements"Budget-style validity: shrink when the budget is exceeded
"count the subarrays that…"Counting window: every valid right pointer contributes right − left + 1
The disqualifier: the answer must be contiguous. If the problem allows picking non-adjacent elements (a subsequence), the window's incremental add/remove logic has nothing to maintain — reach for DP or greedy instead.

4   Mental Model — a caterpillar, not a snapshot

The one idea
The window is a caterpillar crawling along the array: the head (right) pulls forward one step every iteration; the tail (left) catches up only when the body breaks a rule. The body never teleports and never backs up — and that is the entire reason the algorithm is linear.

Watch the longest-window variant find the longest substring without repeating characters in "abcabcbb":

Traceexpand on every step; shrink only when a repeat appears
right=0 'a'  window="a"      valid   best=1
right=1 'b'  window="ab"     valid   best=2
right=2 'c'  window="abc"    valid   best=3
right=3 'a'  window="abca"   repeat 'a' → shrink: drop 'a' (left=1)
             window="bca"    valid   best=3
right=4 'b'  window="bcab"   repeat 'b' → shrink: drop 'b' (left=2)
             window="cab"    valid   best=3
right=7 'b'  …window="b"     left chased right; total pointer moves ≤ 2n

Each character enters the window once (right passes it) and leaves at most once (left passes it). Sixteen pointer moves for eight characters — never more than 2n.

5   Pseudocode Primer

Every sliding window solution is this loop. Variants differ only in when the left pointer moves and where the answer is recorded.

Pseudocodethe canonical (longest) form
FUNCTION slidingWindow(input):
    windowState ← empty
    left ← 0
    best ← 0

    FOR right FROM 0 TO length(input) − 1:
        ① EXPAND — fold input[right] into windowState
        add input[right] to windowState

        ② RESTORE — while the window violates the rule, shrink from the left
        WHILE windowState is invalid:
            remove input[left] from windowState
            left ← left + 1

        ③ RECORD — the window is valid here, by construction
        best ← max(best, right − left + 1)

    RETURN best
Why it works: after the while loop the window is always valid, so step ③ never records garbage — the loop is a self-restoring invariant. For shortest-window problems the invariant flips: the while loop runs while valid, and the record moves inside it (see 8.3).

6   Iterative-Only Pattern — no recursion, ever

Sliding window is inherently iterative: the entire algorithm is one for loop with a nested while, and the only "data structure" is the window state itself. There is no recursive formulation worth writing — a recursive substring enumeration is exactly the O(n²) brute force the window exists to delete.

  • No call stack — state lives in two integers and one small map; nothing to overflow at n = 10⁵.
  • Explicit invariant — "after the while loop, the window is valid" is visible in the code, where a recursive version would hide it in call frames.
  • Single direction — both pointers only increase. If you ever feel the need to move right backwards or restart the window, you have left the pattern (and probably its O(n) guarantee).

7   Swift Translation

Swiftannotated longest-window template — compilable starting point
func longestValidWindow(_ s: String) -> Int {
    // Swift String has no O(1) integer indexing — convert once, up front.
    let characters = Array(s)
    var windowCounts: [Character: Int] = [:]
    var leftPointer = 0
    var bestLength  = 0

    for (rightPointer, rightCharacter) in characters.enumerated() {
        // ① EXPAND
        windowCounts[rightCharacter, default: 0] += 1

        // ② RESTORE validity — replace the condition per problem
        while windowCounts[rightCharacter]! > 1 {
            let leftCharacter = characters[leftPointer]
            windowCounts[leftCharacter]! -= 1
            leftPointer += 1
        }

        // ③ RECORD — window is valid here by construction
        bestLength = max(bestLength, rightPointer - leftPointer + 1)
    }
    return bestLength
}
Note: the Swift-specific costs to say out loud in an interview: Array(s) is O(n) once and buys O(1) indexing forever; dictionary subscript with default: is O(1) amortized; never call .max() or rebuild a map inside the loop — that silently reintroduces O(n²).

8   Variant Catalog

Variant 8.1 Fixed-size window
O(n)O(k)
When to use: the problem fixes the window size k (max average of k elements, anagrams of a fixed pattern). No validity check exists — the left pointer is always exactly right − k + 1.
FOR right FROM 0 TO n − 1:
    add input[right]
    IF right ≥ k − 1:
        record(windowState)
        remove input[right − k + 1]
Variant 8.2 Longest window — grow, shrink on invalid
O(n)O(k)
When to use: "longest substring/subarray such that…". The while loop moves you toward validity, so the answer is recorded after it. This is the §5 primer verbatim.

Examples: Longest Substring Without Repeating Characters (LC 3), Longest Repeating Character Replacement (LC 424), Max Consecutive Ones III (LC 1004).

Variant 8.3 Shortest window — shrink while valid
O(n)O(k)
When to use: "shortest/minimum window such that…". The while loop now runs while the window is valid and moves you away from validity, so the answer is recorded inside it, before each shrink step.
FOR right FROM 0 TO n − 1:
    add input[right]
    WHILE windowState is valid:
        record(right − left + 1)      ← inside: window is valid right now
        remove input[left]
        left ← left + 1
The inversion rule: ask "does my while loop move me toward validity or away from it?" Toward → record after the loop (8.2). Away → record inside the loop (8.3). Getting this placement wrong is the #1 sliding window bug, and it is the entire difficulty of LC 76.

9   Complexity Analysis

Why the nested while loop is still O(n). Count work per element, not per iteration: each element is added to the window exactly once (when right passes it) and removed at most once (when left passes it). Total pointer movement is therefore at most 2n across the whole run, regardless of how the shrink steps cluster. A window that shrinks 50 times in one iteration paid for those 50 steps by not shrinking in the previous 50.

VariantTimeSpaceWhy
Fixed (8.1)O(n)O(k)One add and one remove per step; state holds at most k elements.
Longest (8.2)O(n)O(k)Amortized: ≤ 2n pointer moves; map holds the window's distinct elements.
Shortest (8.3)O(n)O(k)Same amortization; validity check must be O(1) (see matched count) or the bound degrades.
Hidden trapO(n²)—Any per-iteration full scan of the state (comparing whole dictionaries, calling .max()) silently destroys the bound.

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Recording in the wrong place. Longest records after the shrink loop; shortest records inside it. Swapping them returns answers off by an arbitrary amount, not by one — re-derive with the inversion rule (8.3).
▸Resetting the window state on success. The window slides; it never restarts. A full reset throws away accumulated counts and breaks the ≤ 2n movement argument.
▸O(n) validity checks. Comparing two dictionaries per step is O(k) hidden inside the loop. Maintain a matchedCount integer so validity is one comparison.
▸Window length off-by-one. The length is right − left + 1. Forgetting the +1 passes small tests (length 1 windows) and fails everything else.
▸Indexing a Swift String directly. s[s.index(...)] in a loop is quadratic. Convert with Array(s) before the loop, every time.
▸Leaving zero-count keys in the map. If validity checks windowCounts.count, a key with value 0 must be removed, or the distinct-count lies.
