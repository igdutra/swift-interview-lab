The hash map pattern converts a brute-force O(n²) "check all previous elements" search into a single O(n) pass by storing previously seen values in a constant-time lookup structure. The core trade is always the same: spend O(n) space now, eliminate an inner loop forever.

Four distinct interview patterns use this trade: checking whether a value has appeared before (set membership), tracking whether an algorithm has revisited a state (cycle detection), storing a complement to find a matching pair (complement map), and recording the last index at which a value appeared to check proximity (sliding index map). All four are one-pass O(n) algorithms; they differ only in what is stored in the map and what is looked up.

4   Mental Model — the phone book

The one idea
A hash map is a phone book: you look up a name (key) in O(1) and get its number (value) instantly, no matter how big the book is. The hash function is the alphabetical index — it tells you which page to open. The entire pattern is: build the phone book as you walk the array, and look things up in it before you add to it.

Watch the complement map (Variant 8.3) find a pair in [2, 7, 11, 15] with target 9:

Tracecomplement map — each step: look up first, then store
i=0  nums[0]=2   complement=9−2=7   map={}              7 not in map → store {2:0}
i=1  nums[1]=7   complement=9−7=2   map={2:0}           2 IS in map → return [map[2], 1] = [0, 1]

Two steps, zero inner loop. The map holds the "other half" of every pair we've seen so far. When the complement of the current element is already in the map, we're done.

And the simpler set membership pattern on [1, 2, 3, 1]:

Traceset membership — insert and check in one pass
i=0  nums[0]=1   seen={}     1 not in seen → insert → seen={1}
i=1  nums[1]=2   seen={1}    2 not in seen → insert → seen={1,2}
i=2  nums[2]=3   seen={1,2}  3 not in seen → insert → seen={1,2,3}
i=3  nums[3]=1   seen={1,2,3} 1 IS in seen → return true

5   Pseudocode Primer

Two canonical templates cover all four variants. The only differences are what is stored and what is looked up.

Pseudocodetemplate A — set membership / seen-state (LC217 · LC202 shape)
FUNCTION setMembership(input):
    seen ← empty set
    FOR each element IN input:
        IF element IN seen:
            RETURN true     // found a repeat / cycle
        seen.insert(element)
    RETURN false
Pseudocodetemplate B — map with associated data (LC1 · LC219 shape)
FUNCTION mapLookup(input, target):
    map ← empty map          // key: value seen; value: index (or complement)
    FOR i FROM 0 TO length(input) − 1:
        query ← derive_query(input[i], target)   // e.g. target − input[i]
        IF query IN map:
            RETURN answer using map[query] and i
        map[input[i]] ← i    // store AFTER the lookup — order matters
    RETURN not found
Why the lookup comes before the store: if you store first and then look up, a single-element answer will always find itself (e.g., LC1 with input [0, 0] target 0 would falsely match index 0 with itself). Look up first, store after.

6   Iterative-Only Pattern

All four hash map variants are inherently iterative: a single forward-pass for loop. There is no recursive structure to consider — hashing is a lookup operation, not a decomposition of the problem into subproblems. A recursive formulation would just be a manual simulation of the same loop, with call-stack overhead added for no benefit.

  • No divide-and-conquer: you are not splitting the array; you are accumulating state as you scan forward, left to right.
  • No backtracking: you never revisit an element. The map holds everything you need from the past in O(1) reachable form.
  • Single direction: the index only ever increases. If you feel the need to scan backwards or restart the loop, you have likely confused this with a two-pointer problem.

7   Swift Translation

SwiftLC1 — Two Sum, complement map, fully annotated
func twoSum(_ nums: [Int], _ target: Int) -> [Int] {
    // key: a number we've seen; value: the index we saw it at
    var map: [Int: Int] = [:]

    for (i, num) in nums.enumerated() {
        let complement = target - num

        // ① LOOK UP before storing — avoids matching an element with itself
        if let j = map[complement] {
            return [j, i]
        }

        // ② STORE after the lookup
        map[num] = i
    }
    return []   // problem guarantees a solution exists; this is unreachable
}
SwiftLC219 — sliding index map: check proximity before updating
func containsNearbyDuplicate(_ nums: [Int], _ k: Int) -> Bool {
    // key: a value we've seen; value: the most recent index we saw it at
    var lastSeen: [Int: Int] = [:]

    for (i, num) in nums.enumerated() {
        // ① CHECK first: if we've seen this value and it was within k steps
        if let j = lastSeen[num], i - j <= k {
            return true
        }
        // ② UPDATE: always overwrite with the most recent index
        lastSeen[num] = i
    }
    return false
}
Swift-specific costs to say out loud: Dictionary subscript returns an Optional — always unwrap safely with if let. The default: 0 subscript (map[key, default: 0]) is idiomatic for frequency counts. Set.contains and Dictionary lookup are both O(1) amortized — but hash collisions can degrade to O(n) in the theoretical worst case.

8   Variant Catalog

Variant 8.1 Hash Set Membership
O(n)O(n)
When to use: "does any element appear more than once?" The answer is purely yes/no; no associated data is needed. A Set is correct — using a Dictionary here is needless complexity.

Pattern: insert each element into a set. Before inserting, check if it is already there. If yes, return true immediately.

FOR each value IN array:
    IF value IN seen: RETURN true
    seen.insert(value)
RETURN false

Problems: LC217 — Contains Duplicate (the warm-up for this entire block).

Variant 8.2 Seen-State Tracking (Cycle Detection)
O(n)O(n)
When to use: "does this process / sequence eventually repeat a state?" The algorithm generates states one at a time; a set tracks every state visited. When a state appears that's already in the set, a cycle is detected.

Pattern: identical to 8.1 in structure, but the "value" is the state of the algorithm at each step — not necessarily an array element. For LC202, the state is the sum of squared digits.

WHILE state ≠ terminal:
    IF state IN seen: RETURN false   // cycle detected
    seen.insert(state)
    state ← next(state)
RETURN true

Problems: LC202 — Happy Number.

Alternative — Floyd's cycle detection: two pointers (slow/fast) can detect cycles in O(1) space. The set approach is simpler to explain in an interview; mention Floyd's as a follow-up optimization.
Variant 8.3 Complement Map (Pair Lookup)
O(n)O(n)
When to use: "find two elements that sum to a target" and the array is unsorted (or sorting would destroy index information). Store the complement target − x for each element; look it up at the next element.

Pattern: for each element, compute its complement. If the complement is in the map, return the pair. Otherwise, store the current element mapped to its index. Always look up before storing.

FOR i, value IN array:
    IF (target − value) IN map: RETURN [map[target−value], i]
    map[value] ← i
RETURN not found

Problems: LC1 — Two Sum (the canonical complement-map problem).

Variant 8.4 Sliding Index Map (Proximity Check)
O(n)O(n)
When to use: "are there two equal elements whose indices are at most k apart?" The map stores value → most recent index, and you check the distance on each lookup.

Pattern: for each element, check if its value is already in the map and whether the stored index is within k. Then update the map with the current index — even if a match was found (to keep "most recent" accurate for future checks).

FOR i, value IN array:
    IF value IN map AND i − map[value] ≤ k: RETURN true
    map[value] ← i     // always update to the most recent index
RETURN false

Problems: LC219 — Contains Duplicate II.

Note — update after the check, not before: the check must use the previous index; update first and you'll always see a distance of 0.

9   Complexity Analysis

Why O(n) time. Each element is visited exactly once in the for loop. Each operation inside the loop — a set contains, a map lookup, or a map insert — is O(1) amortized. The total work across all n elements is therefore O(n). There is no inner loop, no sorting, no recursion.

Amortized O(1) vs worst-case O(n). A hash map occasionally resizes its internal array when the load factor exceeds ~0.75. The resize copies all entries — O(n) work. But this happens rarely enough that the average cost per insert across n insertions is still O(1). In interview contexts, always say "O(1) amortized" rather than "O(1)".

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Storing before looking up (LC1 / LC219 bug). If you insert into the map before querying it, an element can match itself — e.g., [0, 4, 3, 0] target 0 would return [0, 0] at i=0 if you store first. Always look up, then store.
▸Using a map when a set suffices (LC217). If you only need to answer "has this value appeared?" you never use the stored index. Using a Dictionary<Int, Int> wastes memory and adds noise. Use Set<Int>.
▸Forgetting to update the index in the sliding index map (LC219). Even after finding a valid match and returning true, if you continued, you would need to update the map with the current index — to keep "most recent" accurate for future checks. Not updating leaves stale indices that produce wrong results on future lookups.
▸Nil-unwrap crash on uninitialized count. map[key]! += 1 crashes if key is not in the map. Always use map[key, default: 0] += 1 for frequency counting.
▸Leaving zero-count keys in a frequency map. If validity depends on map.count or iterating over keys, a key with value 0 corrupts the count. Remove the key when its count reaches zero: if map[key] == 0 { map.removeValue(forKey: key) }.
