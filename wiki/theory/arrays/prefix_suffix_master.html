Prefix & Suffix Precomputation

Prefix and suffix precomputation is a technique that eliminates repeated linear work by storing, at each index pos, the accumulated result of every element to its left (the prefix) and every element to its right (the suffix). A single left-to-right pass builds the prefix array; a single right-to-left pass builds the suffix array. Any query that asks "what is the aggregate of some range, or of everything except position pos" then combines prefix[pos] and suffix[pos] in one step — O(1) per query regardless of range length.

The "payload" stored at each position varies by problem — a running sum, a running product, a running maximum, a boolean flag, even a set of day indices — but the two-pass shape is always the same. This uniformity is the technique's power: recognizing any of these payload types as "something a left pass accumulates" immediately hands you an O(n) algorithm where the naive alternative scans the array again for every query, costing O(n²) in total.

Where prefix/suffix meets Dynamic Programming — and where it doesn't.

Think of it as a Venn diagram. DP is the paradigm: break a problem into subproblems, store the answers, reuse them instead of recomputing. Prefix/suffix is a technique: two passes over an array that store an accumulated value at each index. They are not the same thing — but they overlap.

In the intersection (prefix/suffix AND DP): any variant where each cell is built from the previous cell using a recurrence. leftMaxAt[pos] = max(height[pos], leftMaxAt[pos−1]) in LC 42 is a recurrence — you're computing cell pos from the stored answer at cell pos−1 instead of rescanning. prefix[pos+1] = prefix[pos] + nums[pos] in range-sum is identical. These are DP: overlapping subproblems, stored answers, reuse. Variants 8.1, 8.2, and 8.3 all qualify.

Outside the intersection (prefix/suffix shape, but NOT DP): Variant 8.4 — Boolean Coverage — sweeps the array and records "has this day been covered so far?" Each cell doesn't depend on the previous cell via a recurrence; it just accumulates a set. The two-pass shape is the same, but there are no overlapping subproblems being memoized. It's a sweep, not DP.

The clean rule: if building the array requires a recurrence (cell[pos] defined in terms of cell[pos−1]), it's DP. If the array just records running state without that dependency, it's not. When a dedicated DP theory page exists on this wiki, Variants 8.1–8.3 will be cross-referenced there.

2   Recognition Signals — when to reach for it

The problem rarely says "prefix array." It says one of these:

When the problem says…It maps to…
"range sum query" / "sum of elements from left to right"Variant 8.1 — prefix sum; answer is prefix[right+1] − prefix[left]
"product of all elements except self" / "no division"Variant 8.2 — prefix product × suffix product at each index
"water trapped" / "amount each bar can hold"Variant 8.3 — prefix max and suffix max; water at pos = min(leftMax[pos], rightMax[pos]) − height[pos]
"contiguous coverage" / "do segments cover every day"Variant 8.4 — prefix boolean OR; check that every position is covered from the left by some predecessor and from the right by some successor
"repeated range aggregates" / "multiple queries on one array"Any variant — build once O(n), answer each query O(1)

The disqualifier: if the array mutates between queries, a static prefix array is invalidated on every update. Reach for a Fenwick tree or segment tree instead — they support point updates in O(log n).

4   Mental Model — two spotlights meeting in the middle

The one idea
Imagine two spotlights: one starts at the left and sweeps right, recording what it has seen so far at every step; the other starts at the right and sweeps left, doing the same. After both passes, every position holds two receipts — "everything to my left" and "everything to my right" — and the answer for that position is simply combining those two receipts in one operation. The expensive part (accumulation) is done once; every subsequent query spends nothing.

Trace the prefix-sum variant on [2, 3, 1, 4]. The sentinel prefix[0] = 0 expresses "nothing has been accumulated before the first element." Each subsequent cell adds the next element:

Traceleft-to-right pass building prefix sum (length n+1)
index   nums:   [  2,   3,   1,   4  ]
prefix  build:  [  0,   2,   5,   6,  10  ]
                  ↑
                  sentinel (sum of zero elements)

prefix[pos+1] = prefix[pos] + nums[pos]. The sentinel at index 0 means every range query uses the same formula: sum(left, right) = prefix[right+1] − prefix[left], with no special-casing for left = 0.

Now verify two range queries against the table: sum(1, 2) (indices 1 and 2, values 3 and 1) = prefix[3] − prefix[1] = 6 − 2 = 4. Correct. sum(0, 3) = prefix[4] − prefix[0] = 10 − 0 = 10. Correct. Both answers read from two table cells — no loop.

For the "combine at index" variants (product except self, trapped water), the mental model is the same two sweeps, but the combine step uses both arrays simultaneously:

Traceproduct-except-self on [1, 2, 3, 4] — left and right passes, then combine
index      0    1    2    3
nums:      1    2    3    4

left pass  (prefix product, sentinel = 1)
prefix:    1    1    2    6       prefix[pos] = product of nums[0..pos-1]

right pass (suffix product, sentinel = 1)
suffix:    24   12   4    1       suffix[pos] = product of nums[pos+1..n-1]

combine    prefix[pos] * suffix[pos]
result:    24   12   8    6       each cell = product of everything EXCEPT nums[pos]

Division is never needed. The "gap" at position pos is bridged by placing the left product in prefix[pos] and the right product in suffix[pos], then multiplying — neither array knows about position pos itself.

4.1   The one question to ask

The single interview question
"Am I computing the same thing from scratch for every position or pair?" If yes — you have repeated work, and precomputation eliminates it.

Every prefix/suffix problem is the same diagnosis in a different costume. The brute force always contains a nested loop where the inner loop restarts from the beginning or the end for each outer element. That restart is the signal. Below is every problem on this page, reduced to that single question:

Problem What you'd recompute naively (the repeated work) What you precompute instead Where to store it
LC 42 — Trapping Rain Water For each bar: scan left for tallest wall, scan right for tallest wall → O(n) per bar One left pass → leftMax[]; one right pass → rightMax[]; combine in O(1) Array indexed by position
LC 238 — Product Except Self For each index: multiply all other elements → O(n) per index Left products pass + right products pass; combine at each index Array indexed by position
CUSTOM 1 — Relay Handoff Coverage For each team pair: recheck whether coverage ranges connect → O(teams²) Left pass: furthest day each team reaches forward; right pass: earliest day reachable backward; combine at each split Map indexed by team / day
LC 336 — Palindrome Pairs For each word: check every other word as a partner → O(w²·L) For each word at every split point: precompute the complement string that would complete a palindrome; store it so lookups are O(1) Hashmap indexed by the complement string
Why the storage shape differs: if retrieval is by position in the array ("what is the max to my left at index 3?"), an array is the right structure — O(1) access by index. If retrieval is by the value or content of what you're looking for ("does the reverse of this string exist as a word?"), a hashmap is the right structure — O(1) lookup by key. The precomputation logic is the same; only the index changes.
The follow-up question: once you've confirmed "yes, repeated work," ask "what am I indexing by when I retrieve the precomputed answer?" Position → array. Value/content → hashmap.

5   Pseudocode Primer

The two-pass shape is identical across all variants. Only the accumulation operation and the combine step change.

Pseudocodecanonical two-pass prefix/suffix build + combine
FUNCTION prefixSuffixSolve(nums):
    n ← length(nums)

    ① LEFT PASS — build prefix array of length n+1 (sentinel at index 0)
    prefix ← array of size n + 1
    prefix[0] ← IDENTITY              e.g. 0 for sum, 1 for product, false for OR
    FOR pos FROM 0 TO n − 1:
        prefix[pos + 1] ← ACCUMULATE(prefix[pos], nums[pos])

    ② RIGHT PASS — build suffix array of length n+1 (sentinel at index n)
    suffix ← array of size n + 1
    suffix[n] ← IDENTITY
    FOR pos FROM n − 1 DOWNTO 0:
        suffix[pos] ← ACCUMULATE(suffix[pos + 1], nums[pos])

    ③ COMBINE — answer each query in O(1)
    result ← array of size n
    FOR pos FROM 0 TO n − 1:
        result[pos] ← COMBINE(prefix[pos], suffix[pos + 1])

    RETURN result
Why it works: prefix[pos] holds the aggregate of nums[0..pos-1] (nothing from position pos itself), and suffix[pos+1] holds the aggregate of nums[pos+1..n-1] (nothing from position pos itself). Combining them yields "everything except position pos" — the gap is the element at pos, and neither array filled it.

For range queries (not "combine at index") the query formula instead of combine loop is:

Pseudocoderange-sum query using the prefix array
FUNCTION query(prefix, left, right):
    RETURN prefix[right + 1] − prefix[left]
    # left and right are inclusive 0-based indices into the original array

6   Iterative-Only Pattern — no recursion, ever

Prefix/suffix precomputation is inherently iterative: each pass is a single for loop over a flat array, accumulating one scalar. There is no recursive formulation — a recursive aggregate over a range is exactly the O(n) per-query work the precomputed array exists to eliminate.

  • No call stack — state is two arrays of length n+1 and two loop counters. Nothing to overflow at n = 10⁵.
  • No divide-and-conquer needed — segment trees and Fenwick trees exist for the mutable-array case; for static arrays, two flat passes dominate them on constant factors.
  • Direction is deliberate — the left pass always runs 0 → n−1 and the right pass always runs n−1 → 0. A reversed right pass is a bug, not a style choice.

7   Swift Translation

Swiftannotated prefix-sum template — compilable starting point
func buildPrefixSum(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var prefix = [Int](repeating: 0, count: count + 1)  // length n+1
    // prefix[0] = 0 is the sentinel — sum of zero elements
    for index in 0 ..< count {
        prefix[index + 1] = prefix[index] + nums[index]
    }
    return prefix
}

func rangeSum(_ prefix: [Int], left: Int, right: Int) -> Int {
    // Both left and right are inclusive 0-based indices.
    prefix[right + 1] - prefix[left]
}
Swiftproduct-except-self template
func productExceptSelf(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var prefixProduct = [Int](repeating: 1, count: count + 1)
    var suffixProduct = [Int](repeating: 1, count: count + 1)

    for index in 0 ..< count {
        prefixProduct[index + 1] = prefixProduct[index] * nums[index]
    }
    for index in stride(from: count - 1, through: 0, by: -1) {
        suffixProduct[index] = suffixProduct[index + 1] * nums[index]
    }

    return (0 ..< count).map { index in
        prefixProduct[index] * suffixProduct[index + 1]
    }
}
Swift-specific cost to say out loud: [Int](repeating:count:) is O(n) and allocates a contiguous block — far cheaper than appending in a loop. stride(from:through:by:) is the idiomatic descending range in Swift; for index in (0..<count).reversed() is equally valid and more readable to some reviewers.

Enumerating all splits with prefix(_:) / suffix(_:)

The one idea — split enumeration vs. sentinel
Looping 0...word.count is not a sentinel pattern. It is an enumeration of split points. An n-character word has exactly n+1 ways to split into a left part and a right part: before the first character, between every adjacent pair, and after the last character. The loop visits all of them. The empty string on either side is not a degenerate edge case — it is the valid "take nothing from this side" split, and Swift's prefix(_:) / suffix(_:) produce it for free at splitPoint = 0 and splitPoint = count respectively.
Swiftsplit-point enumeration — all n+1 splits of a word
let word = Array("lls")   // count = 3 → 4 split points: 0, 1, 2, 3

for splitPoint in 0...word.count {
    let leftPart  = String(word.prefix(splitPoint))            // "" | "l" | "ll" | "lls"
    let rightPart = String(word.suffix(word.count - splitPoint)) // "lls" | "ls" | "s" | ""
}
Traceall four split points for word = "lls"
splitPoint   leftPart   rightPart
──────────────────────────────────
    0            ""        "lls"     ← splitPoint=0  → prefix(0)=""   suffix(3)="lls"
    1            "l"       "ls"      ← splitPoint=1  → prefix(1)="l"  suffix(2)="ls"
    2            "ll"      "s"       ← splitPoint=2  → prefix(2)="ll" suffix(1)="s"
    3            "lls"     ""        ← splitPoint=3  → prefix(3)="lls" suffix(0)=""
Why prefix(_:) / suffix(_:) — not subscript ranges.

Subscript ranges (array[...(-1)], array[index-1...]) require the index to be valid. At the boundary split points (0 and count), the math produces out-of-bounds indices — behavior becomes undefined. prefix(_:) and suffix(_:) clamp silently: prefix(0) returns an empty slice, prefix(999) on a 3-element array returns all 3 elements. Safe by design. Always prefer them for split enumeration.

The subscript trap — why it "works" until it doesn't
▸array[...(-1)] compiling does NOT mean it is safe. Negative indices land in undefined behavior territory in Swift's Array subscript. It may return an empty slice, it may crash, or it may return garbage — the outcome depends on the runtime's internal bounds-check path, which is not a contract. array[...(-6)] hitting a different path and crashing while array[...(-1)] silently "works" is a perfect illustration: you got lucky, not correct. Use prefix(splitPoint) and this edge case disappears entirely.
Naming the pattern: this loop is called split-point enumeration or split-position scan in interview settings — not "sentinel" and not "prefix array." The sentinels on this page (prefix[0] = 0, prefix[0] = 1) are a different concept: they are the identity element placed at index 0 of a precomputed accumulation array. A split-point loop and a prefix-sum array both use the word "prefix," but they are different tools. LC 336 (Palindrome Pairs) is where you will see the split-point loop most commonly — every word is split at each of its n+1 positions, and each half is checked against a dictionary.

8   Variant Catalog

Variant 8.1 Prefix Sum — range query (LC 303)
O(n) buildO(n) space
When to use: the problem issues multiple "sum of range [left, right]" queries against a static array. Build once, answer each query in O(1). The sentinel at prefix[0] = 0 is not optional — it is what makes the query formula prefix[right+1] − prefix[left] work for all ranges including left = 0.

Payload: integer sum. Identity: 0. Combine: subtraction (prefix[right+1] − prefix[left]).

Demonstrated in: LC 303 — Range Sum Query - Immutable (coming); LC 560 — Subarray Sum Equals K (solo problem below).

Variant 8.2 Prefix × Suffix Product — product except self (LC 238)
O(n) buildO(n) space
When to use: "return an array where each element is the product of all other elements, without using division." Division would be O(n) and fails on zeros. The two-pass approach bypasses division entirely: for index pos, prefix[pos] accumulates nums[0..pos-1] and suffix[pos+1] accumulates nums[pos+1..n-1]; multiplying them leaves exactly nums[pos] excluded.

Payload: integer product. Identity: 1. Combine: multiplication (prefix[pos] × suffix[pos+1]).

Why no division: the "gap" at position pos is structural — neither the left pass nor the right pass includes nums[pos]. The element is excluded by construction, not by inverting the total.

Demonstrated in: LC 238 — Product of Array Except Self (coming).

Variant 8.3 Prefix Max + Suffix Max — physical simulation (LC 42, two-array solution)
O(n) buildO(n) space
When to use: problems where each position's answer depends on the largest (or smallest) value to its left and the largest (or smallest) value to its right — most commonly physical simulations. Trapping Rain Water is the canonical example: the water at position pos is min(leftMax[pos], rightMax[pos]) − height[pos], clamped to 0.

Payload: running maximum. Identity: 0 (or −∞). Combine: min(prefixMax[pos], suffixMax[pos]) − height[pos] per position.

Relation to the two-pointer solution (LC 42): the two-pointer approach is the O(1)-space compression of this two-array solution — it avoids materializing both arrays by observing that the bottleneck side is always determined by min(leftMax, rightMax). If you understand this variant first, the two-pointer version follows immediately. The walkthrough at LC42_master.html covers both.

Demonstrated in: LC 42 — Trapping Rain Water (the alternate two-array solution is discussed in that walkthrough's §9 follow-up section).

Variant 8.4 Prefix/Suffix Boolean Coverage
O(t·d) buildO(t·d) space
When to use: given a set of segments (e.g. athletes covering day-ranges), determine whether a valid handoff split point exists — i.e. a position where one group covers everything to its left and another group covers everything to its right. The "payload" is a set of covered days rather than a scalar, and the identity is the empty set.

Payload: set of covered indices (or a boolean flag per day). Identity: empty set / all-false. Combine: check that the prefix set and the suffix set together cover all required days exactly, with the handoff condition (≤ vs ==) enforced correctly.

Why not brute force: checking all O(t²) split pairs would cost O(t²·d) where t = team count and d = number of days. Building prefix and suffix coverage arrays reduces this to O(t·d) build + O(t) to scan split points, because each candidate split reads two precomputed sets in O(d) rather than reconstructing them.

Demonstrated in: CUSTOM 1 — Relay Handoff Coverage (coming walkthrough).

9   Complexity Analysis

(a) Building both arrays is O(n) time and O(n) space — nothing more. The left pass visits each element exactly once; the right pass visits each element exactly once. Two flat loops, no nesting, no recursion. The prefix and suffix arrays each require n+1 slots, so total space is 2n+2 — linear. There is no hidden work: each accumulation step is a single arithmetic or logical operation.

(b) Why this beats brute force per query. The naive approach re-scans the range for every query: k queries on a length-n array costs O(k·n). After O(n) preprocessing, each query costs O(1) — look up two table entries and combine them. For k queries the total cost becomes O(n + k). When k is even 2, precomputation wins. When k = n (a common pattern), the improvement is O(n²) → O(n).

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Off-by-one on the sentinel — forgetting prefix[0] = 0 and why it exists. The sentinel is not a convenience; it is what makes the formula prefix[right+1] − prefix[left] work uniformly when left = 0. Without it, querying the range [0, right] requires a special case (prefix[right] with no subtraction) — a guaranteed source of bugs. Always allocate the prefix array at length n+1, not n, and initialize index 0 to the identity.
▸Wrong index formula: prefix[right] − prefix[left] instead of prefix[right+1] − prefix[left]. When the prefix array has length n+1 with a sentinel, prefix[k] stores the aggregate of nums[0..k-1]. So "sum of nums[left..right] inclusive" is prefix[right+1] − prefix[left]. Using prefix[right] silently drops nums[right] from every range result. Derive the formula from the invariant, not from memory.
▸Assuming the pattern requires numeric data. The payload can be any monoid — booleans (OR), sets (union), strings (concatenation), even sorted lists. The two-pass shape does not care what "accumulate" means, only that it is associative and has an identity. A boolean prefix-OR array is as valid as a numeric prefix-sum array. When a coverage or membership problem feels quadratic, ask: "can I express the left-side aggregate as an accumulation?"
▸Materializing both arrays when one can be eliminated (space optimization). For the "combine at each index" variants (8.2, 8.3), the suffix array can be computed on a second pass while applying the combine immediately — no need to store it separately. This halves the allocation without changing time complexity. Only matters if the problem constrains space; don't premature-optimize the code's clarity before the algorithm is correct.
▸Using the pattern on a mutable array. Any update to nums[pos] invalidates every prefix cell from index pos+1 onward — an O(n) repair. For arrays that change between queries, a prefix array is wrong; use a Fenwick tree (O(log n) update and query) instead.

14   Solo Problem — no solution on this page

Solo problem — walkthrough available at LC560_master.html

LC 560 — Subarray Sum Equals K (Medium). Given an array of integers and an integer k, return the total number of subarrays whose sum equals k. The array may contain negative numbers. This problem combines Variant 8.1 (prefix sum) with a hash map to count prefix-sum occurrences — the natural next step after LC 303. Build the prefix sum mentally first, then ask: "what does prefix[right] − prefix[left] = k tell me about what to store in the map?"

  • Can you express "subarray sum from left to right" as a difference of two prefix values?
  • What does the hash map store, and what does each lookup mean?
  • Why is the sentinel prefix[0] = 0 (count 1 in the map) essential for subarrays starting at index 0?
  • Why does the approach still work when the array contains negative numbers, unlike a sliding window?
  • State the time and space complexity before coding. Is it the same as a plain prefix-sum build?
Hint 1 — constraint

Negative values mean a sliding window can't decide when to shrink. Prefix sums don't have this problem — the difference formula works regardless of sign.

Hint 2 — pattern

You want to count pairs (left, right) where prefix[right] − prefix[left] = k. Rearranging: you want prefix[left] = prefix[right] − k. As you build the prefix sum left to right, keep a frequency map of prefix values seen so far.

Hint 3 — sentinel

Initialize the map with { 0: 1 } before the loop. This represents the virtual position before index 0 — without it, subarrays starting from index 0 whose sum equals k are never counted.
