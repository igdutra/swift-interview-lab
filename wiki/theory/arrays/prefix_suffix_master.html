Prefix & Suffix Precomputation

Theory Masterfile· Language: Swift· Range Sum · Product · Physical Sim · Coverage

Prefix and suffix precomputation is a technique that eliminates repeated linear work by storing, at each index i, the accumulated result of every element to its left (the prefix) and every element to its right (the suffix). A single left-to-right pass builds the prefix array; a single right-to-left pass builds the suffix array. Any query that asks "what is the aggregate of some range, or of everything except position i" then combines prefix[i] and suffix[i] in one step — O(1) per query regardless of range length.

The "payload" stored at each position varies by problem — a running sum, a running product, a running maximum, a boolean flag, even a set of day indices — but the two-pass shape is always the same. This uniformity is the technique's power: recognizing any of these payload types as "something a left pass accumulates" immediately hands you an O(n) algorithm where the naive alternative scans the array again for every query, costing O(n²) in total.

2   Recognition Signals — when to reach for it

The problem rarely says "prefix array." It says one of these:

When the problem says…It maps to…
"range sum query" / "sum of elements from i to j"Variant 8.1 — prefix sum; answer is prefix[j+1] − prefix[i]
"product of all elements except self" / "no division"Variant 8.2 — prefix product × suffix product at each index
"water trapped" / "amount each bar can hold"Variant 8.3 — prefix max and suffix max; water at i = min(leftMax[i], rightMax[i]) − height[i]
"contiguous coverage" / "do segments cover every day"Variant 8.4 — prefix boolean OR; check that every position is covered from the left by some predecessor and from the right by some successor
"repeated range aggregates" / "multiple queries on one array"Any variant — build once O(n), answer each query O(1)
product / sum / max "without" a specific elementPrefix × suffix combine; the excluded element falls exactly at the join point
The disqualifier: if the array mutates between queries, a static prefix array is invalidated on every update. Reach for a Fenwick tree or segment tree instead — they support point updates in O(log n).

4   Mental Model — two spotlights meeting in the middle

The one idea
Imagine two spotlights: one starts at the left and sweeps right, recording what it has seen so far at every step; the other starts at the right and sweeps left, doing the same. After both passes, every position holds two receipts — "everything to my left" and "everything to my right" — and the answer for that position is simply combining those two receipts in one operation. The expensive part (accumulation) is done once; every subsequent query spends nothing.

Trace the prefix-sum variant on [2, 3, 1, 4]. The sentinel prefix[0] = 0 expresses "nothing has been accumulated before the first element." Each subsequent cell adds the next element:

Traceleft-to-right pass building prefix sum (length n+1)
index   nums:   [  2,   3,   1,   4  ]
prefix  build:  [  0,   2,   5,   6,  10  ]
                  ↑
                  sentinel (sum of zero elements)

prefix[i+1] = prefix[i] + nums[i]. The sentinel at index 0 means every range query uses the same formula: sum(i, j) = prefix[j+1] − prefix[i], with no special-casing for i = 0.

Now verify two range queries against the table: sum(1, 2) (indices 1 and 2, values 3 and 1) = prefix[3] − prefix[1] = 6 − 2 = 4. Correct. sum(0, 3) = prefix[4] − prefix[0] = 10 − 0 = 10. Correct. Both answers read from two table cells — no loop.

For the "combine at index" variants (product except self, trapped water), the mental model is the same two sweeps, but the combine step uses both arrays simultaneously:

Traceproduct-except-self on [1, 2, 3, 4] — left and right passes, then combine
index      0    1    2    3
nums:      1    2    3    4

left pass  (prefix product, sentinel = 1)
prefix:    1    1    2    6       prefix[i] = product of nums[0..i-1]

right pass (suffix product, sentinel = 1)
suffix:    24   12   4    1       suffix[i] = product of nums[i+1..n-1]

combine    prefix[i] * suffix[i]
result:    24   12   8    6       each cell = product of everything EXCEPT nums[i]

Division is never needed. The "gap" at position i is bridged by placing the left product in prefix[i] and the right product in suffix[i], then multiplying — neither array knows about position i itself.

5   Pseudocode Primer

The two-pass shape is identical across all variants. Only the accumulation operation and the combine step change.

Pseudocodecanonical two-pass prefix/suffix build + combine
FUNCTION prefixSuffixSolve(nums):
    n ← length(nums)

    ① LEFT PASS — build prefix array of length n+1 (sentinel at index 0)
    prefix ← array of size n + 1
    prefix[0] ← IDENTITY              e.g. 0 for sum, 1 for product, false for OR
    FOR i FROM 0 TO n − 1:
        prefix[i + 1] ← ACCUMULATE(prefix[i], nums[i])

    ② RIGHT PASS — build suffix array of length n+1 (sentinel at index n)
    suffix ← array of size n + 1
    suffix[n] ← IDENTITY
    FOR i FROM n − 1 DOWNTO 0:
        suffix[i] ← ACCUMULATE(suffix[i + 1], nums[i])

    ③ COMBINE — answer each query in O(1)
    result ← array of size n
    FOR i FROM 0 TO n − 1:
        result[i] ← COMBINE(prefix[i], suffix[i + 1])

    RETURN result
Why it works: prefix[i] holds the aggregate of nums[0..i-1] (nothing from position i itself), and suffix[i+1] holds the aggregate of nums[i+1..n-1] (nothing from position i itself). Combining them yields "everything except position i" — the gap is the element at i, and neither array filled it.

For range queries (not "combine at index") the query formula instead of combine loop is:

Pseudocoderange-sum query using the prefix array
FUNCTION query(prefix, left, right):
    RETURN prefix[right + 1] − prefix[left]
    # left and right are inclusive 0-based indices into the original array

6   Iterative-Only Pattern — no recursion, ever

Prefix/suffix precomputation is inherently iterative: each pass is a single for loop over a flat array, accumulating one scalar. There is no recursive formulation — a recursive aggregate over a range is exactly the O(n) per-query work the precomputed array exists to eliminate.

  • No call stack — state is two arrays of length n+1 and two loop counters. Nothing to overflow at n = 10⁵.
  • No divide-and-conquer needed — segment trees and Fenwick trees exist for the mutable-array case; for static arrays, two flat passes dominate them on constant factors.
  • Direction is deliberate — the left pass always runs 0 → n−1 and the right pass always runs n−1 → 0. A reversed right pass is a bug, not a style choice.

7   Swift Translation

Swiftannotated prefix-sum template — compilable starting point
func buildPrefixSum(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var prefix = [Int](repeating: 0, count: count + 1)  // length n+1
    // prefix[0] = 0 is the sentinel — sum of zero elements
    for index in 0 ..< count {
        prefix[index + 1] = prefix[index] + nums[index]
    }
    return prefix
}

func rangeSum(_ prefix: [Int], left: Int, right: Int) -> Int {
    // Both left and right are inclusive 0-based indices.
    prefix[right + 1] - prefix[left]
}
Swiftproduct-except-self template
func productExceptSelf(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var prefixProduct = [Int](repeating: 1, count: count + 1)
    var suffixProduct = [Int](repeating: 1, count: count + 1)

    for index in 0 ..< count {
        prefixProduct[index + 1] = prefixProduct[index] * nums[index]
    }
    for index in stride(from: count - 1, through: 0, by: -1) {
        suffixProduct[index] = suffixProduct[index + 1] * nums[index]
    }

    return (0 ..< count).map { index in
        prefixProduct[index] * suffixProduct[index + 1]
    }
}
Swift-specific cost to say out loud: [Int](repeating:count:) is O(n) and allocates a contiguous block — far cheaper than appending in a loop. stride(from:through:by:) is the idiomatic descending range in Swift; for index in (0..<count).reversed() is equally valid and more readable to some reviewers.

8   Variant Catalog

Variant 8.1 Prefix Sum — range query (LC 303)
O(n) buildO(n) space
When to use: the problem issues multiple "sum of range [i, j]" queries against a static array. Build once, answer each query in O(1). The sentinel at prefix[0] = 0 is not optional — it is what makes the query formula prefix[j+1] − prefix[i] work for all ranges including i = 0.

Payload: integer sum. Identity: 0. Combine: subtraction (prefix[j+1] − prefix[i]).

Demonstrated in: LC 303 — Range Sum Query - Immutable (coming); LC 560 — Subarray Sum Equals K (solo problem below).

Variant 8.2 Prefix × Suffix Product — product except self (LC 238)
O(n) buildO(n) space
When to use: "return an array where each element is the product of all other elements, without using division." Division would be O(n) and fails on zeros. The two-pass approach bypasses division entirely: for index i, prefix[i] accumulates nums[0..i-1] and suffix[i+1] accumulates nums[i+1..n-1]; multiplying them leaves exactly nums[i] excluded.

Payload: integer product. Identity: 1. Combine: multiplication (prefix[i] × suffix[i+1]).

Why no division: the "gap" at position i is structural — neither the left pass nor the right pass includes nums[i]. The element is excluded by construction, not by inverting the total.

Demonstrated in: LC 238 — Product of Array Except Self (coming).

Variant 8.3 Prefix Max + Suffix Max — physical simulation (LC 42, two-array solution)
O(n) buildO(n) space
When to use: problems where each position's answer depends on the largest (or smallest) value to its left and the largest (or smallest) value to its right — most commonly physical simulations. Trapping Rain Water is the canonical example: the water at position i is min(leftMax[i], rightMax[i]) − height[i], clamped to 0.

Payload: running maximum. Identity: 0 (or −∞). Combine: min(prefixMax[i], suffixMax[i]) − height[i] per position.

Relation to the two-pointer solution (LC 42): the two-pointer approach is the O(1)-space compression of this two-array solution — it avoids materializing both arrays by observing that the bottleneck side is always determined by min(leftMax, rightMax). If you understand this variant first, the two-pointer version follows immediately. The walkthrough at LC42_master.html covers both.

Demonstrated in: LC 42 — Trapping Rain Water (the alternate two-array solution is discussed in that walkthrough's §9 follow-up section).

Variant 8.4 Prefix/Suffix Boolean Coverage
O(t·d) buildO(t·d) space
When to use: given a set of segments (e.g. athletes covering day-ranges), determine whether a valid handoff split point exists — i.e. a position where one group covers everything to its left and another group covers everything to its right. The "payload" is a set of covered days rather than a scalar, and the identity is the empty set.

Payload: set of covered indices (or a boolean flag per day). Identity: empty set / all-false. Combine: check that the prefix set and the suffix set together cover all required days exactly, with the handoff condition (≤ vs ==) enforced correctly.

Why not brute force: checking all O(t²) split pairs would cost O(t²·d) where t = team count and d = number of days. Building prefix and suffix coverage arrays reduces this to O(t·d) build + O(t) to scan split points, because each candidate split reads two precomputed sets in O(d) rather than reconstructing them.

Demonstrated in: CUSTOM 1 — Relay Handoff Coverage (coming walkthrough).

9   Complexity Analysis

(a) Building both arrays is O(n) time and O(n) space — nothing more. The left pass visits each element exactly once; the right pass visits each element exactly once. Two flat loops, no nesting, no recursion. The prefix and suffix arrays each require n+1 slots, so total space is 2n+2 — linear. There is no hidden work: each accumulation step is a single arithmetic or logical operation.

(b) Why this beats brute force per query. The naive approach re-scans the range for every query: k queries on a length-n array costs O(k·n). After O(n) preprocessing, each query costs O(1) — look up two table entries and combine them. For k queries the total cost becomes O(n + k). When k is even 2, precomputation wins. When k = n (a common pattern), the improvement is O(n²) → O(n).

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Off-by-one on the sentinel — forgetting prefix[0] = 0 and why it exists. The sentinel is not a convenience; it is what makes the formula prefix[j+1] − prefix[i] work uniformly when i = 0. Without it, querying the range [0, j] requires a special case (prefix[j] with no subtraction) — a guaranteed source of bugs. Always allocate the prefix array at length n+1, not n, and initialize index 0 to the identity.
▸Wrong index formula: prefix[j] − prefix[i] instead of prefix[j+1] − prefix[i]. When the prefix array has length n+1 with a sentinel, prefix[k] stores the aggregate of nums[0..k-1]. So "sum of nums[i..j] inclusive" is prefix[j+1] − prefix[i]. Using prefix[j] silently drops nums[j] from every range result. Derive the formula from the invariant, not from memory.
▸Assuming the pattern requires numeric data. The payload can be any monoid — booleans (OR), sets (union), strings (concatenation), even sorted lists. The two-pass shape does not care what "accumulate" means, only that it is associative and has an identity. A boolean prefix-OR array is as valid as a numeric prefix-sum array. When a coverage or membership problem feels quadratic, ask: "can I express the left-side aggregate as an accumulation?"
▸Materializing both arrays when one can be eliminated (space optimization). For the "combine at each index" variants (8.2, 8.3), the suffix array can be computed on a second pass while applying the combine immediately — no need to store it separately. This halves the allocation without changing time complexity. Only matters if the problem constrains space; don't premature-optimize the code's clarity before the algorithm is correct.
▸Using the pattern on a mutable array. Any update to nums[i] invalidates every prefix cell from index i+1 onward — an O(n) repair. For arrays that change between queries, a prefix array is wrong; use a Fenwick tree (O(log n) update and query) instead.

Solo problem — no solution on this page

LC 560 — Subarray Sum Equals K (Medium). Given an array of integers and an integer k, return the total number of subarrays whose sum equals k. The array may contain negative numbers. This problem combines Variant 8.1 (prefix sum) with a hash map to count prefix-sum occurrences — the natural next step after LC 303. Build the prefix sum mentally first, then ask: "what does prefix[j] − prefix[i] = k tell me about what to store in the map?"

  • Can you express "subarray sum from i to j" as a difference of two prefix values?
  • What does the hash map store, and what does each lookup mean?
  • Why is the sentinel prefix[0] = 0 (count 1 in the map) essential for subarrays starting at index 0?
  • Why does the approach still work when the array contains negative numbers, unlike a sliding window?
  • State the time and space complexity before coding. Is it the same as a plain prefix-sum build?
Hint 1 — constraint

Negative values mean a sliding window can't decide when to shrink. Prefix sums don't have this problem — the difference formula works regardless of sign.

Hint 2 — pattern

You want to count pairs (i, j) where prefix[j] − prefix[i] = k. Rearranging: you want prefix[i] = prefix[j] − k. As you build the prefix sum left to right, keep a frequency map of prefix values seen so far.

Hint 3 — sentinel

Initialize the map with { 0: 1 } before the loop. This represents the virtual position before index 0 — without it, subarrays starting from index 0 whose sum equals k are never counted.
