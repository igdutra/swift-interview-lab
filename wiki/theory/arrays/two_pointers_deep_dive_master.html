Two Pointers — Deep Dive

Theory Masterfile· Language: Swift· Converging sum · 3Sum anchor · Palindrome · Left/right-max

Two pointers is a traversal technique that places one index at each end of a sorted array and moves them toward each other, making a greedy steering decision at every step. Because the array is sorted, an element that is too large can only be reduced by moving the right pointer left, and an element that is too small can only be increased by moving the left pointer right. This replaces the O(n²) "check all pairs" brute force with a single O(n) convergence walk.

Four distinct interview shapes use this idea: converging toward a target sum, fixing one element as an anchor and converging an inner window (3Sum), comparing from both ends to check symmetry (palindrome), and accumulating the running maximum from each side to compute a bounded quantity (trapping rain water). All four share the same invariant: the answer, if it exists, always lies within the range [left, right].

When NOT to use two pointers: if the array cannot be sorted without destroying the answer (e.g., when indices must be preserved, as in Two Sum LC1 unsorted), use a hash map instead. Two pointers require sorted order to make the steering decision meaningful.
Two pointers vs sliding window: two pointers converge (l moves right, r moves left). Sliding window expands (both l and r only move right). If both pointers must move rightward, it's a window — not two pointers.

4   Mental Model — two people walking toward each other

The one idea
Imagine two people standing at opposite ends of a sorted hallway of numbered doors. They shout their door number to each other and compare their sum to a target. If the sum is too big, the person on the right takes one step inward (to a smaller number). If too small, the person on the left takes a step inward. They stop when they either find the target or meet. This works because the hallway is sorted — stepping inward is always the right direction.

Watch the converging sum find a pair in [-4, -1, 0, 3, 5, 8] with target 4:

Traceconverging two pointers — steer by comparing sum to target
array:  [-4, -1,  0,  3,  5,  8]
         l=0              r=5

l=0 r=5  sum = −4 + 8 = 4 ✓  → found! return [−4, 8]

(if target were 5:)
l=0 r=5  sum = −4 + 8 = 4 < 5  → move l right (sum too small)
l=1 r=5  sum = −1 + 8 = 7 > 5  → move r left (sum too big)
l=1 r=4  sum = −1 + 5 = 4 < 5  → move l right
l=2 r=4  sum =  0 + 5 = 5 ✓  → found! return [0, 5]

Each step eliminates one element as a candidate. Moving l right eliminates nums[l] from all future pairs (no right partner can be small enough). Moving r left eliminates nums[r]. At most n − 1 steps total.

5   Pseudocode Primer

Two canonical templates cover all four variants.

Pseudocodetemplate A — converging sum / min-side steering (LC11 · LC15 inner window shape)
FUNCTION twoPointers(input):
    sort(input)                      // O(n log n) — required for steering
    l ← 0,  r ← length(input) − 1
    best ← initial_sentinel

    WHILE l < r:
        candidate ← evaluate(input[l], input[r])
        best ← update(best, candidate)
        IF sum_too_small: l ← l + 1     // left element is too small — move right
        ELSE:             r ← r − 1     // right element is too large — move left

    RETURN best
Pseudocodetemplate B — left/right-max accumulator (LC42 shape)
FUNCTION leftRightMax(heights):
    l ← 0,  r ← length(heights) − 1
    leftMax ← 0,  rightMax ← 0,  total ← 0

    WHILE l < r:
        IF heights[l] ≤ heights[r]:
            leftMax ← max(leftMax, heights[l])
            total ← total + (leftMax − heights[l])   // water bounded by the left
            l ← l + 1
        ELSE:
            rightMax ← max(rightMax, heights[r])
            total ← total + (rightMax − heights[r])  // water bounded by the right
            r ← r − 1

    RETURN total
Why the left/right-max works: water at any position is bounded by min(leftMax, rightMax) − height[i]. When heights[l] ≤ heights[r], we know the left side is the binding constraint — the rightMax is at least as tall — so we can safely compute and accumulate water for position l without knowing the final rightMax.

6   Iterative-Only Pattern

Two pointers is inherently iterative: a single while l < r loop. There is no recursive structure to consider:

  • No divide-and-conquer: you are not splitting the array and combining results. You are eliminating candidates one at a time from each end.
  • No backtracking: once a pointer moves, it never moves back. The invariant guarantees that the discarded elements contain no valid answers.
  • The anchor pattern (3Sum): the outer for loop is not recursion — it is simply applying the inner two-pointer sweep to each successive subarray. It is still O(n²) total: n anchors × O(n) inner sweep each.
Why pointers can't move backward: the correctness argument rests on the invariant "the answer lies in [l, r]." If a pointer could move backward, it would re-examine eliminated candidates, destroying the invariant and the O(n) bound.

7   Swift Translation

SwiftLC11 — Container With Most Water, fully annotated
func maxArea(_ height: [Int]) -> Int {
    var l = 0, r = height.count - 1
    var best = 0

    while l < r {
        // area = width × min height
        let area = (r - l) * min(height[l], height[r])
        best = max(best, area)

        // min-side rule: moving the taller side inward can only shrink area
        if height[l] < height[r] {
            l += 1
        } else {
            r -= 1
        }
    }
    return best
}
SwiftLC42 — Trapping Rain Water, left/right-max accumulator
func trap(_ height: [Int]) -> Int {
    var l = 0, r = height.count - 1
    var leftMax = 0, rightMax = 0
    var total = 0

    while l < r {
        if height[l] <= height[r] {
            leftMax = max(leftMax, height[l])
            total += leftMax - height[l]   // left side is the binding constraint
            l += 1
        } else {
            rightMax = max(rightMax, height[r])
            total += rightMax - height[r]  // right side is the binding constraint
            r -= 1
        }
    }
    return total
}
Swift-specific notes: nums.sorted() returns a new array (O(n log n), O(n) space); nums.sort() sorts in-place. For in-place pointer problems (LC125 on a character array), use var chars = Array(s) first — Swift strings are not subscriptable with integers. while l < r (strict) handles the single-element and two-element cases correctly.

8   Variant Catalog

Variant 8.1 Converging Sum / Min-Side Steering
O(n)O(1)
When to use: sorted array + "find a pair with a property" (sum, area, distance). The array's sorted order means: if the current pair is too small, you can only make it larger by moving the left pointer right; if too large, only by moving the right pointer left.

Steering rule (LC11 — Container With Most Water): area = width × min(height[l], height[r]). Moving the taller side inward reduces width while the min height stays the same or gets smaller — guaranteed worse. Always move the shorter side.

WHILE l < r:
    evaluate(arr[l], arr[r])
    IF arr[l] < arr[r]: l += 1   // left is the weak side
    ELSE:               r -= 1   // right is the weak side

Problems: LC11 — Container With Most Water.

Variant 8.2 Fixed Anchor + Inner Converging Window (3Sum)
O(n²)O(1)
When to use: "find three (or k) numbers that sum to a target." Sort first, then fix each element as the anchor with an outer loop and run Variant 8.1 on the remaining subarray. O(n²) total — O(n) anchors × O(n) inner sweep.

Critical — duplicate skipping: the outer loop must skip duplicate anchors; the inner pointers must skip duplicates after a valid triplet is found. Without these skips, the result contains duplicate triplets.

SORT array
FOR k FROM 0 TO n − 3:
    IF k > 0 AND array[k] == array[k−1]: CONTINUE   // skip duplicate anchor
    l ← k + 1,  r ← n − 1
    WHILE l < r:
        sum ← array[k] + array[l] + array[r]
        IF sum == 0:
            record triplet
            WHILE l < r AND array[l] == array[l+1]: l += 1   // skip dup
            WHILE l < r AND array[r] == array[r−1]: r -= 1   // skip dup
            l += 1; r -= 1
        ELSE IF sum < 0: l += 1
        ELSE:            r -= 1

Problems: LC15 — 3Sum.

Variant 8.3 Palindrome Check (Converging Compare)
O(n)O(1)
When to use: "is this string a palindrome?" — or any problem where you compare elements symmetrically from both ends. The array does not need to be sorted; the convergence is driven by the structure of the problem (left must equal right).

LC125 twist: the problem defines a palindrome over only alphanumeric characters, ignoring case. Skip non-alphanumeric characters at each pointer before comparing.

WHILE l < r:
    WHILE l < r AND NOT alphanumeric(s[l]): l += 1   // skip junk
    WHILE l < r AND NOT alphanumeric(s[r]): r -= 1   // skip junk
    IF lowercase(s[l]) ≠ lowercase(s[r]): RETURN false
    l += 1; r -= 1
RETURN true

Problems: LC125 — Valid Palindrome (warm-up for the Two Pointers block).

Variant 8.4 Left/Right-Max Accumulator (Trapping Rain Water)
O(n)O(1)
When to use: "how much water / space is bounded between a left wall and a right wall at each position?" Water at position i equals min(maxHeightToLeft, maxHeightToRight) − height[i]. The two-pointer version computes this in O(n) time and O(1) space by maintaining running maxima.

Key insight: when height[l] ≤ height[r], the left side is the binding constraint regardless of what lies between l and r. We can safely accumulate leftMax − height[l] for position l without knowing the final rightMax — because rightMax is already ≥ leftMax.

WHILE l < r:
    IF height[l] ≤ height[r]:
        leftMax ← max(leftMax, height[l])
        total += leftMax − height[l]
        l += 1
    ELSE:
        rightMax ← max(rightMax, height[r])
        total += rightMax − height[r]
        r -= 1

Problems: LC42 — Trapping Rain Water.

O(n) space alternative: precompute leftMax[] and rightMax[] arrays, then iterate once. Easier to reason about but uses O(n) space. Mention it first in an interview, then optimize to the two-pointer O(1) version.
Variant 8.5 Floyd's Cycle Detection (Slow & Fast Pointers)
O(log n)O(1)
When to use: "does this process eventually repeat a state?" — the algorithm generates a sequence of values one at a time, and you need to detect whether it cycles without storing every value seen. Slow moves one step; fast moves two. If there is a cycle, fast laps slow and they meet. If there is no cycle, fast reaches the terminal value first.

Key difference from Seen-State Tracking (hash set): the set approach is O(n) space — it stores every visited state. Floyd's uses O(1) space by exploiting the fact that a cycle guarantees the two pointers will eventually land on the same value. The trade-off: Floyd's is slightly harder to explain; mention it as a follow-up optimization in interviews.

FUNCTION floydCycle(start):
    slow ← next(start)
    fast ← next(next(start))

    WHILE fast ≠ terminal AND slow ≠ fast:
        slow ← next(slow)
        fast ← next(next(fast))

    RETURN fast == terminal   // true → no cycle; false → cycle detected
Why they must meet: in a cycle of length k, fast gains one step on slow per iteration. After at most k iterations, fast has lapped slow — they land on the same node. The cycle is guaranteed to be entered because the sequence is deterministic and the state space is finite.
Happy Number (LC202) application: the "next state" function is the sum of squared digits. The sequence either reaches 1 (happy) or falls into the cycle 4 → 16 → 37 → 58 → 89 → 145 → 42 → 20 → 4. Floyd's detects the cycle in O(log n) time and O(1) space.
func isHappyNumber(_ number: Int) -> Bool {
    func nextSquaresSum(_ current: Int) -> Int {
        var remaining = current, sum = 0
        while remaining > 0 {
            let digit = remaining % 10
            sum += digit * digit
            remaining /= 10
        }
        return sum
    }

    var slow = number
    var fast = nextSquaresSum(number)

    while fast != 1 && slow != fast {
        slow = nextSquaresSum(slow)
        fast = nextSquaresSum(nextSquaresSum(fast))
    }
    return fast == 1
}

Problems: LC202 — Happy Number.

9   Complexity Analysis

Why the pointer walk is O(n). At each step of the while l < r loop, exactly one pointer moves — either l increases by 1 or r decreases by 1. The two pointers start n − 1 steps apart and converge to the same index. Total steps: at most n − 1. One element is permanently eliminated per step — the invariant holds because the eliminated element is proven to contain no valid answer given the sorted order.

Why sort dominates. The pointer walk is O(n) but sorting is O(n log n). For problems that require sorting (pair/triple sum), the total is O(n log n). For problems that don't require sorting (palindrome, rain water), the total is O(n).

10   Common Pitfalls

Pitfalls — bugs this pattern invites
▸Forgetting to sort first (sum variants). Without sorted order, moving l right or r left has no predictable effect on the sum. The algorithm will produce wrong or incomplete answers silently. Always sort before the while l < r loop for sum problems.
▸Moving both pointers on equality. On finding a valid pair, moving only one pointer is correct — the other might still form additional valid pairs with new elements. Skipping both loses valid results. In 3Sum, after recording a triplet, skip duplicates and then move both.
▸Missing duplicate skips in 3Sum. Without skipping duplicate anchors and duplicate inner pairs, the result list contains identical triplets. The skip loops — while nums[l] == nums[l+1] { l += 1 } — must run after recording a triplet, not before the next iteration.
▸Using while l <= r instead of while l < r. When l equals r, both pointers are at the same element — no valid pair can exist. The <= variant processes a single element as a "pair with itself," producing off-by-one errors in palindrome checks and incorrect area calculations in LC11.
▸Subscripting a Swift String with integers. s[i] is not valid on Swift strings. Convert with let chars = Array(s) first. This is O(n) once and buys O(1) indexing for the rest of the algorithm.
▸Confusing two pointers with sliding window. If you catch yourself moving both l and r in the same direction (both right), you're writing a sliding window, not a converging two-pointer. Two pointers always converge — l moves right OR r moves left at each step, never both in the same direction.
