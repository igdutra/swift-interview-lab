Strings & Simulation

Theory Masterfile · Overview· Language: Swift· Produce · Simulate · Parse
At a Glance
Type
Linear single-pass processing of a character sequence or a small state
Core idea
Walk an index once; at each step either emit/mutate (produce) or fold the char into a structure (consume) — never re-scan
Time
O(n) for one pass; O(n·k) when each step does bounded extra work; watch for hidden O(n²)
Space
O(n) output buffer or stack; O(1) running state
Signal words
"build", "simulate", "step by step", "parse", "evaluate", "format", "without using built-ins"
The fork
Producing a string/state → Text Formatting; consuming one → String Parsing
Walkthroughs
LC415 · LC1041 · LC755 · LC68 · LC227 · LC385 · LC336

Strings & simulation problems ask you to process a character sequence — or a small evolving state described by one — in a single left-to-right pass, doing explicit, mechanical work the language's built-ins would normally hide. There is rarely a clever asymptotic trick; the difficulty is in the bookkeeping: carrying a digit, tracking a facing direction, distributing leftover spaces, or deciding exactly when a token ends.

Every problem in this block splits along one axis. You are either producing output — assembling characters into a buffer, or stepping a state machine forward one instruction at a time — or you are consuming input, scanning the characters and folding them into a structure: a number, a stack of operands, a nested list, or a map of tokens. This page teaches you to name which side you are on, because that choice picks the deep dive and, with it, the data structure. The Text Formatting deep dive covers producing and simulating; the String Parsing deep dive covers consuming.

The tell: if the answer is itself a string or a final state you assembled, you are producing. If the answer is a value or structure extracted from the input string, you are consuming. A few problems do both (parse, then reformat) — split them into two passes and treat each half on its own axis.

4   Mental Model — one index, one decision per character

The one idea
Make exactly one pass with one index. At every character you make a single, local decision — emit it, fold it into a running value, push it, or skip it — using only O(1) state plus one growing buffer or stack. If you ever find yourself wanting to "go back and look again," you have picked the wrong structure: the right one remembers what you need.

The two families differ only in what the per-character decision does. Watch the same single-pass shape on each side. First, producing — adding "45" + "8" digit by digit, right to left, carrying:

Traceproduce: walk right-to-left, carry, append to buffer
a="45"  b="8"  carry=0  buf=[]

i→'5', j→'8'   sum = 5 + 8 + 0 = 13 → digit 3, carry 1
               buf=[3]
i→'4', j past  sum = 4 + 0 + 1 = 5  → digit 5, carry 0
               buf=[3,5]
both past, carry 0 → stop
               result = reverse(buf) = "53"

Producing: the buffer grows; the carry is the only running state. One pass, no re-reading.

Now consuming — scanning "3+4" into a value, folding each character into a running number:

Traceconsume: scan left-to-right, fold digits, defer the operator
num=0  lastOp='+'  stack=[]

'3'  digit → num = 0*10 + 3 = 3
'+'  operator boundary → apply lastOp '+' to num: push +3
     reset num=0, lastOp='+'   stack=[3]
'4'  digit → num = 0*10 + 4 = 4
end  flush: apply lastOp '+' to num: push +4   stack=[3,4]
     answer = sum(stack) = 7

Consuming: the structure (here a stack) grows; num and lastOp are the running state. Same one-pass shape, opposite direction of data flow.

Recognizing that both sides are the same loop with a different body is what lets you start coding immediately: set up the index, the O(1) state, and the buffer-or-stack, then fill in the one decision.

5   Pseudocode Primer

Both families are this skeleton. The DECIDE line is the only part that changes between problems — and the choice of state vs structure is the choice of deep dive.

Pseudocodethe universal single-pass shape
FUNCTION process(s):
    chars  ← Array(s)            O(1) indexing — Swift strings aren't indexable
    state  ← initial running values (carry, position, num, lastOp…)
    output ← empty buffer OR empty stack/map

    i ← 0
    WHILE i < chars.length:
        c ← chars[i]
        DECIDE on c:                the one per-character decision
            produce: append to output, update state
            consume: fold c into state; at a boundary, commit state to output
        i ← i + 1

    finalize output (flush last token / reverse buffer / sum stack)
    RETURN output
Why a single pass is enough: the running state carries forward everything a later character could need — the carry already holds the overflow, the stack already holds the deferred operands. Nothing earlier in the string needs to be re-read, so a second pass would be redundant work. When a problem genuinely needs two passes (e.g. justify computes line membership first, then spacing), each pass is still a clean single scan.

6   Iterative-Only Pattern — explicit index, explicit stack

This block is iterative by nature, and deliberately so. Two rules keep it that way:

  • Walk a manual index, not the language. Many of these problems need two indices moving at different speeds (two number strings of unequal length), look-ahead (is the next char a digit?), or back-tracking the buffer. A for c in s loop hides the index you need; an explicit while i < n exposes it. Convert to [Character] first so chars[i] is O(1).
  • Make the stack explicit. Nested-structure parses (brackets, expressions) are recursive in spirit, but an explicit stack is safer than recursion: it cannot overflow the call stack on a deeply nested input, and the parse state is a value you can print and inspect mid-loop. Recursion belongs only as an appendix contrast, never as the main solution.
Why this matters here specifically: the entire difficulty of this block is bookkeeping, and bookkeeping is exactly what an explicit index and an explicit stack make visible. The moment your control flow hides the index or the stack, the off-by-one and the dropped-final-token bugs become invisible too.

7   Swift Foundations

Three Swift facts underlie every problem in this block. Say them out loud before coding — they are the source of most "why is this O(n²)?" surprises.

Swiftthe three string-handling rules for this block
// 1. Strings are NOT randomly indexable. s[2] does not compile.
//    Convert once to [Character] for O(1) indexing.
let chars = Array(s)          // O(n) once; chars[i] is O(1) after

// 2. Build into an array, join at the end. Repeated String += is O(n) each.
var buf: [Character] = []
buf.append("x")               // O(1) amortized
let result = String(buf)      // one O(n) join

// 3. Character ↔ digit: there is no implicit cast. Use wholeNumberValue,
//    or subtract the ASCII zero via asciiValue.
let d = c.wholeNumberValue!     // '7' → 7  (nil for non-digits — guard it)
let back = Character(String(d)) // 7 → '7'
The O(n²) trap. result += String(c) inside a loop reallocates the whole string every iteration — O(n²) total. Always accumulate into [Character] (or [String] for line-level output) and String(...) / .joined() once at the end. Likewise s.removeFirst() is O(n); walk an index instead.

8   The Two Families

This overview catalogs the block at the family level; each deep dive expands its family into concrete variants with full pseudocode, Swift, and worked traces.

Family 8.1 Produce / simulate — build output or step a state forward
O(n)–O(n·k)O(n)
When to use: the answer is a string you assemble or a final state you reach by applying rules. Carry a small O(1) state (carry digit, robot pose, water level, current line) and append to a buffer as you go. Covered in full by the Text Formatting deep dive.

Four variants, in increasing bookkeeping difficulty:

  • Digit-buffer build — two indices walk two strings right-to-left, summing with a carry (LC415). Full walkthrough.
  • State-machine simulation — apply an instruction sequence to (position, direction); read the final pose (LC1041). Full walkthrough.
  • Physics / grid simulation — drop a unit, settle it by a local lowest-reachable rule (LC755). Frame + pitfalls only.
  • Greedy line-packing + spacing — pack words into width-N lines, then distribute spaces (LC68). Frame + spacing edge cases, no code.
Family 8.2 Consume / parse — scan input into a structure
O(n)–O(n·k)O(n)
When to use: the answer is a value or structure extracted from the characters. Fold each character into a running token; at a boundary, commit it to a stack, map, or nested object. Covered in full by the String Parsing deep dive.

Four variants:

  • Tokenize + map — split on a delimiter, discard empties, build an ordered map (Decode Runtime Filters). Short, edge-case-focused.
  • Stack-based expression parse — scan numbers and operators, honor precedence with a stack (LC227). Full walkthrough.
  • Nested-structure stack parse — push on open-bracket, pop on close, accumulate between (LC385). Frame + hints only.
  • Reversed-prefix matching — map reversed words, test prefix/suffix palindrome remainders (LC336). Signal + frame, no code.

9   Complexity Analysis

The baseline is O(n) — your job is to not lose it. A single pass that does O(1) work per character is O(n). The complexity questions in this block are almost never "can I get below O(n)?" — they are "did I accidentally go above it?" The three ways people do: rebuilding a string with += in the loop (O(n²)), re-scanning earlier characters instead of carrying state (O(n²)), or calling an O(n) operation (removeFirst, contains) once per character.

Family / variantTimeSpaceWhy
Digit-buffer build (8.1)O(n)O(n)One pass over the longer string; buffer holds the result.
State-machine simulation (8.1)O(n)O(1)Apply each instruction once; only the pose is stored.
Physics / grid simulation (8.1)O(v·n)O(n)Each of v drops can scan up to the terrain width settling.
Greedy line-packing (8.1)O(n)O(n)Each word is placed once; spacing is O(line width) per line.
Tokenize + map (8.2)O(n)O(n)One scan to split; map holds every value seen.
Stack expression parse (8.2)O(n)O(n)One scan; stack holds at most all deferred operands.
Nested-structure parse (8.2)O(n)O(n)One scan; stack depth bounded by nesting depth.
Reversed-prefix matching (8.2)O(w·L²)O(w·L)w words, each of length L; for each word, test each of its L split points for a palindrome remainder.
BottleneckO(n²)—String += in a loop, removeFirst() per char, or re-scanning earlier characters instead of carrying state — any of these silently turns an O(n) pass quadratic.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Indexing a Swift String directly. s[i] does not compile, and s[s.index(s.startIndex, offsetBy: i)] is O(i) — quadratic in a loop. Convert to [Character] once up front.
▸Forgetting the final flush. The last token, the last carry, the last line, the last operand — almost every consume/produce loop commits on a boundary, and the input ends without a trailing boundary. You must finalize after the loop. This is the single most common bug in the block.
▸Re-scanning instead of carrying state. If you reach for "let me look back at the previous characters," your state is incomplete. Add a variable (carry, lastOp, runningNum) so the past is summarized, not re-read.
▸Building strings with +=. O(n²). Accumulate into an array and join once.
▸Char/digit conversion mistakes. '7' is not 7. Use wholeNumberValue (and guard the nil for non-digits) or subtract the ASCII zero — but be consistent.

Solo problem — no solution on this page

LC 385 — Mini Parser (Medium). You are given a string encoding a nested list of integers, such as "[123,[456,[789]]]". Parse it into the corresponding nested structure. Decide your axis first (you are consuming), then build the structure with one scan and an explicit stack — before opening anything.

  • Which family is this — produce or consume — and what structure are you building?
  • What does each character type do: a digit, a comma, an open bracket, a close bracket, a minus sign?
  • Why an explicit stack rather than recursion — and what does the stack hold?
  • When do you commit an accumulated number, and what is the final-flush case?
  • What are the edge cases: a bare integer with no brackets, an empty list, a negative number?
