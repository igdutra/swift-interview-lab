Strings & simulation problems ask you to process a character sequence — or a small evolving state described by one — in a single left-to-right pass, doing explicit, mechanical work the language's built-ins would normally hide. There is rarely a clever asymptotic trick; the difficulty is in the bookkeeping: carrying a digit, tracking a facing direction, distributing leftover spaces, or deciding exactly when a token ends.

Every problem in this block splits along one axis: you are either producing output — assembling characters into a buffer, or stepping a state machine forward — or you are consuming input, folding the characters into a structure: a number, a stack of operands, a nested list, or a map of tokens. This overview is the hub's decision layer and Swift toolbox: the four loop skeletons that implement the whole block, the language mechanics they lean on, and the fork that picks the right deep dive. The per-family theory lives in the Text Formatting deep dive (produce/simulate) and the String Parsing deep dive (consume/parse).

2   The Four Standard Skeletons

Memorize these loop shapes and the bookkeeping comes for free. A skeleton is a control-flow shape, not a problem: every problem in this block is one of these four shapes, a variant of one, or a composition of two.

SwiftSkeleton 1 · Digit-buffer build — two indices right-to-left, carry folds the flush into the loop (LC415)
var leftIndex = digits1.count - 1          // two inputs, two independent indices
var rightIndex = digits2.count - 1
var carry = 0
var resultBuffer: [Character] = []

while leftIndex >= 0 || rightIndex >= 0 || carry > 0 {   // carry>0 IS the final flush
    let digitA = leftIndex >= 0 ? digits1[leftIndex].wholeNumberValue! : 0
    let digitB = rightIndex >= 0 ? digits2[rightIndex].wholeNumberValue! : 0
    let columnSum = digitA + digitB + carry
    resultBuffer.append(Character(String(columnSum % 10)))   // keep the low digit
    carry = columnSum / 10                                    // carry the high digit left
    leftIndex -= 1; rightIndex -= 1
}
return String(resultBuffer.reversed())     // built low→high — flip once at the end
SwiftSkeleton 2 · State-machine simulation — fold each instruction into O(1) state, no buffer (LC1041)
enum Direction {                       // the state is a TYPE, not index arithmetic —
    case north, east, south, west       // every transition is spelled out and exhaustive

    var delta: (x: Int, y: Int) {
        switch self {
        case .north: (0, 1)
        case .east:  (1, 0)
        case .south: (0, -1)
        case .west:  (-1, 0)
        }
    }

    var turnedLeft: Direction {
        switch self {
        case .north: .west
        case .west:  .south
        case .south: .east
        case .east:  .north
        }
    }
    // turnedRight mirrors it: north → east → south → west → north
}

var position: (x: Int, y: Int) = (0, 0)   // the pose IS the state
var direction: Direction = .north

for instruction in instructions {
    switch instruction {
    case "G": position.x += direction.delta.x
              position.y += direction.delta.y
    case "L": direction = direction.turnedLeft
    case "R": direction = direction.turnedRight
    default: break
    }
}
// read the answer off the final pose — nothing was ever appended anywhere
// LC755 is this shape with a bigger state (the terrain array) + an inner settle scan
SwiftSkeleton 3 · Accumulate-and-commit scan — build a token, commit on a trigger (tokenizer / LC227 / LC68)
var currentToken: [Character] = []   // or a running number, or a line's word list

for (characterIndex, character) in characterArray.enumerated() {
    if character == delimiter || characterIndex == characterArray.count - 1 {
        commit(currentToken)          // fold the finished token into the output
        currentToken = []             // reset EVERY per-token variable, in one place
    } else {
        currentToken.append(character)
    }
}
// The trigger is the only part that changes:
//   delimiter char  → tokenizer (fold the last index into the trigger, as above)
//   operator boundary → LC227 (commit the running number under the DEFERRED operator)
//   overflow ("won't fit") → LC68 (commit BEFORE adding; last batch needs a post-loop flush)
SwiftSkeleton 4 · Nested stack parse — push a scope on open, pop-and-merge on close (LC385 / LC394)
var stack: [NestedContainer] = []

for character in characterArray {
    switch character {
    case "[":
        stack.append(NestedContainer())        // descend — a new open scope
    case ",", "]":
        commitPendingNumber(&stack)             // boundary — attach the number being built
        if character == "]" {
            let finishedScope = stack.removeLast()          // ascend — scope closes…
            stack[stack.count - 1].add(finishedScope)        // …into its PARENT, by index
        }                                       // (empty stack after the pop → it's the root)
    default:
        accumulateDigit(character)              // running number: value * 10 + digit
    }
}
// the innermost bracket always closes first — the stack IS that LIFO fact, made explicit
The shape is the recognition — and skeletons compose. Two number strings → skeleton 1; an agent obeying instructions → skeleton 2; flat tokens with any commit trigger → skeleton 3; open/close scopes → skeleton 4. LC227 (Basic Calculator II) is not a fifth shape: it is skeleton 3 committing into a stack, the way arrays' prefix-sum composes with a hash map. And LC336 (Palindrome Pairs) is not a strings skeleton at all — it is the complement map plus split-point enumeration wearing string clothes. If you feel the urge to recurse on any of these, the right structure is missing: an explicit index or an explicit stack makes the bookkeeping — and therefore the bugs — visible.

4   Produce or Consume? — routing a problem to its skeleton

The one axis, then one decision per character
If the answer is itself a string or a final state you assembled, you are producing. If the answer is a value or structure extracted from the input, you are consuming. Then make exactly one pass with one index: at every character you make a single, local decision — emit it, fold it into a running value, push it, or skip it — using only O(1) state plus one growing buffer or stack. If you ever want to "go back and look again," your state is incomplete: add a variable (carry, lastOperator, runningNumber) so the past is summarized, not re-read.

Family 1 — Produce / simulate

Family 1 Produce / simulate — build output or step a state forward
O(n)–O(n·k)O(n)
Reach for it when: the answer is a string you assemble or a final state you reach by applying rules. Carry a small O(1) state (carry digit, robot pose, water level, current line) and append to a buffer as you go. Covered in full by the Text Formatting deep dive.

Four variants, in increasing bookkeeping difficulty — each is a §2 skeleton or a variant of one:

  • Digit-buffer build (skeleton 1) — two indices walk two strings right-to-left, summing with a carry (LC415). Full walkthrough.
  • State-machine simulation (skeleton 2) — apply an instruction sequence to (position, direction); read the final pose (LC1041). Full walkthrough.
  • Physics / grid simulation (skeleton 2, terrain-array state) — drop a unit, settle it by a local lowest-reachable rule (LC755). Frame + pitfalls only.
  • Greedy line-packing + spacing (skeleton 3, overflow trigger) — pack words into width-N lines, then distribute spaces (LC68). Frame + spacing edge cases, no code.

Family 2 — Consume / parse

Family 2 Consume / parse — scan input into a structure
O(n)–O(n·k)O(n)
Reach for it when: the answer is a value or structure extracted from the characters. Fold each character into a running token; at a boundary, commit it to a stack, map, or nested object. Covered in full by the String Parsing deep dive.

Four variants:

  • Tokenize + map (skeleton 3, delimiter trigger) — split on a delimiter, discard empties, build an ordered multi-map (Decode Runtime Filters). Short, edge-case-focused.
  • Stack-based expression parse (skeletons 3 + 4 composed) — scan numbers and operators, honor precedence with a stack (LC227). Full walkthrough.
  • Nested-structure stack parse (skeleton 4) — push on open-bracket, pop on close, accumulate between (LC385). Frame + hints only.
  • Reversed-prefix matching (arrays complement map + split enumeration) — map reversed words, test prefix/suffix palindrome remainders (LC336). Signal + frame, no code.

5   Terminology Decoder

TermPlain meaningExample
SimulationFaithfully executing the rules of a process step by step, with no shortcut — the answer is whatever the rules produceApply each robot move, then read the final position
State machineA small set of variables (position, direction, carry) that fully describe "where you are" between steps(x, y, facing) for a robot; (carry) for digit addition
Buffer / accumulatorA growing array of characters you append to, joined into a string only at the endCollect digits, then String(buffer.reversed())
TokenizeSplitting a string into meaningful pieces on a delimiter, discarding the empties"a&&b" → tokens ["a", "b"]
Commit triggerThe event that ends the current token and folds it into the output: a delimiter, an operator boundary, or an overflow"&" seen → commit key/value, reset
Final flushThe commit that must happen after the loop because the input ends without a trailing triggerThe last operand in "3+4"; the last line in LC68
Scan / passA single index moving left to right exactly once over the charactersfor character in characterArray or a manual index loop
PrecedenceThe rule that some operators bind tighter than others, so they must be applied first× and ÷ before + and − in 3+4*2
Greedy packingFilling a fixed capacity by taking items in order until the next one would overflow, then committingAdd words to a line until the next word won't fit

Complexity at a glance

The baseline is O(n) — your job is to not lose it. The complexity question in this block is almost never "can I get below O(n)?" — it is "did I accidentally go above it?"

VariantTimeSpaceWhy
Digit-buffer build (S1)O(n)O(n)One pass over the longer string; buffer holds the result.
State-machine simulation (S2)O(n)O(1)Apply each instruction once; only the pose is stored.
Physics / grid simulation (S2 variant)O(v·n)O(n)Each of v drops can scan up to the terrain width while settling.
Greedy line-packing (S3)O(n)O(n)Each word is placed once; spacing is O(line width) per line.
Tokenize + map (S3)O(n)O(n)One scan to split; map holds every value seen.
Stack expression parse (S3+S4)O(n)O(n)One scan; stack holds at most all deferred operands.
Nested-structure parse (S4)O(n)O(n)One scan; stack depth bounded by nesting depth.
Reversed-prefix matchingO(w·L²)O(w·L)w words of length L; each word tests L split points for a palindrome remainder.
BottleneckO(n²)—String += in a loop, removeFirst() per char, String.Index math per access, or re-scanning earlier characters instead of carrying state.
Instant disqualifiers — rule the bug out on sight
▸string[integerIndex] anywhere → does not compile, and the String.Index workaround is O(n) per access → Array(string) first, always.
▸result += String(character) in a loop → O(n²) → accumulate into [Character], join once.
▸No commit after the loop → the last token / carry / line / operand is silently dropped. The single most common bug in the block — every trigger-based loop needs its final-flush story.
▸"Let me look back at the previous characters" → your state is incomplete → add a variable; never re-scan.
▸'7' == 7 thinking → there is no implicit cast → wholeNumberValue in, Character(String(…)) out.
▸Per-token variables reset inline, one by one → a forgotten flag bleeds into the next token → one resetState().
▸Dividing by a gap count that can be zero (single-word line) → crash → guard gapCount > 0.
