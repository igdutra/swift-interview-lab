The produce family covers every problem whose answer is something you assemble: a string built character by character, or a final state reached by faithfully applying a sequence of rules. There is no asymptotic cleverness — the work is mechanical — so the entire difficulty is in the bookkeeping: carrying the overflow digit, tracking which way a robot faces, settling a unit of water to its lowest reachable spot, or distributing leftover spaces evenly across a justified line.

All four variants share one shape: a single index, a small O(1) running state, and a growing buffer that you finalize after the loop ends. This page works through the two simplest variants — digit-buffer addition and state-machine simulation — with full traces and Swift, then gives the simulation and line-packing variants as solve-it-yourself frames with their pitfalls and edge-case inventories, because those two are the ones interviews are actually won and lost on.

4   Mental Model — carry the state, append the output

The one idea
Hold the entire "where am I" in a few O(1) variables — a carry, a pose, a water level, a current line. March through the input once, and at each step update that state and append to a buffer. The work is mechanical; correctness comes entirely from (a) the right per-step rule and (b) remembering to flush the last piece after the loop.

Trace the digit-buffer build adding "99" + "1" — the case where the carry survives to the very end and must be flushed:

Traceadd "99" + "1" — carry propagates past both strings
i→last of "99", j→last of "1", carry=0, buf=[]

step  i='9', j='1'  sum = 9 + 1 + 0 = 10 → digit 0, carry 1
                    buf=[0]   i--, j-- (j now past)
step  i='9', j past sum = 9 + 0 + 1 = 10 → digit 0, carry 1
                    buf=[0,0]  i-- (now past)
loop ends, but carry = 1 still set  → must flush
                    buf=[0,0,1]
result = reverse(buf) = "100"   correct

If you stop at the loop you get "00" — wrong. The post-loop carry flush is the whole game.

And the state-machine simulation — a robot at the origin facing North applies "GGLGG" (G=go forward, L=turn left). The question is whether, after one cycle of the instructions, it has either returned to the origin or changed its facing — either means it is bounded forever:

Tracerobot pose after "GGLGG" from (0,0) facing North
pose: x=0, y=0, dir=N (0,+1)

'G'  move: y += 1 → (0,1)
'G'  move: y += 1 → (0,2)
'L'  turn left: N→W, dir=(−1,0)
'G'  move: x −= 1 → (−1,2)
'G'  move: x −= 1 → (−2,2)

end: position (−2,2) ≠ origin, but dir = W ≠ N (rotated)
     → bounded (returns to a cycle). answer: true

The pose (x, y, dir) is the complete state. The bounded test reads two fields of it after one pass.

5   Pseudocode Primer

The produce family is two canonical loops. The digit-buffer build (8.1):

Pseudocodeadd two number strings
FUNCTION addStrings(a, b):
    i ← a.length − 1,  j ← b.length − 1,  carry ← 0
    buf ← []

    WHILE i ≥ 0 OR j ≥ 0 OR carry > 0:   carry keeps the loop alive
        da ← (i ≥ 0) ? digit(a[i]) : 0
        db ← (j ≥ 0) ? digit(b[j]) : 0
        sum ← da + db + carry
        buf.append(sum mod 10)
        carry ← sum / 10
        i ← i − 1,  j ← j − 1

    RETURN reverse(buf) as string
Why OR carry > 0 in the loop condition: it folds the post-loop flush into the loop. As long as a carry remains, the loop runs one more time and emits it — so "99"+"1"="100" works without a separate finalize step. This is the cleanest way to never forget the flush.

The state-machine simulation (8.2):

Pseudocodesimulate an instruction sequence over a pose
FUNCTION simulate(instructions):
    x ← 0, y ← 0
    dirs ← [(0,1),(1,0),(0,−1),(−1,0)]   N, E, S, W — clockwise
    d ← 0                                index into dirs

    FOR each ins IN instructions:
        IF ins = 'G': x += dirs[d].0; y += dirs[d].1
        IF ins = 'L': d ← (d + 3) mod 4    turn left = −1 mod 4
        IF ins = 'R': d ← (d + 1) mod 4

    RETURN read fields of (x, y, d)        problem-specific test

6   Iterative-Only Pattern — explicit indices, explicit state

Every variant here is naturally iterative, and the iteration is the point:

  • Two indices for two inputs. Digit addition walks two strings of possibly different lengths from the right. An explicit i and j with bounds guards (i ≥ 0 ? … : 0) is the clean expression; there is no recursive form that reads better.
  • State lives in plain variables. The pose, the carry, the current line buffer, the water level array — all mutate in place across iterations. Recursion would thread them through call frames for no benefit and a stack-overflow risk on long instruction strings.
  • Two-phase loops stay two loops. Justify packs lines in one loop, then formats each packed line in another. Keeping them as two sequential passes (not one tangled loop) is what makes the spacing logic testable in isolation.

7   Swift Translation

The two full-solution variants, compilable. First, digit-buffer addition (LC415):

SwiftLC 415 — add two number strings
func addStrings(_ num1: String, _ num2: String) -> String {
    let a = Array(num1), b = Array(num2)   // O(1) indexing
    var i = a.count - 1, j = b.count - 1, carry = 0
    var buf: [Character] = []

    while i >= 0 || j >= 0 || carry > 0 {
        let da = i >= 0 ? a[i].wholeNumberValue! : 0
        let db = j >= 0 ? b[j].wholeNumberValue! : 0
        let sum = da + db + carry
        buf.append(Character(String(sum % 10)))
        carry = sum / 10
        i -= 1; j -= 1
    }
    return String(buf.reversed())
}

And state-machine simulation (LC1041 — the bounded test reads the final pose):

SwiftLC 1041 — is the robot's path bounded in a circle?
func isRobotBounded(_ instructions: String) -> Bool {
    var x = 0, y = 0, d = 0           // d: 0=N 1=E 2=S 3=W
    let dirs = [(0,1), (1,0), (0,-1), (-1,0)]

    for ins in instructions {
        switch ins {
        case "G": x += dirs[d].0; y += dirs[d].1
        case "L": d = (d + 3) % 4          // left = −1 mod 4
        case "R": d = (d + 1) % 4
        default: break
        }
    }
    // Bounded iff back at origin OR no longer facing North
    return (x == 0 && y == 0) || d != 0
}
Why the bounded test works: if after one instruction cycle the robot is back at the origin, it loops forever — bounded. If it is not at the origin but is facing a different direction than it started, then four cycles rotate the net displacement around a closed square — also bounded. The only unbounded case is "moved and still facing North," i.e. a straight drift. That is the entire insight; the simulation just computes the final pose.

8   Variant Catalog

Variant 8.1 Digit-buffer build
O(n)O(n)
When to use: arithmetic on numbers given as strings, with no BigInt. Walk both strings from the right, sum digit + digit + carry, append the low digit, keep the carry. The full Swift is in §7; the loop condition i≥0 || j≥0 || carry>0 folds the final flush into the loop.

Canonical problem: LC 415 — Add Strings (warm-up). Multiplication (LC 43) is the same buffer technique applied per digit with shifted offsets.

Variant 8.2 State-machine simulation
O(n)O(1)
When to use: an agent follows an instruction sequence and you need its final state, or a property of it. Encode the pose as plain variables and a direction-vector table; apply each instruction once; read the answer off the final pose. Full Swift in §7.

Canonical problem: LC 1041 — Robot Bounded In Circle (walkthrough). The direction table + (d±1) mod 4 turn is reusable for any grid-walking agent.

Variant 8.3 Physics / grid simulation — settle each unit
O(v·n)O(n)
Solve-it-yourself variant. This is an attempt problem — the frame, the rules, and the pitfalls are below, but no solution. Build it cold.
When to use: drop V units into a terrain (an array of heights); each unit settles by a local rule. The state is the heights array itself; you mutate it one unit at a time.

The settle rule (LC 755 — Pour Water): a unit dropped at index K tries, in this exact order:

  • Flow left to the nearest lower position, but only while the path doesn't first rise. Stop at the lowest reachable point on the left.
  • Else flow right by the same rule.
  • Else stay at K (raise height[K] by 1).
The frame: for each of the V drops, scan outward from K to find the lowest reachable index under the "never climb a wall to get there" constraint, then increment that index's height. The whole solution is two nested ideas — an outer loop over drops, an inner local search for the settle point — and the difficulty is entirely in the inner search's tie-breaks and bounds (see Pitfalls §10 and Edge Cases §11).

Canonical problem: LC 755 — Pour Water (attempt). The full graduated hints live in the walkthrough.

Variant 8.4 Greedy line-packing + space distribution
O(n)O(n)
Solve-it-yourself variant. This is a target problem — the two-phase frame and the full spacing edge-case inventory are below, but no code. This is the hardest bookkeeping in the block.
When to use: pack words into lines of exactly maxWidth characters, fully justified. Two phases, kept as two passes.

Phase 1 — greedy packing. Add words to the current line while (sum of word lengths) + (number of gaps, one per already-added word) ≤ maxWidth. The moment the next word would overflow, commit the current line and start a new one.

Phase 2 — justify each committed line. Distribute maxWidth − totalWordLength spaces across the gaps:

  • Even split, left-biased: base = spaces / gaps, extra = spaces % gaps; the leftmost extra gaps get base + 1, the rest get base.
  • Last line: left-justified — single space between words, all remaining padding on the right.
  • Single-word line: left-justified — the one word, then pad the right to maxWidth (there are no gaps to distribute into).
The frame: phase 1 produces a list of word-groups (one per line); phase 2 turns each group into a width-maxWidth string. Keeping them separate is what makes the three spacing special cases (normal / last / single-word) tractable — each is a small, isolated branch in phase 2. The edge-case inventory in §11 is the real specification here.

Canonical problem: LC 68 — Text Justification (target). Graduated hints in the walkthrough.

10   Common Pitfalls

Pitfalls — bugs this family invites
▸Dropping the final carry (8.1). "99"+"1" must give "100". If the loop stops when both indices run out, the surviving carry is lost. Put carry > 0 in the loop condition.
▸Wrong turn arithmetic (8.2). Left turn is (d + 3) % 4, not (d − 1) % 4 — in Swift (0 − 1) % 4 == −1, an invalid index. Always add the modulus before taking %.
▸Settle tie-break direction (8.3). When left and right are equally low, the unit must settle on the left. Getting the precedence backwards passes simple cases and fails symmetric terrains.
▸Climbing a wall to reach a low spot (8.3). The unit can only flow to a position it can reach without first going up. A lower index beyond a taller wall is not reachable — stop the scan at the first rise.
▸Off-by-one in the line-fit test (8.4). A line of k words has k−1 mandatory gaps. The fit check is wordLenSum + (k−1) ≤ maxWidth — forgetting the gaps overpacks the line by one word.
▸Justifying the last line or a single-word line (8.4). Both are left-justified, not space-distributed. Treating them like normal lines spreads spaces where there should be none and fails the trailing-pad requirement.

Solo problem — no solution on this page

LC 755 — Pour Water (Medium). Given an array of terrain heights, an amount of water V, and a drop index K, simulate each unit falling and settling, then return the final heights. Build it cold from Variant 8.3's frame: an outer loop over drops, an inner local search for the lowest reachable settle point.

  • For one drop, in what order do you try left, right, and stay-in-place?
  • What stops the leftward (or rightward) scan — when have you found the lowest reachable point?
  • How does "cannot climb a wall to reach a lower spot" change the scan?
  • When left and right are equally low, which side wins, and why does that tie-break matter?
  • What are the boundary cases when K is at index 0 or the last index?
