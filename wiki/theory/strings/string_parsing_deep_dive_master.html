String Parsing

The consume family covers every problem whose answer is a value or structure extracted from a character sequence: a number split into key-value tokens, an arithmetic expression evaluated under precedence, a bracketed string turned into a nested object, or a set of words matched against their reverses. You scan once, fold each character into a running token, and at every boundary — a delimiter, an operator, a closing bracket — commit that token to the structure you are building.

The structure is the variant. A delimiter scan builds a map; an expression scan builds a stack of operands; a bracket scan builds a nested object via an explicit stack; a reverse-match builds a hash map of reversed words. This page works the tokenize-and-map variant in full (a compact, deliberately edge-case-driven section) and the stack expression parse in full, then gives the nested-parse and reversed-prefix variants as solve-it-yourself frames. The recurring trap across all four is the final flush: the last token has no trailing boundary, so you must commit it after the loop.

2   Recognition Signals — when to reach for it

When the problem says…It means…

The tell: the input is a string and the output is something pulled out of it — a map, a number, a tree, a list of index pairs. If the characters become a running token that you keep committing to a structure, you are parsing. If instead you are emitting characters into output, you are in the produce family.

4   Mental Model — accumulate a token, commit on a boundary

The one idea
Scan once. Keep a running token (a number, a string buffer). Every character either extends the token or is a boundary that ends it — and on a boundary you commit the token to your structure and reset. The last token has no boundary after it, so commit it once more after the loop. That post-loop flush is the bug everyone forgets.

Watch the stack expression parse evaluate "3+4*2" — the precedence trick is to defer +/− by pushing onto a stack, but apply ×/÷ immediately to the stack top:

Traceevaluate "3+4*2" — defer add, apply multiply to stack top
num=0  lastOp='+'  stack=[]

'3'  digit → num = 3
'+'  boundary: apply lastOp '+' → push +3.  stack=[3], lastOp='+', num=0
'4'  digit → num = 4
'*'  boundary: apply lastOp '+' → push +4.  stack=[3,4], lastOp='*', num=0
'2'  digit → num = 2
end  flush: apply lastOp '*' → pop 4, push 4*2.  stack=[3,8]
     answer = sum(stack) = 11

× binds tighter, so it modifies the top of the stack in place; + just pushes. The sum at the end combines the deferred additions. The final flush after "2" is essential.

And the tokenize-and-map variant on "a=1&&a=2&b" — note how empty tokens are skipped, not stored, and a token with no = becomes a flag:

Tracetokenize "a=1&&a=2&b" → ordered multi-map
split on '&' → ["a=1", "", "a=2", "b"]

"a=1"  split on first '=' → key "a", value "1"   map: a → [1]
""     empty token → skip
"a=2"  key "a", value "2"   map: a → [1, 2]   (order preserved)
"b"    no '=' → flag → value "true"   map: b → [true]

result = { a: [1,2], b: [true] }   empties dropped, repeats kept in order

The empty token between the two & is discarded silently. A bare token defaults to "true". Repeated keys append, preserving order.

5   Pseudocode Primer

The stack expression parse (8.2) is the canonical consume loop — boundary detection plus a deferred operator:

Pseudocodeevaluate +,−,×,÷ without parentheses
FUNCTION calculate(s):
    num ← 0,  lastOp ← '+',  stack ← []

    FOR i FROM 0 TO s.length − 1:
        c ← s[i]
        IF c is a digit: num ← num*10 + digit(c)

        commit on an operator OR at the last character
        IF c is an operator OR i = last index:
            SWITCH lastOp:
                '+': stack.push(num)
                '−': stack.push(−num)
                '×': stack.push(stack.pop() × num)
                '÷': stack.push(stack.pop() ÷ num)
            lastOp ← c,  num ← 0          reset for the next token

    RETURN sum(stack)
Why "operator OR last index": the commit must fire both at every operator (the boundary between tokens) and once more at the end of the string (the final flush for the last number, which has no operator after it). Folding both triggers into one condition is what makes the loop correct without a separate post-loop block. Spaces are simply skipped — they are neither digits nor operators.

The tokenize-and-map variant (8.1) is split-then-fold:

Pseudocodedecode a delimiter-separated key=value line
FUNCTION decode(input):
    s ← input without a leading '?' if present
    map ← ordered map

    FOR each token IN split(s, '&'):
        IF token is empty: CONTINUE      drop empties from && / ends
        IF token contains '=':
            key, value ← split token on the FIRST '=' only
        ELSE:
            key, value ← token, "true"      bare flag
        map[key].append(value)              repeats preserved in order

    RETURN map

6   Iterative-Only Pattern — one scan, an explicit stack

Parsing is where "explicit stack over recursion" matters most:

  • Nested structure ≠ recursion. A bracketed encoding looks recursive, but an explicit stack parses it in one iterative scan with no call-stack risk: push a new container on [, attach the finished container to its parent on ], accumulate digits in between. The stack is the recursion, made into data you can inspect.
  • Boundaries need an explicit index sometimes. Splitting on the first = only, or look-ahead for multi-digit numbers, is awkward in a for c in s loop. When you need that control, scan a manual index over [Character].
  • The deferred-operator trick is inherently iterative. Carrying lastOp and num across iterations and committing on a boundary has no natural recursive form — it is a fold over the character stream.

7   Swift Translation

The stack expression parse (LC227), compilable:

SwiftLC 227 — Basic Calculator II
func calculate(_ s: String) -> Int {
    let chars = Array(s)
    var stack: [Int] = []
    var num = 0
    var lastOp: Character = "+"

    for i in 0..<chars.count {
        let c = chars[i]
        if let d = c.wholeNumberValue {
            num = num * 10 + d
        }
        // Commit on an operator, or at the final character (flush).
        if (!c.isNumber && c != " ") || i == chars.count - 1 {
            switch lastOp {
            case "+": stack.append(num)
            case "-": stack.append(-num)
            case "*": stack.append(stack.removeLast() * num)
            case "/": stack.append(stack.removeLast() / num)
            default: break
            }
            lastOp = c
            num = 0
        }
    }
    return stack.reduce(0, +)
}
Two Swift specifics: integer / truncates toward zero, which matches LC227's required division semantics — no extra rounding. And the commit condition uses i == chars.count - 1 for the final flush; without it, the last operand never reaches the stack.

The tokenize-and-map variant — this is the section built for the custom problem Decode Runtime Filters. It is "Easy," so the code is short; the value is entirely in handling the forgiving edge cases (foregrounded in §11):

SwiftDecode Runtime Filters — tokenize a key=value line into an ordered multi-map
func decodeRuntimeFilters(_ input: String) -> [String: [String]] {
    // Drop a single leading "?" if present.
    var s = Substring(input)
    if s.first == "?" { s = s.dropFirst() }

    var result: [String: [String]] = [:]

    for token in s.split(separator: "&", omittingEmptySubsequences: true) {
        // Split on the FIRST "=" only. No "=" → it's a flag, value "true".
        if let eq = token.firstIndex(of: "=") {
            let key   = String(token[token.startIndex..<eq])
            let value = String(token[token.index(after: eq)...])  // "name=" → ""
            result[key, default: []].append(value)
        } else {
            result[String(token), default: []].append("true")
        }
    }
    return result
}
Why split(omittingEmptySubsequences: true) is the whole trick: it drops the empty tokens from &&, leading, and trailing & in one stroke — exactly the "skip empty tokens" rule. The firstIndex(of: "=") split (not split(separator: "=")) honors "first = only," so "mode=fast=beta" keeps "fast=beta" as its value. And token[index(after: eq)...] yields "" for "name=" rather than dropping it — the empty-value case.

8   Variant Catalog

Variant 8.1 Tokenize + map  (edge-case focused)
O(n)O(n)
When to use: a delimiter-separated line (query string, config, CSV) decoded into a map. Split on the outer delimiter, discard empties, split each token on its inner delimiter (first occurrence only), default bare tokens, and append into an order-preserving multi-map. Full Swift in §7.

The algorithm is trivial; the specification is the edge cases — leading ?, empty tokens, first-=-only, flag defaulting, empty values, repeated keys. They are enumerated in §11. Motivating problem: Decode Runtime Filters (custom, Easy).

Variant 8.2 Stack-based expression parse
O(n)O(n)
When to use: evaluate an arithmetic expression with mixed-precedence operators. Carry a running number and a deferred operator; on each operator boundary, apply the deferred op (push for +/−, modify the stack top for ×/÷); sum the stack at the end. Full Swift in §7.

Canonical problem: LC 227 — Basic Calculator II (walkthrough). With parentheses (LC 224) the same idea recurses via a stack of saved contexts.

Variant 8.3 Nested-structure stack parse
O(n)O(n)
Solve-it-yourself variant. This is an attempt problem — the frame and rules are below, but no solution. Build it cold.
When to use: a string encodes a nested container (a list of integers and sub-lists, e.g. "[123,[456,[789]]]"). Parse it into the corresponding nested object with one scan and an explicit stack.

The stack frame (LC 385 — Mini Parser):

  • Open bracket [ → push a new empty container onto the stack.
  • Digit (and a leading -) → accumulate into a running number.
  • Comma or close bracket → a boundary: if a number was accumulating, attach it to the container on top of the stack, then reset.
  • Close bracket ] → pop the top container; attach it to the new top (its parent). The last pop is the root result.
The frame: the stack top is always "the container I am currently filling." Opening descends (push), closing ascends (pop + attach). The two recurring bugs are the bare-integer case (no brackets at all → the whole input is one number) and the final-number flush before a ]. Both are in §11.

Canonical problem: LC 385 — Mini Parser (attempt). Graduated hints in the walkthrough.

Variant 8.4 Reversed-prefix / palindrome matching
O(w·L²)O(w·L)
Solve-it-yourself variant. This is a target problem — the recognition signal and the three matching cases are below, but no code.
When to use: find all pairs of words whose concatenation is a palindrome. Brute force is O(w²·L); the parsing insight is to index reversed words in a hash map so each word can look up its complement in O(L) instead of scanning all others.

The three cases (LC 336 — Palindrome Pairs) for a word W paired with some other word X:

  • Exact reverse: X is the reverse of W (and they are different words). W + X is a palindrome.
  • W has a palindromic prefix: if W = [palindrome][rest], then reverse(rest) + W is a palindrome — look up reverse(rest) in the map.
  • W has a palindromic suffix: if W = [rest][palindrome], then W + reverse(rest) is a palindrome — look up reverse(rest) in the map.
The frame: build a map from each reversed word to its index. For every word, split it at each of its L positions; if one side is a palindrome, look up the reverse of the other side in the map. The structure you parse into is the reversed-word map; the per-word work is the L split-point checks. The empty-word and self-pairing cases are the classic traps (§11).

Canonical problem: LC 336 — Palindrome Pairs (target). Graduated hints in the walkthrough.

10   Common Pitfalls

Pitfalls — bugs this family invites
▸Forgetting the final flush. The last number / token / line has no trailing boundary. In 8.2 the commit must also fire at the last index; in 8.1 the last token must still be processed. This is the single most common parsing bug.
▸Splitting on every = instead of the first (8.1). "mode=fast=beta" must become key "mode", value "fast=beta". Using split(separator: "=") shreds the value; use firstIndex(of: "=").
▸Storing empty tokens (8.1). &&, a leading ?/&, or a trailing & produce empty tokens that must be skipped, not added as blank keys. omittingEmptySubsequences: true handles all three.
▸Applying ×/÷ to the wrong value (8.2). Multiply/divide must modify the top of the stack (the immediately preceding operand), not the running sum. Pop, combine, push back.
▸Dropping the last number before ] (8.3). A close bracket is a boundary; an accumulating number right before it must be attached to the current container before the pop.
▸Self-pairing and empty words (8.4). A word must not pair with itself; the empty word "" pairs with every palindrome in both orders. Both are easy to double-count or miss.
