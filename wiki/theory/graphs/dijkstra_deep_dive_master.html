Dijkstra's Algorithm

Dijkstra's algorithm finds the shortest path from a single source to every other vertex in a graph whose edge weights are non-negative. It generalizes BFS: where BFS assumes every edge costs 1 and so expands in simple layers with a queue, Dijkstra handles varying weights by always expanding the closest unsettled vertex — pulled from a min-heap (priority queue) keyed by tentative distance. The moment a vertex comes off the heap, its shortest distance is locked in.

The decision that sends a problem to Dijkstra is "cheapest path with non-negative weights that differ." The machinery is always the same three pieces: a distance array (best cost known to each vertex), a min-heap of (cost, vertex) frontier entries, and a relaxation step that, on popping a vertex, tries to improve each neighbor's tentative cost and pushes the improved ones back. What "cost" means can flex — total weight (classic, LC743), the maximum single edge on the path (min-max, LC1631), or cost under a hop/time constraint (LC787) — but the settle-the-cheapest-first skeleton never changes. This deep dive lives under the Graphs hub; it is the weighted sibling of the BFS deep dive.

2   Recognition Signals — what variant is this?

"Cheapest path over weighted edges" is the Dijkstra master signal. What the cost is, and whether there's an extra constraint, picks the variant.

When the problem says…It means…
"minimum total time / cost / distance" on a weighted graphClassic single-source Dijkstra — sum of weights. §8.1 (LC743)
"shortest weighted path in a grid", weighted movesGrid Dijkstra — cells are vertices, move costs are edge weights. §8.2
"minimize the maximum / minimize the largest step / least effort"Custom-cost (min-max) Dijkstra — relax with max, not +. §8.3 (LC1631)
"cheapest path within K stops / within T minutes"Constrained Dijkstra — augment the vertex with the constraint dimension. §8.4 (LC787)
"all edges cost 1 / fewest steps"Not Dijkstra — BFS is the right, cheaper tool.
"edges can be negative"Not Dijkstra — its settled-is-final invariant breaks. Use Bellman-Ford / SPFA.
The tell: the answer is a cost and the edges carry different, non-negative weights. The instant you see "cheapest / minimum-cost path" with varying weights, reach for a min-heap. Then ask: is the cost a sum (§8.1/§8.2), a max along the path (§8.3), or is there an extra budget like stops or time that must travel in the state (§8.4)?

4   Mental Model — settle the cheapest frontier first

The one idea
Dijkstra is BFS with a priority queue. Instead of expanding in equal layers, it always expands the cheapest-reachable vertex next. Because all weights are non-negative, the cheapest unsettled vertex can never later be reached more cheaply — so the first time you pop a vertex off the heap, its distance is final and you never revisit it. That "cheapest-first ⇒ settled-is-final" property is the entire correctness argument, and it is exactly why a negative edge would break it.

Watch Dijkstra from source A. The heap always serves the smallest tentative cost; popping a vertex finalizes it:

Diagramsettle cheapest-first; edge labels are weights
     1      5
  A ---- B ---- D          dist start: A0  B∞  C∞  D∞
  |      |
 4|      |2
  C ------+                heap: [(0,A)]

pop (0,A): settle A=0 ; relax B→1, C→4         heap [(1,B),(4,C)]
pop (1,B): settle B=1 ; relax D→1+5=6, C→1+2=3 heap [(3,C),(4,C-stale),(6,D)]
pop (3,C): settle C=3 ; (4,C) left in heap is stale  heap [(4,C),(6,D)]
pop (4,C): C already settled → SKIP (lazy deletion)  heap [(6,D)]
pop (6,D): settle D=6
final: A0 B1 C3 D6

C is first reached at cost 4 (via A) but later improved to 3 (via B) — the stale (4,C) entry stays in the heap and is skipped when popped. That's lazy deletion: cheaper than decrease-key, and the settled check makes it correct.

The deep idea that unlocks the variants: "cost" need not be a sum. Keep the settle-cheapest-first skeleton and swap only the combine rule. For "minimize the maximum edge on the path" (LC1631), a path's cost is the largest edge it uses, so relaxation becomes max instead of +:

Tracemin-max Dijkstra — relax with max, not +
classic:   newCost = dist[u] + weight(u,v)
min-max:   newCost = max(dist[u], weight(u,v))   ← cost of a path = its worst edge

the heap still serves the smallest newCost; settle-cheapest-first still holds
because max is monotonic and weights are non-negative. Same skeleton, new combine.

Classic, grid, and min-max Dijkstra are the same algorithm; only the cost combine rule (and, for §8.4, the vertex definition) changes. Master the heap-relax loop once and every variant is a re-skinning.

So the family reduces to three questions: what is a vertex, what does one edge cost, and how do costs combine into a path total — sum, max, or under a constraint that rides in the state?

5   Pseudocode Primer

Classic Dijkstra with lazy deletion. The settled-skip line and the relax-then-push are the two that matter.

PseudocodeDijkstra — min-heap, relax, settle cheapest-first
FUNCTION dijkstra(source, adj):
    dist[*] ← ∞ ; dist[source] ← 0
    heap ← min-heap with (0, source)

    WHILE heap not empty:
        (cost, u) ← pop-min(heap)
        IF cost > dist[u]: CONTINUE      stale entry — already settled cheaper

        FOR (v, w) IN adj[u]:
            newCost ← cost + w            the combine rule (sum here; max for min-max)
            IF newCost < dist[v]:
                dist[v] ← newCost
                push(heap, (newCost, v))   improved → re-enter the frontier

    RETURN dist
Why the cost > dist[u] skip is the whole trick: a vertex can sit in the heap several times (once per improvement), but only the first pop — the cheapest — is valid; every later copy has a cost larger than the now-finalized dist[u], so it's skipped. This "lazy deletion" replaces the textbook decrease-key operation, which Swift's lack of an indexed heap makes awkward. The result is the same shortest paths with far simpler code.

6   The Heap in Swift — you must hand-roll it

Dijkstra is inherently iterative (a heap loop, no recursion), but Swift's standard library has no priority queue — so the one piece of real work is supplying a min-heap. Two practical routes:

  • Hand-rolled binary heap (the expected interview answer). A struct over an array with siftUp on insert and siftDown on removal, ordered by a comparator. ~30 lines, O(log n) push/pop. Know it cold — interviewers specifically watch for whether you can produce one.
  • Sorted-structure stand-in (acceptable, slower). For small graphs you can keep the frontier in a structure you re-sort or scan for the min each step — simpler to write but O(V) per extraction, degrading Dijkstra to O(V²). Fine when V is tiny; say so out loud.
Swifta minimal generic min-heap for Dijkstra frontiers
struct Heap<Element> {
    private var items: [Element] = []
    private let isOrderedBefore: (Element, Element) -> Bool
    init(_ areInIncreasingOrder: @escaping (Element, Element) -> Bool) {
        isOrderedBefore = areInIncreasingOrder
    }
    var isEmpty: Bool { items.isEmpty }

    mutating func push(_ value: Element) {
        items.append(value)
        var child = items.count - 1
        while child > 0 {
            let parent = (child - 1) / 2
            if isOrderedBefore(items[child], items[parent]) {
                items.swapAt(child, parent); child = parent
            } else { break }
        }
    }
    mutating func pop() -> Element? {
        guard !items.isEmpty else { return nil }
        items.swapAt(0, items.count - 1)
        let top = items.removeLast()
        var parent = 0
        while true {
            let left = 2*parent + 1, right = 2*parent + 2
            var best = parent
            if left  < items.count, isOrderedBefore(items[left],  items[best]) { best = left }
            if right < items.count, isOrderedBefore(items[right], items[best]) { best = right }
            if best == parent { break }
            items.swapAt(parent, best); parent = best
        }
        return top
    }
}
Why this matters here specifically: the heap is the load-bearing data structure of Dijkstra, and forgetting Swift lacks one is a classic stumble. Bring this struct ready, parameterize it with a comparator ({ $0.cost < $1.cost }), and the rest of every Dijkstra variant is the relax loop on top of it.

7   Swift Foundations

Five Swift facts underlie nearly every Dijkstra. The heap and the lazy-skip are where the algorithm is won or lost.

Swiftthe five Dijkstra building blocks for this block
// 1. Distance array initialized to a sentinel "infinity".
var dist = [Int](repeating: Int.max, count: numNodes)
dist[source] = 0

// 2. The min-heap (see §6), ordered by cost.
var heap = Heap<(cost: Int, node: Int)>({ $0.cost < $1.cost })
heap.push((0, source))

// 3. Lazy-deletion skip: ignore a popped entry that's already been improved.
while let (cost, node) = heap.pop() {
    if cost > dist[node] { continue }      // stale — skip
    // ... relax neighbors ...
}

// 4. Relaxation — the combine rule. Sum for classic; max for min-max (§8.3).
let newCost = cost + weight              // or: max(cost, weight)
if newCost < dist[neighbor] {
    dist[neighbor] = newCost; heap.push((newCost, neighbor))
}

// 5. Early exit: if you only need the target, return the moment you pop it.
if node == target { return cost }     // first pop of target = final distance
The negative-weight trap. Dijkstra's "settled-is-final" guarantee assumes every edge weight is ≥ 0. A single negative edge can make a vertex reachable more cheaply after it's settled, silently producing wrong answers — no crash, just incorrect output. If weights can be negative, Dijkstra is the wrong tool; use Bellman-Ford. Always confirm non-negativity before reaching for the heap.

8   The Four Variants

Ordered by how far they bend the cost model. Variant 8.3 is taught in a full walkthrough (LC1631); the others are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Classic single-source (sum of weights)
O(E log V)O(V+E)
Solve-it-yourself variant. This is an attempt problem — the frame is below, but no solution. It's the canonical Dijkstra to build cold.
When to use: minimize total weight from a source to a target (or to all). The textbook form: adjacency list, min-heap, relax by summing.

The frame (LC 743 — Network Delay Time): build a weighted adjacency list from the edge triples, run Dijkstra from the source, and the answer is the maximum finalized distance over all nodes (the time for the signal to reach the farthest) — or −1 if any node is unreachable (its distance stays ∞).

The frame: this is Dijkstra with nothing added — sum relaxation, lazy-deletion heap, settle-cheapest-first. The only problem-specific twist is reading the answer (max over all dist, and the unreachable check). Master this and the other three are variations. See pitfalls §10.

Canonical problem: LC 743 — Network Delay Time (attempt). Graduated hints in the walkthrough.

Variant 8.2 Grid Dijkstra (cells as vertices)
O(RC log RC)O(RC)
When to use: a weighted grid — each cell is a vertex, each move into a neighbor carries a cost — and you want the cheapest path between two cells. The implicit-graph cousin of §8.1.

The frame: exactly like grid BFS, but the frontier is a min-heap instead of a queue, because moves have different costs. Neighbors come from a direction table; the move cost is the edge weight; dist[cell] is keyed by the encoded cell row·cols+col. Settle-cheapest-first gives the shortest weighted path.

The frame: grid Dijkstra is grid BFS with a priority queue. If every move cost the same you'd use BFS; because they differ, the heap is required so the cheapest cell is always expanded next. The same encoding and direction-table machinery from the BFS deep dive §8.1 carries over. This is the structural base LC1631 (§8.3) builds on.

Canonical framing: weighted-grid shortest path; the structural base for LC 1631.

Variant 8.3 Custom-cost / min-max path
O(RC log RC)O(RC)
When to use: the path's cost isn't a sum — it's the maximum (or minimum) single edge along it. "Minimize the largest step," "least effort," "widest/narrowest path." Full Swift in the walkthrough (LC1631).

The frame (LC 1631 — Path With Minimum Effort): on a grid of heights, a path's effort is the maximum absolute height difference between consecutive cells. You want the path from top-left to bottom-right with the minimum possible effort. Run grid Dijkstra, but relax with max: the cost to reach a neighbor is max(effortSoFar, |heightDiff|). The heap settles the cell of least max-effort first.

The design decision: the only change from §8.2 is the combine rule — max replaces +. The settle-cheapest-first invariant still holds because max is monotonic with non-negative inputs (extending a path can never lower its max edge), so once a cell is popped its minimum-effort is final. Recognizing that "minimize the maximum" is still a Dijkstra with a swapped operator is the whole insight.

Canonical problem: LC 1631 — Path With Minimum Effort (walkthrough). Full walkthrough.

Variant 8.4 Constrained Dijkstra (augmented state)
O(E·K log …)O(V·K)
Solve-it-yourself variant. This is a target problem — the augmented-state model is below, but no code. The hardest Dijkstra framing on the plan.
When to use: the cheapest path is subject to an extra budget — at most K stops, within T minutes. The vertex must carry the constraint, becoming (node, budgetUsed).

The model (LC 787 — Cheapest Flights Within K Stops): minimize ticket cost from src to dst using at most K stops. A plain dist[node] isn't enough — reaching a node cheaply but with too many stops may be useless, while a pricier route with stops to spare may win. So the state is (node, stopsUsed); the heap is keyed by cost, and you only expand while stops remain. (A bounded BFS / Bellman-Ford over K rounds is the common alternative here.)

The frame: once the constraint rides in the vertex, it's plain Dijkstra over the augmented state graph — the new skill is realizing dist must be indexed by both node and remaining budget, or you'll wrongly discard a path that costs more but obeys the limit. Recognizing the extra state dimension is the whole problem. See pitfalls §10.

Canonical problem: LC 787 — Cheapest Flights Within K Stops (target). Graduated hints in the walkthrough.

9   Complexity Analysis

Dijkstra with a binary heap is O(E log V): each edge can trigger one heap push (O(log V)), and each vertex is settled once. The cost shifts with the heap implementation and, for §8.4, with the size of the augmented state space.

VariantTimeSpaceWhy
Classic single-source (8.1)O(E log V)O(V+E)Up to E pushes, each O(log V) on the heap; adjacency list + dist + heap are O(V+E).
Grid Dijkstra (8.2)O(RC log RC)O(RC)V = R·C cells, E = O(RC) edges (constant directions per cell); heap holds up to O(RC).
Custom-cost / min-max (8.3)O(RC log RC)O(RC)Identical to grid Dijkstra — only the combine operator changes, not the work.
Constrained (8.4)O(E·K log(V·K))O(V·K)The state space multiplies by the budget K; each (node, budget) pair is settled once.
Bottleneckheap ops—The log factor is the heap; with a Fibonacci heap it drops to O(E + V log V), but binary-heap Dijkstra is the practical interview answer. Using a sorted-array frontier instead degrades it to O(V²) — fine only for tiny V.
Why Dijkstra costs more than BFS — and when it's worth it. BFS is O(V+E) because a FIFO queue already pops in distance order when edges are equal-cost. Dijkstra pays an extra log V for the heap precisely to handle unequal weights, where FIFO order no longer matches cost order. If all weights are equal, drop the heap and use BFS; the moment they differ, the heap is the price of correctness.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Negative edge weights (all variants). Dijkstra's settled-is-final guarantee assumes weights ≥ 0. A negative edge silently breaks it. Use Bellman-Ford instead — confirm non-negativity first.
▸Forgetting Swift has no priority queue. You must hand-roll a min-heap (§6). Reaching for a non-existent PriorityQueue or using sorted() each step (O(V²)) is the classic stumble.
▸No lazy-deletion skip. Without the cost > dist[u] guard you re-process settled vertices through stale heap entries — wasted work, and with a buggy combine rule, wrong answers. Skip stale pops.
▸Wrong combine rule for the cost model (8.3). Min-max paths relax with max, not +. Summing the height differences answers a different (wrong) question. Match the operator to "what is a path's cost?"
▸Single-dimension dist on a constrained problem (8.4). With a K-stop or time budget, dist[node] alone discards valid pricier-but-within-budget paths. Index dist by (node, budget).
▸Settling on push instead of on pop. A vertex is final only when popped from the heap, not when first pushed. Marking it done on push can finalize a non-minimal distance.

Swiftcopy-paste classic Dijkstra (assumes the Heap from §6)
func dijkstra(_ numNodes: Int, _ adjacency: [[(node: Int, weight: Int)]], _ source: Int) -> [Int] {
    var dist = [Int](repeating: Int.max, count: numNodes)
    dist[source] = 0
    var heap = Heap<(cost: Int, node: Int)>({ $0.cost < $1.cost })
    heap.push((0, source))
    while let (cost, node) = heap.pop() {
        if cost > dist[node] { continue }          // stale entry — skip
        for (next, weight) in adjacency[node] {
            let newCost = cost + weight              // min-max: max(cost, weight)
            if newCost < dist[next] {
                dist[next] = newCost
                heap.push((newCost, next))
            }
        }
    }
    return dist
}

LC 743 — Network Delay Time (Medium). A signal starts at node k in a directed weighted graph; times[i] = [u, v, w] is an edge u→v taking w time. Return the time for all nodes to receive the signal, or −1 if some never do. Recognize this as classic single-source Dijkstra and build the heap-relax loop before writing any code.
