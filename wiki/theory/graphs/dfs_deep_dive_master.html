Depth-First Search

Depth-first search explores a graph by committing to one branch and following it as far as it goes before backing up to try alternatives. It is the mirror image of BFS: where BFS fans out in layers using a queue, DFS plunges down using a stack — usually the implicit call stack of recursion. That "go deep, then backtrack" rhythm is exactly what you want when the question is about whole paths, connected regions, every combination of choices, or the cheapest total over a sequence of decisions.

DFS is less about shortest paths and more about exhaustive structure: is this region all one piece, does a cycle exist, what's the best outcome over every way of choosing? When the choices form an implicit state graph — each decision leading to a new state — DFS becomes top-down dynamic programming: recurse over the choices, and memoize each state so overlapping subproblems are solved once. That is the pattern LC638 — Shopping Offers instantiates. This deep dive lives under the Graphs hub, which carries the general graph vocabulary; here we go deep on DFS and its four recurring shapes.

2   Recognition Signals — what variant is this?

"Explore everything down a branch" is the DFS master signal. What you're exploring — a region, a tree of choices, a state space, or edges for a cycle — picks the variant.

When the problem says…It means…
"count islands / connected components / flood fill a region"Component DFS — sweep cells/nodes, DFS each unvisited one to mark its whole region. §8.1 (LC200)
"generate all permutations / combinations / subsets / valid paths"Recursion-tree DFS (backtracking) — choose, recurse, un-choose. §8.2 (LC46, LC78)
"minimize / count over a sequence of choices", greedy overshootsDFS over states + memoization (top-down DP) — recurse on the state, cache it. §8.3 (LC638)
"is there a cycle / can all tasks be ordered / detect a deadlock"Cycle-detection DFS — three-color marking or recursion-stack tracking. §8.4 (LC207)
"does a path exist / is everything reachable"Plain reachability — DFS or BFS both work; you don't need the shortest-path property.
"fewest steps / minimum moves" on equal-cost edgesNot DFS — that's BFS; its layer order gives shortest paths for free.
The tell: you need to explore all the way down — every node of a region, every branch of choices, or every state of a decision sequence — not just the nearest layer. The instant you read "all paths / all combinations," "whole region," "cycle," or "best over every choice," reach for recursion. Then ask: am I marking a region (§8.1), enumerating choices (§8.2), optimizing over states (§8.3), or hunting a cycle (§8.4)?

4   Mental Model — plunge, then back up

The one idea
DFS is exploring a maze by always taking a corridor as far as it goes, dropping a breadcrumb at every junction, and only backing up to the last junction when you hit a dead end. The call stack is your trail of junctions; the visited set is your breadcrumbs. Because you finish a whole branch before touching the next, DFS naturally answers "what is the entire structure below here?" — a region's full extent, every leaf of a choice tree, or the best total over all decisions.

Watch DFS traverse a small graph from A. It dives down the first edge each time, only backing up at a dead end. The number beside each node is the order it's first visited:

DiagramDFS visit order from A — deep before wide
      A          edges: A-B, A-E, B-C, B-D
     / \
    B   E        DFS(A): visit A(1)
   / \             → B(2) → C(3) dead end, back up
  C   D                  → D(4) dead end, back up to A
                    → E(5) dead end, done

visit order: A B C D E      (one branch fully, then the next)
call stack peaks at [A,B,C] — depth 3, the longest root-to-leaf path

DFS exhausts B's whole subtree (C, D) before ever touching E. Contrast BFS, which would visit A, then B and E together, then C and D — layer by layer.

The leap that unlocks the hardest variant: the "graph" can be an implicit space of decisions. Each recursive call is a state; each choice is an edge to a new state. DFS explores the whole tree of choices — and when many choice-orders reach the same state, you cache it so it's solved once. That's top-down DP:

TraceDFS over states with memo — the LC638 shape
state  = what's left to decide (e.g. remaining needs vector)
choice = apply one option → a smaller state

solve(state):
    if state in memo: return memo[state]        ← overlapping subproblem
    best = baseline(state)                       ← the "stop choosing" option
    for choice in options(state) that fit:
        best = min(best, cost(choice) + solve(next(state, choice)))
    memo[state] = best ; return best

DFS the choices, memoize the state → exponential search becomes polynomial.

Plain region/path DFS and DFS-over-states are the same recursion; only "what is a vertex" and "is there a value to optimize and cache" change. The memo is what separates top-down DP from naive backtracking.

So the family reduces to three questions: what does one recursive call explore (a region, a choice, a state?), what's the work — mark, collect, or minimize — and is the state reached many ways (if so, memoize)?

5   Pseudocode Primer

Two shapes you must know cold: the plain visited-set DFS (reachability/components) and the memoized state DFS (top-down DP).

Pseudocodeplain DFS — mark visited, recurse on neighbors
FUNCTION dfs(node):
    IF node in visited: RETURN
    visited.insert(node)
    // pre-order work here (mark region, record entry)
    FOR next IN neighbors(node):
        dfs(next)
    // post-order work here (combine children, record exit)
Pseudocodememoized DFS over states — top-down DP
FUNCTION solve(state):
    IF state in memo: RETURN memo[state]    overlapping subproblem
    IF isBase(state): RETURN baseValue       smallest state, direct answer

    result ← baseline(state)                 the "do nothing more" option
    FOR choice IN validChoices(state):
        result ← combine(result, value(choice) + solve(apply(state, choice)))

    memo[state] ← result
    RETURN result
Why the visited set (plain DFS) and the memo (state DFS) are the same idea in different clothes: both stop you from re-doing a node/state you've already handled. The visited set answers a yes/no ("have I been here?") for traversal; the memo stores a value ("what was the answer here?") for optimization. Drop either and a cyclic or overlapping problem explodes — into an infinite loop, or into exponential recomputation.

6   Iterative DFS — the explicit stack

DFS reads most naturally as recursion, but the call stack has a depth limit; for very deep graphs (a 10⁵-node path, a skewed tree) you convert the recursion to an explicit stack:

  • Plain DFS → push the start, loop: pop a node, if unvisited mark it and push its neighbors. This is the literal de-recursion — the explicit stack replaces the call stack. Order differs slightly from recursive DFS (neighbors come off in reverse push order) but the set of visited nodes is identical.
  • Memoized state DFS is usually left recursive — its depth is bounded by the number of decisions (small), and the recursion reads far clearer than a hand-rolled stack of frames. The memo, not the stack form, is what makes it efficient. If depth is a genuine risk, the bottom-up DP table (filling states in dependency order) is the iterative counterpart.
Why this matters here specifically: the standard prefers iterative solutions because recursion hides the data structure and risks stack overflow. For component/reachability DFS on large grids, prefer the explicit stack. For top-down DP (§8.3), recursion + memo is the idiomatic and readable form, and overflow is rarely the constraint — but know the bottom-up table as its iterative twin.

7   Swift Foundations

Five Swift facts underlie nearly every DFS in this family. The memo-key and visited-set ones are where state-DFS problems are won.

Swiftthe five DFS-handling rules for this block
// 1. Recursive DFS is a nested func capturing shared state (visited / memo).
func explore(_ grid: [[Int]]) {
    var visited: Set<Int> = []
    func dfs(_ r: Int, _ c: Int) { /* uses visited */ }
    dfs(0, 0)
}

// 2. Memo key: any Hashable that fully captures the state.
//    Swift arrays ARE Hashable when their elements are — [Int] is a legal key.
var memo: [[Int]: Int] = [:]            // state vector → answer

// 3. The memo guard returns early; missing → compute, store, return.
if let cached = memo[state] { return cached }

// 4. Explicit stack when recursion depth is a risk:
var stack: [(Int, Int)] = [(0, 0)]
while let (r, c) = stack.popLast() { /* mark, push neighbors */ }

// 5. Backtracking: mutate, recurse, then UNDO — leave state as you found it.
path.append(choice); dfs(rest); path.removeLast()    // un-choose
The shared-mutable-state trap. A recursive DFS that mutates a shared array (a path, a board) must undo the mutation on the way out, or sibling branches inherit corruption. For memoized state DFS, build a fresh leftover state per branch (don't mutate the caller's) so siblings don't interfere — the memo assumes each state value is computed in isolation.

8   The Four Variants

Ordered by what one recursive call explores. Variant 8.3 is taught in a full walkthrough (LC638); the others are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Connectivity / components (flood fill)
O(V+E)O(V)
Solve-it-yourself variant. This is an attempt problem — the sweep-and-flood frame is below, but no solution.
When to use: count or label maximal connected regions — islands in a grid, friend circles, provinces. DFS from each unvisited node marks its entire component in one dive.

The frame (LC 200 — Number of Islands): sweep every cell; when you hit an unvisited land cell, increment the island count and DFS-flood the whole connected land region (4-directionally), marking each cell visited so it's never counted again. The number of DFS launches = the number of components.

The frame: the outer sweep finds one seed per region; the inner DFS consumes the whole region so the sweep never re-counts it. Marking visited (or sinking land to water) is what guarantees each cell is processed once → O(V+E). See pitfalls §10.

Canonical problem: LC 200 — Number of Islands (attempt). Graduated hints in the walkthrough.

Variant 8.2 Recursion-tree / backtracking (enumerate choices)
O(branching^depth)O(depth)
Solve-it-yourself variant. This is an attempt problem — the choose/recurse/un-choose frame is below, but no solution.
When to use: generate or count all valid arrangements — permutations, subsets, combinations, board placements. The recursion tree's leaves are the complete solutions.

The frame (LC 46 — Permutations / LC 78 — Subsets): at each level, pick an unused element (a choice), recurse to build the rest, then un-pick it (backtrack) and try the next. The path from root to leaf is one solution; the un-choose step is what lets sibling branches start clean.

The frame: no memo here — the goal is to enumerate every leaf, not optimize over overlapping states, so each path is distinct. The discipline is the mutate-recurse-undo trio; forgetting the undo bleeds one branch's choices into the next. See pitfalls §10.

Canonical problems: LC 46 — Permutations, LC 78 — Subsets (attempt). Graduated hints in the walkthrough.

Variant 8.3 DFS over states + memoization (top-down DP)
O(states·work)O(states)
When to use: minimize or count over a sequence of decisions where greedy fails and the same sub-situation recurs via different decision orders. Recurse on the state; cache it. Full Swift in the walkthrough (LC638).

The frame (LC 638 — Shopping Offers): the state is the remaining needs vector. From a state, the options are: apply each special offer that fits (no overshoot) and recurse on the smaller leftover, or buy everything left at base price. Return the minimum. Because many offer-orders reach the same leftover, memoize on the needs vector — that's what turns the exponential search polynomial.

The design decision: the only new skill versus plain DFS is recognizing the state — the minimal info that determines the remaining answer (here: what's left to buy, since offers are unlimited and order-independent) — and memoizing on it. The recursion explores choices; the memo collapses repeats. This is the bridge from backtracking (§8.2) to dynamic programming.

Canonical problem: LC 638 — Shopping Offers (walkthrough). Full walkthrough.

Variant 8.4 Cycle detection / coloring (directed graphs)
O(V+E)O(V)
Solve-it-yourself variant. This is a target problem — the three-color model is below, but no code. (Topological sort gets its own deep dive; this is the DFS-flavored detector.)
When to use: decide whether a directed graph has a cycle — "can all courses be finished," "is there a dependency deadlock." A cycle means an ordering is impossible.

The model (LC 207 — Course Schedule): DFS each node with three-color marking — White (unseen), Gray (on the current recursion path), Black (fully explored). If DFS reaches a Gray neighbor, that's a back-edge → a cycle exists. If every node finishes Black with no Gray hit, the graph is acyclic and the tasks can be ordered.

The frame: the Gray state — "this node is an ancestor on my current path" — is the whole trick; a plain visited/Black set alone can't tell a cross-edge from a back-edge. Equivalently, track the recursion stack as a set. Topological order falls out of post-order finishing times. See pitfalls §10 and the (coming) Topological Sort deep dive.

Canonical problem: LC 207 — Course Schedule (target). Graduated hints in the walkthrough.

9   Complexity Analysis

Plain DFS is O(V + E): the visited set guarantees each vertex is entered once and each edge examined once. The state variants shift the cost into "how many states exist" and "is there a memo."

VariantTimeSpaceWhy
Component DFS (8.1)O(V+E)O(V)Each cell/node visited once across all DFS launches; visited set + recursion depth are O(V).
Backtracking (8.2)O(b^d)O(d)The recursion tree has one leaf per solution — inherently exponential (there are exponentially many permutations/subsets); space is the path depth d.
Memoized state DFS (8.3)O(states·work)O(states)Each distinct state solved once (memo); work = choices tried per state. The memo is what removes the exponent.
Cycle detection (8.4)O(V+E)O(V)One DFS pass; three-color array + recursion stack are O(V).
Bottleneckstate/leaf count—DFS itself is linear in the graph it walks; the cost is how big that graph is. Backtracking is exponential because the answer set is; memoization tames the state variant by reusing repeats. Without the memo, §8.3 would also be exponential.
Why memoization is the headline. Naive DFS over a decision space recomputes every state once per path that reaches it — exponential. Caching each state's answer collapses those repeats into one solve per distinct state, so time becomes "reachable states × work per state." That single change is what separates top-down DP (§8.3) from the exponential backtracking of §8.2 — same recursion, one dictionary added.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸No visited set on a cyclic graph (8.1/8.4). DFS without marking revisits nodes forever — infinite recursion / stack overflow. Mark a node before (or as) you recurse into it.
▸Forgetting to un-choose in backtracking (8.2). Mutating a shared path/board and not undoing it on return leaks one branch's state into its siblings. The mutate-recurse-undo trio must be complete.
▸Wrong memo key — missing a state dimension (8.3). If the cached answer actually depends on something not in the key, you return stale results. The key must capture everything that determines the remaining answer (and nothing irrelevant, or you cache too little).
▸Mutating the caller's state instead of a fresh copy (8.3). Build a new leftover/state per branch; mutating in place corrupts sibling branches and poisons the memo, which assumes states are evaluated in isolation.
▸Using Black/visited alone for cycle detection (8.4). A plain visited set can't distinguish a back-edge (cycle) from a cross-edge to an already-finished node. You need the Gray "on current path" state.
▸Reaching for DFS when BFS is required. DFS finds a path, not the shortest. For "fewest steps" on equal-cost edges, use BFS — its layer order is the shortest-path order.
▸Stack overflow on deep recursion. A 10⁵-long path or skewed tree can exceed the call-stack limit. Convert to an explicit stack (§6) when depth is unbounded.

Swiftcopy-paste memoized state-DFS skeleton (top-down DP)
func solveTopDown(_ start: [Int]) -> Int {
    var memo: [[Int]: Int] = [:]
    func solve(_ state: [Int]) -> Int {
        if let cached = memo[state] { return cached }
        var best = baseline(state)                 // the "stop choosing" option
        for choice in validChoices(state) {
            let next = apply(state, choice)         // FRESH state, not a mutation
            best = min(best, cost(choice) + solve(next))
        }
        memo[state] = best
        return best
    }
    return solve(start)
}
