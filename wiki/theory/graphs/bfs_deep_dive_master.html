Breadth-first search explores a graph outward from a starting vertex in layers: first all vertices one edge away, then all vertices two edges away, and so on. It is driven by a queue (first-in-first-out), which is exactly what enforces the layer order — and that order is what makes BFS the canonical solution to unweighted shortest path: the first time BFS reaches a vertex, it has reached it by a path with the fewest possible edges.

The single decision that sends a problem to BFS is "fewest steps where every step costs the same." Whether the graph is an explicit adjacency list, a 2-D grid, or an implicit space of board/bitmask states one move apart, the machinery is identical: a queue of frontier vertices, a visited set to never re-process one, and a distance that increases by one each layer. This deep dive lives under the Graphs hub, which carries the general graph vocabulary; here we go deep on BFS and its four recurring shapes.

2   Recognition Signals — what variant is this?

The phrase "shortest / fewest / minimum number of moves" over equal-cost steps is the master signal for BFS. What the vertices and edges are tells you which variant.

When the problem says…It means…
"shortest path in a grid / matrix", 4- or 8-directional stepsGrid BFS — cells are vertices, adjacent cells are edges. §8.1 (LC1091)
"everything spreads / rots / fills at once per minute"Multi-source BFS — seed the queue with all sources at distance 0. §8.2 (LC994)
"minimum moves to transform one configuration into another", one change per moveState-space BFS — each configuration is a vertex, "one move apart" is an edge. §8.3 (LC1284, LC773)
"reach the goal collecting keys / unlocking boxes along the way"BFS with carried state — the vertex is (position + what you hold). §8.4 (LC1298)
"fewest steps", "minimum number of …", all moves equal costAny BFS — the layer count is the answer.
"cheapest path" with different edge weightsNot BFS — that is Dijkstra (or Bellman-Ford with negatives). BFS assumes equal-cost edges.
"count components / does a path exist / detect a cycle"Plain reachability — BFS or DFS both work; you don't need the shortest-path property.
The tell: the answer is a number of steps and every step costs the same. The instant you read "minimum moves / fewest steps / shortest in an unweighted graph," reach for a queue. Then ask: what is a vertex (a cell? a whole board? a (cell, keys) pair?) and what counts as one edge? That answer picks the variant.

4   Mental Model — a wave spreading outward

The one idea
BFS is water spreading from a source: it floods all the cells one step away, then all the cells two steps away, never skipping a layer. Because the queue is FIFO, vertices come out in non-decreasing distance order — so the first time the wave touches a vertex, it has arrived by a fewest-edges path. That "first touch = shortest path" property is the whole reason BFS exists, and it only holds because every edge costs the same.

Picture a grid, source at the top-left S, walls as #. BFS labels each open cell with its distance from S. Watch the labels grow outward in concentric rings — each ring is one queue frontier:

DiagramBFS distances spread in rings from S
grid:            distances (BFS layers):
S . . #            0 1 2 #
. # . .            1 # 3 4
. . . T            2 3 4 5

layer 0: { S }                 → dist 0
layer 1: { (0,1),(1,0) }       → dist 1
layer 2: { (0,2),(2,0) }       → dist 2
layer 3: { (1,2),(2,1) }       → dist 3
layer 4: { (0,3)wall, (1,3),(2,2) } → dist 4
layer 5: { T }                 → dist 5   ← first touch of T = shortest path

Each ring is exactly one queue frontier. T is first reached at layer 5, so the shortest path is 5 edges — no later, longer path can overwrite it because BFS already marked T visited on first touch.

Now the key abstraction that unlocks the harder variants: the grid is just one kind of graph. Replace "cell" with "any configuration" and "adjacent cell" with "any single legal move," and the same wave finds the fewest moves between two configurations:

Tracestate-space BFS — board configurations as vertices
vertex = a whole board state (encoded as a string / Int)
edge   = "apply one legal move" → a neighboring state

start state ─┬─ move A ─→ state₁ ─┬─ ...
             ├─ move B ─→ state₂ ─┤
             └─ move C ─→ state₃ ─┘
queue holds whole states; visited holds encoded states already seen.
BFS layer k = "reachable in exactly k moves" → first touch of the
goal state is the minimum number of moves. Same algorithm, bigger vertices.

Grid BFS and state-space BFS are the same code; only "what is a vertex" and "what is a neighbor" change. Master the wave once and every BFS variant is a re-skinning of it.

So the entire family reduces to three questions: what is a vertex, what is one edge (one equal-cost move), and what is the goal? Answer those and the queue-plus-visited loop is mechanical.

5   Pseudocode Primer

The canonical BFS. The two non-negotiable lines are mark visited when enqueuing and increment distance once per level.

Pseudocodelevel-counting BFS — distance bumps once per frontier
FUNCTION bfs(start, isGoal, neighbors):
    queue   ← [start]
    visited ← { start }          mark on ENQUEUE, never on dequeue
    distance ← 0

    WHILE queue not empty:
        levelSize ← size(queue)          freeze this layer's count
        REPEAT levelSize times:           drain exactly one layer
            node ← dequeue(queue)
            IF isGoal(node): RETURN distance

            FOR next IN neighbors(node):
                IF next not in visited:
                    visited.insert(next)     mark NOW, before it can be re-queued
                    enqueue(queue, next)
        distance ← distance + 1          one whole layer done

    RETURN -1                          goal unreachable
Why mark visited on enqueue, not dequeue: if you only mark a vertex visited when you pop it, the same vertex can be enqueued many times by different neighbors before any copy is popped — blowing up the queue and, worse, sometimes letting a longer path overwrite the distance. Marking the instant you enqueue guarantees each vertex enters the queue exactly once, which is what makes the whole search O(V + E).
Why the level-size loop: freezing levelSize = size(queue) before draining lets you bump distance exactly once per layer, so you never have to store a distance alongside every queued node. The alternative — enqueue (node, dist) pairs — is equally correct and sometimes clearer; both appear in the variants.

6   The Level-Loop Pattern — BFS is inherently iterative

Unlike DFS, BFS has no natural recursive form — it is a queue loop by construction, which suits the standard's iterative discipline perfectly. Two equivalent ways to track distance:

  • Level-size loop (no per-node distance). Capture queue.count at the top of each while pass, drain exactly that many nodes, then increment one shared distance. Best when you need the distance only at the goal, or need to act "per layer" (e.g. multi-source spread time).
  • Distance carried in the queue. Enqueue (node, distance) tuples, or keep a dist[node] map set when you enqueue. Best when different nodes need their own recorded distance, or the graph is sparse and a level boundary is awkward.
Why this matters here specifically: both forms are O(V + E) and interview-acceptable; pick by what the problem asks. "Minimum moves to reach the goal" → level-size loop (one number). "Time for the last orange to rot" → also level-size loop, but the answer is the final layer index. "Distance to every cell" → carry a dist map. Knowing both means you never fumble the bookkeeping.

7   Swift Foundations

Five Swift facts underlie nearly every BFS in this family. Say the queue-cost one out loud before coding — it is the most common interview ding.

Swiftthe five BFS-handling rules for this block
// 1. The queue: an Array used FIFO. removeFirst() is O(n) — shifts the whole array!
//    For interviews, use a head index instead of removeFirst() to keep BFS O(V+E).
var queue: [State] = [start]
var head = 0
while head < queue.count {
    let node = queue[head]; head += 1   // O(1) dequeue
    // ... expand neighbors, queue.append(next) ...
}

// 2. Level-size loop: freeze the count, drain exactly one layer, bump distance once.
var distance = 0
while !queue.isEmpty {
    let levelSize = queue.count
    for _ in 0..<levelSize { /* dequeue + expand */ }
    distance += 1
}

// 3. Grid neighbors: a direction table makes the four (or eight) moves uniform.
let directions = [(0,1), (0,-1), (1,0), (-1,0)]   // add diagonals for 8-dir

// 4. The visited set keys depend on the vertex type:
var visited: Set<Int> = []                 // grid cell encoded as row*cols+col, or a bitmask state
//   For a board, encode the whole configuration to a String or Int and store THAT.

// 5. Bounds + wall check, in this order — bounds first, or you index out of range.
func inBounds(_ r: Int, _ c: Int) -> Bool {
    r >= 0 && r < rows && c >= 0 && c < cols
}
The removeFirst() trap. Array.removeFirst() is O(n) because it shifts every remaining element. Calling it once per vertex turns an O(V + E) BFS into O(V² ). Use a head index (shown above) or a ring buffer / deque. Mention this out loud — interviewers listen for it.

8   The Four Variants

Ordered by increasing abstraction of "what is a vertex." The first two are taught in full walkthroughs (the grid warm-up and the state-space hard problem); the last two are solve-it-yourself problems — frame and pitfalls here, no solution.

Variant 8.1 Grid BFS
O(rows·cols)O(rows·cols)
When to use: the graph is a 2-D grid — cells are vertices, adjacent open cells are edges — and you want the fewest steps from a start cell to a target. Full Swift in the walkthrough (LC1091).

The frame (LC 1091 — Shortest Path in Binary Matrix): the grid is an implicit graph; never build an adjacency list. From a cell, the neighbors are the up-to-8 in-bounds, open (value-0) cells given by a direction table. Run plain BFS from the start, count layers, return the layer at which you first reach the bottom-right cell.

The design decision: the grid spares you graph construction — neighbors are computed by adding direction offsets and checking bounds + wall in that order. The visited set is just the cells already enqueued (or you mutate the grid in place to mark them). Because BFS first-touch is shortest, the first arrival at the target is the answer.

Canonical problem: LC 1091 — Shortest Path in Binary Matrix (warm-up). Full walkthrough.

Variant 8.2 Multi-source BFS
O(rows·cols)O(rows·cols)
Solve-it-yourself variant. This is an attempt problem — the seeding trick and the level-as-time rule are below, but no solution. Build it cold.
When to use: many things spread simultaneously and you want the time until everything is reached (or to detect something never is). The twist: seed the queue with all sources at once, each at distance 0.

The frame (LC 994 — Rotting Oranges): every rotten orange is a source. Enqueue them all before the loop starts (distance 0), then run a normal level-size BFS. Each layer is one minute; a fresh orange rots when the wave first reaches it. The answer is the last layer index — or −1 if any fresh orange is never visited.

The frame: multi-source BFS is single-source BFS with a fat layer 0. Because all sources start at distance 0, every cell is labeled with the distance to its nearest source automatically — no need to run BFS once per source. Recognizing "seed them all at once" is the whole trick. See pitfalls §10.

Canonical problem: LC 994 — Rotting Oranges (attempt). Graduated hints in the walkthrough.

Variant 8.3 State-space BFS (configurations as vertices)
O(states · movesPerState)O(states)
When to use: "minimum moves to turn configuration X into configuration Y," where one move changes the whole configuration a little. The vertex is an entire state; the edge is "apply one legal move." Full Swift in the walkthrough (LC1284).

The frame (LC 1284 — Min Flips to Convert Binary Matrix): the whole matrix is one vertex, encoded as an Int bitmask (cell r,c → bit r·cols+c). A move flips a cell and its 4 neighbors — i.e. XOR a precomputed mask. BFS from the start matrix; the first layer that reaches the all-zero matrix is the minimum number of flips. The encoding is what makes the visited set possible.

The design decision: the only new skill versus grid BFS is encoding a whole configuration into one hashable key (a string for a board, an Int bitmask for a small matrix) so the visited set can store it. Once states are integers, "states one move apart" is just "XOR with a move-mask," and the BFS loop is unchanged. The state space is finite and small (2^(rows·cols)), so BFS terminates.

Canonical problem: LC 1284 — Min Flips to Convert Binary Matrix (walkthrough); LC 773 — Sliding Puzzle is the same shape with a string-encoded board. Full walkthrough.

Variant 8.4 BFS with carried state (augmented vertex)
O(states · 2^keys)O(states · 2^keys)
Solve-it-yourself variant. This is a target problem — the augmented-vertex model is below, but no code. This is the canonical "BFS where the vertex is more than a position" framing.
When to use: the fewest-steps question depends not just on where you are but on what you have collected (keys, opened boxes, visited set of items). The vertex becomes (position, carried-state).

The model (LC 1298 — Maximum Candies from Boxes): you BFS over boxes you can open, but whether you can open a box depends on the keys you've found so far. The trick is that the reachable frontier grows as you collect keys — a box you couldn't open earlier becomes openable later, so you must revisit "blocked" boxes when new keys arrive. The carried state (set of keys / boxes-in-hand) is part of the vertex; visited must key on the augmented vertex, not the position alone.

The frame: this is plain BFS once you widen the vertex to include the carried state. The classic bug is keying visited on position only — then you never reprocess a box after gaining the key that unlocks its neighbors. Recognizing that the vertex must carry the collected state is the whole problem. See pitfalls §10.

Canonical problem: LC 1298 — Maximum Candies from Boxes (target); LC 773 — Sliding Puzzle also sits between §8.3 and here. Graduated hints in the walkthrough.

9   Complexity Analysis

BFS is O(V + E): every vertex is enqueued once (the visited set guarantees it) and every edge is examined once when its endpoint is expanded. The cost shifts only in how big the graph is — and for state-space BFS, the vertex count can be exponential in the configuration size.

VariantTimeSpaceWhy
Grid BFS (8.1)O(R·C)O(R·C)Each of R·C cells is enqueued once; up to 8 neighbor checks each is a constant. Queue + visited hold up to R·C cells.
Multi-source BFS (8.2)O(R·C)O(R·C)Same as single-source — fat layer 0 doesn't change the total; every cell still enters the queue once.
State-space BFS (8.3)O(S·M)O(S)S = number of reachable states (≤ 2^(R·C) for a bitmask matrix), M = moves per state. Each state enqueued once; visited holds encoded states.
BFS with carried state (8.4)O(S·2^K)O(S·2^K)The vertex is (position, key-set); the key-set multiplies the state count by 2^K. Still each augmented vertex once.
Bottleneckstate count—BFS itself is linear in the graph; the cost is how many vertices the graph has. State-space and augmented-vertex BFS explode because the vertex set is exponential in the configuration — that, not the loop, is what to watch.
Why BFS beats Dijkstra when edges are unweighted. Dijkstra needs a priority queue (O(log V) per operation) to handle varying edge weights. When every edge costs 1, the FIFO queue already pops vertices in distance order for free — so BFS is O(V + E), strictly better than Dijkstra's O(E log V). Use BFS whenever edges are equal-cost; reach for Dijkstra only when they differ.

10   Common Pitfalls

Pitfalls — bugs this block invites
▸Marking visited on dequeue instead of enqueue (all variants). If you mark a vertex visited only when you pop it, the same vertex gets enqueued many times before it's popped — the queue bloats and distances can be wrong. Insert into visited the instant you enqueue.
▸Using removeFirst() as the dequeue (Swift-specific). It's O(n); doing it per vertex makes BFS O(V²). Use a head index or a deque. Say this out loud in an interview.
▸Forgetting the start might already be the goal (8.1/8.3). If start == target (or the start cell is blocked), handle it before the loop — e.g. LC1091 returns −1 if the start cell is a wall, and 1 if start == end and open.
▸Bumping distance per node instead of per layer (8.1/8.2). Incrementing the shared distance inside the inner loop counts edges wrong. Freeze levelSize, drain one full layer, then increment once — or carry distance in the queue.
▸Keying visited on position only when the vertex carries state (8.4). If the answer depends on collected keys, two visits to the same cell with different key-sets are different vertices. Key visited on the full augmented state, or you'll wrongly skip a now-openable box.
▸Bounds check after the wall/grid lookup (grid variants). Always test in-bounds before reading grid[r][c], or you index out of range on edge cells.
▸Reaching for BFS on weighted edges. BFS's shortest-path guarantee only holds when every edge costs the same. Different weights → Dijkstra. A "BFS" over weighted edges silently returns wrong answers.

Swiftcopy-paste grid-BFS skeleton (head-index queue, level loop)
func shortestPath(_ grid: [[Int]], start: (Int,Int), goal: (Int,Int)) -> Int {
    let rows = grid.count, cols = grid[0].count
    let directions = [(0,1),(0,-1),(1,0),(-1,0)]
    var queue = [start], head = 0, distance = 0
    var visited: Set<Int> = [start.0 * cols + start.1]
    while head < queue.count {
        let levelSize = queue.count - head
        for _ in 0..<levelSize {
            let (r, c) = queue[head]; head += 1
            if (r, c) == goal { return distance }
            for (dr, dc) in directions {
                let nr = r + dr, nc = c + dc
                guard nr >= 0, nr < rows, nc >= 0, nc < cols,
                      grid[nr][nc] == 0,
                      !visited.contains(nr * cols + nc) else { continue }
                visited.insert(nr * cols + nc)        // mark on enqueue
                queue.append((nr, nc))
            }
        }
        distance += 1
    }
    return -1                                          // goal unreachable
}

LC 994 — Rotting Oranges (Medium). Given a grid where 0 = empty, 1 = fresh orange, 2 = rotten, each minute every fresh orange adjacent (4-dir) to a rotten one rots. Return the minutes until no fresh orange remains, or −1 if some never rot. Recognize this as multi-source BFS and build the seeding + level-counting before writing any code.

  • What are the vertices, what is one edge, and what does "one minute" correspond to in BFS terms?
  • Why do you enqueue all rotten oranges before the loop, all at distance 0?
  • Where does the answer live — is it the last layer index, and how do you read it off the level loop?
  • How do you detect that a fresh orange is never reached (the −1 case)?
  • What is the answer when there are zero fresh oranges to begin with?
