LC 303 — Range Sum Query - Immutable

Walkthrough Masterfile· Language: Swift· Easy · Prefix sum · Build once, query in O(1)
Problem at a Glance

Theory
prefix_suffix_master.html

Given an integer array nums that does not change after initialization, design a data structure that answers repeated "sum of all elements from index left to index right (inclusive)" queries in constant time.

InputOutputWhy
nums = [-2,0,3,-5,2,-1]
sumRange(0, 2)		1−2 + 0 + 3 = 1
sumRange(2, 5) (same array)-13 + (−5) + 2 + (−1) = −1
sumRange(0, 5) (same array)-3Sum of the entire array = −3

The twist that makes it a precomputation problem: the query is called multiple times on the same static array. Recomputing the sum from scratch on every call costs O(n) per query — acceptable for a single query, catastrophic for k queries on a large array. The key observation is that any range sum can be expressed as a difference of two prefix sums, and those prefix sums can be built once in O(n) and reused forever.

2   Reading the Problem — insights from the statement

"Immutable" array: the array never changes after initialization. This is the single most important word in the problem title — it is what makes precomputation legal. If the array could change between queries, every update would invalidate every prefix value at or after the changed index, requiring O(n) repair. The constraint guarantees the prefix array we build in the initializer will remain correct forever.
"Multiple calls to sumRange": LeetCode's NumericArray class is initialized once and queried many times. This is the "pay once, reuse forever" pattern. Even if sumRange is called only twice, the O(n) preprocessing already pays for itself — and with k queries, the total work is O(n + k) instead of O(n·k).
"left to right, inclusive": both endpoints are included in the sum. The prefix array of length n+1 with a sentinel at index 0 is what makes the formula prefix[right+1] − prefix[left] work uniformly for all ranges — including those starting at index 0 — with no special case.
Negative numbers allowed: nums[i] can be negative. This makes no difference to the prefix-sum approach — addition and subtraction work identically over negative integers. Sliding window would be disqualified here (no monotonic shrink condition), but prefix sums are unaffected.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the naïve approach and its problem

The obvious solution: in sumRange, loop from left to right and accumulate. This is O(n) per call. With k calls this is O(n·k). What information could you compute once at init time and reuse on every query?

Hint 2 — prefix sums

Define prefix[i] = sum of nums[0..i−1]. Then sum(left, right) = prefix[right+1] − prefix[left]. Can you see why? Build prefix during init and answer every query with one subtraction.

Hint 3 — the sentinel and why the array is length n+1

Setting prefix[0] = 0 (the sentinel) is what makes the formula work when left = 0. Without it, sumRange(0, j) would need a special case. Try tracing sumRange(0, 2) on nums = [1, 2, 3] with and without the sentinel to see why.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can the array contain negative numbers?"
A: Yes. → Prefix sums work fine with negatives — I won't assume non-negative values anywhere in the design.
Q: "Is the array truly immutable — it will never be updated between queries?"
A: Correct, it never changes. → This means I can safely precompute a prefix sum array in the constructor and reuse it for every query. If the array could mutate, I'd need a Fenwick tree instead.
Q: "Can left equal right — a single-element range?"
A: Yes. → My formula must handle this without special-casing. The sentinel at prefix[0] = 0 is what makes it uniform.
Q: "How many times will sumRange be called, roughly?"
A: Up to 10⁴ times. → That confirms the O(1)-per-query target. With 10⁴ calls on a 10⁴-element array, O(n) per query would be 10⁸ operations — O(1) per query is the right bar.
Step 2 — The plan, before any code
The plan, spoken

"This is a prefix sum problem — specifically Variant 8.1, the range-query variant. The signal is the word 'immutable' combined with 'multiple calls': I need to answer arbitrary [left, right] range sums in O(1), which means I need to precompute something. The natural choice is a prefix sum array."

"Let me state the governing condition. A range sum sum(left, right) can always be decomposed as: everything from the start up through right, minus everything from the start up through left−1. That decomposition is exact — no approximation. Formalizing it: if prefix[k] = nums[0] + nums[1] + … + nums[k−1], then sum(left, right) = prefix[right+1] − prefix[left]."

"The sentinel is non-optional: I'll allocate the prefix array at length n+1, not n, and set prefix[0] = 0. That sentinel is what makes the formula prefix[right+1] − prefix[left] work uniformly when left = 0 — without it, sumRange(0, j) requires a special case that's a guaranteed bug vector. With it, every range query uses the same formula."

"Data structure: one [Int] array of length n+1 stored as an instance variable. The left pass builds it in a single for loop during init: prefix[index+1] = prefix[index] + nums[index]. The sumRange method is then one line: return prefix[right+1] − prefix[left]."

"Expected complexity: O(n) time and space for the constructor — one linear pass; O(1) time per sumRange call — two array lookups and one subtraction. I'll trace the canonical example to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: sum from left to right in sumRangeO(1) init, O(n) per queryWorks for a single query; with k=10⁴ calls on n=10⁴ elements this is 10⁸ operations — too slow
Sliding window—Wrong tool. Sliding window requires a monotonic condition to decide when to shrink. Range sums with negative numbers have no such condition — you cannot infer the sum of [left, right] from the sum of [left−1, right] without recomputing. Negative values break the "if sum is too big, shrink" invariant.
Prefix sum array (length n+1, sentinel at 0)O(n) init, O(1) per queryChosen. One linear pass at construction. Every query is two array reads and one subtraction. Total cost for k queries: O(n + k). Sentinel eliminates all edge cases.
Fenwick tree (Binary Indexed Tree)O(n) init, O(log n) update and queryOverkill for a static array — a Fenwick tree exists to handle point updates in O(log n). With no updates, a plain prefix array dominates on all metrics: simpler, faster queries, same space.
THE ONE IDEA: Any range sum [left, right] is the difference of two prefix sums: prefix[right+1] − prefix[left]. The sentinel prefix[0] = 0 makes this formula hold for all ranges, including those starting at index 0, without any special case. Build the prefix array once; query forever.
When to reach for prefix sum Variant 8.1: the array is static (immutable between queries) and you must answer multiple "sum from i to j" queries. As soon as you hear "multiple queries" on a non-mutating array, the prefix sum is the answer.
PITFALLS
▸Allocating the prefix array at length n instead of n+1. With length n, there is no room for the sentinel at index 0 — you are forced into one-off index arithmetic on every query. Always allocate n+1 and let prefix[0] = 0 carry the sentinel.
▸Off-by-one in the query formula: using prefix[right] instead of prefix[right+1]. With the sentinel convention, prefix[k] = sum of nums[0..k−1]. So the sum of nums[left..right] is prefix[right+1] − prefix[left]. Using prefix[right] silently drops nums[right] from every query result — a constant under-count that won't always surface in tests with positive integers.
▸Re-initializing the prefix array on every sumRange call. The whole point of the design is to compute prefix once at init and reuse it. If the array is rebuilt inside sumRange, you pay O(n) per query and the design buys nothing over brute force.

7   Data Structures — choice and justification

VariableHoldsWhy this and not something else
prefix: [Int] (length n+1)prefix[k] = sum of nums[0..k−1]; prefix[0] = 0 is the sentinelA plain [Int] array gives O(1) random access — the only operation needed in sumRange. A dictionary would work but costs more memory and has pointer indirection. A stack or linked list cannot be random-accessed.
Length n+1 (not n)The extra slot at index 0 holds the sentinelThe sentinel is not a convenience — it is what makes the query formula work uniformly for all ranges without branching. Without it, sumRange(0, j) is a special case.
Why [Int] and not [Double]: the problem guarantees integer inputs and integer sums. Using Int avoids floating-point error and is the exact type the query result must return.

8   Pseudocode Dry Run

Pseudocodeprefix sum — init and query
CLASS NumericArray:
    prefix: Array[n+1]

    INIT(nums):
        n ← length(nums)
        prefix ← array of size n+1, all zeros
        prefix[0] ← 0                             // sentinel
        FOR index FROM 0 TO n−1:
            prefix[index + 1] ← prefix[index] + nums[index]

    FUNCTION sumRange(left, right) → Int:
        RETURN prefix[right + 1] − prefix[left]

9   Swift Solution

A.1 Prefix sum — build once, query in O(1)
O(n) init · O(1) queryO(n)
Plan
In the initializer, build a prefix array of length n+1 with a sentinel of 0 at index 0. Each cell i+1 holds the running sum through nums[i]. In sumRange, return prefix[right+1] − prefix[left]. No loops at query time. 
class NumericArray {
    private let prefix: [Int]

    init(_ nums: [Int]) {
        var buildPrefix = [Int](repeating: 0, count: nums.count + 1)
        // buildPrefix[0] = 0 is the sentinel — sum of zero elements.
        for index in 0 ..< nums.count {
            buildPrefix[index + 1] = buildPrefix[index] + nums[index]
        }
        prefix = buildPrefix
    }

    func sumRange(_ left: Int, _ right: Int) -> Int {
        // Both left and right are inclusive 0-based indices.
        prefix[right + 1] - prefix[left]
    }
}
A.2 Idiomatic Swift — scan + zip
O(n) init · O(1) queryO(n)
Plan
Replace the manual for-loop with nums.reduce(into: [0]) to build the prefix array — the sentinel [0] is the starting accumulator, and each step appends acc.last! + value. sumRange stays identical. 
class NumericArray {
    private let prefix: [Int]

    init(_ nums: [Int]) {
        prefix = nums.reduce(into: [0]) { acc, value in
            acc.append(acc.last! + value)
        }
    }

    func sumRange(_ left: Int, _ right: Int) -> Int {
        prefix[right + 1] - prefix[left]
    }
}
What changed vs A.1: the [Int](repeating: 0, count: n+1) allocation + index-arithmetic loop is replaced by reduce(into: [0]). The starting accumulator [0] is the sentinel — you never write prefix[0] = 0 explicitly because it is the identity you hand to reduce. Each step appends the running sum, so the array grows to length n+1 exactly as before. reduce(into:) mutates in place (no copies per step), so performance is identical to A.1.

Brute force sumRange (no precomputation)O(n) per queryO(1)Must sum all elements in the range on every call; k calls cost O(n·k)

Bottleneck: you must read every element of nums at least once to know any prefix sum — O(n) for init is optimal. The query at O(1) is also optimal (you must at minimum read two cells and compute a result). Total cost for k queries: O(n + k) vs O(n·k) for brute force.

13   Follow-Up Bank

Follow-up: "Before you push this to production — can you see any improvements you'd make?"

The solution is already optimal for a static array: O(n) init, O(1) query, O(n) space — no algorithmic improvement exists. One constant-factor note: I store the prefix array as let, which signals immutability clearly and prevents accidental mutation. If memory were extremely constrained (unlikely at n ≤ 10⁴), a space-O(1) approach would require accepting O(n) per query — an unavoidable tradeoff once you give up precomputation.

Follow-up: "What if the array can be updated between queries?"

Then precomputing a static prefix array is invalid — any update at index i would invalidate every prefix cell from i+1 onward, requiring an O(n) rebuild. The right data structure becomes a Fenwick tree (Binary Indexed Tree), which supports both point updates and prefix-sum queries in O(log n) per operation. LC 307 — Range Sum Query - Mutable — is exactly this problem with updates enabled.

Follow-up: "Can you generalize this to a 2D matrix — sum of a rectangular subregion?"

Yes — this is LC 304 (Range Sum Query 2D - Immutable). Build a 2D prefix sum prefix[row][col] = sum of the rectangle from (0,0) to (row−1, col−1). The query formula extends to inclusion-exclusion: prefix[row2+1][col2+1] − prefix[row1][col2+1] − prefix[row2+1][col1] + prefix[row1][col1]. The sentinel is now a zero-filled first row and first column instead of a single zero cell.

Follow-up: "What if you need the product of a range instead of the sum?"

Prefix products work for ranges where no zero exists — divide prefix[right+1] by prefix[left]. But integer division loses information if any element in the range is zero, and division is not always exact for integers. For "product except self" (LC 238), the standard approach avoids division entirely using separate prefix and suffix product arrays — the two-pass pattern is cleaner and handles zeros correctly.

14   Theory Decoder — concepts used here

ConceptHow it appears in LC 303
Prefix sum (Variant 8.1)prefix[k] = sum of nums[0..k−1]. Built once in the constructor with a left-to-right pass: prefix[index+1] = prefix[index] + nums[index].
Sentinel valueprefix[0] = 0 — the identity element of addition. It represents "the sum of zero elements" and is what makes the query formula uniform for all ranges including those starting at index 0.
Range query in O(1)sumRange(left, right) = prefix[right+1] − prefix[left]. The sentinel shifts the indexing so left is always subtracted, never special-cased.

Immutability as prerequisiteThe entire design depends on the array not changing. The prefix array is stored as let to communicate that it is computed once and thereafter read-only. A mutable array between queries would require a Fenwick tree.
