LC 362 — Design Hit Counter

Walkthrough Masterfile· Language: Swift· Medium · Sliding-window queue · Full interview simulation
Problem at a Glance
Difficulty
Medium
Pattern
Sliding-window queue — enqueue arrivals, evict expired front items
Tools
A single FIFO queue of timestamps
Key insight
The queue front always holds the oldest hit — dequeue while it is outside the 300-second window
Time / Space
O(n) amortized per call / O(w) where w = hits in the window
Theory
queue_master.html

Design a hit counter that supports two operations: hit(timestamp) records that a request arrived at a given second, and getHits(timestamp) returns how many requests have arrived in the last 300 seconds — that is, in the range [timestamp − 299, timestamp], inclusive. Timestamps are always positive integers and calls to hit and getHits are guaranteed to arrive in non-decreasing order.

OperationsOutputWhy
hit(1), hit(2), hit(3), getHits(4)3All three hits are within [4−299, 4] = [−295, 4]
hit(300), getHits(300)4Hits at 1, 2, 3, 300 — all within [1, 300]
getHits(301)3Hit at t=1 is now at 301−300=1 distance, just outside [2, 301] — evicted

The twist: the window is time-based, not index-based. You are not sliding over an array — you are maintaining a live count of events inside a moving 300-second interval. That means stale hits must be evicted lazily, at query time, using the timestamp of the current call.

2   Reading the Problem — insights from the statement

"Last 300 seconds": this is not a fixed-size index window — it is a time range that moves with each call. The window at getHits(301) is [2, 301], not [0, 300]. Any structure that indexes by position rather than by time needs extra bookkeeping; a queue of raw timestamps is the most natural fit.
"Non-decreasing order": hits always arrive at or after the previous timestamp. This is the key that makes the queue correct: the oldest element is always at the front. You never need to search or sort — just check the front and dequeue while it is expired.
"At most 300 hits per second": not stated explicitly, but the problem is framed as a rate-limiter. The real-world analogy is an API rate-limit check: "did this user exceed 300 requests in the last 5 minutes?" The queue models the sliding window of live requests.

"last 300 seconds"A time-based window — the expiry condition is timestamp − 300 >= front, not an index bound
"non-decreasing order"The queue front is always the oldest hit — no sorting, no search, O(1) eviction check
"at most 10⁴ calls"O(n) amortized is fine; each timestamp is enqueued once and dequeued at most once

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

The window moves with time, not with index. What structure holds items in arrival order and lets you remove the oldest efficiently?

Hint 2 — the pattern

Sliding-window queue. On each query, evict from the front while the front timestamp is older than currentTime − 299. The answer is then just the size of the queue.

Hint 3 — the structure

A plain [Int] with a head-index for O(1) dequeue, or a Deque. Enqueue on hit, evict-then-count on getHits. No extra accumulator needed — queue size is the count.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Are calls guaranteed to arrive in non-decreasing timestamp order — could a hit at t=5 arrive after a hit at t=10?"
A: Guaranteed non-decreasing. → The queue front is always the oldest hit. No sorting or random access needed.
Q: "Is 'last 300 seconds' inclusive on both ends — so getHits(301) includes a hit at t=2?"
A: Inclusive. The range is [timestamp−299, timestamp]. → The eviction condition is front < timestamp − 299, or equivalently front + 300 <= timestamp.
Q: "Can multiple hits arrive at the exact same timestamp?"
A: Yes. → Each call to hit enqueues independently — same timestamp appears multiple times.
Q: "Should I optimize for time or space? Could the window hold millions of hits?"
A: Optimize for time; window size is bounded by calls made. → O(1) amortized per operation is the target; space is O(hits in window), acceptable.
Step 2 — The plan, before any code
The plan, spoken

"This is a sliding-window queue problem. The signal is 'last 300 seconds' — a time-based window that moves with each call. The queue models the live window directly: every hit is a timestamp enqueued at the back, and stale hits are dequeued from the front at query time. That FIFO discipline is what makes the front always hold the oldest hit."

"Let me pin the condition before I touch state. The validity condition is: a timestamp is inside the window if it satisfies t >= currentTime − 299. The invalid condition is the mirror: a timestamp at the front is stale when front < currentTime − 299, or equivalently front + 300 <= currentTime. The eviction loop runs while that condition holds — dequeue from the front — and once it exits, every remaining element is in-window. The answer is then simply the queue's count."

"From those conditions, the state is minimal: one queue of Int timestamps. On hit, enqueue the timestamp — O(1) amortized. On getHits, evict from the front while the front is stale, then return count. Each timestamp is enqueued once and dequeued at most once, so the total work across all calls is O(n) — amortized O(1) per call."

"One thing worth noting: in Swift, Array.removeFirst() is O(n). I'll use a head-index approach — advance an index instead of shifting the array — to keep dequeue O(1). Let me code it and trace the example sequence to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: store all hits, count in range on each queryO(n) per getHitsCorrect but wasteful — counts stale hits that will never contribute again
Fixed-size circular buffer of 300 bucketsO(1) per operationValid optimization: bucket[t % 300] stores (timestamp, count). O(1) and O(300) space. Named in follow-up; queue is clearer for the first pass.
Sorted set / binary search for range countO(log n) per operationStrictly worse than the queue — non-decreasing order makes binary search redundant
Sliding-window queue + head-indexO(1) amortizedChosen. Non-decreasing order means front = oldest; FIFO eviction is O(1) amortized. Count is always queue.count − headIndex.
The deciding signal: "arrivals in non-decreasing order" + "count events in the last T seconds." Non-decreasing order is the invariant that makes the queue front always the oldest item — which is exactly what makes the eviction loop correct and amortized O(1).

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker
timestamps: [Int]Every hit timestamp in arrival orderArray gives O(1) amortized append and O(1) index read; with head-index pattern, dequeue is also O(1) — Array.removeFirst() would be O(n)
headIndex: IntThe logical front of the queueAdvancing an index instead of shifting the array keeps every eviction O(1); the live count is timestamps.count − headIndex
Swift cost note: Array.removeFirst() is O(n) — it shifts every remaining element. The head-index pattern avoids this. The array grows without bound over a session; if that matters, compact it periodically or switch to a true ring buffer.

Pseudocodesliding-window queue with head-index
HitCounter:
  timestamps ← []
  head ← 0

  hit(timestamp):
    timestamps.append(timestamp)

  getHits(timestamp):
    WHILE head < timestamps.count AND timestamps[head] + 300 <= timestamp:

    RETURN timestamps.count − head

9   Swift Solution — iterative

A.1 Sliding-window queue with head-index
O(1) amortizedO(w)
Plan
Enqueue every hit timestamp at the back. On getHits, advance the head index past any timestamp where front + 300 <= currentTime (expired). Return timestamps.count − headIndex — the live count — without any shifting. 
class HitCounter {
    private var timestamps: [Int] = []
    private var headIndex = 0

    func hit(_ timestamp: Int) {
        timestamps.append(timestamp)
    }

    func getHits(_ timestamp: Int) -> Int {
        // evict timestamps that have fallen outside the 300-second window
        while headIndex < timestamps.count,
              timestamps[headIndex] + 300 <= timestamp {
            headIndex += 1
        }
        return timestamps.count - headIndex
    }
}

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
Two things: first, the head-index approach lets the array grow unboundedly — compacting it periodically (replace timestamps with the live slice) recovers the memory at O(w) cost. Second, if the timestamp range is known to be bounded (say, always within a 24-hour window), a fixed-size circular buffer of 300 buckets is O(1) time and O(300) space — the bucket holds (lastTimestamp, count) and a stale bucket resets on write. That eliminates the growing array entirely.
Follow-up: "What if hits can arrive out of order — say due to network delay?"
The queue assumption breaks because the front is no longer always the oldest. You would need a sorted structure (a min-heap or a sorted deque) so eviction always removes the globally oldest timestamp. Hit becomes O(log w), getHits stays O(k + log w). If timestamps are bounded (e.g. at most 5 seconds of jitter), a small buffer sorted on flush is more practical than a full heap.
Follow-up: "What if there are multiple users — each needs their own 300-second window?"
Each user gets their own HitCounter instance, keyed in a dictionary by user ID. The per-user cost stays the same; the total space scales with active users × hits in their windows. This is the standard per-key rate-limiter shape used in API gateways.
Follow-up: "Can you do O(1) time and O(1) space for this specific 300-second window?"
Yes — circular buffer of 300 buckets, each bucket is a (timestamp, count) pair. On hit: bucket = timestamp % 300; if bucket.timestamp == timestamp increment count, else reset to (timestamp, 1). On getHits: sum all buckets where timestamp − bucket.timestamp < 300. Time O(300) = O(1), space O(300) = O(1). The trade-off is it aggregates hits per second rather than tracking individual timestamps — fine if sub-second granularity is not required.
