You are given an integer array height where each element represents the height of a vertical line drawn at that index. Two lines, together with the x-axis, form a container. Find the pair of lines that holds the maximum amount of water — the container's capacity is (right − left) × min(height[left], height[right]).

InputOutputWhy
height = [1,8,6,2,5,4,8,3,7]49Lines at index 1 (h=8) and index 8 (h=7): width 7 × min(8,7) = 49
height = [1,1]1Only one pair: width 1 × min(1,1) = 1

The twist is a proof obligation, not a data-structure question. The optimal algorithm is only two pointers and one running maximum — but knowing which pointer to move requires understanding why moving the taller wall is provably suboptimal. Without that proof, the steering rule feels like a trick. With it, the whole algorithm falls out from first principles.

2   Reading the Problem — insights from the statement

"Container with most water": the word "most" tells you this is a maximization problem over all valid pairs. You need to evaluate every meaningful pair — but not by brute force. The O(n²) check-all-pairs baseline must be beaten.
Area formula — width × min(height): the area is bounded by the shorter of the two walls — water spills over the shorter side. Width is the index distance right − left. This formula is the algorithm's engine: it makes the shorter wall the bottleneck, and that bottleneck is what the steering rule exploits.
"You may not slant the container": the container is always axis-aligned — you pick two vertical lines and water fills the rectangle between them at the height of the shorter. There is no diagonal geometry. This rules out any approach that treats the problem as 2D geometry rather than array indexing.
No sorting needed: unlike pair-sum problems (LC167, LC15), the array is not sorted and must not be sorted — sorting destroys the index positions that define the container width. The two-pointer technique applies here for a different reason: the steering rule proves we can safely discard one endpoint at each step, giving O(n) without needing order.
n ≥ 2 guaranteed: the constraints promise at least two elements, so the pointers always start with a valid pair and the answer is always a positive area. No empty-array guard is needed.

Equal-height tie-breaking. When height[left] == height[right], it does not matter which pointer moves — moving either is safe. Moving both simultaneously would skip valid pairs and is wrong. The standard implementation advances left (or retreats right) — pick one and be consistent.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

n up to 10⁵ rules out checking all O(n²) pairs. The array is not sorted. Think about starting with the widest possible container and asking: can we do better by narrowing it?

Hint 2 — the pattern

Converging two pointers. Start with left = 0 and right = n − 1. At each step you narrow the window by one. The question is: which end do you move?

Hint 3 — the steering rule

Consider what happens when you move the taller wall inward: width decreases by 1, and the new min height can only stay the same or decrease (because it's now bounded by the same shorter wall or something even shorter). The area can only get worse. So you must always move the shorter wall.

Step 2 — The plan, before any code
The plan, spoken

"This is a converging two-pointer problem — Variant 8.1. The signal is 'maximize over all pairs in an unsorted array of positions'. The O(n²) brute force evaluates every pair; I need to prune that search to O(n) by proving, at each step, that one endpoint can be permanently discarded."

"Before I write any code, let me state the condition that drives the pointer movement, because everything follows from it. The area formula is width × min(height[left], height[right]). The validity condition here is: 'is this pair the maximum so far?' I record at every step, inside the outer while loop. The steering condition — the one that decides which pointer moves — is: 'which wall is shorter?' That wall is the binding constraint on the container's height."

"Here is the proof that moving the taller wall is always wrong: suppose height[left] < height[right]. If I move right inward, the width decreases by 1. The new height of the container is min(height[left], height[right']) for some new right'. Since the min is capped by height[left] — the shorter side — and the width just shrank, the area is strictly no better than the current pair. I have gained nothing. But if I move left inward, I eliminate the known-short side and may encounter a taller one, which could increase the min height enough to overcome the reduced width. This is the only move with upward potential, so it is the only move we make."

"The invariant: after each step, every pair involving the eliminated endpoint has been provably dominated — the running maximum already holds the best of all pairs we will ever need that endpoint for. The while loop maintains this invariant. Each element is visited at most once."

"Data structures: two index variables leftPointer and rightPointer, one integer maxWater initialized to 0. No maps, no stacks, no auxiliary array."

"Time complexity: O(n) — each pointer moves at most n steps total. Space: O(1) — three scalar variables. I'll code it now and trace [1,8,6,2,5,4,8,3,7] → expected 49."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check every pairO(n²)10⁵ elements → 10¹⁰ pairs. Dead on arrival — but name it for the baseline and to show you know what you're beating.
Sort then two-pointer (sum-style)O(n log n)Breaks the algorithm: sorting destroys the index positions that define width. The area formula is (right − left), not a value sum.
Divide and conquerO(n log n)Correct but overcomplicated; the two-pointer argument makes the linear solution straightforward once the steering proof is stated.
Converging two pointers — min-side steeringO(n)Chosen. Proof shows moving the taller wall can never improve the area — so we safely eliminate one endpoint per step and traverse the whole array in one pass.
The deciding signal: "maximize over all pairs" + "pair property depends on both index distance and element values" + "unsorted array where index encodes width". Any candidate algorithm that changes the index ordering is immediately wrong.
Why this isn't the same as pair-sum two-pointer. In LC167 (sorted pair sum), the array is sorted and the steering rule is "move the side whose contribution to the sum is too small or too large." Here the array is unsorted and the rule comes from the min-height proof, not from sorted order. Both are Variant 8.1 in form; the proof justifying the rule is different.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
leftPointer: IntLeft boundary of current candidate pairDirect index into height. Starts at 0 — the leftmost position. Advances by 1 when height[left] <= height[right].
rightPointer: IntRight boundary of current candidate pairStarts at height.count − 1 — the rightmost position. Retreats by 1 when height[right] < height[left].
maxWater: IntBest area seen across all evaluated pairsInitialized to 0. Updated at every step before any pointer moves. Safe sentinel because all heights are ≥ 1 and n ≥ 2, so the first area is always ≥ 1.
Why no auxiliary storage. The steering proof guarantees we never need to revisit an eliminated endpoint. We do not need to remember which pairs we evaluated — the running max is the only state we carry forward. Compare to problems that require a stack (monotone next-greater-element) or a map (pair-with-given-difference): here the two-pointer invariant makes those structures unnecessary.

Pseudocodeconverging two pointers, min-side steering
leftPointer  ← 0
rightPointer ← height.count − 1
maxWater     ← 0

WHILE leftPointer < rightPointer:
    width  ← rightPointer − leftPointer
    minH   ← min(height[leftPointer], height[rightPointer])
    area   ← width × minH
    maxWater ← max(maxWater, area)

    IF height[leftPointer] <= height[rightPointer]:
        leftPointer  += 1    // left is the shorter (or equal) wall — advance it
    ELSE:
        rightPointer -= 1    // right is the shorter wall — retreat it

RETURN maxWater

9   Swift Solution — iterative

A.1 Converging two pointers — min-side steering
O(n)O(1)
Plan
Start with the widest possible container (outermost pointers). At each step: compute area, update the running max, then move the shorter wall inward. The proof: moving the taller wall guarantees a smaller or equal area (width shrinks, min height can only stay the same or fall). Repeat until pointers meet. 
func maxArea(_ height: [Int]) -> Int {
    var leftPointer  = 0
    var rightPointer = height.count - 1
    var maxWater     = 0

    while leftPointer < rightPointer {
        let width        = rightPointer - leftPointer
        let shorterWall  = min(height[leftPointer], height[rightPointer])
        maxWater = max(maxWater, width * shorterWall)

        // Advance the shorter wall — the only move that can improve area
        if height[leftPointer] <= height[rightPointer] {
            leftPointer  += 1
        } else {
            rightPointer -= 1
        }
    }

    return maxWater
}

12   Complexity

PhaseTimeSpaceWhy
Pointer walkO(n)O(1)At each iteration exactly one pointer moves — either left advances (+1) or right retreats (−1). They start n − 1 steps apart and converge. Total iterations ≤ n − 1. All operations inside the loop are O(1).
TotalO(n)O(1)Reading the input is the bottleneck. No auxiliary structure grows with n — three scalar variables regardless of input size.
Why we don't need to try all O(n²) pairs. The steering proof shows that at each step, every untried pair involving the eliminated endpoint is provably dominated by the current maximum or by pairs the algorithm will still visit. Formally: if we eliminate index l (because height[l] <= height[r]), then for any r' < r, the pair (l, r') has width < width of (l, r) and the same or smaller min height (bounded by height[l]). So area(l, r') <= area(l, r), and (l, r) already represents the best any pair involving l can achieve. We can drop l permanently.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The algorithm is already optimal at O(n) time and O(1) space — no asymptotic improvement is possible since you must read every element at least once. A minor constant-factor improvement: you can add an early skip inside the loop — after computing the area for a given shorter wall height, advance through all subsequent positions with the same or shorter height in a tight inner loop before recomputing. In practice the compiler's branch predictor handles this well, so it rarely matters unless the array is pathologically flat.
Follow-up: "Can you prove the steering rule is correct — not just intuitively, but formally?"
Yes. Suppose height[left] <= height[right]. For any right' < right, the pair (left, right') has width = right' − left < right − left and min height = min(height[left], height[right']). Since height[left] <= height[right], and height[left] remains the binding constraint regardless of height[right'], the area = (right' − left) × min(height[left], height[right']) <= (right' − left) × height[left] < (right − left) × height[left] <= area(left, right). Every pair with this left endpoint and any right' < right is strictly dominated. QED — we can eliminate left permanently.
Follow-up: "What if you need the pair of indices, not just the area?"
Track two additional variables alongside maxWater: bestLeft and bestRight. Whenever area > maxWater, update all three. Return (bestLeft, bestRight, maxWater). The same O(n) time, O(1) space — three extra scalar assignments per improvement step.
Follow-up: "What if the problem becomes 3D — three walls forming a box — maximize volume?"
The 2D two-pointer trick doesn't generalize cleanly to three dimensions because the steering rule's proof relies on the pair being fully determined by two endpoints on a 1D axis. In 3D you'd need to consider all triplets, and the equivalent of the steering proof doesn't exist for three walls in general. You'd likely fall back to O(n³) brute force or look for problem-specific structure (e.g., if two dimensions are fixed and only height varies).
Follow-up: "How does this differ from LC 42 — Trapping Rain Water?"
Both use two converging pointers with a min-side decision, but the quantity they compute is different. LC 11 computes the area of a single rectangle: one container between two chosen walls. LC 42 computes the total water trapped at every cell simultaneously: each position traps water up to min(leftMax, rightMax) − height[i]. LC 42 needs to maintain running left and right maxima across the whole traversal; LC 11 only needs the current pair's values. The steering rule in LC 42 comes from the same min-side invariant, but the accumulation logic is richer.
