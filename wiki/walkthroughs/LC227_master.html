LC 227 — Basic Calculator II

Given a string representing an arithmetic expression with non-negative integers and the four operators +, -, *, / (no parentheses, spaces allowed), evaluate and return the integer result. Division truncates toward zero.

InputOutputWhy
"3+2*2"7Multiplication first: 2*2=4, then 3+4=7
" 3/2 "1Integer division: 3÷2=1 (truncates toward zero)
" 3+5 / 2 "5Division first: 5÷2=2, then 3+2=5; spaces are ignored

The twist is precedence handling without parentheses. Addition and subtraction can be deferred — their operands are separated by other operations. Multiplication and division must bind their operands immediately. A single O(n) scan with a stack and one deferred-operator variable handles both cases cleanly.

2   Reading the Problem — insights from the statement

"No parentheses": this is the key constraint that makes the stack approach tractable. With parentheses, you need recursive descent or nested stack contexts. Without them, the entire expression can be handled in one scan with a single stack level and a single deferred operator.
"Non-negative integers": no leading minus signs to treat as unary negation, so the only negative values on the stack come from the − operator, handled by pushing -num. Multi-digit integers are accumulated digit by digit with num = num * 10 + digit.
"Spaces allowed": spaces are neither digits nor operators. They must be skipped — but the commit condition catches this correctly because a space satisfies neither branch. The character check !c.isNumber && c != " " handles it.
"Division truncates toward zero": Swift's integer / operator already truncates toward zero (not floor division). No special rounding logic is needed — the language semantics match the problem requirement.
Expression is guaranteed valid: the constraints promise a valid expression. No empty string, no double operators, no trailing operators — no input validation required.

3   Edge Case Inventory

InputExpectedHandled by
Single number "42"42Loop accumulates num=42; final flush at last index pushes +42; sum = 42
Leading/trailing spaces " 3 "3Spaces skipped by commit condition; final index triggers flush regardless

Division truncation "7/2"3Swift Int division: 7/2 = 3 (floor 3.5 toward zero)
Negative division result "-7/2"Not possibleInputs are non-negative integers; subtraction can produce negatives on stack but the dividend is always non-negative
GOTCHA: last number never gets committed by an operatorCorrect flush requiredCommit condition includes i == chars.count - 1 — the last character triggers a flush even if it's a digit, not an operator
Embedded spaces "3 + 5"8Spaces are not digits and not operators: they pass through both digit-accumulation and commit-check branches without action
The final flush is the hardest edge case. The last number in any expression has no operator after it. The commit must fire at the last character index regardless of what that character is — including when it's a digit. The condition i == chars.count - 1 is the flush; omitting it yields the wrong answer for every expression whose last token is a number (which is every valid expression).

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

There are no parentheses. The only precedence issue is that × and ÷ bind tighter than + and −. Think about how you would evaluate this manually, left to right. How do you defer an addition while waiting to see if there's a multiplication next?

Hint 2 — the pattern

Stack-based expression parse. Use a stack of operands. Carry a running number and remember the last operator seen. At each operator boundary, decide whether to defer (push for + and −) or apply immediately (× and ÷ modify the stack top).

Hint 3 — the final flush

When do you commit the last number to the stack? There's no operator after it. You need your commit to fire both at every operator AND at the last index of the string. The final sum of the stack is the answer — because all deferred additions become a linear sum.

The plan, spoken

"This is the stack-based expression parse — Variant 8.2 of the String Parsing pattern. The signal is 'evaluate an expression with mixed-precedence operators, no parentheses.' The key difficulty is that + and − can't be applied immediately — I don't know yet if the next token will be multiplied into their operand."

"Before I write any data structures, let me state the governing conditions. The token-type condition: each character is either a digit (extend the running number), a space (skip), or an operator-or-end-of-string (boundary — commit the current number using the last-seen operator). The deferred-vs-immediate condition: for + and −, push the signed number onto the stack because I can always sum the stack at the end to combine all deferred additions. For × and ÷, apply immediately to the stack top — that operand must be bound now before any other operation can separate it."

"The data structures follow: a running num accumulator (reset to 0 after each commit), a lastOp character initialized to '+' (treating the first number as if preceded by a '+'), and an operand stack. The sum of the stack at the end is the answer."

"The commit fires in two situations: when the current character is an operator, and when the current index is the last index of the string. Folding both into one condition avoids a separate post-loop flush block. Spaces are neither digits nor operators so they pass through cleanly without action."

"Time is O(n) — one pass over the string, O(1) per character. Space is O(n) in the worst case for the stack — an expression of all additions pushes every operand. I'll trace \"3+2*2\" → expected 7 after coding."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute-force: split on operators, evaluate in two passesO(n)Correct but fragile with spaces and multi-digit numbers — parsing the split tokens cleanly requires as much work as the stack approach with none of its elegance.
Recursive descent parserO(n)Correct and generalizes to parentheses (LC 224), but overkill here. Without parentheses there is only one precedence level of deferral, and a single stack plus a deferred operator is simpler.
Convert to Reverse Polish Notation then evaluateO(n)Two-pass: shunting-yard to RPN, then RPN evaluation. Works, but two passes and two stacks when one suffices.
Single-pass stack + deferred lastOpO(n) / O(n)Chosen. One scan; one stack; one deferred variable. The "last index flush" trick folds the post-loop commit into the loop. Clear, minimal, and directly extensible to parentheses if asked.
The deciding signal: "evaluate expression, no parentheses, mixed precedence." This exact shape — no nesting, binary operators only — is the canonical deferred-operator stack problem. The minute you see parentheses you upgrade to recursive descent or a stack-of-contexts.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
stack: [Int]Deferred operands waiting to be summedAn array with append and removeLast — both O(1) amortized. A simple Int accumulator would lose track of which operands are still pending. A deque is unnecessary (only the top is touched for × and ÷).
num: IntRunning accumulation of the current multi-digit numberRebuilt digit by digit with num = num * 10 + digit. Reset to 0 on each commit. A String buffer would require an Int conversion at each commit — the arithmetic accumulation is one instruction per digit.
lastOp: CharacterThe most recently seen operator, initially '+'Initialized to '+' so the first number is treated as positively pushed — no special case for the very first token. A separate first-token flag would double the branch count.
chars: [Character]Random-access view of the input stringConverting to [Character] first gives O(1) index access for the last-index check (i == chars.count - 1). Iterating a Swift String directly doesn't give cheap index arithmetic.
Why lastOp = '+' at initialization. The first number has no operator before it. By pretending the expression is prefixed with an implicit +, the commit logic for the first number follows the same branch as all subsequent additions — push +num. This eliminates a special-case branch for the first token and makes the loop uniform from the very first character.

8   Pseudocode Dry Run

Pseudocodesingle-pass stack parse with deferred operator
chars ← Array(s)
num ← 0,  lastOp ← '+',  stack ← []

FOR i FROM 0 TO chars.length − 1:
    c ← chars[i]
    IF c is a digit: num ← num * 10 + digit(c)

    IF (c is an operator) OR (i == last index):
        SWITCH lastOp:
            '+': stack.push(num)
            '−': stack.push(−num)
            '×': stack.push(stack.pop() × num)
            '÷': stack.push(stack.pop() ÷ num)   // truncates toward zero
        lastOp ← c
        num ← 0

RETURN sum(stack)

9   Swift Solution — iterative

A.1 Single-pass stack parse — deferred operator trick
O(n)O(n)
Plan
Scan the expression left to right with three variables: a running number accumulator, a deferred lastOp character, and an operand stack. At each operator or the final index, apply lastOp to commit the current number — push for +/−, modify the stack top for ×/÷. Sum the stack for the result. 
func calculate(_ s: String) -> Int {
    let chars = Array(s)
    var stack: [Int] = []
    var num = 0
    var lastOp: Character = "+"

    for i in 0..<chars.count {
        let c = chars[i]
        if let d = c.wholeNumberValue {
            num = num * 10 + d
        }
        // Commit on an operator, or at the final character (the flush).
        if (!c.isNumber && c != " ") || i == chars.count - 1 {
            switch lastOp {
            case "+": stack.append(num)
            case "-": stack.append(-num)
            case "*": stack.append(stack.removeLast() * num)
            case "/": stack.append(stack.removeLast() / num) // truncates toward zero
            default: break
            }
            lastOp = c
            num = 0
        }
    }
    return stack.reduce(0, +)
}

12   Complexity

PhaseTimeSpaceWhy
Character scanO(n)O(1) work per charOne pass over the n-character string. Each character triggers at most one stack push or pop (O(1) amortized on Swift Array). Digit accumulation is O(1) per digit.
Stack—O(n)Worst case: all operators are + or −, so every operand is pushed and never popped until the final reduce. The stack holds at most ⌊n/2⌋ operands.
Final sumO(k)O(1)k = stack size ≤ n/2. reduce is a single linear pass.
TotalO(n)O(n)The character scan dominates time. The stack dominates space. Both are linear in the expression length. String building with += inside the loop would make this O(n²) — avoided by the digit accumulation approach.
Why the final reduce doesn't change the O(n) bound. The stack holds at most O(n) elements, so reduce is O(n). It does not add a second pass to the asymptotic cost — the overall algorithm is already O(n) for the scan, and the reduce folds into that bound.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The stack can be replaced by a single running sum and a pending multiplier/divisor, bringing space from O(n) to O(1). Maintain runningTotal and lastTerm: on + or −, add lastTerm to runningTotal and update lastTerm = ±num; on × or ÷, update lastTerm = lastTerm × num (or ÷). At the end, return runningTotal + lastTerm. This is the O(1)-space optimization worth mentioning once the stack approach is accepted.
Follow-up: "Now add support for parentheses — how does your approach extend to LC 224?"
When you encounter (, push the current runningTotal and lastOp onto a context stack and reset for the sub-expression. When you encounter ), complete the sub-expression, then apply the saved lastOp to combine the sub-result with the saved context. The stack you used for operands becomes a stack of saved contexts. Time stays O(n); space stays O(n) but the stack now holds (total, lastOp) pairs instead of just integers.
Follow-up: "What if the numbers can be very large — say 10⁵⁰?"
Swift's Int overflows at 2⁶³−1. For arbitrary-precision integers, swap the stack to hold BigInt (or use a string-arithmetic library). The algorithm is identical — only the numeric type changes. Digit accumulation would then be num = num * BigInt(10) + BigInt(digit). Mention that the LC227 constraints keep values within Int range, so this is only a concern if constraints change.
Follow-up: "Can you handle exponentiation — ^ operator with right-to-left associativity?"
Right-associative operators need a different strategy: you must defer them until you know the right operand. The standard approach is Pratt parsing (top-down operator precedence), which assigns each operator a left-binding power and right-binding power. For left-associative operators (all four in LC227), left- and right-binding power are equal. For right-associative ^, the right-binding power is slightly higher, forcing the parser to collect the right operand before applying. The single-deferred-operator trick no longer suffices — you'd need the full Pratt framework.
