Theory
string_parsing_deep_dive_master.html

You are given a string that encodes a nested list of integers — for example "[123,[456,[789]]]" — and you must parse it into the corresponding nested structure. Each element is either a plain integer or another nested list. The string is well-formed: brackets are balanced, integers may carry a leading minus sign, and commas separate siblings within a list. Return the parsed object: a NestedInteger that is either a single integer or an ordered list of NestedIntegers.

InputOutput (structure)Why
"324"integer 324No brackets at all — the whole input is one bare integer, not a list
"[123,[456,[789]]]"list [123, [456, [789]]]Outer list of two elements; the second is itself a list nesting one more level
"[]"empty list []Open immediately followed by close — a container with no children

The twist is that the input is a recursive structure delivered as a flat string. You could parse it recursively, but the clean iterative answer is a single left-to-right scan driven by an explicit stack of partially-built containers — every character either opens a level, closes a level, accumulates a digit, or marks a boundary.

2   Reading the Problem — insights from the constraints

"A nested list of integers": the result type is recursive — a NestedInteger is either a single integer or a list of NestedIntegers. That shape dictates the data structures: a stack whose elements are containers, where opening a bracket pushes a new container and closing one attaches the finished container to its parent.
"May be a bare integer with no brackets": "324" and "-12" are valid inputs with no list at all. This is the first special case — if the string does not start with [, the entire string is a single integer and there is no container to build. Handle it before the scan.
Integers can be negative: a leading - precedes the digits. The number accumulator must track a sign separately (or detect the - and negate when committing), because num = num * 10 + digit only handles magnitude.
Commas and close-brackets are both boundaries: a number being accumulated is finished either by a , (more siblings follow) or by a ] (the list ends). Both must flush the running number into the top container — but only if a number was actually being accumulated, so [] and [[]] don't attach a spurious zero.
The structure is well-formed: balanced brackets, valid syntax. So you never validate — no unmatched-bracket guard, no malformed-input branch. Every ] has a matching [ already on the stack, which is what makes the pop-and-attach safe.

"[123]" — single elementlist [123]The closing ] flushes the accumulated 123 before popping the container.

GOTCHA: "[123,[456]]" — number right before ]list [123, [456]]The inner ] must flush 456 before popping, or the last number of every list is lost.
GOTCHA: "[-1]" — negative inside listlist [-1]The - sets the sign flag; the flush negates the magnitude. Forgetting the flag yields +1.
"Flush only if a digit was seen" is the quiet correctness rule. Both , and ] trigger a flush, but firing it unconditionally attaches a phantom 0 to empty lists and double-counts after a just-popped sub-list (e.g. "[[1],2]" — the ] after 1 pops, then the , must not flush). Guard the flush with a "currently accumulating a number" flag.

4   Try It Yourself First

Attempt this cold — it is the canonical nested-parse drill. Open hints one at a time.

Hint 1 — the constraint

The input encodes a recursive structure (lists inside lists) as a flat string. First handle the degenerate case: if the string doesn't begin with [, it's a single integer — return it directly. Everything else is a list to parse.

Hint 2 — the pattern

Nested-structure stack parse. Keep a stack of containers; the top is always the list you're currently filling. Scan left to right: [ pushes a new container, a digit accumulates into a running number, a boundary (, or ]) flushes that number into the top container, and ] additionally pops the top and attaches it to its parent.

Hint 3 — the two bugs everyone hits

(1) On a closing ], flush the running number before you pop — otherwise the last element of each list disappears. (2) Only flush when you actually accumulated a digit; track that with a flag, or empty lists get a phantom 0 and post-sub-list commas double-count. Negatives: when you see -, set a sign flag and apply it at flush time.

Step 1 — Clarifying questions
Q: "Is the input always well-formed — balanced brackets, valid syntax, no trailing garbage?"
A: Yes, always valid. → No validation; every ] is guaranteed a matching container on the stack, so pop-and-attach is always safe.
Q: "Can the whole input be a single integer with no brackets, like "42"?"
A: Yes. → I branch on the first character: if it's not [, return the integer directly without entering the scan.
Q: "Can integers be negative, and is there any whitespace between tokens?"
A: Negatives yes (leading -), no whitespace. → I carry a sign flag with the number accumulator; no whitespace means every non-digit, non-sign char is a structural token.
Q: "What's the return type — is there a NestedInteger API with add and setInteger?"
A: Yes, NestedInteger supports holding an Int or a list you append to. → My stack holds NestedInteger containers; closing a bracket appends the finished one to its parent via add.
Step 2 — The plan, before any code
The plan, spoken

"This is a nested-structure stack parse — Variant 8.3 of the String Parsing family. The signal is 'a flat string encoding a recursive container, parse it into the structure.' I do it in one left-to-right scan with an explicit stack, no recursion."

"Let me pin the governing conditions before the data structures. The invariant is: the top of the stack is always the container I am currently filling. Opening a bracket descends a level — push a new empty container; closing a bracket ascends — pop the finished container and attach it to the new top, its parent. The boundary condition is when a number ends: that happens on either a comma or a close-bracket, and at that moment I flush the running number into the top container — but the flush fires only if I was actually accumulating a digit, which is the rule that keeps empty lists and post-sub-list commas clean. The transition out of the scan: the very first character decides everything — if it isn't [, the whole input is a bare integer and I return immediately."

"The structures follow from those conditions. I need a stack of NestedInteger containers because the invariant is about 'the container I'm currently filling' and nesting is last-in-first-out. I need an Int number accumulator built with num = num*10 + digit, a Bool sign flag for the leading minus, and a Bool hasDigit flag so the flush is conditional. That's it — no recursion stack, because the explicit stack is the recursion made visible."

"The walk: convert to a character array for O(1) indexing. If the first char isn't [, return the integer. Otherwise scan: [ → push a new container; - → set sign; digit → accumulate; , → flush-if-accumulating, reset; ] → flush-if-accumulating, reset, then pop the top and (if the stack is non-empty) attach it to the new top, else it's the result. Time O(n), space O(n) for the stack and the output."

6   Why This Approach — and not the others

CandidateCostVerdict
Recursive descent — a parse function that calls itself on each [O(n) time, O(depth) call stackCorrect and natural for a recursive grammar, but the call stack can blow on deep nesting, and the index must be passed by reference or returned — fiddly in Swift. The iterative stack is the editorial-preferred form and the one this block teaches.
Strip outer brackets, split on top-level commas, recurse on each pieceO(n·depth) — repeated scanning to find top-level commasSplitting on top-level commas requires tracking bracket depth anyway, so you've done the hard part and then added re-scanning on top. Strictly worse than one pass.
Single scan + explicit stack of containersO(n) / O(n)Chosen. One pass, no recursion-depth risk, the stack makes the nesting explicit and the four character cases are a clean switch.
The deciding signal: a flat string encoding a balanced, nested structure. Whenever brackets nest and you must rebuild the tree, an explicit stack of partially-built nodes is the iterative counterpart of recursive descent — [ pushes, ] pops-and-attaches. Same skeleton parses JSON-ish lists, nested expressions, and directory trees.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
stack: [NestedInteger]The chain of containers from the root down to the one currently being filledNesting is last-in-first-out: the most recently opened bracket is the first to close. A stack models exactly that. Its top is the active container; popping attaches a finished child to its parent.
numberValue: IntThe integer currently being accumulated, digit by digitBuilt with numberValue = numberValue * 10 + digit as each digit arrives — the standard running-number trick. Reset to 0 after every flush.
isNegative: BoolWhether the current number had a leading minusThe accumulator only builds magnitude; the sign is tracked separately and applied at flush. Cheaper and clearer than re-parsing a substring.
hasDigit: BoolWhether a digit has been seen since the last resetGuards the flush so boundaries that follow a ] or open an empty list [] don't attach a phantom 0. The single most important flag for correctness.
characters: [Character]The input string as a random-access arraySwift String indexing is O(n); materializing once gives O(1) access for the single forward scan.
Why a stack and not recursion. The grammar is recursive, so recursion is the "obvious" tool — but the explicit stack makes the descent/ascent visible, removes the call-stack depth limit, and keeps the index a plain loop variable instead of a value threaded through recursive calls. The stack is the recursion, turned inside out.

8   Pseudocode Dry Run

Pseudocodesingle scan with a stack of containers
IF first char ≠ '[':
    RETURN integer(parse whole string)   // bare integer

stack ← empty
num ← 0, neg ← false, hasDigit ← false

FOR each ch IN string:
    SWITCH ch:
        '[':  push new empty container onto stack
        '-':  neg ← true
        digit: num ← num*10 + value(ch); hasDigit ← true
        ',':  IF hasDigit: attach (neg ? −num : num) to top; reset num,neg,hasDigit
        ']':  IF hasDigit: attach (neg ? −num : num) to top; reset num,neg,hasDigit
              finished ← pop stack
              IF stack not empty: attach finished to new top
              ELSE: result ← finished

RETURN result

9   Swift Solution — iterative

LeetCode supplies the NestedInteger class; the relevant API is init() (empty list), init(_ value: Int), setInteger(value:), and add(elem:). The solution below uses it directly.

A.1 Single scan with an explicit container stack
O(n)O(n)
Plan
Bare integer? Return it. Otherwise scan once: [ pushes a new container, digits accumulate (with a sign flag), and a comma or close-bracket flushes the accumulated number into the top container — but only if a digit was seen. A close-bracket also pops the finished container and attaches it to its parent, or returns it as the result when the stack empties. 
func deserialize(_ input: String) -> NestedInteger {
    let characters = Array(input)

    // Bare integer — no list to build.
    if characters.first != "[" {
        return NestedInteger(Int(input)!)
    }

    var stack: [NestedInteger] = []
    var numberValue = 0
    var isNegative = false
    var hasDigit = false
    var result = NestedInteger()

    // Invariant: stack.last is the container currently being filled.
    func flushNumberIfNeeded() {
        if hasDigit {
            let value = isNegative ? -numberValue : numberValue
            stack.last?.add(elem: NestedInteger(value))
            numberValue = 0
            isNegative = false
            hasDigit = false
        }
    }

    for character in characters {
        switch character {
        case "[":
            stack.append(NestedInteger())          // descend: new empty container
        case "-":
            isNegative = true
        case "0"..."9":
            numberValue = numberValue * 10 + character.wholeNumberValue!
            hasDigit = true
        case ",":
            flushNumberIfNeeded()
        case "]":
            flushNumberIfNeeded()                    // flush BEFORE popping
            let finished = stack.removeLast()       // ascend
            if let parent = stack.last {
                parent.add(elem: finished)            // attach to parent
            } else {
                result = finished                    // root list closed
            }
        default:
            break
        }
    }

    return result
}

12   Complexity

PhaseTimeSpaceWhy
Array materializationO(n)O(n)Array(input) copies the string into a character array once for O(1) indexing.
Single scanO(n)O(d)Each character is handled by one switch branch in O(1). The stack holds at most d containers, where d is the maximum nesting depth.
TotalO(n)O(n)One linear pass dominates. Space is O(n) overall because the output structure itself holds all n′ integers, which is irreducible — the parsed tree must exist.
Why one pass is enough. Every character is consumed exactly once and triggers a constant-time action: push, accumulate, or flush-and-maybe-pop. There is no backtracking and no re-scanning, because the stack remembers all the partially-built ancestor containers — the information a recursive parser would keep on the call stack lives explicitly here instead.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The algorithm is already O(n) single-pass, which is optimal. The main robustness concern is the two force-unwraps: Int(input)! on the bare-integer branch and character.wholeNumberValue! on the digit branch. Both are safe given the well-formed-input guarantee, but in production I'd validate or use guarded optionals so malformed input fails gracefully rather than crashing. I'd also reserveCapacity on the stack if a depth bound is known.
Follow-up: "Write the recursive version instead — does it have any drawback here?"
Recursive descent reads cleanly: a parse(from index) that, on seeing [, loops calling itself for each element until the matching ]. The drawback is the call-stack depth equals the nesting depth, so a pathological input like "[[[[…]]]]" with thousands of levels risks a stack overflow. The explicit-stack iterative form moves that depth onto the heap and has no recursion limit — which is why it's the preferred answer.
Follow-up: "Now serialize a NestedInteger back into this string format — the inverse."
That's a recursive (or stack-based) traversal: if the node is an integer, emit its digits; if it's a list, emit [, recurse on each child joining with ,, then ]. It's the mirror of parsing — the same tree, walked instead of built. Together they form a round-trip you can test by asserting serialize(deserialize(s)) == s on well-formed inputs.
Follow-up: "Once parsed, flatten it with an iterator that returns the integers in order — like LC 341."
That is exactly LC 341 — Flatten Nested List Iterator: take the parsed NestedInteger and expose next()/hasNext() with lazy stack-based descent. Parsing builds the tree; the iterator walks it on demand. The two problems compose directly — parse with this solution, then feed the result into LC 341's iterator.
