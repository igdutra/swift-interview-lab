Theory

You are given an array of distinct words. Return every pair of indices (i, j) with i ≠ j such that the concatenation words[i] + words[j] is a palindrome (reads the same forwards and backwards). The order matters: (i, j) and (j, i) are different pairs, and both should be reported when both concatenations are palindromes.

InputOutput (pairs)Why
["abcd","dcba","lls","s","sssll"][[0,1],[1,0],[3,2],[2,4]]"abcd"+"dcba"="abcddcba"; "s"+"lls"="slls"; "lls"+"sssll"="llssssll" — all palindromes
["bat","tab","cat"][[0,1],[1,0]]"bat"+"tab"="battab" and "tab"+"bat"="tabbat"; "cat" pairs with nothing
["a",""][[0,1],[1,0]]The empty string pairs with any palindrome ("a") in both orders — "a"+"" and ""+"a"

The twist is the size: up to ~5·10⁴ words. The brute-force "try every ordered pair and test the concatenation" is O(w²·L) and times out. The parsing insight rescues it — index every word's reverse in a hash map, then for each word check its L+1 split points, turning the search for a partner into an O(L) lookup. That drops the whole algorithm to O(w·L²).

2   Reading the Problem — insights from the constraints

"words[i] + words[j] is a palindrome": for a concatenation to be a palindrome, one word must "answer" the other's exposed end. If words[i] is longer, its extra prefix must itself be a palindrome and the rest must be reverse(words[j]). This decomposition into "a palindromic part + a reversible part" is the entire idea.
Up to ~5·10⁴ words — brute force is out: O(w²) pair tests will not pass. You must make "find a partner for this word" cheaper than scanning every other word. A hash map of reversed words turns that scan into a constant-ish lookup per split point.
The three matching cases. For a word W paired with some X: (1) exact reverse — X = reverse(W), so W+X is a palindrome; (2) palindromic prefix — if W = [palindrome][rest], then reverse(rest) + W is a palindrome, so look up reverse(rest); (3) palindromic suffix — if W = [rest][palindrome], then W + reverse(rest) is a palindrome, so look up reverse(rest).
"Distinct words": no duplicates, so the reversed-word map has unique keys — a plain [String: Int] suffices, no list of indices per key. The only self-collision to guard is matching a word with itself (a found index equal to the current index).
The empty string is a word: "" is a palindrome and pairs with every palindromic word in both orders. It falls out of the split-point logic if you scan split points 0…L inclusive (allowing an empty side), but it's the classic trap — many solutions miss (palindrome, "") by stopping at L−1.

The two empty-string traps are the difference between AC and WA. First: split points must allow an empty side so (palindrome, "") pairs are found. Second: because an empty remainder is checkable as both a "prefix" and a "suffix" split, you must exclude the empty remainder from one of the two cases (commonly: skip it in the suffix case) or every exact-reverse pair is emitted twice.

4   Try It Yourself First

This is the block's hardest problem. Attempt it cold; open hints one at a time.

Hint 1 — the constraint

With tens of thousands of words, testing all ordered pairs (O(w²·L)) is too slow. You need a way to ask "is there a word that completes this word into a palindrome?" without scanning every other word. A hash map keyed on something derived from each word is the move.

Hint 2 — what to put in the map

Build a map from each word's reverse to its index. Then, for a word W, if you split it as left + right, a partner exists when one side is itself a palindrome and the reverse of the other side is a key in the map. Looking up reverse(other side) is O(L); doing it for all L+1 splits is O(L²) per word.

Hint 3 — the three cases, precisely

For each split W = left + right: (a) if left is a palindrome and reverse(right) is in the map at index k ≠ i, then words[k] + W is a palindrome → emit (k, i); (b) if right is a palindrome and reverse(left) is in the map at index k ≠ i, then W + words[k] is a palindrome → emit (i, k). The exact-reverse pairing is just the split where one side is empty.

Hint 4 — the empty-string traps

Let split positions run from 0 to L inclusive so an empty side is allowed (that's how (palindrome, "") is found). And to avoid reporting exact-reverse pairs twice, skip the empty-remainder check in exactly one of the two cases — e.g. in case (b) require right non-empty. Verify with ["abc","cba"]: you must get [[0,1],[1,0]], not four entries.

The plan, spoken

"This is reversed-prefix / palindrome matching — Variant 8.4 of the String Parsing family. The signal is 'find pairs of words whose concatenation is a palindrome, at a scale where O(w²) is too slow.' The parsing insight is to index reversed words in a hash map so each word finds its complement by lookup."

"Let me pin the conditions before structures. The governing condition is the palindrome decomposition: words[i] + words[j] is a palindrome exactly when, splitting the longer word into left + right, one side is itself a palindrome and the other side is the reverse of the partner. That yields two checkable cases per split point of each word — a palindromic prefix means a partner attaches in front, a palindromic suffix means a partner attaches behind. The invariant I maintain over split points is that I never emit a self-pair (found index ≠ current index) and never emit the same pair twice (I exclude the empty remainder from one of the two cases). The stop condition per word is simply scanning all L+1 split points."

"The structures follow. I need a hash map [String: Int] from each word's reverse to its index — that's what makes 'is there a partner?' an O(L) lookup instead of an O(w) scan. I need a helper isPalindrome(of:) on a substring. And I need the result array of index pairs. No tries here — a reversed-word hash map is enough at this scale and far simpler to get right."

"The walk: build the reversed-word map. For each word at index i, for each split point p from 0 to L: let left = word[0..<p], right = word[p...]. Case A — if left is a palindrome and map[reverse(right)] exists at k ≠ i, emit (k, i). Case B — if right is non-empty and a palindrome and map[reverse(left)] exists at k ≠ i, emit (i, k). The non-empty guard in Case B prevents double-counting exact reverses. Time O(w·L²) — w words, L split points, each with an O(L) palindrome check and reversal; space O(w·L) for the map."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force — test every ordered pair's concatenationO(w²·L)Times out. With ~5·10⁴ words, w² is 2.5·10⁹ pair tests before the per-pair palindrome cost. Correct but far too slow.
Trie of reversed words, walk each word down itO(w·L²)Optimal asymptotics and elegant, but a palindrome-aware trie (storing, at each node, which words have a palindromic remainder below) is intricate and bug-prone under interview time. Same big-O as the hash-map form.
Hash map of reversed words + per-word split checksO(w·L²) / O(w·L)Chosen. Same complexity as the trie, far simpler to implement correctly. The two split cases and the two empty-string guards are the whole subtlety.
The deciding signal: "pairs whose concatenation is a palindrome" at a scale that kills O(w²). The reflex is to precompute reversed words into a lookup so each word's partner is found by hashing rather than scanning — and to decompose the palindrome condition into "a palindromic part fixed to the word + a reversible part that must be the partner." That decomposition is the reusable idea.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
reversedIndex: [String: Int]Each word's reverse → that word's indexTurns "find a word equal to reverse(remainder)" into an O(L) hash lookup instead of an O(w) scan. Words are distinct, so each reversed key is unique — a single Int value suffices, no [Int] per key.
characters per word (via Array(word))The word as a random-access array for slicing and palindrome checksSwift String slicing and indexing is O(n) per access; an array gives O(1) indexing so the inner palindrome check is genuinely O(L).
result: [[Int]]The emitted ordered index pairsThe required output shape. Appended to as each valid pair is discovered.
helper isPalindrome(_:from:to:)(no storage) two-pointer check on a sub-rangeChecking a sub-range in place avoids allocating substrings just to test them — keeps the constant factor down across the L checks per word.
Why a hash map beats a trie here. Both are O(w·L²). The trie is the "textbook optimal" structure, but encoding "does the remaining suffix below this node form a palindrome?" requires augmenting every node and is a classic source of off-by-one bugs. The reversed-word hash map achieves the same complexity with a data structure you can write correctly in two minutes — the right engineering trade for an interview.

9   Swift Solution — iterative

A.1 Two-map solution — Understand this first
O(w·L²)O(w·L)
Plan
Build two maps: leftWordToRightIndex (key = string to prepend, value = list of word indexes that need it) and rightWordToLeftIndex (key = string to append, value = list of word indexes that need it). For each word, scan all L+1 split points. When a side is a palindrome, the reverse of the other side is the key. Duplicates are handled by a Set on the output — no guard needed on the split loop itself. This is the solution arrived at by solving the problem from scratch. 
func palindromePairs(_ words: [String]) -> [[Int]] {
    func isPalindrome(_ characters: [Character]) -> Bool {
        var left = 0, right = characters.count - 1
        while left <= right {
            if characters[left] != characters[right] { return false }
            left += 1
            right -= 1
        }
        return true
    }

    // leftWordToRightIndex[key] = [i]: word[i] needs `key` prepended to form a palindrome
    // rightWordToLeftIndex[key] = [i]: word[i] needs `key` appended to form a palindrome
    var leftWordToRightIndex: [String: [Int]] = [:]
    var rightWordToLeftIndex: [String: [Int]] = [:]

    for (wordIndex, word) in words.enumerated() {
        let wordArray = Array(word)
        for splitPoint in 0...wordArray.count {
            let leftSplit = Array(wordArray.prefix(splitPoint))
            let rightSplit = Array(wordArray.suffix(wordArray.count - splitPoint))

            // if left is a palindrome, we need reverse(right) to appear as a word to our LEFT
            if isPalindrome(leftSplit) {
                let targetLeft = String(rightSplit.reversed())
                if word != targetLeft {
                    leftWordToRightIndex[targetLeft, default: []].append(wordIndex)
                }
            }

            // if right is a palindrome, we need reverse(left) to appear as a word to our RIGHT
            if isPalindrome(rightSplit) {
                let targetRight = String(leftSplit.reversed())
                if targetRight != word {
                    rightWordToLeftIndex[targetRight, default: []].append(wordIndex)
                }
            }
        }
    }

    var outputIndexes: Set<[Int]> = [] // Set handles duplicates from overlapping split paths
    for (currentIndex, word) in words.enumerated() {
        for rightIndex in leftWordToRightIndex[word, default: []] where rightIndex != currentIndex {
            outputIndexes.insert([currentIndex, rightIndex])
        }
        for leftIndex in rightWordToLeftIndex[word, default: []] where leftIndex != currentIndex {
            outputIndexes.insert([leftIndex, currentIndex])
        }
    }
    return Array(outputIndexes)
}
Why [String: [Int]] and not [String: Int]: The same key can be produced by multiple split points of different words. For example, if two different words both have an empty palindromic prefix at split 0, they both store reverse(word) in leftWordToRightIndex. A single Int value would silently overwrite the first entry — the bug that breaks ["a","abc","aba",""]. A [Int] per key stores all of them.
A.2 Single-map space-optimized variant
O(w·L²)O(w·L)
Plan
Collapses both maps into one [String: Int] by exploiting the fact that words are distinct — each reversed key is unique, so a single Int per key suffices. Uses a non-empty suffix guard (p < length) to prevent double-counting exact reverses instead of a Set. This is the cleaner production form; understand A.1 first.
Why this works with [String: Int] when A.1 needs [String: [Int]]: Words are distinct, so each word's reverse is unique in the map — no two words share the same lookup key. The double-count problem (exact reverses found twice) is handled by the p < length guard instead of a Set. A.1 generalizes to duplicate words; A.2 does not.
func palindromePairs(_ words: [String]) -> [[Int]] {
    var reversedIndex: [String: Int] = [:]
    for (wordIndex, word) in words.enumerated() {
        reversedIndex[String(word.reversed())] = wordIndex
    }

    func isPalindrome(_ characters: [Character], _ leftBound: Int, _ rightBound: Int) -> Bool {
        var leftPointer = leftBound, rightPointer = rightBound
        while leftPointer < rightPointer {
            if characters[leftPointer] != characters[rightPointer] { return false }
            leftPointer += 1; rightPointer -= 1
        }
        return true
    }

    var result: [[Int]] = []

    for (wordIndex, word) in words.enumerated() {
        let characters = Array(word)
        let wordLength = characters.count

        // splitPoint: left = characters[0..<splitPoint], right = characters[splitPoint..<wordLength]
        for splitPoint in 0...wordLength {
            // Case A: prefix [0..<splitPoint] is a palindrome → partner attaches in FRONT
            if isPalindrome(characters, 0, splitPoint - 1) {
                let rightRemainder = String(characters[splitPoint..<wordLength])
                if let partnerIndex = reversedIndex[rightRemainder], partnerIndex != wordIndex {
                    result.append([partnerIndex, wordIndex])
                }
            }
            // Case B: suffix [splitPoint..<wordLength] is a NON-EMPTY palindrome → partner attaches BEHIND
            if splitPoint < wordLength, isPalindrome(characters, splitPoint, wordLength - 1) {
                let leftRemainder = String(characters[0..<splitPoint])
                if let partnerIndex = reversedIndex[leftRemainder], partnerIndex != wordIndex {
                    result.append([wordIndex, partnerIndex])
                }
            }
        }
    }

    return result
}
On the two lookups. Case A looks up rightRemainder directly (not its reverse) because the map is already keyed on reversed words — reversedIndex[rightRemainder] finds the word whose reverse equals rightRemainder. Same logic in Case B for leftRemainder. The splitPoint < wordLength guard in Case B is the non-empty-suffix rule that stops exact reverses being emitted twice.

12   Complexity

PhaseTimeSpaceWhy
Build reversed-word mapO(w·L)O(w·L)Reverse and hash each of w words of length up to L. The map stores all w reversed strings.
Per-word split scanO(w·L²)O(L)For each word: L+1 split points, each doing an O(L) palindrome check plus an O(L) substring build for the lookup key.
TotalO(w·L²)O(w·L)The split scan dominates. This beats brute force's O(w²·L) whenever L ≪ w — exactly the given regime (many short words). The map is the irreducible space.
Why L² and not L per word. There are L+1 split points, and at each one the palindrome check and the substring extraction for the hash key are each O(L). Multiplying gives O(L²) per word, O(w·L²) overall. A trie reaches the same bound by amortizing the character walks; the hash-map form pays it explicitly but is far simpler to implement without bugs.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
Two constant-factor wins. First, building String(chars[…]) for every lookup key allocates per split point; I could hash sub-ranges or cache reversed substrings to cut allocations. Second, I'd precompute, per word, which prefixes and suffixes are palindromes in a single O(L) pass using the symmetry of the checks, instead of re-running the two-pointer check independently at every split. Both keep the O(w·L²) bound but shrink the constant.
Follow-up: "Can you hit the same complexity with a trie, and when is it worth it?"
Yes — build a trie of reversed words where each node records which complete words have a palindromic remainder beneath it. Walking each word down the trie finds partners in O(L) per word for the exact/prefix cases. It's worth it when memory locality and avoiding repeated substring hashing matter at very large scale, but the augmentation (tracking palindromic remainders per node) is intricate; in an interview the hash map is the safer call for the same big-O.
Follow-up: "What changes if the words are NOT distinct — duplicates allowed?"
The reversed-word map can no longer store a single index per key; switch to [String: [Int]] and, on a match, emit a pair for every stored index (still skipping self-pairs by index). The palindrome logic is unchanged — only the lookup result becomes a list, and you must be careful that two equal words that are palindromes pair with each other in both orders.
Follow-up: "How would you verify correctness on large random inputs?"
Write the O(w²·L) brute force as an oracle — it's trivially correct — and run a property test: generate random word lists, compute both results as sets of pairs, and assert equality. The brute force is too slow for production but perfect as a reference to fuzz the optimized solution against, especially around the empty-string and exact-reverse edge cases.

Note: A.1 (two-map + Set) reached 134/136 test cases before hitting Time Limit Exceeded on a pathological input of 5000 long repeated words. This is not a WA — the algorithm is correct. The TLE comes from Set<[Int]> hashing overhead on large outputs. A.2 (single-map + array dedup via the splitPoint < wordLength guard) avoids the Set entirely and passes all 136 test cases. In an interview, A.1 is the solution to present — correctness is the goal. A.2 is the production-ready form.

The exact code submitted for A.1 that reached 134/136:

Swiftsubmitted to LeetCode — 134/136, TLE on pathological input only
class Solution {
    func palindromePairs(_ words: [String]) -> [[Int]] {
        func isPalindrome(_ characters: [Character]) -> Bool {
            var leftPointer = 0, rightPointer = characters.count - 1
            while leftPointer <= rightPointer {
                if characters[leftPointer] != characters[rightPointer] { return false }
                leftPointer += 1
                rightPointer -= 1
            }
            return true
        }

        var leftWordToRightIndex: [String: [Int]] = [:]
        var rightWordToLeftIndex: [String: [Int]] = [:]

        for (wordIndex, word) in words.enumerated() {
            let wordArray = Array(word)
            for splitPoint in 0...wordArray.count {
                let leftSplit = Array(wordArray.prefix(splitPoint))
                let rightSplit = Array(wordArray.suffix(wordArray.count - splitPoint))
                if isPalindrome(leftSplit) {
                    let targetLeft = String(rightSplit.reversed())
                    if word != targetLeft { leftWordToRightIndex[targetLeft, default: []].append(wordIndex) }
                }
                if isPalindrome(rightSplit) {
                    let targetRight = String(leftSplit.reversed())
                    if targetRight != word { rightWordToLeftIndex[targetRight, default: []].append(wordIndex) }
                }
            }
        }

        var outputIndexes: Set<[Int]> = []
        for (currentIndex, word) in words.enumerated() {
            for rightIndex in leftWordToRightIndex[word, default: []] where rightIndex != currentIndex {
                outputIndexes.insert([currentIndex, rightIndex])
            }
            for leftIndex in rightWordToLeftIndex[word, default: []] where leftIndex != currentIndex {
                outputIndexes.insert([leftIndex, currentIndex])
            }
        }
        return Array(outputIndexes)
    }
}
