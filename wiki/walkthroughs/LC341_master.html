Theory
iterator_design_deep_dive_master.html

You are given a nested list of integers — each element is either an integer or another nested list — and you must implement an iterator that flattens it into a single in-order stream. Implement NestedIterator(nestedList), next() returning the next integer, and hasNext() returning whether one remains. The list is exposed through the NestedInteger API: isInteger() tells you which kind an element is, getInteger() reads the integer, and getList() reads the sub-list.

Inputnext() streamWhy
[[1,1],2,[1,3]]1, 1, 2, 1, 3In-order flatten — descend each sub-list as it is reached
[1,[4,[6]]]1, 4, 6Deeper nesting: 6 sits two levels down, surfaced when reached
[[]](empty)Only an empty sub-list → hasNext() is false from the start

The twist that makes it more than a tree walk is the iterator contract: you cannot flatten everything in the constructor (that defeats laziness and breaks on huge inputs). You must produce one integer per next(), descending into nesting only when a call actually needs the next value — and keep hasNext and next perfectly consistent.

2   Reading the Problem — insights from the constraints

"Implement next and hasNext": this is a pull interface — the caller drives. Your object must pause between integers and resume on demand. A recursive flatten runs to completion and can't pause, so the recursion must become an explicit stack you can stop and restart across calls.
"Each element is an integer or a nested list": the structure is a tree. An in-order flatten is a depth-first traversal that yields integers at the leaves. The stack holds the frontier of not-yet-processed elements, with the next one to emit nearest the top.
Laziness is the real requirement: flattening the whole tree in the constructor is O(total) up front and pointless if the caller stops early — or impossible if the structure is enormous. The intended design pushes only the top level and descends into a sub-list exactly when hasNext needs to expose its first integer.
Stacks are LIFO, so order must be reversed: if you push a list front-to-back, popping yields it back-to-front. To emit in original order, push each list's elements reversed, so the front element ends up on top and pops first.
Empty and nested-empty lists yield nothing: [[]] or [[],[]] contain no integers. hasNext must drain those lists (popping and pushing their empty contents) and correctly conclude there is nothing to return — without next ever being called.

GOTCHA: repeated hasNext() callssame answer; no consumptionhasNext only descends (pops lists, pushes contents) and inspects the top — it never pops an integer, so it is idempotent.
Putting the descent in hasNext is what makes the iterator correct. hasNext unwinds lists until the top is an integer (or the stack empties), but it never removes that integer. So calling hasNext any number of times leaves the same integer ready, and next can safely pop it. If the descent lived in next instead, a hasNext-then-next sequence on nested data would misbehave.

4   Try It Yourself First

Attempt the lazy design before reading on — eager flattening is the trap. Open hints one at a time.

Hint 1 — the constraint

You must return one integer per next() and be able to pause between them. Recursion can't pause mid-traversal, so turn it into an explicit stack you control. Do not flatten everything in the constructor.

Hint 2 — the pattern

Push the top-level elements onto a stack. To get original order out of a LIFO stack, push them reversed (front element on top). In hasNext, while the top is a list, pop it and push its contents reversed — keep going until the top is an integer or the stack is empty.

Hint 3 — keeping next/hasNext consistent

Let hasNext do all the descending (it's idempotent — it never removes an integer, only unwinds lists). next first calls hasNext to guarantee an integer is on top, then pops and returns it. That single rule keeps the two methods in lockstep and makes repeated hasNext calls harmless.

Step 1 — Clarifying questions
Q: "Should the iterator flatten lazily, or is flattening everything up front in the constructor acceptable?"
A: Lazily — that's the point of an iterator. → I push only the top level and descend on demand; the constructor stays cheap.
Q: "Can sub-lists be empty or arbitrarily deeply nested?"
A: Yes to both. → hasNext must drain empty lists to conclude false, and unwind multiple levels in one call for deep nesting.
Q: "Is next() only ever called when hasNext() is true?"
A: Yes, the caller checks first. → next can assume an element is available, but I'll still route it through hasNext to position the cursor.
Q: "What's the NestedInteger API — how do I tell an integer from a list?"
A: isInteger(), getInteger(), and getList(). → My stack holds NestedInteger values; I branch on isInteger() when inspecting the top.
Step 2 — The plan, before any code
The plan, spoken

"This is lazy flattening with a stack — Variant 8.2 of the Iterator Design family. The signal is 'flatten a nested list behind next/hasNext,' and the constraint that makes it interesting is laziness: I descend only when asked."

"Let me pin the conditions before the structure. The invariant I maintain is: the stack holds the not-yet-emitted elements with the next one to produce nearest the top, and after hasNext runs, the top is always an integer (or the stack is empty). The governing condition lives in hasNext: while the stack is non-empty and its top is a list, pop that list and push its elements reversed — stop when the top is an integer or the stack empties. It must be a while, not an if, because a list can contain lists, and empties must be fully drained. The consistency condition: hasNext never pops an integer, so it's idempotent; next calls hasNext first, then pops and returns the integer now guaranteed on top."

"The structures follow: a single stack of NestedInteger. I push the top-level list reversed in the constructor so the front element is on top — that's what turns LIFO popping back into original order. I need nothing else; the stack is the paused depth-first traversal."

"The walk: constructor pushes the input reversed. hasNext runs the unwind loop and returns whether the stack is non-empty. next calls hasNext, then pops the top and returns its getInteger(). Each element is pushed once and popped once across the whole traversal, so per-call cost is amortized O(1); space is O(depth + width) for the stack."

6   Why This Approach — and not the others

CandidateCostVerdict
Flatten everything into a flat array in the constructor, then indexO(total) up front, O(1)/call afterSimple and passes, but eager: it does all the work even if the caller takes one element, and stores every integer. Breaks the spirit of an iterator and the "huge/lazy source" follow-up.
Recursive generator / iteratorO(1) amortizedElegant in languages with native generators, but Swift has none here, and a hand-rolled recursive iterator must still externalize its call stack — which is exactly the explicit-stack design, just obscured.
Explicit stack, push reversed, descend in hasNextamortized O(1)/call, O(depth+width)Chosen. Truly lazy, pausable, no recursion-depth limit, and hasNext/next stay consistent by construction.
The deciding signal: a next/hasNext contract over a nested or generated structure. Whenever you must pause a traversal between elements, externalize the recursion into an explicit stack and prepare the next element lazily in hasNext. The same shape iterates trees, merges streams, and flattens any nested container.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
stack: [NestedInteger]The frontier of not-yet-emitted elements, next-to-emit on topA stack is the externalized DFS call stack — it lets the traversal pause and resume between next calls. append/removeLast are O(1). A flat pre-built array would be eager; a queue would give the wrong order without reversing anyway.
push-reversed discipline(no storage) the order conventionPushing each list's elements reversed makes the front element pop first, recovering original order from LIFO. Without it the stream comes out per-list-reversed.
the hasNext unwind loop(no storage) the laziness mechanismA while that pops lists and pushes their contents until an integer is on top. Centralizing descent here keeps it idempotent and keeps next trivial.
Why the stack holds NestedInteger, not Int. The stack must remember unexpanded elements — some are integers, some are still-nested lists awaiting descent. Storing the raw NestedInteger lets hasNext decide, lazily, whether each is a leaf to emit or a list to unwind. A stack of Int would require pre-flattening, which is the eager design we are avoiding.

9   Swift Solution — iterative

LeetCode supplies the NestedInteger class with isInteger() -> Bool, getInteger() -> Int, and getList() -> [NestedInteger]. The iterator below uses it directly.

A.1 Explicit stack, push reversed, descend in hasNext
amortized O(1)/callO(depth+width)
Plan
Push the input list reversed so its front element is on top. hasNext unwinds lists — while the top is a list, pop it and push its contents reversed — until an integer is on top or the stack empties, then reports whether anything remains. next calls hasNext to position the cursor, then pops and returns the integer. 
class NestedIterator {
    private var stack: [NestedInteger] = []

    init(_ nestedList: [NestedInteger]) {
        // Push reversed so the front element ends up on top.
        for element in nestedList.reversed() {
            stack.append(element)
        }
    }

    func next() -> Int {
        // Precondition assured by hasNext: the top is an integer.
        _ = hasNext()
        return stack.removeLast().getInteger()
    }

    func hasNext() -> Bool {
        // Unwind lists until an integer is on top (or the stack is empty).
        while let top = stack.last, !top.isInteger() {
            stack.removeLast()
            for element in top.getList().reversed() {
                stack.append(element)
            }
        }
        return !stack.isEmpty
    }
}
On the unwind loop. while let top = stack.last, !top.isInteger() peeks (does not pop) the top and continues only while it is a list. Inside, it pops that list and pushes its reversed contents. Because it never removes an integer, calling hasNext repeatedly is harmless — it stops the instant an integer surfaces, leaving it ready for next.

12   Complexity

OperationTimeSpaceWhy
initO(top-level count)O(top-level count)Pushes only the top-level elements reversed — not the whole tree. Lazy by design.
hasNext / nextamortized O(1)—Each element is pushed once and popped once over the iterator's life. A single call may unwind several frames, but the total across all calls is O(total elements).
Whole traversalO(total elements)O(depth + width)Every node is touched a constant number of times. Stack size peaks at the nesting depth plus the widest sibling group pushed at once — never the full flattened size.
Why "amortized" O(1) and not strict O(1). One next on a deeply nested element (like [[[[6]]]]) unwinds several list frames in a single call — that call is O(depth). But each list is unwound exactly once ever, so summed across all calls the work is O(total elements); dividing by the number of next calls gives O(1) on average. This is the eager-vs-lazy trade: we never pay O(total) up front, only as elements are consumed.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The design is already lazy and amortized O(1). The one robustness note is the precondition on next: it assumes hasNext is true. I route next through hasNext to position the cursor, but if a caller ignores the contract and calls next on an empty iterator, removeLast would crash — in production I'd return an optional or trap with a clear message. Otherwise it's optimal.
Follow-up: "Why push reversed — what breaks if you push in order?"
The stack is LIFO, so the last thing pushed pops first. If I push a list front-to-back, the back element is on top and comes out first — the stream would be each list reversed. Pushing reversed puts the front element on top so it pops first, recovering original order. It's the single most important detail and the most common bug if missed.
Follow-up: "Add a peek() that returns the next integer without consuming it."
That's the LC 284 — Peeking Iterator idea applied here: call hasNext to position the cursor, then return stack.last!.getInteger() without popping. Because hasNext already guarantees an integer on top and is idempotent, peek is a one-liner that reads the top; next still does the popping. The two compose cleanly.
Follow-up: "Compare this to first parsing the structure — how does it relate to LC 385?"
LC 385 — Mini Parser builds the NestedInteger tree from a string; this problem walks an already-built tree on demand. They compose: parse with LC 385, then feed the result into this iterator to stream the integers. Parsing is the producer, the iterator is the lazy consumer — together a full round trip from text to a pull-based integer stream.
