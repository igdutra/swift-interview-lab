LC 560 — Subarray Sum Equals K

Walkthrough Masterfile· Language: Swift· Medium · Prefix sum + hash map · Count complement pairs in one pass
Problem at a Glance
Difficulty
Medium

Tools

Key insight

Theory
prefix_suffix_master.html

Given an integer array nums and an integer k, return the total number of contiguous subarrays whose elements sum to exactly k. The array may contain negative numbers and zeros.

InputOutputWhy
nums=[1,1,1], k=22Subarrays [1,1] at indices (0,1) and (1,2) both sum to 2
nums=[1,2,3], k=32Subarrays [3] at index (2,2) and [1,2] at indices (0,1) both sum to 3
nums=[1,-1,1], k=13[1] at index 0, [1,-1,1] at indices (0,2), and [1] at index 2 — negatives enable longer subarrays to still sum to k

The twist: negative numbers rule out sliding window. A sliding window works by shrinking when the sum exceeds a threshold — but with negatives, adding more elements can decrease the sum, so there is no monotonic shrink condition. The solution instead recasts the problem algebraically: a subarray from index i to j sums to k exactly when prefix[j+1] − prefix[i] = k, which rearranges to prefix[i] = prefix[j+1] − k. As you build the prefix sum left to right, every value you have already seen and stored in a frequency map is a potential left endpoint — one hash map lookup counts all valid left endpoints for the current right endpoint in O(1).

2   Reading the Problem — insights from the statement

"Contiguous subarray": the subarray must be a contiguous slice of nums — not a subsequence, not a subset. This is the prefix-sum fingerprint: any contiguous subarray sum can be expressed as a difference of two prefix values, which is what makes the complement-lookup trick work.
"May contain negative numbers": this single phrase kills sliding window. Sliding window works when adding an element always increases the window sum and removing one always decreases it — a monotonic relationship. Negative numbers break this: adding a negative element decreases the sum, so there is no well-defined point to shrink. Prefix sums do not rely on monotonicity — the difference formula works for any sign.
"Return the total number" (count, not find): you are counting valid pairs (i, j), not returning the subarrays themselves. This shifts the design: instead of constructing subarrays, you need to count how many left endpoints i satisfy prefix[i] = currentPrefix − k. A frequency map gives you that count in O(1) per right endpoint.
The algebraic flip — the central move: subarray sum from i to j equals k means prefix[j+1] − prefix[i] = k. Rearranging: prefix[i] = prefix[j+1] − k. Instead of searching for pairs, for each new right endpoint j you ask "how many times has the value currentPrefix − k appeared as a prefix sum so far?" The frequency map answers this in O(1).

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — why not sliding window?

Try to define a "valid" window for this problem. When is the window too large? Can you always shrink by moving the left pointer? What happens when nums contains negative numbers — can adding more elements to the window ever bring the sum back to k after exceeding it?

Hint 2 — prefix sums

Express "sum of subarray from i to j" in terms of prefix values. You should get prefix[j+1] − prefix[i] = k. Now rearrange for prefix[i]: what are you looking for in a hash map?

Hint 3 — the frequency map

As you walk left to right computing prefix sums, store each prefix value in a map with its frequency. At each right endpoint j, look up currentPrefix − k in the map. Each hit is a valid left endpoint i that, paired with j, gives a subarray summing to k. What happens if the same prefix value appears multiple times?

Hint 4 — the sentinel

Initialize the map with {0: 1} before the loop begins. Why? Consider a subarray starting at index 0 — its prefix[i] is the virtual "prefix before any element," which has value 0. Without the sentinel, you'd never count those subarrays. Trace nums=[2], k=2 with and without the sentinel.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can the array contain negative numbers and zeros?"
A: Yes. → This rules out sliding window immediately. With negatives, there is no monotonic condition to shrink on — the window sum can go up and down. I need a prefix-sum approach that works regardless of sign.
Q: "Can k itself be negative or zero?"
A: Yes, k can be any integer. → The complement lookup formula currentPrefix − k is sign-agnostic — it works identically for negative k. No special casing needed.
Q: "Are there duplicate values in nums? Could there be many subarrays with the same prefix sum?"
A: Yes to both. → The frequency map must store counts, not just presence. A Set would miss the case where the same prefix value appears multiple times — each occurrence represents a distinct valid left endpoint, so I need map[prefixValue] = numberOfTimesItHasOccurred.
Q: "Is there a minimum array length?"
A: At least 2 elements. → So I always have at least one subarray to check. The loop always runs at least once.
Step 2 — The plan, before any code
The plan, spoken

"This is a prefix sum plus frequency map problem. The signal is 'contiguous subarray count' combined with 'negative numbers allowed' — that dual constraint immediately eliminates sliding window and points to prefix sums."

"Let me state the central condition. A subarray from index i to j sums to k exactly when prefix[j+1] − prefix[i] = k. Rearranging: prefix[i] = prefix[j+1] − k. So for any right endpoint j, I need to count how many left endpoints i have already produced the prefix value prefix[j+1] − k. That is a frequency map lookup."

"The validity condition for adding to the count: the map contains the complement currentPrefix − k. Each occurrence in the map is a distinct valid subarray — if the complement appears three times, that means three different starting positions all produce a subarray ending at j that sums to k. I add the map's count for that complement to my result in one step."

"The sentinel is mandatory: I initialize the map with {0: 1} before the loop. That represents the virtual 'prefix before any element' — the empty prefix whose value is 0. Without it, any subarray starting at index 0 whose sum equals k would look for complement 0 in the map and find nothing. With it, it finds count 1 and is correctly counted."

"Data structures: one running integer for the current prefix sum; one Dictionary<Int, Int> mapping prefix values to how many times they've appeared. I do not need to store the prefix array itself — I only ever need the running total and the count of each past value."

"The walk: initialize prefixCounts = {0: 1}, runningSum = 0, count = 0. For each number in nums: add it to runningSum; look up runningSum − k in prefixCounts and add the result to count; increment prefixCounts[runningSum] by 1. Return count."

"Expected O(n) time — one pass; O(n) space — the map holds at most n+1 entries. I'll trace the first example to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check all (i, j) pairsO(n²) time, O(1) spaceCorrect but quadratic — n=2×10⁴ gives 4×10⁸ operations. Compute sum(i,j) in an inner loop or with prefix sums in O(1) per pair — still O(n²) pairs total.
Sliding window—Wrong tool. Requires a monotonic condition to shrink — "if the window sum exceeds k, move the left pointer." With negative numbers the window sum is not monotonic: adding more elements can decrease it. There is no valid shrink point. This is the most common wrong instinct on this problem.
Sort then two pointersO(n log n) timeSorting destroys subarray order — subarrays must be contiguous slices of the original array, and sort breaks that invariant. Not applicable.
Prefix sum + frequency mapO(n) time, O(n) spaceChosen. One pass. Algebraic rearrangement turns "find pairs (i,j) where prefix[j]−prefix[i]=k" into "look up prefix[j]−k in a map of past prefix values." Each lookup is O(1); the sentinel {0:1} handles subarrays that start at index 0.
THE ONE IDEA: Instead of searching for two prefix values that differ by k, fix the right endpoint and ask "how many left endpoints already in the map give me a difference of exactly k?" The algebraic flip — prefix[i] = currentPrefix − k — converts a pair-search problem into a single-key lookup, reducing the inner work from O(n) to O(1). The frequency map is what enables counting multiple valid left endpoints in that single lookup.
When to reach for prefix sum + frequency map: "count subarrays with a target sum" where negatives are allowed (ruling out sliding window). The complement-lookup pattern is reusable: LC 974 (subarray sums divisible by k) and LC 523 (continuous subarray sum) use the same algebraic rearrangement with modular arithmetic instead of exact equality.
PITFALLS
▸Forgetting the sentinel {0: 1}. Without it, every subarray starting at index 0 is missed. The sentinel represents the virtual "prefix before any element" — a zero-valued prefix that has occurred once. Trace nums=[3], k=3: after processing index 0, runningSum=3, complement=0 — if the map doesn't contain 0, count stays 0 and the answer is wrong.
▸Using a Set instead of a Dictionary. A Set records whether a prefix value has occurred, not how many times. If prefix value 5 appears at three different positions, each is a distinct valid left endpoint. Storing just presence misses the second and third occurrences — a silent undercount.
▸Adding runningSum to the map before the complement lookup. The lookup must happen before the update — otherwise the current position counts itself as a valid left endpoint for its own right endpoint, producing subarrays of length 0. The correct order is: (1) look up complement, (2) add current prefix to map.
▸Reaching for sliding window because the problem mentions "subarray." Sliding window applies when the validity condition is monotonic. The moment you see negative numbers in a sum-target problem, rule sliding window out immediately and switch to prefix sum.

7   Data Structures — choice and justification

VariableHoldsWhy this and not something else
prefixCounts: [Int: Int]Maps each prefix sum value seen so far to how many times it has appeared. Initialized with [0: 1] (the sentinel).Dictionary gives O(1) average lookup and update. A Set would record presence but not multiplicity — misses cases where the same prefix value appears at multiple indices (each is a distinct valid left endpoint). A sorted array would cost O(log n) per lookup.
runningSum: IntThe current prefix sum after processing each element — equivalent to prefix[index+1] in the explicit prefix array.A single integer replaces the entire prefix array. We never need to revisit past prefix values by index — only by value. The map stores the values; the running integer produces the next one.
count: IntAccumulator for valid subarrays found.Plain integer — each map lookup returns a count (≥ 0) and we add it directly. The result fits in an Int because at most n(n+1)/2 subarrays exist.
Why we don't need the explicit prefix array: in LC 303 we needed random O(1) access to any prefix value by index (to answer arbitrary range queries). Here we only need to count past occurrences of a specific value — a hash map is the right structure for that lookup pattern. The running integer produces prefix values in order; the map stores their frequencies. No array required.

8   Pseudocode Dry Run

Pseudocodeprefix sum + frequency map — one pass
prefixCounts ← { 0: 1 }    // sentinel: "empty prefix with value 0 has appeared once"
runningSum   ← 0
count        ← 0

FOR number IN nums:
    runningSum += number

    // How many left endpoints i give runningSum − prefix[i] = k?
    // Equivalently: prefix[i] = runningSum − k
    complement ← runningSum − k
    count += prefixCounts[complement] ?? 0    // add 0 if complement not seen

    // Record this prefix value for future right endpoints to look up.
    prefixCounts[runningSum] = (prefixCounts[runningSum] ?? 0) + 1

RETURN count

9   Swift Solution

A.1 Prefix sum + frequency map — single pass
O(n)O(n)
Plan
Initialize a frequency map with sentinel {0: 1}. Walk left to right, maintaining a running prefix sum. At each step, look up the complement (runningSum − k) in the map and add the count to the result — each hit is a valid subarray ending here. Then record the current prefix sum in the map for future steps to find. Never store the full prefix array — the running integer and the map are sufficient. 
func subarraySum(_ nums: [Int], _ k: Int) -> Int {
    // Sentinel: the empty prefix (value 0) has been "seen" once before the loop.
    // This lets subarrays starting at index 0 be counted when their sum equals k.
    var prefixCounts: [Int: Int] = [0: 1]
    var runningSum = 0
    var count = 0

    for number in nums {
        runningSum += number

        // Count all left endpoints i where prefix[i] = runningSum − k,
        // meaning subarray(i..current) sums to k.
        count += prefixCounts[runningSum - k, default: 0]

        // Record this prefix value so future right endpoints can find it.
        prefixCounts[runningSum, default: 0] += 1
    }

    return count
}

11   Question → Code Map

Question / decisionCode it became
"Why can't I use sliding window?"No code — negative numbers break the monotonic shrink invariant. The entire prefix-sum approach exists because of this constraint.
"How do I express 'subarray from i to j sums to k'?"runningSum - k — the complement we look up; derived from prefix[j+1] − prefix[i] = k → prefix[i] = prefix[j+1] − k
"What does the map store?"var prefixCounts: [Int: Int] — maps each prefix sum value to how many times it has appeared (frequency, not presence)
"Why a Dictionary and not a Set?"prefixCounts[runningSum - k, default: 0] — adding the count, not just 1. A Set can only contribute 0 or 1; multiple occurrences of the same prefix value would be silently merged.
"What is the sentinel and why?"[0: 1] — the virtual empty prefix before index 0; without it, subarrays starting at index 0 are never matched
"Which comes first — the lookup or the map update?"Lookup first: count += prefixCounts[runningSum - k, default: 0], then update: prefixCounts[runningSum, default: 0] += 1. Reversing the order would let the current position count itself as a valid left endpoint.

12   Complexity

SolutionTimeSpaceWhy
Brute force (all pairs, recompute sum)O(n³)O(1)O(n²) pairs × O(n) to sum each subarray from scratch
Brute force with prefix arrayO(n²)O(n)O(n²) pairs, O(1) query per pair via prebuilt prefix array — still quadratic pairs
Prefix sum + frequency map (this solution)O(n)O(n)One pass through nums; each step is one hash map lookup + one hash map update, both O(1) average; the map holds at most n+1 entries
Bottleneck: O(n) time is optimal — you must read every element at least once to know the answer. The O(n) space for the map is unavoidable: in the worst case (all distinct prefix values), you store n+1 entries. There is no O(1)-space solution for this problem unlike LC 303, because here you need to recall specific past values by key, not just the most recent one.

13   Follow-Up Bank

Follow-up: "Before you push this to production — can you see any improvements you'd make?"

The solution is already optimal: O(n) time and O(n) space — nothing can be improved asymptotically. One constant-factor note: Swift's Dictionary subscript with default: is slightly cheaper than a two-step ?? 0 chain for repeated access. I'd also add the comment about lookup-before-update order in production code — that invariant is subtle and the silent bug it prevents is hard to spot in review.

Follow-up: "What if you needed to return the subarrays themselves, not just the count?"

The map would need to store indices (or lists of indices) rather than counts. Instead of [Int: Int], use [Int: [Int]] mapping each prefix value to all indices where it occurred. At each hit, record the pair (storedIndex, currentIndex) in the result. This doesn't change the O(n) time — but the output size can be O(n²) in the worst case (e.g., all zeros, k=0), so the overall cost becomes O(n + output size), which is unavoidable.

Follow-up: "What if instead of sum = k, you needed sum divisible by k?"

This is LC 974 — Subarray Sums Divisible by K. The same pattern applies with one change: instead of storing prefix values, store prefix values modulo k. A subarray sum is divisible by k when two prefix values have the same remainder mod k — so the map key becomes runningSum % k. Swift's modulo for negative numbers can produce negative remainders, so normalize: ((runningSum % k) + k) % k. The sentinel is still {0: 1}.

Follow-up: "What if all numbers are positive — would sliding window work then?"

Yes — with positive numbers only, the window sum is strictly monotonic: adding elements always increases the sum, removing always decreases. This restores the sliding window shrink condition. Sliding window would give O(n) time and O(1) space — better space than the hash map approach. But since the problem explicitly allows negatives, the prefix-sum approach is the only safe choice here.

14   Theory Decoder — concepts used here

ConceptHow it appears in LC 560
Prefix sum (Variant 8.1)runningSum accumulates prefix values exactly as the explicit prefix array in LC 303 — it is just the rolling version, discarding the array allocation. At step j, runningSum equals what prefix[j+1] would contain.
Sentinel valueprefixCounts = [0: 1] before the loop. Represents the virtual empty prefix — "before index 0, the prefix sum is 0, and this has occurred once." Without it, subarrays starting at index 0 are never counted.
Complement lookup (algebraic flip)Instead of searching for pairs, rearrange prefix[j] − prefix[i] = k to prefix[i] = prefix[j] − k. Fix the right endpoint (current), look up the left-endpoint value in the map. This converts O(n) pair search into O(1) map lookup.
Frequency map (counts, not presence)prefixCounts: [Int: Int] maps each past prefix value to how many times it has appeared. Multiple occurrences mean multiple valid left endpoints at that prefix value — a Set would merge them and produce a wrong count.
Lookup before update (order invariant)The complement lookup (count += prefixCounts[runningSum - k]) must happen before the map update (prefixCounts[runningSum] += 1). This ensures the current position is never counted as a valid left endpoint for itself — which would imply a subarray of length 0.
