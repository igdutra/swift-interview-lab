Theory
stateful_system_design_deep_dive_master.html

Design a tiny file system that supports two operations. createPath(path, value) creates a new path and associates a value with it, returning true on success and false on failure. get(path) returns the value associated with a path, or -1 if the path does not exist. A path is a string like "/a/b/c". Creation succeeds only if the path does not already exist and its parent path does exist — the root "/" is the implicit base that always exists, but you cannot create a path whose immediate parent was never created.

CallsResultWhy
createPath("/a", 1)trueParent is "/" (root, always exists); "/a" is new
createPath("/a/b", 2) then get("/a/b")true, then 2Parent "/a" exists; value stored and read back
createPath("/c/d", 3)falseParent "/c" was never created → creation rejected

The twist is that this looks like it wants a tree of folders, but none of the operations ever walk children — every call names a full path. So the clean design is a flat dictionary from path string to value, and the only real logic is the two-part validity check on creation.

2   Reading the Problem — insights from the constraints

"Returns the value associated with a path, or −1": the answer to a get is a stored value — there is nothing to compute or aggregate. That immediately says map-backed state: a dictionary keyed by the path, and the query is one lookup with a sentinel default.
"Its parent path must exist": this is the entire algorithmic content. Creating "/a/b/c" requires "/a/b" to be present. So creation is a two-part check — the path itself is absent, and its parent is present — and deriving the parent is pure string work: drop everything after the last "/".
The root "/" always exists: a top-level path like "/a" has parent "/", which is the implicit base. You never store the root; you special-case it so that creating a depth-1 path only needs the "absent" check, its parent being trivially present.
No operation walks children: there is no "list directory," no "delete subtree," no traversal. Every operation is keyed by a full path. A real tree of nodes would let you walk children but buys nothing here — the flat map is simpler and O(1) per lookup. Model the representation to the operations, not the metaphor.
"Create a new path" — no overwrite: if the path already exists, creation must fail and leave the existing value untouched. The "absent" half of the check protects against clobbering.

The parent-of-a-top-level-path case is the one trap. Splitting "/a" on "/" and dropping the last piece yields the empty string, not "/". 4   Try It Yourself First

This is the family's warm-up — the design is small but the validity rule has an edge. Try it cold, then open hints one at a time.

Hint 1 — the constraint

Every operation names a full path, and get returns a stored value. Nothing walks children. So you don't need a tree — you need a fast lookup from a path string to its value. What single data structure gives O(1) lookup by a string key?

Hint 2 — the pattern

Map-backed state: a [String: Int] dictionary. get(path) is store[path] ?? -1. createPath stores store[path] = value — but only after a validity check.

Hint 3 — the validity rule

Creation succeeds iff (a) the path is not already a key, and (b) its parent path is a key — except a top-level path like "/a" whose parent is the root "/", which always exists. Get the parent by dropping the substring after the last "/"; if that leaves the empty string, the parent is the root and condition (b) is automatically satisfied.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Does any operation need to list or traverse the children of a path?"
A: No — every call names a full path. → I don't need a folder tree; a flat dictionary keyed by the full path is enough and gives O(1) lookups.
Q: "On creation, what exactly makes it fail — only if the path exists, or also if the parent is missing?"
A: Both — fail if the path already exists, or if its parent path was never created. → A two-part validity check; and I must derive the parent path from the string.
Q: "Is the root "/" considered to always exist, so a top-level path's parent is satisfied?"
A: Yes, root always exists. → I special-case a one-component path: its parent is the root, so I skip the parent-presence lookup.
Q: "Can values be any integer, and is a path guaranteed well-formed (leading slash, no trailing slash)?"
A: Values are integers; paths are well-formed with a leading slash and no trailing slash. → No path normalization; I can split on "/" directly.
Step 2 — The plan, before any code
The plan, spoken

"This is map-backed state — Variant 8.1 of the Stateful System Design family. The signal is 'get returns a stored value and nothing traverses children,' so the answer is a dictionary, not a tree."

"Let me pin the conditions before the structure. The invariant is simply: the dictionary holds exactly the paths that have been successfully created, each mapped to its value. The governing condition is the creation validity rule, and I'll state it exactly as it'll be coded: a createPath(path, value) succeeds if and only if the path is not already a key AND its parent path either is the root or is already a key. The transition case inside that rule is the top-level path: if the path has a single component, its parent is the root, which always exists, so I skip the parent lookup. On any failure I return false and mutate nothing — the dictionary is left exactly as it was."

"The structure follows directly: a single [String: Int] store. I need nothing else — no tree, no per-directory child lists — because the conditions only ever ask 'is this exact path present?' which a dictionary answers in O(1). To derive the parent I split the path on "/" and rejoin all but the last component; an empty result means the parent is the root."

"The walk: get(path) returns store[path] ?? -1. createPath first checks the path is absent, then derives the parent and checks it's the root or present; if both pass, it stores the value and returns true, else false. Each operation is O(L) in the path length for the string work and O(1) for the dictionary; space is O(total stored paths)."

6   Why This Approach — and not the others

CandidateCostVerdict
Tree of folder nodes, each with a children mapO(depth) per opFaithful to the "file system" metaphor, but every operation would walk from the root down the path components — more code, more allocations, and slower than a flat lookup. No operation needs the tree's child-listing power.
Store paths as a set + a separate values dictionaryO(1) per opWorks, but redundant — the values dictionary already encodes presence (a key exists iff the path was created). Two structures to keep in sync where one suffices.
Single [String: Int] keyed by full pathO(L) string work, O(1) map / opChosen. One structure, O(1) lookups, presence and value in the same map. The only logic is the two-part creation check.
The deciding signal: "get returns a stored value, keyed by an identifier, with no traversal." Whenever a design's queries are all "give me the value for this exact key," reach for a dictionary keyed by that identifier and resist building the richer structure the problem's metaphor suggests. Fit the representation to the operations.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
store: [String: Int]Every created path → its valueO(1) average lookup, insert, and presence test. A key's existence doubles as "this path was created," so one structure encodes both presence and value. A tree would add traversal power no operation uses.
parent derivation (string split)(no storage) the parent path of a given pathComputed on demand by dropping the last "/"-component. Storing parent pointers would duplicate information already implicit in the path string.
the -1 sentinel(no storage) the "absent" answer for getstore[path] ?? -1 turns a missing key into the required −1 in one expression — no separate contains-check.
Why presence and value share one map. Because creation forbids overwrite and get returns −1 for missing paths, "is this path a key?" and "what is its value?" are answered by the same dictionary. There is never a created path without a value or a value without a created path, so a second presence structure would only risk drifting out of sync.

9   Swift Solution — iterative

A.1 Flat path dictionary with the two-part creation check
O(L) / opO(paths)
Plan
Keep one [String: Int] mapping each created path to its value. get is a lookup defaulting to −1. createPath rejects an existing path, derives the parent by dropping the last component, accepts when the parent is the root or already present, then stores the value. 
class FileSystem {
    private var store: [String: Int] = [:]

    init() {}

    func createPath(_ path: String, _ value: Int) -> Bool {
        // No overwrite: the path must be new.
        if store[path] != nil { return false }

        // Parent = everything up to the last "/". Empty means the root.
        let components = path.split(separator: "/", omittingEmptySubsequences: true)
        let parent = "/" + components.dropLast().joined(separator: "/")
        let parentIsRoot = components.count == 1

        // Valid iff the parent is the root or an existing path.
        if !parentIsRoot && store[parent] == nil { return false }

        store[path] = value
        return true
    }

    func get(_ path: String) -> Int {
        return store[path] ?? -1
    }
}
On parent derivation. Splitting with omittingEmptySubsequences turns "/a/b" into ["a","b"]; dropping the last and rejoining with a leading "/" rebuilds "/a". When there is a single component ("/a" → ["a"]), parentIsRoot is true and the parent-presence check is skipped — the root always exists.

12   Complexity

OperationTimeSpaceWhy
createPathO(L)O(L)L = path length. The dictionary ops are O(1) average; the cost is the string split/join to derive the parent, linear in the path.
getO(L)O(1)One dictionary lookup, O(1) average; hashing the path key is O(L).
Overall stateO(L) / opO(Σ paths)The store grows by one entry per successful create. No operation rescans the state, so there is no hidden quadratic — the only per-op cost is hashing/deriving on the path string.
Why there is no aggregate to maintain here. Unlike the tic-tac-toe variant, get returns a value that is simply stored — there is no summary to derive or keep consistent. That makes this the family's simplest member: the design is entirely "choose the key," and the dictionary does the rest in O(1).

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The operations are already O(1) on the dictionary; the only per-call cost is the string split to find the parent. If creates dominate, I could cache the parent derivation or store paths pre-split, but for typical inputs the split is cheap and not worth the complexity. The bigger production concern is input validation — guarding against a malformed path (no leading slash, trailing slash, double slashes) rather than assuming well-formed input.
Follow-up: "Now add a deletePath(path) that removes a path and all its descendants."
Deletion is where the flat map starts to hurt: to remove a subtree I'd have to scan every key for those starting with path + "/" — O(total paths) per delete. If deletes become frequent, that's the point to switch to an actual tree of nodes with children maps, where deleting a subtree is just dropping one node's reference. The representation should follow the operation set: add subtree deletion and the tree earns its keep.
Follow-up: "Make it thread-safe for concurrent creates and gets."
Wrap the store in a serial dispatch queue (or an actor in Swift concurrency) so mutations and reads are serialized, or use a reader-writer lock to allow concurrent gets while serializing creates. The validity check and the store mutation must happen atomically as one critical section — otherwise two concurrent creates of sibling paths could both pass the parent check and interleave inconsistently.
Follow-up: "What if values can be updated after creation — add a setValue?"
That's a one-liner: setValue(path, value) succeeds iff the path exists (store[path] != nil), then overwrites. It doesn't disturb the design — the dictionary already supports overwrite; we deliberately forbade it only in createPath. The two methods just differ in their precondition: create requires absence, set requires presence.
