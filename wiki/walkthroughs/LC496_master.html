LC 496 — Next Greater Element I

Walkthrough Masterfile· Language: Swift· Easy · Monotonic Stack · Decreasing stack + lookup map
Problem at a Glance
Difficulty
Easy
Pattern
Monotonic Stack — decreasing stack sweeping right-to-left, result stored in a map
Tools
A monotonically decreasing stack · a value → nextGreater dictionary · a final map lookup
Key insight
Elements smaller than the new arrival can never be the next-greater for anything to the left — evict them. Whatever remains on top after eviction is the answer.
Time / Space
O(n + m) / O(n)
Theory
stack_master.html

You are given two arrays, nums1 and nums2, where nums1 is a subset of nums2 (all values are unique). For each element in nums1, find the first element to its right in nums2 that is strictly greater. Return −1 if no such element exists. The relative positions in nums1 don't matter — only the position of each value inside nums2 determines the answer.

InputOutputWhy
nums1=[4,1,2], nums2=[1,3,4,2][-1,3,-1]4 has nothing greater to its right; 1 → 3 is the first greater; 2 has nothing
nums1=[2,4], nums2=[1,2,3,4][3,-1]2 → 3 is the first greater; 4 is the last element, nothing to its right
nums1=[1], nums2=[1][-1]Only one element, nothing to its right at all

The twist: a brute-force suffix scan is O(n·m), which works but misses the point of the problem. The real question is: can you compute the next-greater for every element in nums2 in a single O(n) pass — and the answer is yes, via a monotonic stack.

2   Reading the Problem — insights from the constraints

"First element to its right that is greater" — first is the tell. Not the maximum. Not any element. The nearest qualifying element. Any time a problem asks for the nearest element satisfying a condition, a monotonic stack is the O(n) answer — it processes each element exactly once while maintaining a structure that resolves "nearest" queries for free.

4   Try It Yourself First

Hint 1 — constraint

The constraints say "first element to its right" and all values are unique. What does "first" tell you about the kind of scan you need — and does O(n·m) brute force actually count as solving the problem the way an interviewer intends?

Hint 2 — pattern

You want the nearest element greater than each value. Think about what kind of stack stays useful: if a small element is sitting on the stack and a large element arrives, can the small element ever be the next-greater for anything to the left of the large element?

Hint 3 — structure

Sweep nums2 right-to-left. Maintain a stack that is always decreasing from bottom to top. Before pushing a new value, pop everything smaller — they are dominated and useless. Whatever is on top after popping is the next-greater for the current value. Store that in a dictionary keyed by value, then answer nums1 in one final map lookup.

5.2   The Plan, Spoken

"This is a monotonic stack problem. The signal is 'first element to its right that is greater' — whenever a problem asks for the nearest qualifying element, that phrase maps to a monotonic stack, which resolves nearest-greater queries in O(n) by reusing prior comparisons. The sequence goes like this: I sweep nums2 right-to-left. I maintain a stack that is always decreasing from bottom to top — meaning the top is always the smallest thing on the stack. When I visit a new element, I pop everything off the top that is smaller than or equal to it. Those smaller elements are dominated: if they cannot be the next-greater for the current element, they cannot be the next-greater for anything further to the left either — because the current element is already blocking them. After the pops, whatever is left on top is the first element to the right that is greater than the current one. I record that mapping in a dictionary: nextGreater[currentValue] = stack.last ?? -1. Then I push the current value onto the stack, because it could be the next-greater for something to its left. After the full sweep I answer nums1 with a single map lookup per value. The state I need: a stack of Ints, decreasing bottom-to-top; a dictionary of [Int: Int] mapping each value to its next-greater. The invariant: after processing each element, the stack contains, in decreasing order, the candidates that could still be the next-greater for elements further to the left. The condition: pop while stack.last <= currentValue — those elements are dominated. Record stack.last ?? -1 before pushing. Let me walk nums2 = [1, 3, 4, 2] right-to-left. Start at 2: stack empty → nextGreater[2] = -1, push 2. Stack: [2]. Move to 4: pop 2 (2 ≤ 4), stack empty → nextGreater[4] = -1, push 4. Stack: [4]. Move to 3: 4 > 3, don't pop → nextGreater[3] = 4, push 3. Stack: [4, 3]. Move to 1: 3 > 1, don't pop → nextGreater[1] = 3, push 1. Stack: [4, 3, 1]. Done. Then for nums1 = [4, 1, 2]: nextGreater[4]=-1, nextGreater[1]=3, nextGreater[2]=-1 → [-1, 3, -1]. Correct. Time is O(n + m) — each element is pushed and popped at most once during the sweep (O(n)), then one lookup per nums1 element (O(m)). Space is O(n) for the stack and dictionary."

6   Why This Approach

ApproachTimeSpaceVerdict
Brute force — for each nums1 element, scan right in nums2O(n · m)O(1)Correct but misses the insight. Interviewer will ask "can you do better?"
Suffix scan — precompute next-greater for all of nums2 by scanning rightO(n²)O(n)Still O(n²); you build the full table but pay O(n) per element to find the answer
Monotonic stack (right-to-left sweep)O(n + m)O(n)Optimal. Each element pushed and popped at most once.
Why not two pointers? Two pointers converge from the ends of a sorted array, or both run rightward in a sliding window. Neither motion answers "what is the first thing greater than me to my right?" for multiple unsorted elements simultaneously. Two pointers track a pair, not a pending set of unanswered queries. The monotonic stack is exactly the structure that holds "elements waiting for their answer" and resolves them the instant a qualifying element arrives.
Why not suffix precomputation (prefix/suffix)? A suffix-max pass tells you the maximum to the right, not the nearest element that is greater. If you want nearest, you'd still need to scan rightward per element — back to O(n²). The monotonic stack is the O(n) optimization over that suffix scan: instead of rescanning, it keeps the candidates alive on the stack and discards them exactly when they are proven useless.
The routing signal for monotonic stack: "next greater / next smaller / nearest element satisfying condition X." If the problem asks for the nearest qualifying neighbor (left or right), that phrase alone selects a monotonic stack. If it asked for the maximum to the right → suffix precomputation. If it asked for any matching pair → hash map.

7   Data Structures

StructureWhat it storesWhy this one
Monotonically decreasing stack ([Int])Candidate next-greater values, bottom-to-top decreasingLIFO lets us resolve queries in arrival order; the decreasing invariant ensures the top is always the smallest (nearest) candidate, so the first pop check is always the answer
Dictionary [Int: Int] — nextGreaterMaps each value in nums2 to its next-greater (or −1)Values are unique, so value is a safe key; O(1) lookup; built during the single sweep so the final nums1 pass is just map lookups
Why decreasing, not increasing? We want the stack to hold candidates for "next greater to the right." When a new element arrives, it beats any smaller elements — they are no longer candidates for anything to the left. Popping them (maintaining the decreasing invariant) is the same as saying: "the new element is a better answer than you for any future query." An increasing stack would be the right choice for "next smaller" problems.

8   Pseudocode Dry Run

Pseudocode
function nextGreaterElement(nums1, nums2):
    nextGreater ← empty map
    stack       ← empty stack          // decreasing bottom-to-top

    for value in nums2.reversed():
        while stack is not empty AND stack.top ≤ value:
            stack.pop()                // dominated — evict
        nextGreater[value] ← stack.top ?? -1
        stack.push(value)

    return nums1.map { nextGreater[$0] ?? -1 }

9   Swift Solution

A.1 Monotonic Stack — right-to-left sweep
O(n + m)O(n)
PLAN: Sweep nums2 reversed. Maintain a decreasing stack — pop everything ≤ current before recording and pushing. The top after popping is the next-greater. Answer nums1 via dictionary lookup.
Swift
private func nextGreaterElement(_ nums1: [Int], _ nums2: [Int]) -> [Int] {
    var nextGreater: [Int: Int] = [:]
    var stack: [Int] = []          // monotonically decreasing, bottom → top

    for value in nums2.reversed() {
        while let top = stack.last, top <= value {
            stack.removeLast()       // top is dominated — it can never be next-greater for anything further left
        }
        nextGreater[value] = stack.last ?? -1
        stack.append(value)
    }

    return nums1.map { nextGreater[$0] ?? -1 }
}

13   Follow-Up Bank

Follow-up: Before you push this to production — can you see any improvements you'd make?

The dictionary allocates O(n) space to store a mapping we only query once at the end. If nums1 is much shorter than nums2, we could filter to only track values that appear in nums1 — building a Set(nums1) and only inserting into the dictionary for those values. Same O(n) sweep, but the dictionary stays smaller: O(m) instead of O(n). The stack itself stays O(n) in the worst case regardless.

Follow-up: What if nums2 is circular — after the last element, the array wraps around and you look again from the start?

That is LC 503 — Next Greater Element II. The fix is to sweep the array twice (using index % n to wrap), which simulates the circular view without duplicating the array. Everything else stays the same — same monotonic stack, same eviction rule.

Follow-up: What if you want the next smaller element instead of the next greater?

Flip the stack to increasing (pop everything ≥ current instead of ≤). The structure is symmetric — a monotonically increasing stack resolves "nearest smaller to the right" in the same O(n) sweep. The eviction condition is the only line that changes.

Follow-up: What if nums1 and nums2 can have duplicates?

The value-keyed dictionary breaks — two identical values would overwrite each other. You'd need to key by index instead. Find each nums1 element's position in nums2 first (O(n) with a value→index map), then build the next-greater by index. Or store a list of indices per value. The monotonic stack logic is unchanged; only the lookup key shifts from value to index.

14   Theory Decoder

This problem uses one concept from the parent stack theory: the monotonic stack — Pattern 4 of the stack theory page.

Theory conceptHow it appears in LC 496
Monotonically decreasing stackThe stack holds values in decreasing order bottom-to-top. The top is always the smallest current candidate for "next-greater." Evicting elements ≤ the new arrival maintains this invariant and simultaneously answers their pending queries.
Push / pop at most once per elementEach of the n elements in nums2 is appended once and removed at most once during the sweep — the amortized O(n) argument. The loop body runs O(1) amortized even though the inner while loop can pop multiple times per step.
Stack as "pending queries"Every element on the stack is an element that has not yet found its next-greater. It stays on the stack until a larger element arrives to its left (in the right-to-left sweep) and resolves it by eviction.
Dictionary lookup decouples the two arraysBy precomputing a value→nextGreater map from nums2, the nums1 query becomes a pure O(m) lookup — the monotonic stack does all the work upfront, independent of nums1.
The eviction IS the answer. When you pop an element from the stack because it is dominated, you are not discarding it — you have just determined that the current element is its next-greater, and you already recorded that before popping (via nextGreater[value] = stack.last ?? -1 for the incoming value). The pop is the resolution of a pending query, not a discard.
