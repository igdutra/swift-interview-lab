Theory
intervals_overview_master.html

A mission window spans [startDay, endDay] inclusive. You are given a map of teams, each with a set of available days. Find every ordered pair (A, B) of distinct teams where A covers a contiguous left segment and B covers the remaining contiguous right segment, together covering every day in the window with no gap. Order matters: (A, B) and (B, A) are different pairs.

The twist: teams provide raw day sets, not interval objects. The brute force tries every split point for every ordered pair — O(t² · d). The insight is that you only need to extract one number per team per role: how far it reaches contiguously from startDay (its prefixEnd), and from where it can start while still reaching endDay contiguously (its suffixStart). These two numbers per team reduce pair-checking from O(d) to O(1).

2   Reading the Problem — insights from the constraints

"Exactly two teams": the split is binary — one left segment, one right segment. You are not finding a minimum cover or merging k schedules. Every valid answer is a single handoff point between exactly two teams.
"Contiguous segment": team A must cover every day from startDay to some splitDay with no gaps. Team B must cover every day from splitDay + 1 to endDay with no gaps. A team that covers, say, days {3,4,6} cannot anchor the left segment past day 4 even though it also has day 6.
"Order matters": (A, B) is distinct from (B, A). A is always the left team; B is always the right team. This means a team that can serve as either a left or right anchor may appear in pairs in both roles.
Raw day sets, not intervals: the input is a dictionary of [String: [Int]], not [String: [(start, end)]]. The interval objects in Variants 8.1–8.4 don't exist here. You must derive the effective interval for each team's role — that derivation is the prefix/suffix precompute.
startDay < endDay required for a valid split: a single-day window cannot be divided between two teams. Guard this at the top and return empty immediately.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the brute force

For each ordered pair (A, B), try every possible split point in [startDay, endDay). For each split, verify A covers [startDay…split] and B covers [split+1…endDay]. This is O(t² · d). It works — but can you avoid repeating the coverage walk for every pair?

Hint 2 — the key observation

Each team's usefulness as a first team is fully captured by one number: the furthest day it reaches contiguously from startDay (its prefixEnd). Its usefulness as a second team is fully captured by one number: the earliest day it can start while still reaching endDay contiguously (its suffixStart). Compute these once per team, not once per pair.

Hint 3 — the valid pair condition

Pair (A, B) is valid when B's contiguous reach to endDay starts at or before the handoff point: suffixStart[B] <= prefixEnd[A] + 1. Note: it's ≤ not ==. A team covering a wider suffix than needed is still a valid second team.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Does 'contiguous' mean no gaps — every single day between startDay and the split must be covered?"
A: Yes, every day. → I need to walk each team's day set and stop at the first missing day. One gap kills the contiguity.
Q: "Does order matter — is (A, B) a different answer from (B, A)?"
A: Yes, order matters. → I enumerate ordered pairs. A team may appear in both roles across different pairs.
Q: "Can a single-day window startDay == endDay be split?"
A: No — two teams can't both cover a single day as separate segments. → Guard startDay < endDay at the top and return empty.
Q: "Can the availability lists have duplicates?"
A: Possibly. → Convert each list to a Set immediately — deduplication is free and makes membership lookup O(1).
Q: "Should the output be sorted?"
A: Lexicographically by first team name, then second. → Iterate sorted team names for A; sort the candidate B list within each A bucket.
Step 2 — The plan, before any code
The plan, spoken

"This is a split/handoff coverage problem — Variant 8.5. The input is raw day sets, not interval objects, so I can't directly apply the sweep variants. The key observation is that each team's role is fully determined by one number per role. As a first team, the only thing that matters is how far it reaches contiguously from startDay — I'll call this prefixEnd. As a second team, the only thing that matters is the earliest day it can start while still covering contiguously to endDay — I'll call this suffixStart."

"The conditions. prefixEnd[A] is defined only if the team covers startDay. I walk forward from startDay, stopping at the first gap. suffixStart[B] is defined only if the team covers endDay. I walk backward from endDay, stopping at the first gap. The valid pair condition is: suffixStart[B] <= prefixEnd[A] + 1. This means B's contiguous reach to endDay starts at or before the handoff point — B already covers day prefixEnd[A]+1 and everything through endDay. Critically, the condition is ≤ not == — a team covering a wider suffix than needed is still valid."

"The algorithm is three steps. Step 1: compute prefixEnd and suffixStart for every team — O(t · d). Step 2: build a lookup from suffixStart values to team names — O(t). Step 3: for each team A with a prefixEnd, collect all teams B where suffixStart[B] ≤ prefixEnd[A]+1, filter out B==A, and append sorted pairs."

"Data structures: two dictionaries of size t (prefixEnd and suffixStart), one grouping dictionary for the suffix lookup (also size t), one result array. No sort of the full team list needed beyond the initial name sort. Time: O(t · d). Space: O(t). I'll trace Example 2 now."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: every ordered pair × every split pointO(t² · d)Correct and simple to reason about. For t=10 teams and d=365 days: ~13,000 operations — fine in practice. But it recomputes coverage walks for the same team against every partner. Name it, implement it first, then explain the optimization.
Convert to interval objects, apply sweep variantsO(t · d) convert + O(t log t) sortNot directly applicable — each team has potentially many disjoint intervals (not one), and the question is about two-team handoff, not merging. You'd have to derive the same prefix/suffix numbers anyway.
Prefix-suffix precompute (Variant 8.5)O(t · d)Chosen. Compute one number per team per role (O(t · d) total), then pair-check in O(t) using a suffix lookup. Eliminates the per-pair coverage walk entirely.
The deciding signal: "two teams cover the window" + raw day sets. You can't sort-and-sweep because there are no interval objects. The prefix/suffix precompute derives the one interval that matters per team (its effective left-anchor or right-anchor reach) before any pair comparison.
Why this isn't the same as the merge variants (8.1–8.4). In 8.1–8.4, the input is already a list of [start, end] interval objects. Here, the "interval" each team represents must be derived from a raw set of days. The derivation is the prefix/suffix walk — once done, the pair-checking logic is simple index arithmetic.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
availability: [String: Set<Int>]Each team's days as a set for O(1) membership lookupArray lookup would be O(n) per day check — O(d) per walk becomes O(d · a) where a is list length. Set gives O(1) contains, making each walk O(d).
prefixEnd: [String: Int]For teams that cover startDay: the furthest day reachable contiguously from startDayOnly teams covering startDay are valid first-team candidates. Teams not in this dict are skipped in Step 3.
suffixStart: [String: Int]For teams that cover endDay: the earliest day from which they cover contiguously to endDayOnly teams covering endDay are valid second-team candidates. Teams not in this dict never enter suffixMap.
suffixMap: [Int: [String]]Groups teams by their suffixStart value for range-query in Step 3Lets Step 3 collect all Bs with suffixStart ≤ handoffPoint in one iteration over the map's keys — avoids rescanning all teams for each A.
Why Set for availability. The core operation in both the prefix and suffix walks is "does this team cover day X?" on potentially thousands of days. A Set gives O(1) for that lookup; an Array would require O(a) binary search (even if sorted) or O(a) linear scan. The Set construction is a one-time O(a) cost per team — amortized away across d lookups per walk.

8   Pseudocode Dry Run

Pseudocodeprefix-suffix precompute + pair lookup (Variant 8.5)
IF startDay >= endDay: RETURN []

Step 1 — derive the one interval that matters per team per role
FOR each team:
    days ← Set(team.availability)

    IF days contains startDay:
        reachDay ← startDay
        WHILE reachDay < endDay AND days contains (reachDay + 1):
            reachDay += 1
        prefixEnd[team] ← reachDay

    IF days contains endDay:
        reachDay ← endDay
        WHILE reachDay > startDay AND days contains (reachDay - 1):
            reachDay -= 1
        suffixStart[team] ← reachDay

Step 2 — build suffix lookup: suffixStart value → [team names]
FOR each (team, startValue) in suffixStart:
    suffixMap[startValue].append(team)

Step 3 — collect valid pairs in lex order
FOR each firstTeam in sortedTeamNames:
    IF prefixEnd[firstTeam] undefined: CONTINUE
    handoffPoint ← prefixEnd[firstTeam] + 1
    candidates  ← all teams B where suffixStart[B] ≤ handoffPoint   ← ≤ not ==
    FOR each secondTeam in sorted(candidates) where secondTeam ≠ firstTeam:
        result.append([firstTeam, secondTeam])

RETURN result

9   Swift Solution — iterative

A.1 Brute force — every pair × every split point
O(t²·d)O(t·a)
Plan
For every ordered pair (A, B), try every split point in [startDay, endDay). Verify A covers [startDay…split] and B covers [split+1…endDay]. Correct and simple — establish this baseline before the optimization. 
func relayPairsBruteForce(startDay: Int, endDay: Int, teams: [String: [Int]]) -> [[String]] {
    guard startDay < endDay else { return [] }
    let sortedNames  = teams.keys.sorted()
    let availability = Dictionary(uniqueKeysWithValues: teams.map { ($0.key, Set($0.value)) })
    var result: [[String]] = []

    for firstTeam in sortedNames {
        for secondTeam in sortedNames where secondTeam != firstTeam {
            guard let firstDays  = availability[firstTeam],
                  let secondDays = availability[secondTeam] else { continue }

            for splitDay in startDay..<endDay {
                let firstCovers  = (startDay...splitDay).allSatisfy { firstDays.contains($0) }
                let secondCovers = ((splitDay+1)...endDay).allSatisfy { secondDays.contains($0) }
                if firstCovers && secondCovers { result.append([firstTeam, secondTeam]); break }
            }
        }
    }
    return result
}
A.2 Optimal — prefix-suffix precompute (Variant 8.5)
O(t·d)O(t)
Plan
Compute prefixEnd (how far A reaches from startDay) and suffixStart (from where B reaches endDay) once per team. Pair (A,B) is valid when suffixStart[B] ≤ prefixEnd[A]+1. Build a suffix lookup for efficient candidate collection. 
func relayPairs(startDay: Int, endDay: Int, teams: [String: [Int]]) -> [[String]] {
    guard startDay < endDay else { return [] }

    let sortedTeamNames = teams.keys.sorted()
    var prefixEnd:   [String: Int] = [:]
    var suffixStart: [String: Int] = [:]

    // Step 1 — precompute one number per team per role.
    for teamName in sortedTeamNames {
        let days = Set(teams[teamName] ?? [])

        if days.contains(startDay) {
            var reachDay = startDay
            while reachDay < endDay && days.contains(reachDay + 1) { reachDay += 1 }
            prefixEnd[teamName] = reachDay
        }

        if days.contains(endDay) {
            var reachDay = endDay
            while reachDay > startDay && days.contains(reachDay - 1) { reachDay -= 1 }
            suffixStart[teamName] = reachDay
        }
    }

    // Step 2 — group teams by suffixStart value.
    var suffixMap: [Int: [String]] = [:]
    for (teamName, startValue) in suffixStart {
        suffixMap[startValue, default: []].append(teamName)
    }
    for key in suffixMap.keys { suffixMap[key]?.sort() }

    // Step 3 — for each valid first team, collect Bs where suffixStart[B] ≤ handoffPoint.
    var result: [[String]] = []
    for firstTeam in sortedTeamNames {
        guard let handoffDay = prefixEnd[firstTeam] else { continue }
        let handoffPoint = handoffDay + 1

        var candidates: [String] = []
        for (startValue, bucket) in suffixMap where startValue <= handoffPoint {
            candidates.append(contentsOf: bucket)
        }
        candidates.sort()

        for secondTeam in candidates where secondTeam != firstTeam {
            result.append([firstTeam, secondTeam])
        }
    }
    return result
}

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The main constant-factor improvement: instead of scanning all suffixMap entries for each first team (O(t) per first team in Step 3), sort the suffixMap keys and binary-search for all entries ≤ handoffPoint. This reduces Step 3 from O(t²) to O(t log t + output) — meaningful when t is large and most teams are invalid second-team candidates. Asymptotically the precompute step still dominates at O(t · d).
Follow-up: "What if the mission could be covered by exactly k teams instead of two?"
For k=2 the prefix/suffix trick works because there is exactly one handoff point. For k>2 you'd need k−1 handoff points. The natural extension is dynamic programming: define reach[i][team] = the furthest day coverable contiguously by exactly i teams ending at this team. Fill left to right, k layers. This is O(k · t² · d) — the two-team prefix/suffix trick doesn't generalize cleanly beyond k=2.
Follow-up: "What if a team's availability is given as intervals instead of raw day sets?"
The prefix/suffix precompute becomes even simpler: sort each team's intervals by start, then walk forward checking for gaps between consecutive intervals. prefixEnd is the end of the last interval before the first gap after startDay. suffixStart is the start of the first interval after the last gap before endDay. The set construction step is eliminated — you walk O(i) intervals per team instead of O(d) days, where i is the number of intervals per team (usually i << d).
Follow-up: "The output requires sorted pairs — does your approach guarantee that without a final sort?"
Yes, by construction. The outer loop in Step 3 iterates sortedTeamNames for first teams — so pairs are emitted in first-team alphabetical order. Within each first-team bucket, candidates.sort() before the inner loop ensures second teams are also alphabetical. No final sort of the result array is needed.
