You are given an array of intervals where each interval is a pair [start, end]. Merge all overlapping intervals and return the minimal non-overlapping list that covers exactly the same range as the input.

InputOutputWhy
[[1,3],[2,6],[8,10],[15,18]][[1,6],[8,10],[15,18]][1,3] and [2,6] overlap (2 ≤ 3) → merge to [1,6]; the others are disjoint
[[1,4],[4,5]][[1,5]]Adjacent intervals touch at 4 — the overlap condition uses ≤, so they merge
[[1,4]][[1,4]]Single interval — nothing to merge, return as-is

The twist: "overlapping" includes touching endpoints. Two intervals [a,b] and [c,d] overlap whenever c ≤ b — not just c < b. Getting the inequality wrong produces an extra interval in the output for every adjacent pair, and it is the single most common bug.

2   Reading the Problem — insights from the constraints

"Merge all overlapping intervals": the word "all" means you cannot stop after one merge — the result of a merge may itself overlap the next interval. This rules out any per-pair fix-up. You need a single pass that always holds the current merged state.
"Return an array of non-overlapping intervals": the output must cover the same points as the input, with no gaps introduced and no redundant coverage. This tells you the merge rule precisely: when two intervals overlap, the merged interval's end is max(b, d) — never min, never the average.
Input is unsorted: the problem does not guarantee sorted input. An unsorted sweep would require checking every past interval for overlap at each step — O(n²). Sorting by start first reduces this to one comparison per step: if the new interval's start is after the last merged end, it cannot possibly overlap anything earlier.
1 ≤ intervals.length ≤ 10⁴: n up to 10,000 means O(n²) is borderline (10⁸ ops) and O(n log n) is comfortably fast. The sort is the bottleneck — the sweep itself is O(n).
0 ≤ start ≤ end ≤ 10⁴: starts and ends are non-negative and bounded, so no integer overflow. Also: start ≤ end guarantees each interval is well-formed — you never need to guard against inverted intervals.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

The input is unsorted and you need to merge overlapping intervals. O(n²) pair-checking is too slow at n = 10⁴. What single step makes the sweep O(n)?

Hint 2 — the pattern

Sort by start. After sorting, any interval that overlaps the "current" merged interval must start ≤ its end. An interval starting after the current end is guaranteed to be disjoint from everything already processed.

Hint 3 — the structure

Seed a result array with the first sorted interval. For each subsequent interval: if its start ≤ result.last's end, extend result.last's end to max(lastEnd, incomingEnd). Otherwise append it. Be careful about the overlap condition — is it < or ≤?

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Is the input guaranteed to be sorted, or can it arrive in any order?"
A: Any order. → I need to sort by start first — the sweep's correctness depends on sorted order.
Q: "Do touching intervals like [1,3] and [3,5] count as overlapping?"
A: Yes, they should be merged. → The overlap condition is incoming.start ≤ lastEnd — not strict less-than. This is the single most common bug.
Q: "Can a smaller interval be entirely contained within a larger one?"
A: Yes. → I must use max(lastEnd, incomingEnd) when merging — not just set end to incomingEnd. A contained interval must not shrink the merged end.
Q: "Is the input guaranteed non-empty?"
A: Yes, at least one interval. → I can seed my result array with the first sorted interval without a guard check.
Step 2 — The plan, before any code
The plan, spoken

"This is a sort-then-sweep merge problem — Variant 8.1 of the interval pattern. The signal is 'merge all overlapping intervals' on an unsorted input. The O(n²) brute force checks every pair; I beat it by sorting once and then sweeping in a single O(n) pass."

"Let me state the conditions first. The overlap condition is: incoming.start ≤ lastEnd. If that holds, the two intervals overlap and I merge them. The merge rule is: keep the same start, extend the end to max(lastEnd, incoming.end). The non-overlap condition is: incoming.start > lastEnd — the new interval starts strictly after the current merged end, so it is disjoint. I append it fresh."

"The invariant the sort buys: after sorting by start, if an incoming interval's start is after the current merged end, it cannot overlap any earlier interval in the result — because all earlier intervals have an even smaller end. This is why I only need to compare against result.last, never against earlier entries."

"Data structures: a sorted copy of the input (O(n log n)), a mutable result array seeded with the first interval (O(n) space). I mutate result[result.count - 1][1] in place when extending — I never remove and re-append, which would be two allocations."

"Time: O(n log n) — the sort dominates; the sweep is O(n). Space: O(n) — the result holds at most n intervals. I'll trace [[1,3],[2,6],[8,10],[15,18]] → expected [[1,6],[8,10],[15,18]]."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: compare every pairO(n²)For each interval, scan all others for overlap. At n = 10⁴ that is 10⁸ comparisons — technically feasible but far from the expected solution. Name it to establish the baseline.
Event-sweep (start/end events)O(n log n)Correct and elegant, but produces the same asymptotic cost with more bookkeeping. The sort-then-sweep approach is simpler and is what interviewers expect.
Interval treeO(n log n) build, O(log n) queryCorrect but far more complex to implement. Appropriate for online queries against a dynamic set, not for one-shot batch merging.
Sort by start + sweep mergeO(n log n)Chosen. Sort once, then a single linear pass with one comparison and one optional mutation per interval. Simple, correct, and the canonical answer.
The deciding signal: "merge all overlapping intervals" on unsorted input. The sort buys the invariant that lets each step look only at the last merged interval. Without it, every step would need O(n) comparisons.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
sorted: [[Int]]Input intervals sorted by startA new sorted array via .sorted { $0[0] < $1[0] }. Allocates once. If input were guaranteed sorted we'd skip this, but we never assume that.
result: [[Int]]Accumulated merged intervalsSeeded with sorted[0]. Mutated in place at the last index: result[result.count - 1][1] = max(...). Never remove-and-re-append — that is two allocations for one logical operation.
result[result.count - 1][1]The end of the current last merged intervalDirect index mutation is O(1). Prefer this over result.last which is read-only in Swift — you cannot mutate through an optional.
Why no stack or map. The sweep's invariant — "only the last merged interval can overlap the next incoming one" — means we never need to look back further than one step. A stack would work but adds no value over a plain array. A map would be O(n) overhead with no benefit for this problem.

8   Pseudocode Dry Run

Pseudocodesort + sweep merge
sorted ← sort intervals by start value
result ← [sorted[0]]            // seed with the first interval

FOR each interval IN sorted[1..]:
    lastEnd ← result[last index][1]

    IF interval[0] ≤ lastEnd:        // overlap (including touching)
        result[last index][1] ← max(lastEnd, interval[1])   // extend in place
    ELSE:
        result.append(interval)   // disjoint — start a new interval

RETURN result

9   Swift Solution — iterative

A.1 Sort by start + sweep merge
O(n log n)O(n)
Plan
Sort by start. Seed result with the first interval. For each subsequent interval: if its start ≤ result.last's end, extend result.last's end to max(lastEnd, incomingEnd). Otherwise append. Return result. 
func merge(_ intervals: [[Int]]) -> [[Int]] {
    let sorted = intervals.sorted { $0[0] < $1[0] }
    var result = [sorted[0]]

    for interval in sorted.dropFirst() {
        let lastEnd = result[result.count - 1][1]

        if interval[0] <= lastEnd {
            // Overlap — extend the current merged interval's end if needed.
            result[result.count - 1][1] = max(lastEnd, interval[1])
        } else {
            result.append(interval)
        }
    }

    return result
}

PhaseTimeSpaceWhy

SweepO(n)O(1) extraEach interval is visited exactly once. Every step is one comparison and one mutation or one append — all O(1).
Output—O(n)Result holds at most n intervals (the all-disjoint case).

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The asymptotic complexity is optimal — O(n log n) is the lower bound because merging requires sorted order and sorting requires O(n log n) comparisons. A constant-factor improvement: if the input is already sorted, skip the sort step. You could add a linear pre-check pass, but for n ≤ 10⁴ it is rarely worth the extra code. More practically: use reserveCapacity(intervals.count) on the result array to avoid repeated reallocations when the input is mostly disjoint.
Follow-up: "What if the input arrives as a stream and you need to maintain the merged set incrementally?"
The batch sort-then-sweep doesn't apply — you can't sort a stream. You'd maintain a sorted data structure (a balanced BST or sorted array) of non-overlapping intervals. On each new interval: binary-search for the insertion point, then merge leftward and rightward until no more overlaps. Each insertion is O(log n + k) where k is the number of intervals merged away. The total work is still O(n log n) amortized because each interval is merged at most once.
Follow-up: "How does this change if intervals represent half-open ranges [start, end)?"
The overlap condition changes from incoming.start ≤ lastEnd to incoming.start < lastEnd. Adjacent intervals like [1,3) and [3,5) are now disjoint — [3,5) starts exactly where [1,3) ends, with no shared point. Everything else — sort, seed, max-extend — stays the same. Half-open ranges appear in calendar-event problems; always clarify which convention the problem uses.
Follow-up: "What is the next problem in this pattern family?"
LC 57 — Insert Interval applies the same sort-then-sweep insight but to an already-sorted list with a new interval to insert. Because the list is pre-sorted, the sort step is free and the whole solution is O(n). It splits into three regions: before the new interval, overlapping it, and after — see LC57_master.html.
