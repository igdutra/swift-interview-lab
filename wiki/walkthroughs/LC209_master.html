You are given an array of positive integers and a target. Find the length of the shortest contiguous subarray whose sum is greater than or equal to the target — and return 0 if no such subarray exists.

InputOutputWhy
target = 7, nums = [2,3,1,2,4,3]2[4,3] sums to 7 — the tightest valid subarray
target = 4, nums = [1,4,4]1[4] alone already meets the target
target = 11, nums = [1,1,1,1,1,1,1,1]0Entire array sums to 8 — no valid subarray exists

This is the purest form of the minimum-window sliding window: the window state is a single running sum, validity is a single integer comparison, and no frequency maps or auxiliary structures are needed at all. What it does share with its harder siblings — LC 76 especially — is the inversion that trips people up: you shrink the window while it is valid, recording at each step, rather than after it breaks. That word "minimum" is the entire signal.

2   Reading the Problem — insights from the statement

"Minimal length": the word "minimal" does all the work. You are searching for the smallest valid window — so once the running sum reaches the target you must keep shrinking, recording every valid length as you go. This is the exact opposite of longest-window problems, where you only shrink when broken and record after the repair.
"Sum ≥ target": the validity condition is a single integer comparison. There is no character set to track, no frequency map to maintain — just a scalar currentSum that updates in O(1) as the window expands or contracts. This is what makes LC 209 the simplest member of the minimum-window family.
"Positive integers": all elements are strictly positive (nums[i] ≥ 1). This guarantees that shrinking the window always reduces the sum, so the shrink loop terminates correctly. If negatives were allowed, a window could grow valid again after shrinking — the two-pointer approach would break.
"Return 0 if no such subarray": there is a possibility of no answer. The sentinel for "not found" must be chosen carefully — initializing the best-length variable to 0 would corrupt the answer for a real length-1 result. Initialize to Int.max and convert to 0 at the return.
Phrase / constraintWhat it forces
"minimal length"Variable window, shrink while valid, record inside the shrink loop
"sum ≥ target" (single comparison)No frequency map needed — one running sum integer is enough
"positive integers only"Shrinking always reduces the sum → two-pointer works; no negative-element trap
"return 0 if impossible"Init best with Int.max, not 0; convert at the return
n up to 10⁵, values up to 10⁴O(n²) brute force reaches 10¹⁰ — one-pass O(n) required

The Int.max gotcha. Initializing minLength = 0 instead of Int.max silently corrupts every test case: min(0, realAnswer) always returns 0. The debug move that caught this during development: print leftPointer and currentSum at each shrink iteration — the window is visibly valid but the recorded length never updates.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the constraint

n up to 10⁵ rules out checking all O(n²) subarrays. The array has only positive integers — what does that let you do with two pointers over a running sum?

Hint 2 — the pattern

Sliding window — the minimum-length variant. Ask: when the sum reaches the target, do I stop or keep going? Which direction does my while loop run, and where do I record the answer?

Hint 3 — the gotcha

What value do you initialize the best-length variable to? Think about what happens when no valid window exists — and what happens to min(yourInitialValue, 1) if you get that initialization wrong.

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Are all the integers in the array strictly positive, or can I have zeros or negatives?"
A: Strictly positive, at least 1. → Shrinking always reduces the sum, so a two-pointer window is safe. If negatives were allowed I'd need a different approach.
Q: "If no subarray reaches the target sum, should I return 0?"
A: Yes, return 0. → I need a sentinel that doesn't collide with a real length. I'll initialize the answer to Int.max and convert to 0 at the return.
Q: "Is the target inclusive — sum ≥ target, or strictly greater than?"
A: Greater than or equal. → The validity condition is currentSum >= target.
Q: "How large can the array get — should I optimize for time?"
A: Up to 10⁵ elements. → O(n²) brute force is out. I'm aiming for a single O(n) pass.
Step 2 — The plan, before any code
The plan, spoken

"This is a sliding window problem — specifically the minimum-length variant. The signal is the word 'minimal' over a contiguous subarray: I need the shortest window whose sum reaches the target, so once a window becomes valid I have to keep shrinking it to find the tightest possible fit."

"Before I touch a data structure, let me pin the conditions, because they drive everything else. The validity condition is: currentSum >= target. The invalid condition is the mirror: the window goes invalid the moment an eviction drops the sum below the target. The loop runs while valid — shrinking toward a smaller window — and the answer is recorded inside that loop, before each eviction, because that is exactly the moment I hold a valid window. The invariant: every pass through the shrink loop holds a valid window. This is the inversion that trips people up: in longest-window problems you shrink while invalid and record after. Here it flips."

"Those conditions tell me exactly what I need to track. The window state is a single running sum — I add the new right element on expand, subtract the left element on eviction. No frequency map, no set, just three scalars: leftPointer, currentSum, and minSubArrayLength. I initialize minSubArrayLength to Int.max — not 0 — so that min() works correctly on the very first valid window. At the return, if it is still Int.max, I return 0."

"The walk is simple: advance the right pointer over the array, accumulate the sum. The moment currentSum >= target, enter the shrink loop: record the current window length if it beats the best, subtract the left element, advance the left pointer. Repeat until the window goes invalid. Each element enters the window once and leaves at most once."

"Time complexity is O(n) — each element is processed at most twice. Space is O(1) — four scalar variables regardless of input size. I'll code it now and trace target = 7, nums = [2,3,1,2,4,3] to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check every subarrayO(n²)10⁵ input reaches 10¹⁰ operations — dead on arrival, but name it for the baseline
Prefix sums + binary searchO(n log n)Works and handles negative numbers too, but strictly worse here since all values are positive and the two-pointer beats it
Sliding window — shrink while validO(n)Chosen. Positive integers guarantee shrinking always reduces the sum; the minimum-length record inside the while gives the answer in one pass
The deciding signal: "minimum contiguous subarray" + "positive integers" — contiguity makes the window applicable, positivity makes shrinking monotone, and "minimum" flips the record placement inside the while.
Why prefix sums still matter. If the problem asked for the minimum-length subarray summing to exactly a target, or if negatives were allowed, the sliding window breaks and prefix sums with a hash map become the correct approach. The positive-integer constraint is load-bearing.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
currentSum: IntRunning sum of the current windowA single integer suffices because all values are positive and validity is one comparison. No map or set needed.
minSubArrayLength: IntBest (smallest) valid window length seenInitialized to Int.max so min() works on the first valid window and the "no answer" case is distinguishable from a real length of 0.
leftPointer: IntLeft boundary of the current windowIndex into nums, advanced one step per eviction. No deque or buffer needed — direct index access is O(1).
Why this is the simplest minimum-window problem. LC 76 needs two frequency maps and a matchedCount integer to answer "is the window valid?" in O(1). LC 209 answers that same question with a single scalar comparison. Same variant, same control structure — O(1) window state instead of O(k).

Pseudocodeshrink-while-valid, minimum length
minLength ← Int.max; leftPointer ← 0; currentSum ← 0

FOR (rightPointer, rightNumber) IN nums.enumerated():
    currentSum += rightNumber

    WHILE currentSum >= target:          // window is VALID
        minLength ← min(minLength, rightPointer - leftPointer + 1)
        currentSum -= nums[leftPointer]
        leftPointer += 1

RETURN minLength == Int.max ? 0 : minLength

9   Swift Solution — iterative

A.1 Shrink-while-valid window — running sum
O(n)O(1)
Plan
One pass of the right pointer over nums. Expand: add the right element to currentSum. While valid (sum ≥ target): record the window length if it beats the best, subtract the left element, advance leftPointer. Return the best length, or 0 if no valid window was found. 
func minSubArrayLen(_ target: Int, _ nums: [Int]) -> Int {
    var leftPointer = 0
    // GOTCHA: init with Int.max, not 0 — lets min() work on the first valid window
    var minSubArrayLength = Int.max
    var currentSum = 0

    for (rightPointer, rightNumber) in nums.enumerated() {
        currentSum += rightNumber

        // Shrink from the left while the window is valid; record at every valid size
        while currentSum >= target {
            minSubArrayLength = min(minSubArrayLength, rightPointer - leftPointer + 1)
            currentSum -= nums[leftPointer]
            leftPointer += 1
        }
    }

    // GOTCHA: no valid window found → return 0 instead of Int.max
    return minSubArrayLength == Int.max ? 0 : minSubArrayLength
}

12   Complexity

PhaseTimeSpaceWhy
Window passO(n)O(1)Right pointer advances n times; left pointer advances at most n times total (amortized — each element is evicted at most once). Every operation inside the loop is O(1).
TotalO(n)O(1)The bottleneck is reading the input once. No auxiliary structure grows with n.
Why the nested while doesn't make it O(n²). The left pointer only moves forward, never backward. Across the entire run, it advances at most n times — the same as the right pointer. The shrink loop's total work, summed across all outer iterations, is therefore O(n), not O(n) per outer step.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "What if the array can contain negative numbers?"
The sliding window breaks. Shrinking no longer guarantees a smaller sum — you might need to grow again after shrinking. The correct approach switches to prefix sums with a hash map: compute prefix[i], then for each i find the smallest j such that prefix[i] - prefix[j] ≥ target. That runs O(n log n) with a sorted structure or O(n) with a monotone deque.
Follow-up: "Can you solve it in O(n log n) using a different technique — without the sliding window?"
Yes — prefix sums plus binary search. Build prefix[i] = sum of nums[0..i]. For each right index, binary-search the prefix array for the smallest left such that prefix[right] - prefix[left] ≥ target. This generalizes to non-positive inputs and is the fallback when the window doesn't apply, at the cost of an extra log-n factor.
Follow-up: "How would the code change for 'longest subarray with sum at most k'?"
That flips the variant: the while loop becomes "while sum > k" (shrink while invalid), and the record moves to after the while, where the window is guaranteed valid again. The initialization of the best-length variable also flips — start at 0, not Int.max, since 0 is the correct answer when no valid window exists for a maximum problem. Saying this out loud shows ownership of the pattern, not just this problem.
Follow-up: "What if instead of the minimum length you needed the minimum sum that is ≥ target?"
That's just the target itself if a valid subarray exists — any sum ≥ target is valid, and you want the smallest one, which by definition is the first sum that hits the target exactly. If the problem asked for the exact minimum achievable sum ≥ target, you'd track the minimum currentSum at each record step inside the while, rather than the window length.
