Theory
text_formatting_deep_dive_master.html

You are given two non-negative integers represented as strings, num1 and num2. Return their sum, also as a string. The catch is the one that makes it a real problem: you may not convert the strings to Int (or any built-in big-integer type) and you may not use any library that does the addition for you. You add them the way you learned in grade school — column by column, right to left, carrying the overflow.

InputOutputWhy
num1 = "11", num2 = "123""134"11 + 123; the shorter number contributes 0 in its missing high columns
num1 = "456", num2 = "77""533"6+7=13 → write 3, carry 1; 5+7+1=13 → write 3, carry 1; 4+0+1=5
num1 = "0", num2 = "0""0"Degenerate case — the loop must still run once and not produce ""

The twist is that the numbers can be up to 10⁴ digits long — far beyond what any fixed-width integer holds. The whole point is to simulate the arithmetic on the characters directly, which is exactly why the Int conversion is banned.

2   Reading the Problem — insights from the constraints

"You must not convert the inputs to integers directly": this is the constraint that defines the problem. It forces character-level arithmetic — you subtract the ASCII value of "0" from each character to get its numeric value, add, and convert back. The entire technique exists because the natural shortcut is forbidden.
Lengths up to 10⁴: the two numbers need not be the same length. The shorter one runs out of digits before the longer one. Rather than left-padding the shorter string with zeros (an extra allocation), you guard each read: if a pointer has fallen off the left edge, that column contributes 0.
"Non-negative integers": no minus signs, no subtraction. You only ever add, and the carry is only ever 0 or 1. This keeps the column math to a single sum = digit1 + digit2 + carry; write sum % 10; carry = sum / 10.
"No leading zeros, except the number 0 itself": the inputs are clean, so you never strip leading zeros on the way in. But the output must obey the same rule — and the only way to accidentally violate it is the all-zeros case, which the example table flags.
Addition is right-to-left: the least significant digit is at the end of each string. So you walk both strings backward with two independent pointers, and you build the result backward too — appending the low digit each step, then reversing once at the end.

3   Edge Case Inventory

InputExpectedHandled by
"0" + "0""0"The loop condition fires once (both pointers at index 0); writes a single "0". Never produces "" or "00" because the loop runs on the digit, not on a carry.
"1" + "9999" — unequal lengths"10000"The short pointer falls off the left edge and contributes 0; the carry propagates through the remaining columns of the long number.
"99" + "1" — carry to a new column"100"After both pointers exhaust, carry > 0 keeps the loop alive one more turn to emit the leading 1.
"9" + "9" — single-column overflow"18"9+9=18 → write 8, carry 1; both pointers exhausted but carry > 0 writes the 1.
"123" + "0" — one operand zero"123"The zero contributes 0 in its single column; remaining columns of num1 copy through with carry 0.
GOTCHA: forgetting the final carrywrong: "99"+"1" → "00"If the loop stops when both indices run out, the surviving carry is dropped. Putting carry > 0 in the loop condition is what handles it.
The carry-in-the-loop-condition trick is the whole elegance. A naive solution loops while either pointer is in range, then adds a special post-loop if carry > 0 { append("1") }. Folding carry > 0 into the loop condition eliminates that branch entirely — the loop naturally runs one extra time to flush a surviving carry, and exactly zero extra times when there is none.

4   Try It Yourself First

This is a warm-up, but the carry handling has a sharp edge. Try it cold, then open hints one at a time.

Hint 1 — the constraint

You cannot turn the strings into Int. That forces you to add them digit by digit, exactly the way you do by hand: start from the rightmost column, add the two digits plus any carry, write the low digit, carry the rest.

Hint 2 — the pattern

Digit-buffer build. Use two read pointers starting at the last index of each string and a result buffer. Each step: read a digit from each pointer (0 if that pointer has run off the left edge), add them with the carry, append sum % 10, set carry = sum / 10, and step both pointers left.

Hint 3 — the loop condition

Don't stop when the pointers run out — stop when the pointers run out and the carry is gone. Use while left >= 0 || right >= 0 || carry > 0. That single condition handles unequal lengths and a final carry-out without any post-loop special case. Build the buffer backward, then reverse it once at the end.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Are both inputs guaranteed to be valid non-negative integers with no leading zeros, no signs, no whitespace?"
A: Yes — clean digit strings, no leading zeros except the literal "0". → No input validation or trimming; I can treat every character as a digit.
Q: "Can I assume ASCII digits only, so I can subtract the code point of '0'?"
A: Yes, only '0'–'9'. → I'll use Character.wholeNumberValue (or ASCII subtraction) to get each digit's value — no Int conversion of the whole string.
Q: "The two numbers might be different lengths — should I handle that without padding the shorter one?"
A: Yes, lengths can differ by a lot. → Independent pointers with a bounds guard that contributes 0 when a pointer falls off, rather than allocating a padded copy.
Q: "How large can the inputs be — is there a risk the sum overflows a machine integer?"
A: Up to 10⁴ digits each — far beyond Int range. → That's exactly why I must stay at the character level; there is no Int that holds these.
Step 2 — The plan, before any code
The plan, spoken

"This is a digit-buffer build — Variant 8.1 of the Text Formatting and Simulation family. The signal is 'do arithmetic on numbers given as strings, with no BigInt allowed.' I simulate grade-school column addition right to left."

"Let me pin the governing conditions before naming any structure. The loop's continuation condition is the heart of the problem: I keep going while there is still a digit to read in either number, or a carry left to flush — left >= 0 || right >= 0 || carry > 0. The invariant each pass maintains is that everything to the right of the current column is already correctly summed and sitting in the buffer, and carry holds exactly the overflow from that already-summed suffix. The stop condition is the negation: both pointers off the left edge AND no carry — at that moment the buffer holds the complete reversed result."

"The structures follow from those conditions. I need two read pointers, left and right, starting at the last index of each string, because the conditions reference 'a digit still to read in either number.' I need one Int carry, because the invariant says I carry exactly the suffix overflow and that is never more than 1. And I need a result buffer — an array of characters I append the low digit to each step, because the loop produces digits in reverse order and I want O(1) appends rather than O(n) string prepends."

"The walk: each pass, read digit1 from left (or 0 if left < 0), digit2 from right (or 0 if right < 0), compute sum = digit1 + digit2 + carry, append sum % 10 to the buffer, set carry = sum / 10, and decrement both pointers. When the loop's stop condition holds, reverse the buffer and join it into a string. Time is O(max(n, m)) — one pass over the longer number; space is O(max(n, m)) for the result buffer."

6   Why This Approach — and not the others

CandidateCostVerdict
Convert both to Int, add, stringifyO(n)Disqualified. Banned by the problem, and impossible anyway — 10⁴-digit numbers do not fit in any fixed-width integer. This is the trap the constraint is built to block.
Left-pad the shorter string with zeros, then add column by columnO(n) time, O(n) extra for the padded copyWorks and is correct, but the padding is a wasted allocation. A bounds guard contributing 0 achieves the same with no extra string.
Build the result with string prepend ("3" + result)O(n²)Correct order without a reverse, but each prepend copies the whole accumulated string — quadratic. Append-then-reverse is linear.
Two pointers, carry in the loop condition, append + reverseO(max(n,m)) / O(max(n,m))Chosen. One pass, no padding, no post-loop carry branch, O(1) amortized appends. The loop condition does all the bookkeeping.
The deciding signal: "numbers as strings, no BigInt." Any time a problem hands you arithmetic on arbitrarily long numeric strings and forbids the integer conversion, reach for the right-to-left digit-buffer walk with a carry. It is the same skeleton for Add Strings, Add Two Numbers (linked-list form), Multiply Strings, and Plus One.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
digits1: [Character], digits2: [Character]The two input strings as random-access arraysSwift String has no O(1) integer subscript — indexing is O(n) per access. Materializing each string into a [Character] once gives O(1) indexing for the pointer walk, turning a potential O(n²) into O(n).
left: Int, right: IntCurrent column index in each number, walking right to leftTwo independent indices, not one shared loop counter, because the numbers differ in length. Each is guarded so a negative index contributes the digit 0.
carry: IntThe overflow from the column just summed (always 0 or 1)A single Int, not a stack or queue — addition of non-negative numbers never carries more than 1. Living in the loop condition, it also drives the final flush.
result: [Character]The output digits, accumulated least-significant firstAn array with O(1) append, reversed once at the end. Building a String by prepending would be O(n²); appending to an array and reversing is O(n).
Why convert String to [Character] first. In Swift, num1[someIndex] requires a String.Index, and computing the index for position k is an O(k) walk from the start. Doing that inside a loop makes the whole algorithm O(n²). Converting to [Character] (or [UInt8] via Array(num1.utf8)) once up front buys true O(1) random access for the rest of the function.

8   Pseudocode Dry Run

Pseudocoderight-to-left column addition with carry folded into the loop
left  ← lastIndex(num1)
right ← lastIndex(num2)
carry ← 0
result ← empty buffer

WHILE left ≥ 0 OR right ≥ 0 OR carry > 0:
    d1 ← (left ≥ 0)  ? digitAt(num1, left)  : 0
    d2 ← (right ≥ 0) ? digitAt(num2, right) : 0
    sum ← d1 + d2 + carry
    append(result, sum mod 10)        // low digit, buffer grows backward
    carry ← sum / 10                  // 0 or 1
    left  ← left  − 1
    right ← right − 1

RETURN reverse(result) as string

9   Swift Solution — iterative

A.1 Two-pointer carry walk — append and reverse
O(max(n,m))O(max(n,m))
Plan
Materialize both strings as character arrays for O(1) indexing. Walk two pointers from the right ends. Each step, read a digit from each (0 if a pointer has fallen off the left edge), add them with the carry, append the low digit to the result buffer, update the carry. Keep looping while either pointer is in range or a carry survives. Reverse the buffer and join it into the answer. 
func addStrings(_ num1: String, _ num2: String) -> String {
    let digits1 = Array(num1)
    let digits2 = Array(num2)
    var leftIndex  = digits1.count - 1
    var rightIndex = digits2.count - 1
    var carry = 0
    var resultDigits: [Character] = []

    // Invariant: everything to the right is summed; carry holds that suffix's overflow.
    while leftIndex >= 0 || rightIndex >= 0 || carry > 0 {
        let digit1 = leftIndex  >= 0 ? digits1[leftIndex].wholeNumberValue! : 0
        let digit2 = rightIndex >= 0 ? digits2[rightIndex].wholeNumberValue! : 0
        let columnSum = digit1 + digit2 + carry
        resultDigits.append(Character(String(columnSum % 10)))
        carry = columnSum / 10
        leftIndex  -= 1
        rightIndex -= 1
    }

    return String(resultDigits.reversed())
}

12   Complexity

PhaseTimeSpaceWhy
Array materializationO(n + m)O(n + m)Array(num1) and Array(num2) copy each string once into character arrays.
Carry walkO(max(n, m))O(max(n, m))The loop runs once per column of the longer number (plus at most one extra turn for a final carry). The result buffer grows to at most max(n,m)+1 digits.
TotalO(max(n, m))O(max(n, m))Dominated by a single linear pass. The result string is irreducible — the output itself is that long, so no algorithm can do better in space.
Why the += -on-String pitfall matters here. If you build the answer as a String by prepending (result = String(columnSum % 10) + result), each prepend copies the entire accumulated string, making the build O(n²). Appending to a [Character] and reversing once at the end keeps it O(n). This is the single performance trap in an otherwise easy problem.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
Two constant-factor wins. First, work on Array(num1.utf8) instead of Array(num1) — UInt8 bytes are cheaper than Character values, and the digit is just byte - 48. Second, I could pre-size the result buffer with reserveCapacity(max(n, m) + 1) to avoid reallocations as it grows. The asymptotics stay O(max(n, m)); these just shave the constant.
Follow-up: "Now the inputs are two linked lists of digits, least-significant digit first — add them."
That is LC 2 — Add Two Numbers, the same carry walk in a different container. The list is already stored least-significant-first, so there's no reversal at all: walk both lists forward in lockstep, build the result list node by node, and the same while l1 != nil || l2 != nil || carry > 0 condition flushes the final carry. The algorithm is identical; only the data structure changes.
Follow-up: "Multiply the two numeric strings instead of adding them."
That's LC 43 — Multiply Strings. The trick is that the product of digit i of num1 and digit j of num2 lands in result positions i+j and i+j+1. Allocate a result array of length n + m, accumulate each pairwise product into those two positions with carry, then strip the leading zero. It reuses this problem's per-digit carry handling, applied on a grid of digit pairs rather than a single column.
Follow-up: "What if the numbers can be negative, given with a leading minus sign?"
Parse the signs first. Same signs → add the magnitudes (this exact routine) and reattach the sign. Different signs → subtract the smaller magnitude from the larger (a borrow walk, the mirror of this carry walk) and take the sign of the larger. The addition core is unchanged; you add a magnitude comparison and a subtract-with-borrow variant.
