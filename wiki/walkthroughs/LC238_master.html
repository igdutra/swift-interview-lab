Theory
prefix_suffix_master.html

Given an integer array nums, return an array result where result[i] equals the product of every element in nums except nums[i]. You must solve it without using division and in O(n) time.

InputOutputWhy
[1, 2, 3, 4][24, 12, 8, 6]result[0]=2×3×4=24; result[1]=1×3×4=12; result[2]=1×2×4=8; result[3]=1×2×3=6
[-1, 1, 0, -3, 3][0, 0, 9, 0, 0]The zero at index 2 makes every other result 0; result[2] = (−1)×1×(−3)×3 = 9
[2, 3][3, 2]Minimal case: result[0]=nums[1], result[1]=nums[0]

The twist: the "no division" constraint rules out the obvious approach of computing the total product and dividing by each element. Division also breaks on zeros — if nums contains a zero, the total product is zero, and dividing by any element is either 0/nums[i] (wrong for the zero's own position) or division-by-zero. The prefix × suffix construction avoids division entirely: for each index i, the left pass delivers the product of everything to the left, and the right pass delivers the product of everything to the right. Neither pass includes nums[i] itself — exclusion is structural, not arithmetic.

2   Reading the Problem — insights from the statement

"Without using division": this single constraint eliminates the naïve O(n) solution (totalProduct / nums[i]) and hints directly at the answer. Without division, you must construct each result[i] from two separate pieces — the product of everything to its left and the product of everything to its right. That two-sided dependency is the prefix/suffix fingerprint.
"Product of all elements except nums[i]": the element at position i is absent. This is not "product of a range" — it is "product of the complement." The complement splits cleanly into left-of-i and right-of-i, and each half can be precomputed in one pass.
Zeros in the array: the constraint says nums[i] can be 0. This would break a division-based solution in two ways: if more than one zero exists, all products are 0; if exactly one zero exists, only the result at the zero's index is non-zero. The prefix × suffix approach handles both cases without any branching — zeros propagate through the product arrays exactly like any other value.
"O(n) time, O(1) extra space" (the follow-up): the output array result is not counted toward the space limit. This is a hint to reuse the output array as the prefix storage in the first pass and update it in-place with a running suffix product in the second pass.

"except nums[i]"Position i is excluded from both the left and right accumulations by construction — no need to track "which index to skip"
zeros allowed in numsDivision would fail; prefix × suffix propagates zeros correctly with no special case

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — why not division?

The naïve solution is: compute the total product, then result[i] = totalProduct / nums[i]. Write down why this fails when nums contains a zero. If you can't use division, what two pieces of information about position i do you need?

Hint 2 — the two-pass shape

result[i] = (product of nums[0..i−1]) × (product of nums[i+1..n−1]). Both halves can be computed in separate passes: a left-to-right pass builds the left products, and a right-to-left pass builds the right products. Neither pass ever includes nums[i] itself. How does this work for the first and last index?

Hint 3 — O(1) extra space

In the left pass, write the prefix products directly into the result array. Then do a single right-to-left pass with a running suffix product (one integer variable), multiplying it into result[i] at each step. You end up with result[i] = prefixProduct[i] × suffixProduct[i] without ever allocating a second array.

5   Interview Simulation

The interviewer has stated the problem. The script starts with your first words — practice it out loud.

Step 1 — Clarifying questions
Q: "Can the array contain zeros? How many?"
A: Yes, zeros are allowed and there can be multiple. → This rules out any division-based approach entirely, even if the constraint didn't — dividing by zero is undefined, and with two or more zeros every result is 0 anyway. My approach must produce correct results without division.
Q: "Can elements be negative?"
A: Yes. → Integer multiplication handles negatives correctly. No sign tracking needed.
Q: "What is the minimum array length?"
A: Two elements. → So every index always has at least one other element contributing to its result. No empty-product edge case at both ends simultaneously — the sentinel handles each end independently.
Q: "The O(1) extra space constraint — does the output array count?"
A: No, the output array is not counted. → I can reuse the output array for the prefix pass and then update it in-place during the suffix pass. That's two passes total, O(1) extra space excluding the output.
Step 2 — The plan, before any code
The plan, spoken

"This is a prefix × suffix product problem — Variant 8.2. The signal is 'except self' combined with 'no division': I need to exclude exactly one element per position, and I cannot invert the total. That forces me to build the result from two independent accumulations."

"Let me pin the conditions. For any index i, result[i] = (product of nums[0..i−1]) × (product of nums[i+1..n−1]). The governing invariant is that index i must be absent from both halves — not skipped, just structurally excluded by the boundary of each pass. The left half stops before i; the right half starts after i."

"The sentinel is identity: the product of zero elements is 1 — the multiplicative identity. So prefix[0] = 1 (there's nothing to the left of index 0) and suffix[n−1] = 1 (there's nothing to the right of index n−1). These sentinels mean the formula works uniformly at the endpoints."

"For O(1) extra space, I'll reuse the output array. First pass left-to-right: result[i] = running prefix product of nums[0..i−1]; starts at 1. Second pass right-to-left: maintain a running suffix product integer, initialized to 1; multiply result[i] by it at each step, then update the running suffix with nums[i]. After the second pass, result[i] holds prefix × suffix — exactly the product of everything except nums[i]."

"Data structures: the output array (not counted toward space) and one integer variable for the suffix pass. No extra arrays. Two linear passes, O(n) time, O(1) extra space."

"I'll trace the canonical example [1,2,3,4] to verify."

6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: for each index, multiply all other elementsO(n²) time, O(1) extraCorrect but quadratic — n=10⁵ gives 10¹⁰ multiplications
Total product then divide by nums[i]O(n) time, O(1) extraFails with zeros (division by zero; wrong result when multiple zeros exist). Explicitly forbidden by the constraint "no division."
Two extra arrays (prefix and suffix)O(n) time, O(n) extraCorrect and readable — the natural two-pass form. Extra space is O(n) because both arrays are stored. Good starting point; the O(1) version is a refinement of this.
Prefix pass into result + rolling suffix variableO(n) time, O(1) extraChosen. Reuse the output array for the prefix pass; a single integer carries the suffix product right-to-left. Achieves the follow-up constraint. Zero-safe, division-free, two passes.
THE ONE IDEA: Position i is excluded from result[i] structurally — the left pass computes the product of nums[0..i−1] and the right pass computes the product of nums[i+1..n−1]. Neither pass ever sees nums[i]. No division, no zero-check, no skip logic. The "gap" at position i is simply where the two passes don't overlap.
When to reach for prefix × suffix product (Variant 8.2): whenever you need "product of all elements except one" and cannot use division. The two sentinels of 1 (multiplicative identity) make the formula work uniformly at both ends without special-casing.
PITFALLS
▸Using additive identity (0) instead of multiplicative identity (1) for the sentinel. The sentinel must be the identity for the accumulation operation. For sums, it's 0. For products, it's 1. Initializing runningSuffix to 0 would zero out every result in the second pass.
▸Updating the running suffix before multiplying into result. In the right-to-left pass, the correct order is: (1) multiply result[index] by runningSuffix, then (2) update runningSuffix with nums[index]. Reversing the order includes nums[index] in result[index], which is exactly what you must avoid.
▸Reaching for division as the "obvious" O(n) solution. It works on arrays with no zeros and the constraint permits it — but both conditions are violated here. The problem explicitly says no division. Always check: are zeros possible? Is division forbidden? If either is true, use two-pass prefix/suffix.

7   Data Structures — choice and justification

VariableHoldsWhy this and not something else
result: [Int] (length n)First: prefix product of everything to the left of each index. After second pass: final result[i] = prefix × suffix at index i.Reusing the output array avoids allocating a separate prefix array — this is the O(1) extra space optimization. The output array is not counted toward the space budget per the problem statement.
runningSuffix: IntRunning product of everything to the right of the current index, maintained right-to-left in the second pass. Initialized to 1 (multiplicative identity).A single integer replaces an entire suffix array. At each step it holds exactly what suffixProduct[index] would contain in the two-array version. It is discarded at the end of the second pass.
The two-array connection: in the O(n) space version you would allocate prefixProduct[index] = product of nums[0..index−1] and suffixProduct[index] = product of nums[index+1..n−1] separately, then combine them. In this solution, result[index] after the first pass contains exactly prefixProduct[index], and runningSuffix before the multiply step equals suffixProduct[index]. Same values, one fewer array.

8   Pseudocode Dry Run

Pseudocodeprefix × suffix product — two-pass O(1) space
n ← length(nums)
result ← array of size n

① LEFT PASS — prefix products into result
runningPrefix ← 1                      // multiplicative identity
FOR index FROM 0 TO n−1:
    result[index] ← runningPrefix      // product of everything LEFT of index
    runningPrefix ← runningPrefix * nums[index]

② RIGHT PASS — multiply suffix products into result
runningSuffix ← 1                      // multiplicative identity
FOR index FROM n−1 DOWNTO 0:
    result[index] ← result[index] * runningSuffix   // combine prefix × suffix
    runningSuffix ← runningSuffix * nums[index]

RETURN result

Symmetrically at index 3, the prefix value is 6=1×2×3 and the suffix is 1 (nothing to the right). The sentinels handle both endpoints uniformly.

9   Swift Solution

A.0 Two explicit arrays — prefix array + suffix array
O(n)O(n) extra
Plan
Allocate two arrays of size n: prefixProducts[i] = product of nums[0..i−1], suffixProducts[i] = product of nums[i+1..n−1]. Fill prefixProducts left-to-right and suffixProducts right-to-left, each initialized with the multiplicative identity 1. Then combine: result[i] = prefixProducts[i] × suffixProducts[i]. Three passes, O(n) extra space. This is the natural reading of the formula — understand it here before optimizing to A.1. 
func productExceptSelf(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var prefixProducts = [Int](repeating: 1, count: count)
    var suffixProducts = [Int](repeating: 1, count: count)

    // Left pass: prefixProducts[i] = product of nums[0..i−1].
    // prefixProducts[0] = 1 (nothing to the left — multiplicative identity).
    var runningPrefix = 1
    for (index, num) in nums.enumerated() {
        prefixProducts[index] = runningPrefix
        runningPrefix *= num
    }

    // Right pass: suffixProducts[i] = product of nums[i+1..n−1].
    // suffixProducts[n−1] = 1 (nothing to the right — multiplicative identity).
    var runningSuffix = 1
    for (index, num) in nums.enumerated().reversed() {
        suffixProducts[index] = runningSuffix
        runningSuffix *= num
    }

    // Combine: result[i] = prefixProducts[i] × suffixProducts[i].
    return zip(prefixProducts, suffixProducts).map(*)
}
A.0 → A.1 connection: in A.1, result[index] after the left pass holds exactly prefixProducts[index] from A.0, and runningSuffix before the multiply step holds exactly suffixProducts[index]. A.1 eliminates the suffix array by computing and consuming each suffix value on the fly, right-to-left. Same correctness, half the extra allocations.
A.1 Prefix pass into result + rolling suffix variable
O(n)O(1) extra
Plan
Left pass: fill result[i] with the running product of all elements to the left of i (result[0] = 1, the multiplicative sentinel). Right pass: walk right-to-left with a single running suffix product, multiplying it into result[i] before updating it with nums[i]. After both passes, result[i] = leftProduct[i] × rightProduct[i] = product of everything except nums[i]. 
func productExceptSelf(_ nums: [Int]) -> [Int] {
    let count = nums.count
    var result = [Int](repeating: 1, count: count)

    // Left pass: result[i] = product of nums[0..i−1].
    // result[0] stays 1 — the multiplicative identity (nothing to the left).
    var runningPrefix = 1
    for (index, num) in nums.enumerated() {
        result[index] = runningPrefix
        runningPrefix *= num
    }

    // Right pass: multiply result[i] by the running product of nums[i+1..n−1].
    // Multiply BEFORE updating runningSuffix so nums[i] is never included in result[i].
    var runningSuffix = 1
    for (index, num) in nums.enumerated().reversed() {
        result[index] *= runningSuffix
        runningSuffix *= num
    }

    return result
}

Bottleneck: you must read every element at least once to compute any result — O(n) is optimal. The O(1) extra space bound is tight: you need at minimum one rolling variable to carry the suffix product without materializing an array.

13   Follow-Up Bank

Follow-up: "Before you push this to production — can you see any improvements you'd make?"

The solution is already optimal: O(n) time, O(1) extra space, zero-safe, division-free. A minor readability note: the comment "Multiply BEFORE updating runningSuffix" is load-bearing — the order of the two lines in the right pass is a correctness constraint, not a style choice. I'd keep that comment in a production codebase. No algorithmic improvement is possible.

Follow-up: "What if the problem allowed division — would you use it?"

Only with caution. The division-based approach (total product / nums[i]) is O(n) and O(1) extra space — same complexity. But it fails when any nums[i] is zero, requiring special-casing: zero count = 0 → all zeros; zero count = 1 → only the position with the zero gets a non-zero result. The two-pass version handles zeros automatically with no branching, so it's strictly simpler and more robust even when division is permitted.

Follow-up: "Can you do this with only one pass?"

No — not for the general case. To compute result[i] you need both the product of all elements to its left and the product of all elements to its right. With only one pass in one direction, you have one piece but not the other. Two passes is the theoretical minimum for this two-sided dependency (same reasoning as LC 42's need for both a prefix-max and a suffix-max).

Follow-up: "What if instead of product you needed the sum — how would the code change?"

Almost nothing changes structurally. Replace the multiplicative identity sentinel (1) with the additive identity (0), replace *= with += in both passes, and replace result[i] *= runningSuffix with result[i] += runningSuffix in the combine step. But for sum-except-self you'd usually just compute total − nums[i] in one pass, which is simpler and works with no special cases. The prefix/suffix structure is only necessary for "product except self" because products have no easy inversion when zeros are present.

14   Theory Decoder — concepts used here

ConceptHow it appears in LC 238
Prefix × suffix product (Variant 8.2)Two passes over nums. Left pass: result[i] = product of nums[0..i−1]. Right pass: multiply result[i] by product of nums[i+1..n−1]. Both passes together give "product except self."
Multiplicative identity sentinel (1)runningPrefix and runningSuffix both start at 1. At index 0, the "product of nothing to the left" is 1 — correct because 1 is the identity for multiplication. Without the sentinel, the first and last elements would require special cases.
Structural exclusion of position inums[i] is never multiplied into result[i]: the left pass stops before updating runningSuffix at i (uses the old value), and the right pass multiplies result[i] by runningSuffix before incorporating nums[i]. The gap is structural, not a skip condition.
Rolling variable as space optimizationrunningSuffix is the O(1)-space replacement for a full suffix product array. At each step it holds exactly what suffixProduct[index] would in the two-array version. Compare to LC 42's two-pointer solution replacing prefix-max arrays with two running scalars.
Zero-safetyThe prefix and suffix products accumulate zeros naturally. No branching is needed — zeros propagate through multiplication exactly like any other value. The division-based approach requires explicit zero-count logic; this approach does not.
