You are given a list of employees, each with a list of non-overlapping work intervals sorted by start. Find all time slots where no employee is working — the common free time across everyone. Return those free intervals sorted by start.

InputOutputWhy
Employee 1: [[1,3],[6,7]]
Employee 2: [[2,4]]
Employee 3: [[2,5],[9,12]]		 [[5,5],[7,8]] Merged union = [1,5],[6,7],[9,12]. Gaps: [5,5] between 5 and 6? Wait — [1,5] ends at 5, [6,7] starts at 6: gap=[5+1,6-1]=[6,5]? No — the gap is the open range between the end of one and the start of the next: gap start = merged[i-1].end+1, gap end = merged[i].start-1. So [1,5]→[6,7]: gap=[6,5]... Let's use example 1 below.
Employee 1: [[1,3],[6,7]]
Employee 2: [[2,4],[9,12]]		 [[4,5],[7,8]] All intervals: [1,3],[6,7],[2,4],[9,12]. Sorted: [1,3],[2,4],[6,7],[9,12]. Merged: [1,4],[6,7],[9,12]. Gaps: [5,5] and [8,8].
Employee 1: [[1,3]]
Employee 2: [[3,6]]		 [] Merged union = [1,6] — continuous coverage, no gaps

The twist: the input is nested — a list of lists — and each employee's schedule is sorted within itself, but schedules across employees may interleave arbitrarily. The difficulty rating comes from recognizing that "free time" is just the complement of the merged union. Once you see that, the algorithm is: flatten all intervals, apply the LC 56 merge, then a second linear pass to collect the gaps.

2   Reading the Problem — insights from the constraints

"List of employees, each with a schedule": the input is nested — [[Interval]], not [Interval]. The first step is always to flatten: discard the employee grouping entirely. For the purpose of finding free time, it does not matter which employee is working — only whether anyone is working.
"Each employee's schedule is sorted and non-overlapping": sorted within each employee, but not across employees. After flattening you have a potentially unsorted array. You must sort before sweeping — this is Variant 8.1 applied after the flatten.
"Time where no employee is working": free time is exactly the complement of the merged union of all schedules. Once you have the merged list, the free time is the gap between every pair of consecutive merged intervals. Gap formula: gapStart = merged[i-1].end + 1, gapEnd = merged[i].start - 1. A gap exists when gapStart ≤ gapEnd.
The gap formula in integer coordinates: because intervals represent discrete time slots (or the problem uses integer endpoints), a gap between [1,4] and [6,7] is [5,5] — the single slot 5. In continuous-time problems the gap would be the open interval (4,6). Always confirm with the interviewer whether endpoints are inclusive integers.
Two-phase structure: Phase 1 is the LC 56 flatten-and-merge (O(n log n)). Phase 2 is a linear gap-collection pass (O(n)). The sort in Phase 1 dominates. The "Hard" difficulty is entirely about recognizing this decomposition — neither phase is algorithmically novel once you see it.

4   Try It Yourself First

Attempt the problem before reading on. Open hints one at a time.

Hint 1 — the structure

The input is a list of lists of intervals. "Free time" means no one is working. Does it matter which employee owns each interval? What would the problem look like if you forgot about employees entirely?

Hint 2 — the reduction

Flatten all schedules into one array. Sort by start. Merge overlapping intervals (LC 56 exactly). Now you have the merged "busy" union. Free time is everything that isn't busy — can you find it from the merged list?

Hint 3 — the gap formula

Walk the merged list. Between each consecutive pair merged[i-1] and merged[i], the gap is [merged[i-1].end + 1, merged[i].start - 1]. Record it when gapStart ≤ gapEnd. The gap pass runs in O(n) after the O(n log n) merge.

The plan, spoken

"This is a flatten-merge-gap problem — Variant 8.4. The key observation: 'free time' is the complement of the merged union of all work intervals. Once I have the merged union, the free time is exactly the gaps between consecutive merged intervals. 6   Why This Approach — and not the others

CandidateCostVerdict
Brute force: check every time slotO(T × n)Walk every possible time unit T; for each, check whether any employee covers it. O(T × n) — impractical when T can be up to 10⁸.
Per-employee free time, then intersectO(n²) in the worst caseCompute each employee's complement, then intersect pairwise. The pairwise intersection requires two-pointer walks — O(n) per pair, O(n × employees) total. More complex and no better asymptotically.
Min-heap k-way mergeO(n log k)Merge k sorted employee lists with a min-heap (k = number of employees). Avoids the flatten-and-sort step — but more complex to implement correctly and offers no asymptotic improvement when k is small.
Flatten + sort + sweep merge + gap passO(n log n)Chosen. Reduces to two well-understood operations (LC 56 merge + linear gap scan). The flatten step is O(n); the sort dominates at O(n log n). Simple, correct, and the canonical expected answer.
The deciding signal: "find time where no employee is working" across nested interval lists. Once you reframe it as "find gaps in the merged union," the algorithm is LC 56 plus one more linear pass. The "Hard" difficulty is entirely in recognizing that reframe.

7   Data Structures — choice and justification

StructureHoldsWhy this and not weaker/stronger
allIntervals: [[Int]]Flattened and sorted array of all work intervals across all employeesschedule.flatMap { $0 } then .sorted { $0[0] < $1[0] }. The flatten discards employee identity; the sort enables the merge sweep.
merged: [[Int]]Result of the LC 56 sweep — minimal non-overlapping union of all busy timeSame structure as the LC 56 result array: seeded with allIntervals[0], mutated in place at the last index. At most n entries.
freeTimes: [[Int]]Output — the gaps between consecutive merged intervalsBuilt by the gap pass. Automatically sorted because the merged list is sorted and we walk it in order.
Why flatten instead of keeping the nested structure. The nested structure encodes employee identity. Once you recognize that "free time" only depends on the union of all busy time — not on who is busy — employee identity becomes irrelevant. The flatten step is the conceptual leap; the merge step is mechanical.

allIntervals ← flatten all employee schedules into one array
sorted       ← sort allIntervals by start

9   Swift Solution — iterative

A.1 Flatten + sort + sweep merge + gap pass
O(n log n)O(n)
Plan
Flatten all employee schedules into one array, sort by start, apply the LC 56 sweep merge to build the busy union. Then walk the merged list collecting gaps between consecutive intervals. The gap formula is [merged[i-1].end+1, merged[i].start-1]. 
func employeeFreeTime(_ schedule: [[[Int]]]) -> [[Int]] {
    // Phase 1 — flatten and merge to build the busy union.
    let allIntervals = schedule.flatMap { $0 }.sorted { $0[0] < $1[0] }

    guard !allIntervals.isEmpty else { return [] }

    var merged = [allIntervals[0]]
    for interval in allIntervals.dropFirst() {
        let lastEnd = merged[merged.count - 1][1]
        if interval[0] <= lastEnd {
            merged[merged.count - 1][1] = max(lastEnd, interval[1])
        } else {
            merged.append(interval)
        }
    }

    // Phase 2 — collect gaps between consecutive merged intervals.
    var freeTimes: [[Int]] = []
    for index in 1..<merged.count {
        let gapStart = merged[index - 1][1] + 1
        let gapEnd   = merged[index][0]   - 1
        if gapStart <= gapEnd {
            freeTimes.append([gapStart, gapEnd])
        }
    }

    return freeTimes
}

12   Complexity

PhaseTimeSpaceWhy
FlattenO(n)O(n)flatMap visits every interval once; allocates a new flat array of n elements.
SortO(n log n)O(n)Swift's Timsort on the flat array — O(n log n) worst case; O(n) auxiliary space.
Sweep mergeO(n)O(n)Each of the n intervals is visited exactly once; one comparison and one mutation or append per step.
Gap passO(m)O(g)m = number of merged intervals (≤ n); one comparison per pair. g = number of gaps ≤ m − 1.
TotalO(n log n)O(n)Sort dominates. n = total intervals across all employees. The flatten, merge, and gap pass are each O(n) — unavoidable since you must read every interval. The sort is irreducible: without sorted order the merge sweep's invariant breaks.

13   Follow-Up Bank — the interviewer's second act

Follow-up: "Before you push this to production — can you see any improvements you'd make?"
The asymptotic complexity is optimal — O(n log n) is unavoidable because the cross-employee intervals are unsorted. A constant-factor improvement: instead of building the flat array and then sorting, use a k-way merge with a min-heap (where k = number of employees). Each employee's schedule is already sorted, so a min-heap of size k produces the globally sorted order in O(n log k) — better when k << n. In the worst case k = n and this degenerates to O(n log n).
Follow-up: "How would you use a min-heap instead of flattening and sorting?"
Seed the heap with the first interval from each employee's schedule (heap key = interval start). Then: pop the minimum interval, apply the merge logic (same as Phase 1 above), and push the next interval from that employee's schedule. Repeat until the heap is empty. This processes intervals in sorted order without materializing the full flat sorted array — O(n log k) time, O(k) extra space for the heap. The merge and gap-pass logic are identical.
Follow-up: "What if employees can also have overlapping intervals within their own schedule?"
The flatten-then-merge approach handles this automatically — the merge sweep doesn't care whether overlapping intervals come from the same employee or different employees. You'd need to also sort within each employee's schedule before flattening, which adds O(d log d) per employee (d = per-employee interval count). The overall complexity stays O(n log n).
Follow-up: "What if the problem returns the busiest time instead of free time?"
The merged array from Phase 1 is the busy union — return it directly, skipping Phase 2 entirely. If the question asks for the slot with the highest number of simultaneous employees (not just busy vs. free), that requires a different approach: event-based sweep counting open/close events, or a difference array over the time axis.
