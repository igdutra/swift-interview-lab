Stages of an Interview

Review & Reference · Process & Soft Skills

Most algorithmic interview rounds are 45–60 minutes. Each stage has specific actions that maximize your chances of success.

1 Introductions

At the start, the interviewer briefly introduces themselves, then asks you to do the same.

Prepare a 30–60 second intro covering education, work experience, and interests. Rehearse it until it feels natural.
  • Smile and speak with a confident voice.
  • Pay attention when the interviewer describes their work — use it to ask better questions at the end.
  • If you share an interest with the interviewer, mention it naturally.

2 Problem Statement

The interviewer reads the problem (often pasting it into a shared editor). Your goal: fully understand it before writing a single line.

Always paraphrase the problem back to the interviewer after they read it. This confirms understanding and shows communication skill.

Clarifying questions to ask:

  • Will the input only have integers, or could there be other types?
  • Will the input be sorted or unsorted?
  • Is the input guaranteed to have elements, or could it be empty?
  • What should I do if an invalid input is given?
  • What is the expected input size? (Small n → backtracking; ~100–1000 → O(n²) may be fine; large → need better than O(n).)
Walk through one of the provided example test cases out loud to confirm your understanding before moving on.

3 Brainstorming DS&A

Figure out which data structure or algorithm applies by breaking down what the problem needs and what can accomplish it efficiently.

Think out loud. Even a wrong idea is valuable — it shows your thought process and opens the door for the interviewer to give hints.
  • Identify the pattern: subarrays → sliding window; graph traversal → DFS/BFS; sorted search → binary search; etc.
  • Be receptive to interviewer hints. They know the optimal solution and want you to succeed. Don't be stubborn.
  • Before coding, explain your approach to the interviewer and get their agreement that it's reasonable.
If the interviewer is hinting at something, follow that thread — they are not trying to mislead you.

4 Implementation

Once the approach is agreed on, write clean, well-explained code.

  • Ask before importing libraries or modules.
  • Narrate decisions as you type: "I'm declaring a Set here to avoid revisiting nodes and prevent cycles."
  • Write clean code following the language's conventions (case, indentation, spacing).
  • Avoid duplicated logic — use helper functions or loop over a directions array instead of repeating the same block 4 times.
Brute force first is fine — acknowledge it's suboptimal, implement it cleanly, then identify the slow steps and optimize. Include the interviewer in that discussion.
Don't panic if you get stuck. Communicate your concern. Silence is the worst thing you can do.

5 Testing & Debugging

After coding, walk through test cases — sometimes the environment runs code, sometimes it doesn't.

EnvironmentApproach
Built-in test cases, code runsFocus on correctness — edge cases are already included. Least pressure on you to generate tests.
You write tests, code runsCall your function in the outer scope and print results. Include edge cases, typical inputs, and potentially invalid inputs.
You write tests, code doesn't runWalk through manually. Track variable values in writing. Condense trivial loops ("after this, the prefix sum will look like…").
If there's a bug and the environment supports running code, add print statements at key points, then walk through a small test case comparing expected vs actual variable values.

6 Explanations & Follow-ups

Be ready to answer these questions after coding:

  • Time and space complexity — worst case, and average if it differs significantly.
  • Why did you choose X? — data structure, algorithm, loop configuration. Know your reasoning.
  • Can the algorithm be improved? — if the interviewer asks this, the answer is almost always yes. Don't assert optimality unless you're certain.
If there's time left, you may get a new problem (restart from §2) or a follow-up with new constraints. This is why understanding solutions matters more than memorizing them.

7 Outro

The last few minutes are for your questions. You rarely improve the outcome here, but you can hurt it.

Interviews are two-way. Use this time to evaluate whether you want to work there.

Good questions to have ready:

  • What does an average day look like?
  • Why did you join this company instead of another?
  • What's your favorite and least favorite thing about the job?
  • What kind of work can I expect to do?
Read the company's tech blog before the interview. Asking specific questions about engineering decisions signals genuine interest.

★ Cheat Sheet

Print this and keep it visible during a remote interview.

Stage 1 — Introductions

  • Rehearsed 30–60 sec intro (education, experience, interests)
  • Smile, confident voice
  • Listen when they describe their work

Stage 2 — Problem Statement

  • Paraphrase the problem back
  • Ask about input types, sorting, empty inputs, invalid inputs, size
  • Walk through one example test case

Stage 3 — Brainstorming DS&A

  • Always think out loud
  • Break down: what do I need to do → what can accomplish it efficiently?
  • Be receptive to hints
  • Explain your approach to the interviewer before coding

Stage 4 — Implementation

  • Narrate decisions as you code
  • Write clean, convention-following code
  • No duplicate logic — use helpers/loops
  • Stuck? Communicate, don't go silent
  • Brute force first is fine — then optimize together

Stage 5 — Testing & Debugging

  • Track variables in writing as you walk through cases
  • Condense trivial steps ("after this loop, prefix sum = …")
  • Use print statements if environment allows

Stage 6 — Follow-ups

  • Time/space complexity (worst case + average if different)
  • Justify your choices
  • If asked "can it be improved?" → answer is usually yes

Stage 7 — Outro

  • Have 2–3 questions ready about the company/role
  • Stay engaged, smile, ask follow-ups
