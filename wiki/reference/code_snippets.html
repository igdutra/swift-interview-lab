1 The big gotcha: string problems are bounds arithmetic — guard relentlessly

Almost every bug in a string-building problem is an off-by-one or an out-of-bounds, not a logic error in the strategy. The strategy is usually right; the arithmetic on lengths, gaps, slices, and ranges is where it breaks. Treat every index expression and every divisor as a potential crash until you've guarded it.

Pattern 1 — guard the divisor (zero-gap line)

A line with a single word has zero gaps. Any division or modulo on gapCount crashes without this guard.

SwiftLC 68 — justify(), guarding the single-word case
guard gapCount > 0 else {
    // single word — left-justify with trailing padding, skip the division entirely
    return lineWords[0] + String(repeating: " ", count: maxWidth - lineWords[0].count)
}
let baseSpacePerGap = totalSpaces / gapCount   // safe: gapCount >= 1 here
let extraSpaceGaps  = totalSpaces % gapCount

Pattern 2 — ternary to distribute remainder left-first

When spaces don't divide evenly, leftmost gaps get one extra. A ternary on the gap index is the cleanest expression of "left-first".

SwiftLC 68 — iterating gaps, giving +1 to the leftmost remainder gaps
for gapIndex in 0..<gapCount {
    let spacesForThisGap = baseSpacePerGap + (gapIndex < extraSpaceGaps ? 1 : 0)
    line += String(repeating: " ", count: spacesForThisGap)
    line += lineWords[gapIndex + 1]
}

Pattern 3 — ternary before building a range whose lower bound can go negative

Constructing 0...(dropIndex - 1) when dropIndex == 0 produces 0...(-1) — a runtime crash. Guard the boundary first and return nil when that side doesn't exist.

SwiftLC 755 — building left/right scan ranges only when in-bounds
let lowestIndexLeft: Int? = dropIndex > 0
    ? getReachableMinIndex(scanning: 0...(dropIndex - 1), leftward: true)
    : nil
let lowestIndexRight: Int? = dropIndex < currentHeightMap.count - 1
    ? getReachableMinIndex(scanning: (dropIndex + 1)...(currentHeightMap.count - 1), leftward: false)
    : nil

Pattern 4 — ternary on an exhausted pointer contributing a default

When two strings have different lengths, the shorter one runs out first. Rather than branching the whole loop, use a ternary to contribute 0 once the index goes negative.

SwiftLC 415 — reading digits from two strings of different lengths
let digit1 = indexNum1 >= 0 ? array1[indexNum1].wholeNumberValue ?? 0 : 0
let digit2 = indexNum2 >= 0 ? array2[indexNum2].wholeNumberValue ?? 0 : 0

The recurring off-by-one: "gap before word" counted unconditionally

There is one off-by-one that appears in greedy line-packing problems more than any other. When checking whether a new word fits on the current line, it is tempting to add +1 for the space that precedes the word — unconditionally. But the space only exists when the line already has at least one word. The first word on a fresh line has no gap before it. When the buffer is empty, the fit check fires one character too early, and the string you build ends up with a leading space.

Swiftstring-buffer variant (bordered-box formatter)
// WRONG — +1 is unconditional, phantom gap fires on the first word of every fresh line
let isOverflow = currentLine.count + 1 + nextWord.count > lineMaxWidth

// CORRECT — gap only exists when the line already has words
let gapCount = currentLineWords.isEmpty ? 0 : currentLineWords.count
let isOverflow = currentLine.count + gapCount + nextWord.count > lineMaxWidth
Swiftrunning-integer variant (LC 68 text justification)
// WRONG — +1 added for every word, including the first on a fresh line
currentLineCount += word.count
currentLineCount += 1
if currentLineCount >= maxWidth { ... }

// CORRECT — use word count as gap count: 0 on an empty line, always n-1 gaps for n words
let widthWithWord = currentLineLength + currentLineWords.count + word.count
if widthWithWord > maxWidth { ... }
The rule: n words have n−1 gaps, never n. The space belongs to the gap between two words, not to the word itself. existingWordCount is naturally 0 on an empty line and always one fewer than the total words after adding the new one — that is the correct gap count, with no conditional needed.
The reflex: Before you write array[i], array[...i], a / b, or a % b, ask: can the index be out of range? can the divisor be zero? In string problems the answer is "yes" far more often than in array problems, because the empty line, the single word, and the last element are all special cases.

Where this appears: LC 68 Text Justification (patterns 1, 2, and the gap off-by-one), LC 755 Pour Water (pattern 3), LC 415 Add Strings (pattern 4), and a bordered-box formatter (the gap off-by-one again — same mistake, two independent problems).

The two operations you need every time you simulate arithmetic on numeric strings without converting to Int.

// Digit math — split a column sum into the kept digit and the carry
let currentDigit = columnSum % 10   // units place: what gets appended

3 Stack-based nesting — it is physically LIFO

When you read a nested structure like [12,[13,14]] left to right, the innermost bracket always closes before the outer one. The thing you opened most recently is always the thing you close next — that is the definition of LIFO, and why a stack is the natural fit for any bracket parser.

CharacterAction
[Push a new empty container onto the stack
digit / -Accumulate into currentNumber
, or ]Commit currentNumber (if any) into the top container, then reset
] onlyAlso pop the top container and add it to the new top (or save as result)
Why it works: the stack is recursion made explicit. The call stack in a recursive DFS does exactly what this bucket-tower does — it remembers "where was I before I went deeper." You could recurse instead; the explicit stack is the iterative form the wiki standard prefers.

Push order vs popLast() — reverse on push only to cancel a reversal

An array used as a stack via append / popLast() retrieves elements last-in-first-out: the most recently pushed comes out first. So whether you reverse before pushing depends entirely on the order you want back out. There are exactly two cases, and confusing them is a silent bug.

You want to process…Push orderWhy
Most-recent first (LIFO is the goal) — bracket nesting, undo, innermost-closes-firstAs-is — no reverseThe input's natural order already is the LIFO order. Push the opening bracket as you see it; popLast() hands back the most recent open to match the next close.
Left-to-right / first-in first (FIFO output from a stack) — flatten a list, emit children in orderReversed — push the last element firstpopLast() reverses you, so you pre-reverse to cancel it. Push [a, b, c] as c, b, a and popLast() yields a, b, c.
The mental model: a stack reverses whatever you put in. If you want the reversal (LIFO), push as-is. If you want to undo the reversal (FIFO order out), reverse on the way in — the two reversals cancel. For LC 20 you push the bracket you just saw, because "last opened closes first" already is LIFO, so no reverse. For LC 341 you reverse the nested list onto the stack so that popLast() emits its elements front-to-back, the order next() must return them in.
PITFALLS
▸Reversing "to be safe" when the input order is already LIFO. In a bracket parser, reversing on push flips the matching order and every pair mismatches. Reverse only when you need FIFO output from the stack.
▸Pushing left-to-right when you need elements out left-to-right. popLast() then yields them backwards. Push the collection reversed so the first element ends up on top.

Where this appears: LC 385 Mini Parser, LC 394 Decode String, LC 227 / LC 224 Basic Calculator — any problem where [ or ( opens a scope and ] or ) collapses it back into the parent. The push-order distinction: LC 20 Valid Parentheses (push as-is — input is already LIFO) vs LC 341 Flatten Nested List Iterator (push reversed — FIFO output from a stack).

4 Centralize multi-variable state in one resetState() helper

THE ONE IDEA: The moment you carry two or more related state variables that must be cleared together between iterations, extract a single resetState() (or resetAccumulators()) helper and call it in one place. Never reset them inline, one by one — the day you add a third variable you will forget to reset one of them, and it bleeds silently into the next iteration. A helper makes "all the state I manage" a single named, auditable list.

The classic case is a character parser with a sign flag: it tracks three things together — the number, its sign, and whether a number was seen at all. Reset them as a unit. And currentNumber == 0 alone cannot distinguish "no number" from "the number zero", so an empty list [] would get a phantom 0 — hence the seen-flag.

Swifta parser — the seen-flag and the single reset point
var currentNumber = 0
var isNegative = false
var hasSeenNumber = false

if let digit = character.wholeNumberValue {
    currentNumber = currentNumber * 10 + digit
    hasSeenNumber = true
}
if character == "-" {
    isNegative = true
    hasSeenNumber = true
}

func commitNumber() {
    guard hasSeenNumber else { return }   // no phantom zero on empty lists
    // commit currentNumber * (isNegative ? -1 : 1) …
    resetAccumulators()
}

func resetAccumulators() {
    currentNumber = 0
    isNegative = false
    hasSeenNumber = false
}

The exact same discipline applies far outside parsers. In greedy line-batching (LC 68) a "line" is also a bundle of state — the words on it and its running character length — that must be cleared together every time a line commits:

Swifta batcher — the line's state cleared as a unit on commit
var currentLineWords: [String] = []
var currentLineLength = 0

func resetLine() {     // one place that knows EVERY per-line variable
    currentLineWords = []
    currentLineLength = 0
}

// on overflow: commit the line, then reset BOTH — never just one
outputLines.append(justify(currentLineWords))
resetLine()
Why it works: a resetState() helper turns "what state do I manage?" into a single readable list. When I solved LC 68 I reset currentLineLength but forgot to clear currentLineWords inline — every line after the first carried words from the previous one. Had both lived inside one resetLine(), the omission would have been impossible. The helper is cheap insurance against the single most common simulation bug: a forgotten reset.
PITFALLS
▸Resetting related variables separately, inline. Extract one resetState() and call it in one place. Forgetting to clear even one (a sign flag, a seen-flag, a line's word list) bleeds it into the next iteration — a silent wrong answer, not a crash.
▸Committing unconditionally on ]. Without the hasSeenNumber guard, an empty list gets a stray 0 appended.

Where this appears: LC 385 Mini Parser, LC 224 / LC 227 Basic Calculator (parser state trio), and LC 68 Text Justification (per-line batch state) — any loop that carries multiple variables it must clear between iterations.

5 stack.last is a copy for structs — mutate by index

When you do var top = stack.last and call a mutating method on it, you may be mutating a throwaway copy — the stack never changes.

Swiftvalue-type vs reference-type top-of-stack
// BUG if the element is a struct (value type): mutates a copy
var lastObject = stack.last
lastObject?.add(something)

// Always safe — works for both class and struct: mutate in place by index
stack[stack.count - 1].add(something)
Note: if the element type is a class (reference type / AnyObject), stack.last hands back a reference and mutation works. If it is a struct, stack.last is a copy — mutate by index. Check the declaration: class Foo / protocol Foo: AnyObject → reference; struct Foo → value.

Where this appears: any stack-based parser that appends into the top container — LC 385 Mini Parser, LC 394 Decode String, calculator problems.

6 Monotonic stack — record AFTER eviction, BEFORE push

The routing signal is "next greater / next smaller / first element to the right satisfying condition X". The word nearest or first is what picks a monotonic stack over a suffix-max pass (which gives the maximum, not the nearest) or two pointers (which converge or both run right, not pending-query resolution).

Swiftdecreasing stack → next greater
var nextGreater: [Int: Int] = [:]
var stack: [Int] = []              // monotonically DECREASING bottom → top

for value in numbers.reversed() {   // right-to-left so the stack holds future candidates
    while let top = stack.last, top <= value {
        stack.removeLast()         // dominated — never the answer for anything further left
    }
    nextGreater[value] = stack.last ?? -1  // record AFTER eviction, BEFORE push
    stack.append(value)
}
// each element pushed once, popped at most once → O(n) amortized
PITFALLS
▸Recording after the push. Reversing the record and push makes the current element its own next-greater — a silent wrong answer. Record between the eviction loop and the append.
▸Flip for next-smaller: evict on top >= value (pop ≥ instead of ≤), keep the same record-then-push order.

Where this appears: LC 496 Next Greater Element I, LC 503 (circular), next-smaller variants, LC 84 Largest Rectangle in Histogram.

7 Palindrome check

The reflex for "is this a palindrome": converge from both ends, return false on the first mismatch. Boolean output means no map and no counting — just compare ends inward. An empty or single-element range returns true because the loop never runs.

Swiftrange check + the LC 125 skip-and-fold flavor
// Clean range, every character matters — reused as a building block everywhere
func isPalindrome(_ characters: [Character], _ startIndex: Int, _ endIndex: Int) -> Bool {
    var leftPointer = startIndex, rightPointer = endIndex
    while leftPointer < rightPointer {
        if characters[leftPointer] != characters[rightPointer] { return false }
        leftPointer += 1
        rightPointer -= 1
    }
    return true
}

// LC 125 flavor — skip non-alphanumeric on the fly, case-insensitive.
// Bounds check FIRST in the inner loops, THEN the character test.
func isValidPalindrome(_ string: String) -> Bool {
    let characters = Array(string)
    var leftPointer = 0
    var rightPointer = characters.count - 1
    while leftPointer < rightPointer {
        while leftPointer < rightPointer && !characters[leftPointer].isLetter && !characters[leftPointer].isNumber {
            leftPointer += 1
        }
        while leftPointer < rightPointer && !characters[rightPointer].isLetter && !characters[rightPointer].isNumber {
            rightPointer -= 1
        }
        if characters[leftPointer].lowercased() != characters[rightPointer].lowercased() {
            return false
        }
        leftPointer += 1
        rightPointer -= 1
    }
    return true
}
Note: Character.lowercased() returns a String, not a Character — the comparison is String-vs-String (correct, but surprising). Array("") is empty, so rightPointer = -1 and the loop never runs — empty string returns true (Swift Int is signed, no underflow). For O(1) space, keep the String and walk String.Index with index(after:) / index(before:).

Where this appears: LC 125 Valid Palindrome (the standalone check) and LC 336 Palindrome Pairs (the range check is the building block whose empty-range-returns-true behavior makes the inclusive split scan correct).

LC 336 gotcha — palindrome definition + dictionary of all split variants

A palindrome obeys one rule: string == String(string.reversed()). That identity is the key: if you split a word into left + right and left is already a palindrome, then prepending reverse(right) gives reverse(right) + left + right — which collapses to a palindrome because the middle reads the same forwards and backwards. The trap is trying to enumerate pairs with a nested loop (O(w²·L)); you beat it by turning the check inside-out: for every word, enumerate every split point, and record what each half needs in a dictionary instead of searching for it.

Core insight: word + word.reversed() is always a palindrome — it reads the same forwards and backwards by construction. LC 336 generalises this: any concatenation of two words forms a palindrome if, at some split point, one half is already a palindrome and the other half matches the reverse of some other word in the list. The dictionary captures all those "reverse of the remainder" keys up front so lookup is O(1).
Swiftenumerate split points, store what each half needs in a dictionary
// Helper — full-array variant (no index params needed when slices are pre-extracted)
func isSlicePalindrome(_ characters: [Character]) -> Bool {
    var leftPointer = 0, rightPointer = characters.count - 1
    while leftPointer < rightPointer {
        if characters[leftPointer] != characters[rightPointer] { return false }
        leftPointer += 1; rightPointer -= 1
    }
    return true
}

// For every word and every split point, record what complement is needed.
// leftNeedsMap[key] = [wordIndex]  →  word[wordIndex] needs `key` prepended
// rightNeedsMap[key] = [wordIndex] →  word[wordIndex] needs `key` appended
var leftNeedsMap:  [String: [Int]] = [:]
var rightNeedsMap: [String: [Int]] = [:]

for (wordIndex, word) in words.enumerated() {
    let wordChars = Array(word)
    for splitPoint in 0...wordChars.count {
        let leftSlice  = Array(wordChars.prefix(splitPoint))
        let rightSlice = Array(wordChars.suffix(wordChars.count - splitPoint))

        // left half is a palindrome → need reverse(right) to arrive from the LEFT
        if isSlicePalindrome(leftSlice) {
            let neededKey = String(rightSlice.reversed())
            if neededKey != word { leftNeedsMap[neededKey, default: []].append(wordIndex) }
        }

        // right half is a palindrome → need reverse(left) to arrive from the RIGHT
        if isSlicePalindrome(rightSlice) {
            let neededKey = String(leftSlice.reversed())
            if neededKey != word { rightNeedsMap[neededKey, default: []].append(wordIndex) }
        }
    }
}

// Now match: for each word, look it up in both maps
var resultPairs: Set<[Int]> = []
for (currentIndex, word) in words.enumerated() {
    for partnerIndex in leftNeedsMap[word, default: []] where partnerIndex != currentIndex {
        resultPairs.insert([currentIndex, partnerIndex])   // current goes LEFT of partner
    }
    for partnerIndex in rightNeedsMap[word, default: []] where partnerIndex != currentIndex {
        resultPairs.insert([partnerIndex, currentIndex])   // partner goes LEFT of current
    }
}
Gotcha — split point range is 0...wordChars.count inclusive: splitPoint = 0 means the entire word is the right slice (left is empty — the empty string is trivially a palindrome). splitPoint = wordChars.count means the entire word is the left slice (right is empty). Both boundary cases are essential: they handle empty-string partners and exact-reverse pairs. A 0..<wordChars.count range silently misses them.

Where this appears: LC 336 Palindrome Pairs.

8 Reachable-min: scan outward, stop at the first wall

You need the lowest index reachable from a pivot moving in one direction — but "reachable" means you can never climb higher than your current footing. A valley hidden behind a taller peak is invisible. .min() over a slice finds the global minimum and is wrong here.

Why .min() failsheights = [0,2,1], pivot=0, scanning right
Global min of right side  →  index 2 (height 1)   ← WRONG (behind a wall)
Reachable min from pivot  →  none (index 1 is higher than pivot, wall blocks)
Swiftbreak the moment terrain rises above the best
func getReachableMinIndex(_ array: [Int], scanning range: ClosedRange<Int>, leftward: Bool) -> Int? {
    var bestIndex: Int? = nil
    let indices = leftward ? Array(range.reversed()) : Array(range)
    for index in indices {
        if let currentBest = bestIndex, array[index] > array[currentBest] { break } // wall — stop
        if bestIndex == nil || array[index] < array[bestIndex!] { bestIndex = index }
    }
    return bestIndex // nil → nothing reachable (pivot is already the local min)
}
Why it works: this scan is O(n) per element → O(n²) total. Recognizing the reachable-scan shape is exactly what tells you to reach for a monotonic stack (§6) instead — the histogram and daily-temperatures stacks exist precisely to kill this O(n²).

The combined helper above handles both directions via the leftward flag. When the two directions are called separately — as in LC 755 where left and right scans are always both needed — you can split them into two named functions. The bodies are identical; only the sequence fed to for changes:

Swiftleft→right using ClosedRange directly (no allocation)
// A ClosedRange<Int> is already a Sequence that counts upward — hand it to for directly.
func getReachableMinIndexLeft(_ array: [Int], scanning range: ClosedRange<Int>) -> Int? {
    var bestIndex: Int? = nil
    for currentIndex in range {
        if let confirmedBest = bestIndex, array[currentIndex] > array[confirmedBest] { break }
        if bestIndex == nil || array[currentIndex] < array[bestIndex!] { bestIndex = currentIndex }
    }
    return bestIndex
}
Swiftright→left using stride — O(1), no Array allocation
// ClosedRange has no built-in reversed() that stays lazy — stride gives O(1) reverse iteration.
// from: upperBound starts at the pivot's edge; through: lowerBound is the far wall.
func getReachableMinIndexRight(_ array: [Int], scanning range: ClosedRange<Int>) -> Int? {
    var bestIndex: Int? = nil
    for currentIndex in stride(from: range.upperBound, through: range.lowerBound, by: -1) {
        if let confirmedBest = bestIndex, array[currentIndex] > array[confirmedBest] { break }
        if bestIndex == nil || array[currentIndex] < array[bestIndex!] { bestIndex = currentIndex }
    }
    return bestIndex
}
Why the bodies are identical: the wall logic and best-tracking don't care about direction — they only care about the sequence of indices they receive. The direction is a property of the iterator, not the algorithm. Splitting into two functions gives each a clear name and removes the leftward: Bool flag, at the cost of one extra function.
stride vs Array(range.reversed()): Array(range.reversed()) allocates the full index list upfront — O(n) memory. stride(from:through:by: -1) is a lazy sequence that yields one index at a time — O(1) allocation. Prefer stride in a tight loop or when the range can be large.

Where this appears: LC 755 Pour Water (scan left/right from drop index, stop at first rise), LC 84 Largest Rectangle (naive nearest-shorter scan), LC 739 Daily Temperatures (naive next-warmer scan).

9 Argmin in a subrange — single pass, real indices

You need both the minimum value and its index within a subrange. .min() + .firstIndex(of:) is two passes and loses index information on slices. Loop the range directly — the loop variable is already the real position in the original array, so no offset math.

Swiftargmin without allocating a slice
func getSmallerIndex(_ array: [Int], range: ClosedRange<Int>) -> Int {
    var currentMin = Int.max
    var currentMinIndex = range.lowerBound
    for index in range {
        if array[index] < currentMin {
            currentMin = array[index]
            currentMinIndex = index
        }
    }
    return currentMinIndex
}
Note: avoid array[range].enumerated() — on a slice, .enumerated() resets the offset to 0, so the returned index is relative to the slice, not the original array. You'd have to add range.lowerBound back: easy to forget, a classic off-by-one.

Where this appears: LC 755 Pour Water — any problem needing the argmin of a subarray without a new slice.

10 Pivot-side nil guard — both sides are independent candidates

Whenever you split a collection at a pivot and scan each side, one or both sides may not exist — at the left edge there is no left side, at the right edge no right side. Calling .min(), iterating a ClosedRange, or subscripting an empty region crashes. Produce a candidate per side only when that side exists; nil means "cannot settle / not a candidate".

Swiftguard each side, then let-bind in the decision
let leftMinIndex  = pivotIndex > 0
    ? getSmallerIndex(array, range: 0...(pivotIndex - 1))
    : nil

let rightMinIndex = pivotIndex < array.count - 1
    ? getSmallerIndex(array, range: (pivotIndex + 1)...(array.count - 1))
    : nil

if let leftIndex = leftMinIndex, array[leftIndex] < array[pivotIndex] {
    array[leftIndex] += 1
} else if let rightIndex = rightMinIndex, array[rightIndex] < array[pivotIndex] {
    array[rightIndex] += 1
} else {
    array[pivotIndex] += 1
}
Note: don't use a guard/continue — you don't want to bail out of the whole iteration, just skip the side that doesn't exist. And prefer nil over a sentinel like Int.max: an optional forces the compiler to make the caller check; a sentinel only hopes they remember.

Where this appears: LC 755 Pour Water — any problem that splits at a pivot and searches each half independently.

11 Iterating a range in reverse — reversed() vs stride

You have a ClosedRange<Int> for the left side of an array and need to walk it right-to-left (closest to the pivot first). A plain for index in range walks left-to-right.

Swifttwo ways to walk outward-leftward
let leftRange = 0...(pivotIndex - 1)

// Clear, but allocates O(range) — ClosedRange.reversed() is not directly for-iterable
for index in Array(leftRange.reversed()) { … }   // pivotIndex-1, …, 0

// Zero-allocation alternative — lazy, same order
for index in stride(from: pivotIndex - 1, through: 0, by: -1) { … }
ApproachAllocationUse when
Array(range.reversed())O(n)Range is small, clarity matters
stride(from:through:by: -1)O(1)Range is large or you're in a tight loop

Where this appears: LC 755 Pour Water (left scan starts nearest the pivot), LC 238 Product of Array Except Self (right-to-left suffix pass), LC 42 Trapping Rain Water (suffix-max precompute).

let wordArray = Array(word)      // O(1) indexing — String is not integer-indexable
for splitPoint in 0...wordArray.count {
    let leftSplit  = Array(wordArray.prefix(splitPoint))

    // splitPoint=n → left=full word, right=[]    (empty right)
}
Note: 0...count is count + 1 iterations — one for each gap between characters, including both ends. 0..<count misses the empty-right split; 1..<count misses both empty sides. Convert with Array(word) first because Swift String is not integer-indexable — subscripting with an Int is a compile error.

Where this appears: LC 336 Palindrome Pairs (check one side palindrome, look up the other's reverse), LC 131 Palindrome Partitioning — any problem decomposing a string into two parts and checking a property of each.

The other two slice patterns — anchor-then-walk

The split-point loop above needs both halves as values because it immediately inspects or transforms each part. But there is a second family: you already know a fixed anchor index (found via firstIndex(of:)) and you just need to walk forward or backward from it. Here you use subscript range operators — and they are safe because the anchor index is always validated before use.

Swiftanchor-then-walk forward — CUSTOM 1 Relay Handoff, prefix phase
// "Walk every date AFTER startDate and stop when continuity breaks."
// The anchor is guarded by the line below — if startDate isn't in the array, we skip entirely.

Swiftanchor-then-walk backward — CUSTOM 1 Relay Handoff, suffix phase
// "Walk every date BEFORE endDate in reverse and stop when continuity breaks."
// The anchor is guarded by the line below — if endDate isn't in the array, we skip entirely.
// [..= 0 is guaranteed).

Why subscript ranges are safe here — but not in split enumeration. In split enumeration, the index is computed from a loop counter that can legally be 0 or count. At those boundaries, array[...(splitPoint - 1)] produces a negative index (undefined behavior) and array[(splitPoint)...] at splitPoint = count is an out-of-bounds crash. prefix(_:) / suffix(_:) clamp silently instead. In anchor-then-walk, the anchor is always a real, validated index from firstIndex(of:) — the guard let means the subscript math can never go out of bounds.

13 Sliding window: two-pointer vs queue — same name, different tool

Both say "sliding window," but they solve different problems with different structures. Route with one question: is your input a static array you can index freely → two-pointer window. Is it a live stream arriving one item at a time → sliding-window queue.

Swifttraditional — static array, two index pointers, no queue
func longestSubarrayWithSumAtMost(_ numbers: [Int], _ limit: Int) -> Int {
    var leftPointer = 0
    var windowSum = 0
    var best = 0
    for rightPointer in numbers.indices {
        windowSum += numbers[rightPointer]
        while windowSum > limit {
            windowSum -= numbers[leftPointer]   // shrink: jump the left index directly
            leftPointer += 1
        }
        best = max(best, rightPointer - leftPointer + 1)
    }
    return best
}
Swiftqueue — live stream, FIFO eviction from the front
var window: [Int] = []
func ping(_ timestamp: Int) -> Int {
    window.append(timestamp)                 // enqueue at the back
    while let oldest = window.first, oldest <= timestamp - 3000 {
        window.removeFirst()                 // evict stale from the front
    }
    return window.count               // the queue IS the window; its count IS the answer
}

14 String ↔ collection conversions (the small casts that bite)

The recurring conversions between String, SubSequence, [Character], and dictionary accumulators — each with one trap.

15 Accumulate-and-commit — overflow, delimiter, or guaranteed close

Every scanner that builds a result token-by-token follows the same skeleton: accumulate characters into a buffer, then commit that buffer when a trigger fires. The trigger varies — overflow, delimiter character, or a structurally guaranteed close — and the trigger determines how you handle the last token, which may never be followed by one.

THE ONE IDEA: a commit fires on a trigger. The last-token problem is: what if the input ends without one? Pick exactly one of the three strategies below and never mix them in the same parser.

Strategy A — overflow trigger, last item different (LC 68 greedy line batching)

Commit fires when the next item won't fit. The last batch never overflows, so it needs an explicit flush — and in LC 68 it needs different treatment (left-justify vs full-justify), which is why post-loop is cleaner than an index check inside the loop.

Swiftoverflow commits, overflowing word starts the NEW line, last batch flushed after
for (index, word) in words.enumerated() {
    let widthWithWord = currentLineLength + currentLineWords.count + word.count
    //                  ^ word chars only   ^ one space per EXISTING word = the gaps

    if widthWithWord > maxWidth {              // overflow → commit what we have
        outputLines.append(justify(currentLineWords))
        currentLineWords = []
        currentLineLength = 0
    }

    currentLineWords.append(word)              // the overflowing word starts the NEW line
    currentLineLength += word.count

    if index == words.count - 1 {              // the last batch never overflows…
        outputLines.append(leftJustify(currentLineWords))   // …so commit it here, differently
    }
}
PITFALLS
▸Resetting the count but not re-adding the overflowing word. After committing, reset the counters and then append the word — its length counts into the fresh line, not lost. Forgetting this undercounts every new line by its first word.
▸Treating the last commit as an if/else with the overflow. One iteration can be both the last word and an overflow; each needs its own rule. Keep them as two separate branches.

Strategy B — delimiter trigger, last item identical (CUSTOM 2 query parser)

Commit fires on a delimiter character (&). The last token has no trailing delimiter, but its commit behavior is identical to a mid-stream commit — so fold end-of-input into the same branch with an index check. No post-loop flush needed.

Swiftdelimiter OR last character — same branch, single commit point
for (index, character) in stringArray.enumerated() {
    if character == "&" || index == stringArray.count - 1 {
        guard !currentKey.isEmpty else { continue }   // skip double-delimiter tokens (e.g. "&&")
        outputMap[String(currentKey), default: []].append(String(currentValue))
        currentKey = []; currentValue = []; hasSeenEqual = false
    } else {
        // accumulate into currentKey / currentValue
    }
}
// no post-loop flush — the last-index branch handles it
Note: requires enumerated() for the index, so iterate a [Character] (not a raw String) to get an Int index. Works cleanly here because "&" is a pure separator — never part of key or value — so the commit branch and the accumulate branch are mutually exclusive. If the last character is a value character (not a separator), accumulate it before the commit fires.

Strategy C — structurally guaranteed closing delimiter, bare-value fallthrough (LC 385 Mini Parser)

The input format guarantees every nested list ends with "]", which commits and pops the stack. No post-loop flush needed for that case. The only exception: a plain number like "324" has no brackets at all, so the loop ends with the accumulator full. One post-loop line catches exactly this edge case.

Swift"]" commits; post-loop is only for bracket-free plain numbers
for character in stringArray {

    case ",", "]":

            stack[stack.count - 1].add(NestedInteger(currentNumber))
            currentNumber = 0; hasSeenNumber = false         // commit on delimiter
        }
        if character == "]" { /* pop and merge stack */ }
    default:
        // accumulate digit
    }
}
// Only reached for plain numbers like "324" — no bracket ever fired a commit

Why hasSeenNumber guards every commit. currentNumber == 0 cannot distinguish "nothing accumulated" from the literal value zero — without the flag, an empty list [] commits a phantom 0 on the "]". See also §4.

16 Distribute N things across K buckets evenly — remainder biased left

You have total things to share across bucketCount buckets and it doesn't divide evenly. The floor division gives every bucket the minimum; the modulo tells you exactly how many buckets get one extra. Hand them left-to-right — the first remainder buckets each get base + 1, the rest get base.

THE ONE IDEA: total % bucketCount is the count of "lucky" buckets. Loop with an index and ask bucketIndex < remainder — while true, you're still in the lucky ones. Once false, everyone gets just the base. No splitting, no slice arithmetic — the modulo is the counter.
Skeletongeneral form — works for spaces, jobs, items, pages
base      ← total / bucketCount       // floor — everyone gets at least this
remainder ← total % bucketCount       // how many buckets get one extra

for bucketIndex in 0..<bucketCount:
    amount ← base + (bucketIndex < remainder ? 1 : 0)
    // buckets 0 … remainder-1 are the lucky ones; the rest get base
SwiftLC 68 — full-justify a line; gapCount = n words − 1
let gapCount = lineWords.count - 1          // NOT lineWords.count — n words = n-1 gaps
let baseSpacePerGap = totalSpaces / gapCount   // everyone gets this many
let extraSpaceGaps  = totalSpaces % gapCount   // the leftmost this-many get +1

var line = lineWords[0]
for gapIndex in 0..<gapCount {
    let spacesForThisGap = baseSpacePerGap + (gapIndex < extraSpaceGaps ? 1 : 0)
    line += String(repeating: " ", count: spacesForThisGap)
    line += lineWords[gapIndex + 1]
}
Trace["example","of","text"], maxWidth=16 → totalSpaces=3, gapCount=2

gapIndex 0: 0 < 1 ✓  spacesForThisGap = 2  →  "example  of"
gapIndex 1: 1 < 1 ✗  spacesForThisGap = 1  →  "example  of text"   ✓
Why it works: the modulo gives you the count of leftmost buckets that need the bonus — and bucketIndex < remainder is just a counter running from zero up to that count. Once the index exceeds the remainder the counter saturates and everyone gets the base. The index itself does the work; no extra variable needed.

17 Prefix / suffix precompute — expensive walk once, O(1) per query

The unifying idea: do the expensive work once per entity and store the result, so every later lookup is O(1). The three variants below all share this shape but differ in what they store and why.

Variant A — running array (LC 42, LC 238)

Scan once forward and once backward, building arrays where each position stores the running best from that direction. A third pass combines both in O(1) per position. The signature: at each position the answer depends on something to the left AND right simultaneously.

Swiftforward prefix max, backward suffix max, combine
// FORWARD — build prefix max
var runningMax = input[0]
let prefixMax = input.map { value in
    runningMax = max(runningMax, value)
    return runningMax
}

// BACKWARD — build suffix max, re-orient so index 0 = leftmost
runningMax = input.last!
let suffixMax = Array(input.reversed().map { value in
    runningMax = max(runningMax, value)
    return runningMax
}.reversed())

// COMBINE — each position uses both arrays in O(1)
for index in input.indices {
    let answer = min(prefixMax[index], suffixMax[index]) - input[index]
}

Variant B — scalar-per-entity map (CUSTOM 1 Relay Handoff)

Walk each entity (team) exactly once forward and once backward, distilling its entire date list into a single scalar. Store those scalars in a dictionary. The pair-matching phase then reads the dictionary — no date arrays touched again. Replaces O(t²·d) brute force with O(t·d) precompute + O(t²) matching.

Swiftone scalar per team, stored in a dict, queried in pair phase
// PHASE 1 — prefix walk: for each team, how far forward can it run from startDate?
var handoffDayByTeam: [String: Int] = [:]
for team in teams {
    guard let startIndex = team.availableDates.firstIndex(of: startDate) else { continue }
    var lastCoveredDay = startDate
    for date in team.availableDates[(startIndex + 1)...] {
        guard date == lastCoveredDay + 1, date < endDate else { break }
        lastCoveredDay = date
    }
    handoffDayByTeam[team.name] = lastCoveredDay + 1   // first day Team B must own
}

// PHASE 2 — suffix walk: for each team, how far back can it run to reach endDate?
var earliestDayByTeam: [String: Int] = [:]
for team in teams {
    guard let endIndex = team.availableDates.firstIndex(of: endDate) else { continue }
    var earliestCoveredDay = endDate
    for date in team.availableDates[..<endIndex].reversed() {
        guard date == earliestCoveredDay - 1 else { break }
        earliestCoveredDay = date
    }
    earliestDayByTeam[team.name] = earliestCoveredDay
}

// PHASE 3 — pair matching: pure dict lookups, no date arrays touched
for startingTeam in teams {
    guard let handoffDay = handoffDayByTeam[startingTeam.name] else { continue }
    for finishingTeam in teams {
        guard finishingTeam.name != startingTeam.name else { continue }
        if let earliestDay = earliestDayByTeam[finishingTeam.name], earliestDay <= handoffDay {
            validPairs.append([startingTeam.name, finishingTeam.name])
        }
    }
}
Why ≤ not ==: if Team B's contiguous run starts at day 3 and handoffDay is 5, Team B still covers day 5 through endDate — extra early coverage is fine. Strict equality would miss valid pairs.

Variant C — reverse-lookup map built from split-point enumeration (LC 336)

For each word, enumerate all n+1 split points, check which half is a palindrome, and store reverse(other half) as a key in a dictionary. The "precomputed suffix" here is the reversed string stored as a lookup key, not a running array. The final pass is one O(1) dict lookup per word — replacing the O(w²·L) brute force.

Swiftbuild a reverse-lookup map, then query it in one pass
// PRECOMPUTE — for every split of every word, register what complement is needed
var complementMap: [String: [Int]] = [:]
for (wordIndex, word) in words.enumerated() {
    let wordArray = Array(word)
    for splitPoint in 0...wordArray.count {
        let leftSplit  = Array(wordArray.prefix(splitPoint))
        let rightSplit = Array(wordArray.suffix(wordArray.count - splitPoint))
        // if left is a palindrome, we need reverse(right) to appear somewhere as a partner
        if isPalindrome(leftSplit) {
            complementMap[String(rightSplit.reversed()), default: []].append(wordIndex)
        }
    }
}

// QUERY — one dict lookup per word, O(1) each
for (currentIndex, word) in words.enumerated() {
    for partnerIndex in complementMap[word, default: []] where partnerIndex != currentIndex {
        outputIndexes.insert([currentIndex, partnerIndex])
    }
}
What makes this "prefix/suffix": for each word you enumerate every prefix (left split) and check if it's a palindrome. If it is, you need the reverse of the suffix (right split) to exist somewhere as a standalone word. You're precomputing "given this suffix, who needs it" — paying O(L) per word up front so the query phase pays O(1) per word instead of O(w·L) per pair.

18 Multi-digit number accumulation

The input is a String, so the integer 142 arrives as three separate characters '1' '4' '2'. Rebuild the number as you scan: multiply the running value by 10 (shifting every digit one decimal place left) and add the new digit on the right.

Swiftaccumulate one digit per character

19 ClosedRange<Int> — create, store, and traverse

A ClosedRange<Int> captures both endpoints as a typed value you can pass around, store as an optional, and iterate directly. It is the right tool when the range itself is part of your algorithm's state — not just a loop bound you inline.

Swiftcreating a ClosedRange
// Literal — both bounds known at the call site
let leftSide: ClosedRange<Int> = 0...(pivotIndex - 1)
let rightSide: ClosedRange<Int> = (pivotIndex + 1)...(array.count - 1)

// Stored as optional — nil means "this side doesn't exist"
let leftRange: ClosedRange<Int>? = pivotIndex > 0 ? 0...(pivotIndex - 1) : nil

// Passed as a function parameter (LC 755 Pour Water, §8–§9)
func getSmallerIndex(_ array: [Int], range: ClosedRange<Int>) -> Int { ... }

// Reconstructed from its bounds (useful after storing lowerBound / upperBound separately)
let range = range.lowerBound...range.upperBound
Swifttraversing — three forms, different allocation and direction
let range: ClosedRange<Int> = 2...5   // lowerBound=2, upperBound=5

// ① Left → right: ClosedRange is a Sequence — hand it directly to for
for currentIndex in range { … }        // 2, 3, 4, 5   — O(1) allocation

// ② Right → left: stride is O(1) and lazy (see §11)
for currentIndex in stride(from: range.upperBound, through: range.lowerBound, by: -1) { … }   // 5, 4, 3, 2

// ③ Right → left: Array(range.reversed()) is simpler to read but allocates O(n)
for currentIndex in Array(range.reversed()) { … }   // 5, 4, 3, 2   — prefer stride in tight loops

// Inspect bounds without iterating
range.lowerBound   // 2
range.upperBound   // 5
range.count        // 4  (upperBound - lowerBound + 1)
range.contains(3) // true
Why store it as ClosedRange<Int> instead of two loose integers? A ClosedRange is a single typed value: you can make it optional (nil = "side doesn't exist"), pass it to a helper as one argument, and iterate it directly without reconstructing the bounds. Two loose integers require you to always carry them together and manually compute the half that's missing — an optional ClosedRange gives you the same nil-safety the compiler enforces on any optional.
