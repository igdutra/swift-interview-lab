Pattern Recognition

Review & Reference · I see X in the problem → reach for Y

The which-pattern-do-I-reach-for layer. Distilled from the problems actually solved in this wiki — not from theory. Read the question verb and the constraint bounds first; they route you faster than anything else. Every row points at a real walkthrough so you can confirm the match against a worked example.

How to use it: when you've read a problem and don't yet know the family, scan the master routing table (§1) for the signal that matches, then jump to the section that explains the call. For the exact Swift idiom once you've picked a pattern, cross to the companion Code Snippets & Gotchas page.

0 The two questions that route almost everything

THE ONE IDEA: Before pattern-matching on structure, ask two questions. (1) What is the question verb? — "is it…", "longest…", "count…", "nearest…", "merge…", "shortest path…". The verb usually names the family. (2) What do the constraint bounds allow? — n ≤ 20 invites exponential search; n ≤ 10⁵ forbids O(n²); a tiny state space invites BFS over states. The verb picks the family; the bounds confirm or veto it.

2 Input size → likely complexity → likely algorithm

The constraint bound is a near-deterministic hint at the intended complexity. Read n from the constraints and work backwards.

n up to…Target complexityLikely family
≤ 12O(n!) / O(2ⁿ·n)Permutations, brute backtracking
≤ 20O(2ⁿ)Subsets, bitmask DP, state-space BFS (LC 1284)
≤ 100 — 1,000O(n²) / O(n³)Interval DP, nested two-pointer, pairwise scans
≤ 10⁵ — 10⁶O(n) / O(n log n)Sliding window, two pointers, sort-sweep, heap, prefix sum
> 10⁶O(n) / O(log n) / O(1)Single pass, binary search, math/formula
Note: the inverse is just as useful — if a problem says n ≤ 10⁵ it is telling you an O(n²) brute force (≈10¹⁰ ops) will time out, so the intended answer is O(n) or O(n log n). Constraint bounds are the interviewer quietly naming the target.

4 Prefix / suffix family — the unified skeleton

All three prefix/suffix problems in this wiki share one skeleton: do the expensive walk once, pay O(1) at query time. The "element" changes shape across problems, but the structure never does.

Skeletonprecompute one summary per direction, then combine
PRECOMPUTE (expensive, done once)
  → for each element, derive a summary going LEFT  → store in prefixArray / leftMap
  → for each element, derive a summary going RIGHT → store in suffixArray / rightMap

COMBINE (cheap, per element or per pair)
  → at each position, query both in O(1) and emit the answer
ProblemThe "element"The precomputed summary
LC 42 Trapping Rain Watera bar heightrunning max wall height each side
LC 238 Product Except Selfan array elementrunning product each side
LC 336 Palindrome Pairsa split point in a wordreversed-word → index map (the "other side")
CUSTOM 1 Relay Handoffa calendar dayhandoff day / earliest reachable day
Note — when it is NOT prefix/suffix: arbitrary subarray [i…j] sums at runtime → cumulative prefix sum array. Left and right depend on each other → two pointers. Each position depends on the previous answer → DP. Nearest smaller/larger → monotonic stack.

5 Two pointers vs sliding window — which motion?

Both use two indices; the motion is the tell.

MotionSignalExample
Converging (ends → middle)Sorted input; answer is a pair/area from the two ends; palindrome checkLC 11, LC 15, LC 125
Both run right (window)"longest/shortest contiguous" with a grow/shrink validity ruleLC 3, LC 76, LC 209
Two inputs, walk bothMerge / intersect two sorted listsLC 986
Why it works: for LC 42 the dual dependency (water needs the tallest wall left AND right) rules out a single-direction window — that's the signal to converge or to precompute both sides. The converging two-pointer is often the O(1)-space compression of a prefix/suffix solution.

6 Monotonic stack — the "nearest" signal

The word nearest or first is what selects a monotonic stack over its neighbours.

You need…UseWhy not the other
The maximum to the rightSuffix-max pass—
The nearest greater/smaller to the rightMonotonic stackSuffix-max gives the max, not the nearest → still O(n²) to find nearest
A pair/area from two converging endsTwo pointersTwo pointers don't track pending unanswered queries for many elements
Why it works: the monotonic stack holds elements waiting for their answer and discharges them the instant a qualifying value arrives — each pushed once, popped once, O(n). Recognizing the reachable-min outward scan (an O(n²) shape) is exactly the cue to switch to the stack. See LC 496.

8 Palindrome routing — the two families in this wiki

A palindrome is never the problem — it is a property the problem hangs another technique on. Read the question verb to route. This wiki holds exactly two palindrome problems, so only those two families are covered here in depth; the broader landscape (longest substring, subsequence DP, partitioning, permutation parity, KMP prepend) exists but has no walkthrough yet.

The prompt says…Family / toolExample
"is s a (valid) palindrome", alphanumeric/case rulesVerify — converging two pointers, skip on the flyLC 125
"pairs of words whose concatenation is a palindrome" at a scale that kills O(w²)Pair-matching — reversed-word hash map + split-point decompositionLC 336
Why it works: both reduce to the same primitive — the two-pointer check isPalindrome(chars, low, high) that converges from the ends and returns true on an empty range. In LC 336 that empty-range-returns-true behavior is exactly what makes the inclusive 0...length split scan correct. The map is keyed on reversed words, so you look up the suffix/prefix directly.
Note — the broader palindrome map (no walkthroughs here yet): "longest palindromic substring / count them" → expand-around-center (mention Manacher for O(n)). "longest palindromic subsequence / min insertions to make a palindrome" → interval DP, answer = n − LPS. "partition into palindromes" → backtracking. "can any permutation be a palindrome" → odd-count parity. "shortest palindrome by prepending" → KMP failure function. Listed for recognition only — solve cold if one appears.

11 Hashmap & pairs matching — the "does the partner exist?" signal

The most easily-missed move in the whole wiki, because it hides inside problems that look like something else. The signal is small but reliable: you are matching one element against another — its complement, its closer, or its transformed twin — and you want that lookup to be O(1) instead of an O(n) rescan. The instant you catch yourself about to write a nested loop "for each element, scan the rest to find its partner," that quadratic shape is the cue to replace the inner scan with a hashmap.

A hashmap shows up in three distinct roles. They feel different but are the same trick — turn a search for the matching element into a constant-time lookup keyed on what makes the match.

The role…What the map holdsRecognition cue · example
Static config map — a fixed, known pairingThe pairing literal itself, written once: [")": "(", "]": "[", "}": "{"]A small finite set of "this matches that" rules (open↔close brackets, char↔char, code↔name). Don't if/else the cases — table them. LC 20
Complement / seen map — built as you scanvalue → index (or count) of elements already passed; you ask "have I seen target − current?""two numbers that sum to k", "a pair with a given difference", "first repeat" — one pass, ask the map for the partner before inserting yourself. 2Sum (the archetype)
Transformed-key map — match on a derived formtransformed(element) → index, e.g. the reversed word, the sorted-letter signature (anagrams), the canonical form"pairs whose concatenation is a palindrome", "group anagrams" — the partner isn't equal to you, it's equal to some function of you, so key the map on that function. LC 336
Why it works: finding a partner by rescanning is O(n) per element → O(n²) overall. A hashmap pre-indexes every candidate by the exact key the match is defined on, so the "is my partner here?" question collapses to one O(1) probe. The art is naming the right key: for 2Sum it's the value, for LC 20 it's the bracket character, for LC 336 it's the reversed word. Pick the key that makes "match" mean "equal."
Note — map vs set vs the alternatives: if you only need "does it exist," a Set is enough; if you need where / how many / which index, you need a Dictionary (the 2Sum answer is the partner's index, so a set won't do). If the input is sorted and you want O(1) space, converging two pointers beats a map for pair-sum problems — the map is what you reach for when the input is unsorted and sorting would cost more than it saves, or when an index/order must be preserved. And for bracket-style nesting the map only names the pairing; the stack still does the matching (most-recent-open closes first).

12 Easily-confused pairs — the tie-breakers

Looks like both…Tie-breaker
Sliding window vs two pointersContiguous range with a grow/shrink validity rule → window. Pair/area from sorted ends → converging two pointers.
Sliding window vs sliding-window queueStatic indexable array → two index pointers. Live stream arriving one at a time → a queue holds the window.
Prefix/suffix vs prefix sumNeed best (max/min) to both sides at a fixed position → prefix/suffix arrays. Need the sum of an arbitrary [i…j] at runtime → cumulative prefix sum.
Hashmap complement vs converging two pointers (pair-sum)Unsorted input, or you must return an index → hashmap (value → index, one pass). Sorted input and O(1) space wanted → converging two pointers.
Hashmap config map vs if/else chainA fixed set of "this matches that" rules → table it in a literal map, don't branch. The map is data; the branch is code that drifts.
Monotonic stack vs suffix-maxNearest qualifying element → stack. The maximum regardless of distance → suffix-max.
BFS vs DijkstraUnweighted (every move costs 1) → BFS. Weighted / min-max edges → Dijkstra.
Backtracking vs DP"list all / enumerate every" → backtracking. "count / min / max" with overlapping subproblems → memoized DP.
Stack parse vs recursionSame algorithm — the explicit stack IS the call stack. Prefer the iterative stack per the wiki standard.
